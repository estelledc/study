# 生态地图与发展现状

## 1. 先建立直觉

把视频 AI 想成一位审阅实验录像的老师：

1. 先拿到录像并确认能播放。
2. 决定看哪些时刻、听哪些片段。
3. 把看到和听到的内容整理成可搜索笔记。
4. 根据问题或评分表定位相关片段。
5. 对不确定结论回看、放大或换一种工具验证。
6. 输出带证据的回答、评分或反馈。
7. 保存教师校正，检查下次是否更好。

“多模态”只是第 2、3 步涉及画面、语音、文字等多种信息。一个真实系统还需要采集、存储、检索、编排、验证和产品工程。

## 2. 六层技术栈

### 2.1 输入与采集层

负责把外部媒体变成可处理的本地或云端对象。

常见输入：

- 本地视频；
- YouTube 或其他网页 URL；
- HLS/DASH 直播流；
- 对象存储键；
- 屏幕录制或浏览器操作录像；
- 图片和多图作业。

代表实现：

- `watch-skill`：yt-dlp、cobalt、ffmpeg 回退链，支持本地、URL 和流。
- `Director`：通过 VideoDB 处理上传、集合、索引和流媒体。
- `DeepVideoDiscovery`：YouTube/本地文件，轻量模式可只取字幕。
- 全智评：浏览器直传 OSS，Worker 下载或读取本地开发文件。

工程关键点不是“支持多少网站”，而是：

- 是否有大小、格式和时长验证；
- 下载或转码失败如何恢复；
- 隐私边界是什么；
- 同一视频能否复用已有结果；
- 大媒体是否会穿过不适合的大消息通道。

### 2.2 感知预处理层

负责把连续媒体转成可计算的观察单元。

主要子任务：

- 帧采样；
- 场景切分；
- 视觉描述；
- OCR（Optical Character Recognition，光学字符识别）；
- ASR（Automatic Speech Recognition，自动语音识别）；
- 对象检测和空间关系；
- 人脸、动作或其他领域工具。

当前项目出现四种选帧路线：

| 路线 | 做法 | 优点 | 代价 |
|---|---|---|---|
| 固定频率 | 每秒或每 N 秒取一帧 | 简单、覆盖稳定 | 大量相似帧，关键瞬间可能仍错过 |
| 场景感知 | 场景起点、中点和补充均匀帧 | 把预算花在画面变化处 | 对细微连续动作不一定敏感 |
| 查询相关 | 用 CLIP 相似度选择与问题相关的帧 | 节省上下文 | 第一次筛选错误会漏掉证据 |
| 主动回看 | 先粗看，推理中再请求时间窗 | 按信息需要花成本 | 需要工具协议、循环和停止条件 |

样本对应关系：

- 全智评：固定频率为主，Phase 2 再按初判时间窗聚焦。
- `watch-skill`：场景感知 + 感知哈希去重 + 聚焦模式。
- `ReAgent-V`：CLIP 相关性 × 图像熵，迭代筛选关键帧。
- `DeepVideoDiscovery`：全局描述和检索后，按时间窗读取最多 50 帧。
- `OmAgent`：场景摘要后由 Rewinder 按需回看。

### 2.3 时序表示与记忆层

视频太长，不能把全部帧一直放进模型上下文。系统必须把连续媒体压缩成分层表示。

常见表示：

1. **原始层**：帧、音频块、短片段。
2. **观察层**：转录、OCR、帧描述、对象关系。
3. **索引层**：文本向量、图像向量、全文索引、时间范围。
4. **摘要层**：场景、章节、人物、事件、协议步骤。
5. **任务层**：与当前问题相关的证据和推理状态。

代表设计：

- `watch-skill`：SQLite 中分开保存视频、转录段、场景、OCR、向量和 FTS5。
- `multimodal-rag-agent`：Pixeltable 用视频表、帧视图、音频视图和 computed columns 统一管理派生数据。
- `OmAgent`：场景摘要进入 Milvus，短期内存保存当前视频和回看图片。
- `DeepVideoDiscovery`：固定时长片段描述进入 NanoVectorDB。
- `VidMentor`：CSV/JSON/NPZ 文件保存 ASR、OCR、摘要、层级段落和向量。
- `Director`：视频索引与搜索主要委托给 VideoDB 云服务。

这里的核心不变量是：

> 每条压缩信息必须能回到原视频的时间位置，否则它只能用于生成流畅答案，不能用于审计。

### 2.4 检索与主动感知层

RAG（Retrieval-Augmented Generation，检索增强生成）先找到相关证据，再交给模型回答。

视频 RAG 比文本 RAG 多三个变量：

- 证据有时间范围；
- 同一事件可能同时出现在画面、语音和 OCR；
- 检索结果不够时，可以重新读取原视频。

本轮样本中的检索方式：

| 项目 | 检索方式 |
|---|---|
| watch-skill | FTS5 BM25 + 本地多语言向量，融合排序 |
| Director | VideoDB spoken-word / scene semantic search |
| DeepVideoDiscovery | 片段描述向量检索 + 全局浏览 |
| OmAgent | 场景摘要向量检索 + 时间过滤 |
| VidMentor | BGE 文本向量，视频级摘要和片段级转录两层检索 |
| multimodal-rag-agent | ASR 文本、帧描述、CLIP 图像三类索引 |
| ReAgent-V | OCR/ASR RAG + 查询相关对象与场景图 |

检索后的三种处理：

1. **直接回答**：检索文本足够就生成结果。
2. **回看确认**：根据检索时间窗读取原始帧。
3. **重新规划**：生成更细的问题或子任务，再选工具。

第 2、3 种更适合评价，因为它们允许系统承认“现有摘要不够”。

### 2.5 推理、反思与领域评价层

这一层决定系统是“问答工具”还是“评价系统”。

#### 通用工具调用

`Director` 的 ReasoningEngine 把每个 Agent 暴露为 LLM tool；`VideoAgent` 先分类意图，再让模型生成 Agent DAG；`multimodal-rag-agent` 用 LangGraph 显式建模路由、选工具、执行和最终响应。

#### 长视频分治

`OmAgent` 的 DnC：

- Conqueror 尝试直接回答或调用工具；
- 任务过于复杂时 Divider 拆子任务；
- 失败时 Rescue；
- TaskTree 保存任务状态和结果。

#### 反思和复核

- `DeepVideoDiscovery`：找到答案后仍要求 `frame_inspect_tool` 确认。
- `ReAgent-V`：初答后生成批评问题，重新选 OCR/ASR/Scene Graph，再生成评价报告和多种修订。
- `watch-skill`：检索置信度不足时，本地重采样、OCR 放大和视觉模型验证。
- `proteomics_lab_agent`：把每个疑似偏差当成假设，回看前后 10 秒尝试反证。
- 全智评：Phase 1、Phase 2 状态冲突时触发 Phase 3 仲裁。

#### 领域评价

通用视频系统通常回答“发生了什么”。垂直评价要回答：

- 标准步骤是什么；
- 视频里是否观察到；
- 做错、漏做、乱序还是额外添加；
- 证据在哪个时间；
- 置信度和拍摄盲点是什么；
- 应扣多少分、是否人工复核。

`proteomics_lab_agent` 和全智评最接近这个层级。区别是：

- 前者主要把完整协议和完整视频交给 Gemini，再靠结构化提示生成 lab note；
- 全智评把评分模板、分阶段取证、后端评分规则和业务持久化拆成独立层。

### 2.6 产品与治理层

这一层常被研究代码忽略，却决定系统能否真实使用：

- 多用户和权限；
- 异步任务与进度；
- 数据库和对象存储；
- 重试、熔断和恢复；
- 成本与资源台账；
- 人工校正和审计；
- 评测集、回归和版本追踪。

本轮样本中：

- 全智评的产品和治理最完整。
- `watch-skill` 的本地索引、错误结构、测试与验证闭环最系统。
- `proteomics_lab_agent` 有较强的领域评测和单元测试。
- `multimodal-rag-agent` 有完整 UI/API/MCP/Compose，但测试证据弱。
- 多数论文型项目仍依赖固定路径、手工配置和单机 GPU。

## 3. 发展路线

### 3.1 第一阶段：端到端“看视频”

早期重点是把视频帧编码进多模态模型，让模型能够描述和问答。

典型问题：

- 可输入帧数有限；
- 长视频成本随时长增长；
- 时间定位不稳定；
- 语音、OCR 和视觉细节容易互相脱节。

### 3.2 第二阶段：视频切片与 RAG

系统把视频切成片段，为每段生成转录、描述和向量。

收益：

- 分析一次，多次查询；
- 支持长视频；
- 可跨视频搜索；
- 可定位时间范围。

代价：

- 预处理可能丢细节；
- 切片边界可能切断动作；
- 摘要错误会进入索引；
- 只检索文本描述时，视觉事实可能被二次压缩。

### 3.3 第三阶段：Agentic Search 与主动感知

Agent 不再一次性消费全部内容，而是：

1. 粗略建立全局地图；
2. 按当前问题搜索；
3. 请求更细的时间窗、帧、音频或工具；
4. 重复直到证据足够或预算耗尽。

DVD 和 OmAgent 是这一阶段的代表。

这条路线把计算复杂度从“跟视频长度走”部分转为“跟问题的信息需求走”。但它也引入新风险：

- Agent 可能搜索错方向；
- 循环可能不收敛；
- 工具结果和时间单位可能不一致；
- 需要限制轮数、深度、帧数和 token。

### 3.4 第四阶段：自我验证与奖励反馈

系统开始显式检查自己的答案：

- 生成批评问题；
- 多视角修订；
- 结构化评价报告；
- 把推理质量转成训练样本或 reward；
- 用人工纠错生成可重放 eval。

ReAgent-V 和 watch-skill 代表两种实现：

- ReAgent-V 偏模型研究：同一多模态模型生成初答、批评、修订和 meta 选择。
- watch-skill 偏工程：先用确定性检索信号和本地工具，必要时才调用模型验证。

### 3.5 第五阶段：领域协议与业务闭环

通用视频智能开始进入实验室、教育、质检、操作指导和 UI 验证。

领域化带来新的事实源：

- 标准操作协议；
- 评分 rubric；
- 设备和专业知识；
- 错误类型；
- 人工校正；
- 合规和隐私要求。

`proteomics_lab_agent` 展示了“视频 + 协议 + lab note + 误差分类 + 评测”的完整方向；全智评则进一步实现多角色 SaaS、模板版本和评分审计。

## 4. 当前常见技术栈

### 4.1 媒体处理

- ffmpeg：转码、抽帧、裁剪、音频提取；
- OpenCV：读取元数据、图像处理；
- MoviePy/PyAV：Python 媒体操作；
- PySceneDetect：场景切分；
- yt-dlp：网页视频采集。

### 4.2 感知模型

- Whisper / faster-whisper：ASR；
- RapidOCR / PaddleOCR：OCR；
- CLIP / ImageBind：跨模态相似度；
- Qwen-VL、Gemini、GPT-4.x、LLaVA-Video：视觉语言理解；
- 对象检测和场景图：补充空间关系。

### 4.3 索引与记忆

- SQLite FTS5 + 向量；
- NanoVectorDB；
- Milvus；
- Pixeltable；
- JSON/NPZ 文件型索引；
- 云端 VideoDB。

### 4.4 Agent 与应用

- 原生 function calling；
- LangGraph；
- Google ADK；
- 自定义 workflow/task tree；
- MCP；
- FastAPI、Flask、Gradio、Streamlit、Next.js。

## 5. 领域的核心矛盾

### 5.1 覆盖率与成本

看得越密，越不容易漏；但帧数、延迟和 token 也越高。

可行的折中不是只选一种抽帧方式，而是分层：

```text
低成本全局覆盖
  -> 查询/步骤相关定位
  -> 不确定项高分辨率回看
```

### 5.2 摘要效率与证据保真

把视频变成摘要有利于检索，却可能把错误固化。评价系统应同时保存：

- 压缩后的描述；
- 原始时间范围；
- 可重新读取的帧或片段；
- 生成描述所用的模型和版本。

### 5.3 语音与动作

“说了要做”不等于“真的做了”。ASR 可用于：

- 定位候选时间；
- 识别口述型步骤；
- 补充视觉不可见信息。

但对动作完成的证明仍应优先依赖视觉或明确的结果状态。

### 5.4 通用能力与领域可靠性

通用模型能识别常见动作，却不自动知道：

- 哪些步骤是评分标准；
- 何种偏差算 minor/major；
- 哪些设备状态是危险；
- 哪些遮挡应标人工复核。

因此领域模板不是普通 prompt，而是评价系统的可版本化规则资产。

### 5.5 本地隐私与云端能力

- 本地处理保护隐私、可控成本，但模型部署重。
- 云端原生视频模型使用简单、能力强，但涉及上传、费用和供应商锁定。

本轮项目覆盖两端：

- `watch-skill` 偏本地优先；
- `proteomics_lab_agent` 明确上传 GCS 并使用 Gemini；
- 全智评使用 OSS、DashScope 和本地/云 ASR 混合；
- `Director` 高度依赖 VideoDB 云服务。

## 6. 截至快照日的发展现状

### 已较成熟

- 视频采集、转码和基础抽帧；
- ASR、OCR 和帧描述；
- 片段级向量检索；
- LLM tool calling；
- 基于时间戳的问答和剪辑；
- FastAPI/MCP 等服务接口。

### 正在快速演进

- 主动感知和长视频工具调用；
- 查询相关选帧；
- 多模态记忆压缩；
- 推理时反思和验证；
- 从人工校正自动生成 eval/reward；
- 视频 Agent 与其他 Agent 框架的连接层。

### 仍没有通用解

- 如何证明一个动作真的完整发生；
- 如何处理遮挡、镜头外动作和极短瞬间；
- 如何校准“置信度”而不是让模型自报数字；
- 如何统一评估时序、空间、语音和领域知识错误；
- 如何在真实业务上证明长期改进；
- 如何让复杂 Agent 在失败、取消和重试时保持一致状态。

## 7. 对全智评的生态定位

全智评已经不缺“能调用多模态模型”的基本能力。它位于：

```text
通用视频理解
  + 领域模板
  + 多阶段评价
  + 教师复核
  + SaaS 业务闭环
```

其差异化不在于模型名字，而在于：

- 可维护的评分模板；
- 逐步骤结构化结果；
- 证据和置信度；
- 教师覆写和审计；
- 样本驱动评价改进。

下一阶段更有价值的技术问题是：

1. 把固定抽帧和主动回看组合成统一预算策略；
2. 建立可跨轮复用、能回到原时间轴的证据索引；
3. 让低置信复核由“重新取证”驱动，而非只让模型重写；
4. 把教师校正转成分类明确、可重放的回归样本；
5. 用真实步骤级 TP/TN/FP/FN 校准策略，而不是只看总分。
