## 3. 四个漂移项目

### 3.1 nanobot：不改变 Agent runtime 结论

唯一提交只改 3 个 WebUI 文件：

- 从 URL fragment 读取 native host port/token；
- 只允许 `127.0.0.1` loopback；
- Bearer token 长度和字符集校验；
- token 从 URL 清除后只保留在当前 tab 的 session storage；
- 通过 authenticated loopback endpoint 打开原生文件夹选择器。

`AgentLoop`、`TurnContext`、message bus 和 tool loop 没有变化。

### 3.2 LightRAG：并发预约从“布尔忙碌”升级为 owner/fence

34 个提交、19 个文件主要补三层可靠性：

1. keyed lock 记录 owner PID 与 process start id；
2. pipeline reservation 使用 owner token，release 必须验证所有权；
3. caller 被取消时，release/finalize 仍通过 join-resistant helper 跑完。

死 worker 恢复不是统一“清掉 busy 再重跑”：

| 死 owner 类型 | 默认动作 | 原因 |
|---|---|---|
| processing / scan | 回收 reservation，可重新执行 | 工作可重新扫描或处理 |
| pending enqueue token | 删除死 token，重算 count | token 本身不是已提交副作用 |
| custom chunks / delete / clear | 设置 `recovery_required` fence | 无法从状态推断外部修改完成到哪一步 |

fence 会拒绝新的 enqueue、graph mutation、reprocess、delete 和 clear；只有显式
force reset 才能清除。它证明了：

> “owner 死了”只说明 lease 可回收，不说明副作用可以安全重放。

本轮只做 upstream E1 diff，没有运行新增的 Linux multiprocess tests。

### 3.3 PageIndex：文档和 LLM 返回值都不再被信任

19 个提交只改 `pageindex/page_index.py` 和 `tests/test_page_index.py`，新增：

- 对文档中的常见 prompt-injection 文本做清洗；
- 用 `<user_document>` 包裹不可信文档，并防止文档伪造结束标签；
- physical index 必须处于真实页范围；
- chunk 生成的页码必须存在于当前 chunk marker 集；
- 更新 TOC 时条目数、`structure` 和 `title` 必须保持身份不变；
- 越界、伪造或无法解析的页码置空，而不是写入索引。

这比“在 system prompt 里说忽略恶意指令”更强，因为输出还要经过结构门。但它仍
是 pattern-based hardening，不代表未知注入形式全部解决。

### 3.4 Mem0：失败可见性、时效与 scope 更明确

161 个提交同时包含大规模 TypeScript OSS SDK 扩展和 Python core 修复。与教学记忆
最相关的变化：

- LLM fact extraction 失败改为抛出 `LLMError`，不再与“没有事实”共同返回 `[]`；
- `user_id`、`agent_id`、`run_id` 在统一入口做字符串化、trim 和校验；
- memory 支持 `expiration_date`，默认 search/get_all 隐藏已过期记录；
- read API 返回 `attributed_to`，避免 assistant 事实被误归因给 user；
- entity link 先做 normalized exact match，再做 semantic fallback；
- metadata-only update 不重建 entity links；
- vector store 增加 identifier/filter escaping，修复多种 SQL、Cypher、NoSQL 和
  search expression injection。

这强化了一个教育场景原则：

```text
no memory extracted
  != memory provider failed
  != memory expired
  != memory belongs to another scope
```

本轮没有运行 161 个提交对应的全套 Python/TypeScript tests。

### 4.2 Mastery Path

```bash
PYTHONDONTWRITEBYTECODE=1 \
/tmp/deeptutor-venv-20260717/bin/python -m pytest -q \
  deeptutor/learning/tests/test_guided_mastery_updates.py \
  deeptutor/learning/tests/test_policy.py \

  deeptutor/learning/tests/test_storage.py
```

结果：


覆盖：

- 单次正确 mastery 上限 0.5；
- 两次正确上限 0.8，三次后才可到 1.0；
- MEMORY/PROCEDURE 用 0.9 quantitative gate；
- CONCEPT/DESIGN 只接受 qualitative pass；
- gate 决定下一 objective，而不是线性 stage cursor；
- pending question、review、probe、practice、assess 的优先级；
- grade → attempt → mastery → scheduler → persistence。

### 4.3 仍不能写成“已证明学习有效”

`compute_mastery()` 只接收最近的布尔正确序列，不接收：

- 是否重复同一道题；
- 是否使用提示或查看答案；
- 题目难度；
- 是否为新场景迁移；
- baseline；
- 反应时间和解释质量。

现有测试还明确用同一 `question_id` 的重复正确答案提升 mastery。它适合作为产品内
progress estimate，不足以单独证明 learning gain。

## 6. 稳定结论

1. RAG 回答“依据什么”，learner model 回答“对谁、怎样讲”。
2. 教学 policy 必须位于可测试的 gate，不应只存在于长 prompt。
3. Mastery 是基于观测的估计，不是学生头脑状态的直接测量。
4. 一次答对要防 lucky guess，多次同题仍不能替代独立 transfer。
5. 学习者记忆必须有 scope、来源、时效、归因、纠错和删除。
6. 系统 E2 测试证明软件合同，不证明教学法带来真实学习增益。
7. 文档检索要把文档文本视为不可信输入，LLM 返回的结构也必须再验证。
8. 长任务恢复要区分可重放状态与不可安全推断的副作用。

## 7. 仍未证明

- DeepTutor 全功能、真实 provider 和 Web/CLI 的端到端部署。
- 14 个项目的统一安装和 benchmark。
- 同一教材在五种 RAG 引擎上的质量、成本与延迟。
- learner profile 长期准确性和用户纠错体验。
- DeepTutor mastery score 与真实 delayed transfer 的相关性。
- LLM qualitative assessment 与人类教师评分的一致性。
