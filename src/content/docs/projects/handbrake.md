## 是什么

**HandBrake** 是面向普通用户和创作者的开源**视频转码器**——把已有视频（DVD、相机文件、录屏、蓝光镜像）转成手机、电视、浏览器能播的 **MP4 / MKV / WebM**。底层调用 [[ffmpeg]]、[[x264]]、[[x265]]、SVT-AV1 等编码器，但用**图形界面和预设**把参数藏起来。

日常类比：[[ffmpeg]] 像**专业厨房**——刀工火候全自己控，强大但学习陡。HandBrake 像**电饭煲菜单**——选「煮饭/粥/蛋糕」对应编码预设，点开始就等。复杂调色仍得回 ffmpeg，但 80%「压个小体积给微信发」用 HandBrake 就够。

命令行等价：

```bash
HandBrakeCLI -i lecture.mov -o lecture.mp4 \
  -e x264 -q 22 --encoder-preset medium
```

`-q 22` 是恒定质量 RF 思路，数越小画质越高、文件越大。

## 为什么重要

不理解 HandBrake，下面这些事讲不清：

- 为什么开源媒体栈里「编码器很强」不等于「普通人能用」——需要产品化壳
- 为什么教程里「用 HandBrake 压 H.264」和「用 ffmpeg 一行命令」说的是同一件事
- 为什么 GPL 转码工具链和 [[x264]] / [[x265]] 授权要一起考虑分发
- 为什么消费级转码预设（Fast / HQ / Super HQ）是 rate-control 教学的入口

## 核心要点

1. **预设系统**：把分辨率、编码器、滤镜、音频码率打包成「Fast 1080p30」等名字，降低试错成本。

2. **编码器后端可切换**：H.264 走 x264，HEVC 走 x265，AV1 可走 SVT-AV1；容器 MP4/MKV/WebM 可选。

3. **滤镜管线**：去隔行、裁剪、缩放、降噪在转码前处理，避免二次编码损失。

4. **队列批处理**：桌面版可排队多个任务过夜跑，适合课程录像批量压缩。

5. **章节/DVD 源支持**：仍保留光盘爱好者工作流，底层 demux 走 libav。

## 实践案例

### 案例 1：课程录像压到 1080p H.264

```bash
HandBrakeCLI -i raw.mkv -o course.mp4 \
  --preset "Fast 1080p30" \
  --audio-lang-list eng --first-audio
```

预设已含缩放与帧率策略；只保留第一条英语音轨减小体积。

### 案例 2：恒定质量 RF 模式

```bash
HandBrakeCLI -i source.mov -o out.mkv -e x264 -q 20 -r 24
```

RF/CQ 让编码器按复杂度分配比特，静态幻灯片场景文件远小于固定码率。

### 案例 3：与 [[ffmpeg]] 分工

| 任务 | HandBrake | [[ffmpeg]] |
|---|---|---|
| 批量预设转码 | 强 | 需自建脚本 |
| 精确滤镜图 | 有限 | 滤镜链无限 |
| 流媒体切片 | 不做 | `ffmpeg -f hls` |
| 学习编码参数 | 预设入口 | 参数全集 |

训练侧 [[decord]] 读的是已转好码的视频；HandBrake 负责**把原始素材变成训练友好 mp4**。

## 踩过的坑

1. **只看预设名不看编码器**——同一「Fast」在 x264 与 x265 下速度差几倍。

2. **VFR 源直接压**——可变帧率录屏易音画不同步，先固定帧率或让 HandBrake 重采样。

3. **RF 太小 + preset 太慢**——RF 18 + veryslow 一夜转不完一小时 4K。

4. **GPL 分发混静态链**——二次打包应用要核对 [[x264]] 等许可证。

## 适用 vs 不适用场景

**适用**：
- 把素材转成通用 MP4 给手机/网页
- 不需要写脚本的一次性批量转码
- 学习 CQ/RF 与 preset 的入门实验

**不适用**：
- 实时直播转码（用 [[ffmpeg]] + 硬件编码）
- 复杂多路流混流、广告插播
- Video-LLM 训练内联解码（用 [[decord]] / [[torchcodec]]）

## 历史小故事（可跳过）

- **2003**：Eric Petit 在 macOS 上做 DVD 转码小工具
- **2006+**：开源社区接手，扩展 Linux/Windows
- **2010s**：随 x264 成熟成为 H.264 消费级标配
- **2020s**：加入 AV1（SVT-AV1）预设，跟进新编码标准

## 学到什么

1. **工具链需要「最后一公里」GUI**——编码器再强也要预设降门槛
2. **转码是重 CPU 批处理**——队列与 preset 比极限画质更影响体验
3. **容器+编码器+音频** 三件套要一次想清楚
4. **与训练管线分工**：HandBrake 产出文件，[[decord]] 负责读帧
5. **GPL 生态要整体看**：分发定制版需连带关注 [[x264]] 等组件许可

## 与同类对比

| 工具 | 界面 | 目标用户 | 灵活性 | 批处理 |
|---|---|---|---|---|
| **HandBrake** | GUI+CLI | 创作者 | 预设为主 | 队列 |
| [[ffmpeg]] | CLI | 工程师 | 无限 | 脚本 |
| [[shotcut]] | NLE | 剪辑 | 时间线 | 导出 |
| 厂商闭源转码 | GUI | 企业 | 许可证贵 | 集群 |

HandBrake 的价值在**把 x264/x265 参数翻译成可点击的预设**，降低「第一次能播」的门槛。
