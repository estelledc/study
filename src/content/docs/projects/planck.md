## 是什么
planck.js 是 Box2D 的 JavaScript 移植，让 2D 物理更易接入浏览器。。
类比：像把桌面物理引擎的规则翻译成浏览器“方言”，不重写原理。

典型入口通常是把上下游系统分成一个清晰的主链路：
- 依赖识别或资源发现
- 核心计算或转发
- 结果输出与可观测。


## 为什么重要
- 希望在浏览器里做确定性 2D 物理。
- 不想引入 WASM 或重度 C++ 构建链。
- 项目更注重轻量可控的刚体系统。
- 与可视化库整合成本低。
- 需要可复用碰撞、关节与传感器模型。

当你只盯最终接口时，你经常会误判是“模型/库本身慢”，实际上瓶颈往往在接入层。

## 核心要点
1) 机制分解：
- 复用 Box2D 的世界/刚体/fixture 抽象。
- Step 更新固定步长与时间积分控制。
- 提供 joints 与 contacts 监听器。
- 社区 API 覆盖常见物理调试工具。

2) 典型收益：
- 可控性提升、吞吐上升、异常定位更快。

- 拖拽交互：利用传感器检测接触而不产生物理反推。


## 踩过的坑
1. 混用像素和世界单位导致比例错乱。
1. dt 不稳定：帧率抖动直接影响仿真。
1. 频繁创建/销毁 body 时未清理 joint。
1. fixture 分类掩码复杂后难 debug。

## 适用 vs 不适用
**适用**：
- Canvas/WebGL 教育或轻游戏。
- 需要跨端部署的 2D 引擎原型。
- 想快速验证物理玩法。
- 强调确定性与可重放。

**不适用**：
- 高精度 3D 物理。
- 超大规模刚体场景。
- 严格实时策略游戏，需要极低抖动。
- 团队缺少物理参数调参经验。


## 历史小故事（可跳过）
关于 planck，更稳妥的使用方式通常经历了：
1. 小规模验证：先把 API 调通。
2. 配置收敛：把关键参数固定为一套默认值。
3. 指标加固：将耗时、错误、资源放入固定大盘。
4. 扩容验证：再验证不同负载窗口。
在社区讨论中，大家更关注“问题可复现性”与“回归成本”。


## 学到什么
1. 复杂系统先找一个稳定基线，再谈优化。
2. 参数不是“越大越好”，目标应是吞吐与稳定平衡。
3. 可观测指标比口头经验更有说服力。
4. 小步迭代能显著降低回归风险。

可以自测的三问：
- 现在的瓶颈到底在 CPU、IO 还是内存？
- 参数改动后是否会引入长尾延迟？
- 回归场景是否覆盖了异常输入？
