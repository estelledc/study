## 是什么

**x264** 是开源世界里** H.264/AVC 视频编码**的事实标准实现——用 C + 汇编把 DCT、运动估计、率失真优化写到极致。[[handbrake]]、[[ffmpeg]]、YouTube 早期转码栈都常默认或可选调用它。

日常类比：H.264 标准像**建筑规范**——规定墙厚、承重，但不告诉你怎么砌最快。x264 像**资深施工队**——在规范内用更聪明的脚手架（多参考帧、自适应量化、psy-RD）用更少砖块（比特）达到同样牢固（画质）。

典型 [[ffmpeg]] 调用：

```bash
ffmpeg -i in.mov -c:v libx264 -preset medium -crf 23 -c:a aac out.mp4
```

`-crf 23` 是恒定质量模式，`-preset` 在速度与压缩率间权衡。

## 为什么重要

不理解 x264，下面这些事讲不清：

- 为什么「H.264 开源编码」几乎默认指 x264——专利与实现成熟度平衡
- 为什么 rate-control 教程用 CRF/ABR/VBV 举例都绕不开 x264 参数名
- 为什么 [[handbrake]] 的 x264 预设能直接映射到 psy-tune/film/animation
- 为什么十年后的 [[x265]] / AV1 仍常拿 x264 做画质-码率基线

## 核心要点

1. **率失真优化（RDO）**：编码器搜索运动矢量、分区、QP，最小化「失真 + λ×比特」代价；psy-RD 加入感知权重。

2. **预设阶梯**：`ultrafast` → `placebo`，越慢相同 CRF 体积越小；直播用 `veryfast`，归档用 `slow`。

3. **CRF vs ABR**：CRF 恒定质量；ABR 目标平均码率；VBV 约束直播缓冲。

4. **多线程与切片**：帧级/帧内并行；切片利于低延迟 RTMP。

5. **tune 选项**：`film` / `animation` / `grain` 针对不同内容纹理优化 psy 模型。

## 实践案例

### 案例 1：课程录像 CRF 归档

```bash
x264 --crf 18 --preset slow -o archive.h264 --input-res 1920x1080 raw.yuv
```

低 CRF + slow preset 给长期保存；实际常经 ffmpeg 封装而非裸 yuv。

### 案例 2：直播低延迟

```bash
ffmpeg -i cam -c:v libx264 -preset veryfast -tune zerolatency \
  -g 60 -keyint_min 60 -f flv rtmp://live.example/app/stream
```

`zerolatency` 关 B 帧lookahead；短 GOP 便于切流恢复。

### 案例 3：与 [[x265]] / AV1 对照

| 编码 | 效率 | 解码普及度 | 编码 CPU | 典型场景 |
|---|---|---|---|---|
| **x264** | 基线 | 极高 | 中 | 通用分发 |
| [[x265]] | ~+40% 效率 | 高 | 高 | 4K/HDR |
| SVT-AV1 | 更高 | 增长中 | 很高 | 新平台 |

Video-LLM 训练更关心**解码速度**（[[decord]]）；x264 因硬解普及仍是兼容首选。

## 踩过的坑

1. **CRF 与 preset 混淆**——CRF 管质量，preset 管搜索深度；只调一个不够。

2. **10-bit 源用 8-bit 编码**——banding 明显；用 `libx264` 10bit 或保持高位深。

3. **B 帧过多拖延迟**——实时场景要 `tune zerolatency` 或限 B 帧。

4. **专利授权**——x264 以 GPL 发布，商业静态链接需合规方案。

## 适用 vs 不适用场景

**适用**：
- 通用 MP4/HLS 分发
- 学习 H.264 编码器内部机制
- 与 [[handbrake]] / ffmpeg 集成的默认高质量路径

**不适用**：
- 需要免 H.264 专利顾虑（考虑 VP9/AV1）
- 极限压缩 4K（[[x265]] / AV1 更省码率）
- 训练管线内联编码（离线批处理即可）

## 历史小故事（可跳过）

- **2004**：Loren Merritt 等启动 x264，迅速超越早期开源实现
- **2000s–2010s**：成为流媒体与蓝光外开源首选
- **2010s**：[[x265]] 承接 HEVC；x264 仍统治 AVC 兼容场景
- **现状**：mirror 组织维护 GitHub 镜像；工业界仍大量调用 libx264

## 学到什么

1. **编码器价值在 RD 搜索与实现细节**，不是标准本身
2. **CRF 是理解率控的第一课**
3. **preset 是 CPU 时间与码率的旋钮**
4. **分发格式要和解码生态一起选**——训练侧 [[decord]] 吃什么都行，播放端不行
5. **开源编码器可以成为事实标准**——实现质量比标准纸面更重要

## 与同类对比

| 编码器 | 标准 | 开源 | 硬解普及 | 编码速度 |
|---|---|---|---|---|
| **x264** | H.264 | GPL | 极高 | 快 |
| [[x265]] | HEVC | GPL/商业 | 高 | 慢 |
| [[libvpx]] VP9 | VP9 | BSD | 中 | 慢 |
| SVT-AV1 | AV1 | BSD | 增长 | 很慢 |

x264 仍是**兼容性终点编码器**：任何怀疑播放端能力时的安全默认。
