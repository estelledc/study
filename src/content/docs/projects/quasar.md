## 是什么

Quasar 是 Vue 生态中的“全端桥梁”：你写一套组件和路由逻辑，可以打出 SPA、SSR、PWA、Electron 桌面 App、移动端应用。日常类比：像买一栋可折叠沙发，一次买材料，平时当客厅沙发，临时改成床或办公桌。

传统开发常见情况是：桌面项目一套链路、H5 一套、移动一套。Quasar 的吸引力就是把这些“多入口”整合到一套约定与 CLI 能力里，减少重复维护。

Quasar 的卖点不是“更强大组件库”，而是“同一工程能跨目标发布”，并且生态支持从 CLI 到 CI 的完整闭环。

## 为什么重要

如果你把 Quasar 只当前端框架，会忽略它真正价值：

- 你会把相同业务逻辑在多个项目重写
- 你会在迁移到 PWA 或 Electron 时推翻原有路由策略
- 团队会把性能优化分叉到每个端单独做
- 你总觉得“全端统一”很虚，但其实是工程成熟度的问题

Quasar 用统一脚手架把“重复造轮子”替换成“统一分发策略”，对新团队非常友好。

## 核心要点

1. **CLI + 配置驱动**：`quasar create`、`quasar dev`、`quasar build` 按目标自动选择模板和打包策略。

```bash
npm init quasar
cd my-app
quasar dev -m pwa  # 同一项目改为 PWA 模式启动
quasar dev -m electron # 桌面预览
quasar build -m ssr    # 服务端渲染构建
```

2. **组件生态与路由一致性**：同一套 UI 组件库可在 web/mobile 视觉一致性下复用。
3. **部署模式可切**：`-m` 参数决定目标（spa/ssr/pwa/electron/cordova/capacitor/cordova/bex）而不是改代码。

## 实践案例

### 案例 1：同一项目从 Web 跳到 PWA

```bash
quasar dev -m pwa
```

流程：
- 保持 router、store 不变
- 仅调整 service worker 与 manifest
- 添加离线缓存策略后，关键页面在弱网可预缓存访问

核心结果：不用再维护第二个仓库。

### 案例 2：同一业务逻辑复用到桌面端

```bash
quasar build -m electron
```

注意三点：
- 桌面端只改变打包与窗口壳
- 同步复用同一份 Vue 组件与状态树
- 插件层按运行时自动注入 Node 环境能力

### 案例 3：SSR + SEO 的最小链路

```bash
quasar dev -m ssr
quasar build -m ssr
```

SSR 下你会得到服务端可渲染页面，适合需要首屏 SEO 的公开内容；同样项目不需要重写内容层。

## 踩过的坑

1. **过度依赖自动配置**：升级依赖后默认配置可能变化，建议先在 `quasar.config.js` 中冻结关键行为。
2. **Bundle 体积上升**：一锅端式支持多端会导致依赖树膨胀，要用生产分析插件定期审计。
3. **移动端插件适配偏移**：某些 web-only 组件在 capacitor/cordova 下行为不一致，需要平台级 conditional rendering。
4. **SSR 与状态初始化边界**：在服务端与客户端共享 store 时，初始化逻辑要避免竞态。

## 适用 vs 不适用场景

**适用**：
- 新项目需要同时覆盖网页、移动、桌面
- 团队重视统一组件和同一套仓库协作
- 产品演进快，需多目标并行验证 MVP

**不适用**：
- 想要只做单一端极致微调的场景
- 对底层打包链路有特殊定制要求（例如非 Vue 生态）
- 技术债已经在别的多端框架中高度绑定

## 历史小故事（可跳过）

- **2016 前后**：Vue 周边工具生态逐步成熟，Quasar 逐渐沉淀成统一框架。
- **2020s**：Quasar 在移动与桌面跨端里形成标准化脚本。
- **近两年**：围绕 build mode 的插件和社区文档补齐，应用规模扩大。
- **今天**：仍保持“同一套工程，多端编译”的核心定位，适合产品主干持续交付。

## 学到什么

- 多端不是一项“库能力”，而是工程治理能力。
- 把变更点推到配置层，能显著降低端间重复。
- 路由、状态、组件只做一遍是高复用的最直接路径。
- 平台特性差异必须显式声明，不然“统一”会变成隐性 bug。

补充说明（上线与排障）

真实项目里，Quasar 的体验问题常出现在“看似没变更核心代码”的情况下。以下顺序通常有效：

- 先确认目标模式是否正确（-m 参数）
- 再确认环境变量是否对齐（SSR/PWA 需要不同）
- 最后再看路由与缓存策略

## 迁移核对清单

1. 统一抽离环境变量：Node / 浏览器代码分离
2. 把 API 层改成可注入服务，不要放在组件顶层直接执行
3. 对 `localStorage` 与 `session` 的边界做显式封装
4. 在 PWA 与 SSR 之间复用同一份 store 初始化策略
5. 打包前运行体积分析，确认首屏资源可接受

如果你在 Electron 模式突然变慢，多半不是框架本体，而是 renderer 进程加载了过多非必需依赖。

另外一个常见误区是把“跨端能力”理解成“无需平台判断”；实际上 `mobile` 与 `web` 常有能力差异，至少要在两个端写一次可用性验收。

后续建议把最关键路由提前放入可回归脚本：

- 首屏路由加载时长
- 首次交互到响应时间
- 多窗口状态下的内存增长

有了这三项，你会更快判断是框架配置问题，还是业务组件本身耗时。
