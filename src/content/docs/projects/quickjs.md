## 是什么

**QuickJS** 是由 Fabrice Bellard 开发的轻量级 JavaScript 引擎。
它强调两个目标：

1. 小、
2. 可嵌入。

官方介绍很明确：单文件 C 代码就能做可执行内核，启动快、依赖少。

如果你想给工具、CLI、游戏脚本、物联网组件提供 JS 执行能力，但又不想带完整运行时负担，QuickJS 是常见选择。

## 为什么重要

很多项目只需要“给应用一个脚本执行层”而不是“完整浏览器”。

QuickJS 的价值点：

- 可直接嵌入 C/C++ 工程；
- 启动快、二进制小、无外部运行时依赖；
- 支持现代语法（至 ES2025 系列）；
- 支持解释执行与离线编译。

它比 `node` 更轻，通常更适合“运行时附加价值”场景，而不是服务端 Web 平台的全栈框架。

## 核心要点

### 1. 小而完整的内核

从工程角度，QuickJS 的设计目标很像“可搬运的引擎积木”。

- 体积小到适合嵌入；
- API 相对集中，不会像大运行时那样分散。

### 2. 解释器 + 编译器 `qjsc`

项目里既有 `qjs`（解释器），也有 `qjsc`（编译器）：

- `qjs`：运行脚本；
- `qjsc`：把脚本编译为可执行体或中间产物，减少部署依赖。

### 3. 内存模型和 GC

QuickJS 使用引用计数（ref-count）为核心思路，并配套循环回收。对于嵌入式场景，这是可预测性较好的权衡。

### 4. 低开销生命周期

启动时间和运行时实例创建时间都很短，适合大量短命脚本运行。

### 5. 特性覆盖

官方文档声称近乎完整的 ES2025 支持（按版本迭代逐步更新），足够覆盖多数脚本语法需求。

## 实践案例

### 案例 1：配置脚本执行

你的主进程只接受 JSON/脚本化策略。

- 主系统通过 QuickJS 加载策略脚本；
- 策略更新时只替换脚本文件，不重启大系统。

### 案例 2：CLI 的自定义扩展

开发 CLI 时让用户写表达式。\n
- 用 `qjs` 解释执行，
- 快速返回结果，减少外部依赖。

### 案例 3：游戏或仿真中的嵌入脚本

核心程序 C/C++ 实现物理和状态机，行为规则交给 JS 编写。

- 行为安全边界可控；
- 热更新更容易。

## 踩过的坑

1. **把它当 Node**：QuickJS 不是 Node 生态，不直接带 `require`/NPM 模块系统。
2. **垃圾回收心智模型误用**：引用计数看似直观，但循环引用需要注意。
3. **标准库差异**：某些 Node 内建模块在 QuickJS 中不存在。
4. **大项目依赖重注入**：你需要显式暴露 C 接口给脚本，不像服务端 JS 有现成模块体系。
5. **跨平台构建细节**：原生嵌入层需要针对不同目标平台写兼容代码。

## 适用场景

- 工具链、CLI、嵌入式应用、IoT/游戏中脚本执行器。
- 需要轻量且低启动延迟的 JS 运行环境。
- 需要可控内存和可裁剪特性的部署模型。

## 不适用场景

- 需要完整 Node.js 生态（例如大量 npm 包兼容）。
- 需要浏览器宿主 API（DOM、Web Crypto 完整集合）。
- 高级模块打包生态要求太强。

## 历史小故事

QuickJS 经常被提到的原因不是“又一个 JS 引擎”，
而是“它重新定义了可嵌入 JS 的成本边界”。

相比传统重运行时，QuickJS 显示：

- 当你的目标是“运行脚本能力”，不一定要引入庞大引擎。

这推动了很多轻量运行时方案，让脚本能力从“重服务”变成“可按需挂件”。

## 学到什么

1. 运行时选择要看问题边界：你是要 JS 平台，还是 JS 能力。
2. 嵌入式项目中，启动时间和体积常比最完整兼容更关键。
3. API 覆盖不是单向线性：少很多能力也能换取更高可控性。
4. QuickJS 是“少即是多”：接口少，但稳定。

补充一段选型判断：

- 如果你只要 `eval`、少量标准库和短生命周期脚本，那么 QuickJS 比 V8 / QuickJS-ng 更易嵌入。
- 如果你要跑复杂大前端应用，QuickJS 的兼容边界会先于打补丁能力成为瓶颈，不妨先用 WASI 或 duktape 做阶段化对比。
- C 端项目里，`qjs`/`qjsc` 的 CLI 能做 smoke test，但上线前一定配合最小 fuzz 测试。
- QuickJS 的优势是体积和可控性，代价是生态兼容和部分现代语法补齐。
- 这条线上的真正成本不是“功能是否有”，而是“你是否允许在每次升级时自己承担更多兼容测试”。

补充建议：

- 如果你希望快速验证脚本能力，先用最小输入跑 20~30 个脚本样例，再决定是否接入完整 CI。
- 生产场景里建议保留回归基线：一组标准语法兼容测试 + 一组边界脚本，避免升级后语义悄悄漂移。
