## 是什么

SuperCollider（SC）是一个面向实时音频合成的开发环境，核心包含两层：服务器端的 DSP 执行引擎 `scsynth` 和语言端 `sclang`。

日常类比：你在厨房里做汤，`sclang` 负责写“配方”，`scsynth` 负责“真正炒出来”。配方写得清楚，口味执行就稳定。

SC 的关键不是“能不能出声”，而是“能不能把声音当可编程对象”持续迭代。你可以实时改动参数、连音色合成、动态调度节奏并在短时间内听见反馈。

## 为什么重要

不理解 SC 的工作模式，以下问题会反复出现：

- 为什么音乐系统里“实时性”比离线渲染更重要。
- 为什么一段算法听起来像“音色”，另一段像“噪声”，其实都在同一套参数图里。
- 为什么同样是循环，节奏抖动通常来自时序调度，不是声卡坏。
- 为什么学电子音乐很需要先学“时间、振幅、包络”三个概念再谈作曲。

SuperCollider 把“音频工程 + 算法思维 + 编程范式”放到同一个体系，适合做算法作曲和实时交互。

## 核心要点

1. **Client/Server 架构**：`sclang` 负责语言层描述，`scsynth` 负责音频图（UGen）运算。
2. **Unit Generator（UGen）图**：所有声音都由 UGen 组合。你不是直接写“播放文件”，而是画出一条 DSP 图。
3. **时间是第一等公民**：`TempoClock` / `SystemClock` / `AppClock` 决定事件何时发生，决定听感是否有拍子。
4. **模式与参数可编程**：用语言构造音色、节奏、调度，参数可以来自函数、随机、规则系统。
5. **可观测反馈回路**：监听器、缓冲、FFT 分析帮助你把“听起来像”转成“可调的证据”。

## 实践案例

### 案例 1：最小音色与触发

```text
// 先启动 server（首次执行后确认 connected）
s.boot;

SynthDef(
  "beep",
  {
    arg freq = 440, amp = 0.2;
    var env = Env.perc;
    Out.ar(0, SinOsc.ar(freq) * EnvGen.kr(env, doneAction: 2) * amp ! 2);
  }
).add;

Synth("beep", [\freq, 880, \amp, 0.1]);
```

这是“语言层描述 + 服务器合成”的最小闭环。

### 案例 2：节奏事件流

```text
Pbind(
  \instrument, \beep,
  \dur, 0.25,
  \freq, Pseq([330, 440, 550, 660], inf),
  \amp, 0.12
).play;
```

`Pbind` 的核心不是复杂代码，而是把“乐句”写成规则流。规则一旦写对，节奏自动稳定。

### 案例 3：给音色加噪声与包络变形

```text
SynthDef(
  "dirty",
  {
    arg freq = 220, amp = 0.15, gate = 1;
    var env = Env.adsr(0.01, 0.2, 0.4, 0.3).ar(doneAction: 2);
    var osc = Saw.ar(freq, 0.3) + (WhiteNoise.ar(0.02));
    var filt = LPF.ar(osc, XLine.kr(8000, 500, 0.4));
    Out.ar(0, filt * env * amp ! 2);
  }
).add;

Synth("dirty", [\freq, 220]);
```

同样的 DSP 模块，通过包络和滤波组合，听感会从“锐利”变为“可控”。

## 踩过的坑

1. **时序漂移**：同一代码在不同系统上延迟不同，先选统一时钟，再做 latency 补偿。
2. **资源未释放**：Demo 里不断创建 synth 而不停止，会累积 CPU 与节点，建议统一管理生命周期。
3. **UGen 名字理解错误**：一个 `SinOsc` 和 `Pulse` 的差别，不只是波形；在频谱上会直接决定可听结果。
4. **Buffer 与采样率**：录音回放或采样类项目若采样率不一致，音高与时长都会偏移。

## 适用 vs 不适用场景

**适用**：
- 需要实时交互音乐、算法作曲、声音可视化实验。
- 想做可重复、可参数化的音频流程（不是单次录音工具）。
- 研究音色合成、音高/节奏系统与认知实验。

**不适用**：
- 只需要简单“播放音频文件”的产品场景。
- 不想维护音频服务器依赖的移动端轻应用。
- 团队对音频 DSP 无基础，但对 UI 有强交付要求的商业快速项目。

## 历史小故事（可跳过）

- **1990s-2000s**：SC 生态在学校和实验音乐圈快速生长。
- **2000s**：server/client 分层逐渐稳定，超越很多“脚本化音频工具”。
- **2010s**：算法作曲与 live coding 社区把 SC 推入更多实践场景。
- **2020s**：和新硬件、机器学习、可视化工具联动，SC 不只写乐器，也写实验平台。

## 学到什么

1. 真正的实时音频不是“发一段字节”，而是构造一个可控的 DSP 计算图。
2. 参数化是 SC 的核心生产力：一行 `Pseq` 可能替代大量手工乐谱。
3. 时钟、缓冲、包络共同决定“听起来像不稳定”，工程上先修时钟后调音色。
4. SC 的开发节奏是先听、再改；不是先建完完整模型再试音。

## 额外补充

- 从教学场景看，先确定时间模型（clock）再谈合成器参数，错误率会明显下降。
- 从工程场景看，SC 的 server/client 边界清晰，能让多人协作更安全。
- 常见节奏和参数 bug，最省时的处理方式是缩小实验窗口并保留监听器快照。
- 若项目以交互表演为核心，建议先固定事件调度，再引入更复杂的图形化控制。
