## 是什么

Boa 是一个用 Rust 实现的 ECMAScript 引擎。它的定位不是直接打败 V8 / SpiderMonkey，而是给 Rust 生态提供可嵌入、可审计、可改造的 JS 运行时。

日常类比：相当于你把一个“黑盒浏览器解释器”改成了可组装的机床：需要多少齿轮自己加，坏了哪颗螺丝也能自己修。

## 为什么重要

Rust 生态里一直缺一个“足够轻量、足够标准、足够易嵌入”的 JS 引擎时，Boa 就很实用。

- 适合在 Rust 应用里跑受控脚本
- 可定制语法/内存策略
- 适合做安全边界（最小 host 接口）

## 核心要点

### 引擎入口与值系统

解析/执行流水线覆盖词法分析到字节码或中间表示。

- ECMAScript 兼容是第一目标。
- Value 系统需要清晰的类型标记，避免运行时错误。

### 垃圾回收与内存模型

安全运行时离不开 GC 与宿主分配边界。

- 与 Rust 所有权模型协作时，边界管理尤为关键。
- 对对象生命周期的静态/动态策略需要配套。

### 执展能力

可通过模块机制提供 host function。

- 主机暴露 API 时要做白名单。
- 错误类型要映射为 Rust 可处理的错误码。

## 实践案例

### 案例 1：配置脚本执行器

把用户策略、过滤规则写为受控脚本。

- 固定上下文注入变量
- 禁止高危全局对象
- 执行超时和资源配额

### 案例 2：插件隔离系统

每个插件单独上下文，失败不污染主系统。

- 载入前做 feature 白名单
- 执行中监控耗时与内存

### 案例 3：教学型 JS 解释器

用于讲解词法作用域、原型链、闭包。

- Boa 代码可读性高于很多“商业引擎”包装层。

## 踩过的坑

1. **兼容与性能平衡**：严格标准和轻量实现难统一。
2. **标准覆盖率认知偏差**：别把“能跑”当“ES 完全兼容”。
3. **宿主对象白名单**：少配一个对象，插件可能越界。
4. **GC 与耗时控制**：脚本注入后最容易失控的是死循环和大数组。
5. **错误回传体验**：JS 异常要映射为清晰可观测的 Rust 错误。

## 适用 vs 不适用场景

适用：

- 希望 Rust 系统内嵌安全脚本的场景
- 需要可维护、可审计、可定制的 JS 层
- 想做高信任边界的规则引擎

不适用：

- 高吞吐浏览器级 JS 执行（应优先 JS 引擎大厂实现）
- 需要完整前端生态工具链支持
- 对标准追赶速度敏感的产品

## 历史小故事（可跳过）

- ECMAScript 引擎常由 C/C++ 历史主导，Rust 生态推动了“安全可控”替代路径。
- Boa 先把核心可用性打稳，再逐步补齐标准。
- 在很多嵌入场景中，它比“全量大引擎”更容易治理。

## 学到什么

1. 嵌入式 JS 最大值不在性能榜第一，而是安全边界可控。
2. 生态完整度和兼容性不能一次到位，版本迭代要清晰。
3. 宿主绑定接口要最小化。
4. 运行时错误与资源控制必须前置。
5. Rust 与 JS 接口设计是全链路工程。

## 运行时与安全补充

### 安全边界

- 只暴露最小能力集给脚本。
- 白名单方法优先于黑名单拦截。
- 为每次执行设置时间、内存与调用深度上限。

### 性能测试建议

1. 构造小脚本基准：算术、对象创建、模块加载。
2. 构造大对象基准：长数组、嵌套对象。
3. 运行时错误回收率统计。
4. 与 baseline（例如 quickjs）对比启动与执行延迟。

### 集成建议

- 主应用与引擎通信采用 DTO，减少不可控共享对象。
- 与日志系统打通异常堆栈，保留原始执行位置。
- 对外模块加载加签与来源校验。

### 兼容性和生态

- 逐步扩展语法特性，避免一次性打满兼容目标。
- 标准覆盖不足的地方明确注释降级策略。
- 社区生态未覆盖的 API 先转到 host function。

### 小结

Boa 的实用价值在于：
- 可控、可查。
- 适合作为规则执行、DSL 扩展、插件沙箱。
- 当你需要在 Rust 中把脚本能力内置进产品时，它是“可审计优先”路线的重要选项。
