## 是什么

Aubio 是音频处理库，常见用途是从原始声音中提取“可计算特征”。

日常类比：你在听歌时可以立刻分辨“鼓点”“音高”“节奏”，
Aubio 就是在代码里做这件事。

你不必先理解所有 DSP 公式，先把目标任务定义清楚：
检测 onset（何时敲击）、检测音高、估算 tempo。

## 为什么重要

- 语音、音乐应用里，前置特征提取比建模更重要。
- 没有稳定的特征提取，后续分类、检索都容易抖动。
- Aubio 覆盖 C 接口且有命令行工具，适合做实验与生产原型。

## 核心要点

1. **onset 检测**：识别声音的“事件边界”。

- 类比：你听到鼓点从“沉默”转到“命中”。
- 对节奏分析、动作同步很关键。

2. **pitch 检测**：估算基频轨迹，得到音高变化。

- 类比：把复杂波形浓缩成一条“音高线”。
- 实战里常用于转调、字幕、音高可视化。

3. **tempo 跟拍估计**：从帧流中恢复每分钟拍点。

- 类比：把混沌音乐节奏转成统一节拍。
- 后续可用于 DAW 或步进播放器。

## 实践案例

### 案例 1：检测鼓点并输出时间戳

```bash
aubio onset -i drum.wav -H 4 -O 10
```

- 输出时刻可用于剪辑或同步动画。
- 先调参数，再看召回率，而不是先追求“看起来很对”。

### 案例 2：估算音频主音高

```bash
aubiopitch -i vocal.wav -p yin -B 256 -H 512
```

- 你会看到一系列 pitch 帧。
- 真正难的是滤除异常跳变和异常短时噪音。

### 案例 3：快速节拍估计

```bash
aubiotrack -i mix.wav
```

- 得到 BPM 与拍点后可以驱动播放器节奏网格。
- 如果场景有变拍（rubato），需要分段处理。

## 踩过的坑

1. **没做前置滤波**：噪声会被误识别成事件。
2. **参数默认值通吃**：不同录音长度和采样率需独立调。
3. **把工具输出当真值**：仍需人工抽样评估。
4. **忽略实时性约束**：离线与实时算法不一定同一套阈值。

## 适用 vs 不适用场景

**适用**：
- 音乐分析、节拍检测、音频可视化。
- 教育产品里做发声反馈。
- 小型推荐/索引系统中的音频特征抽取。

**不适用**：
- 只需播放而不做分析的场景。
- 高度噪声环境下无降噪链路。
- 不想投入 DSP 学习成本的快速原型。

## 学到什么

1. 音频项目先选好特征，不是先堆网络层。
2. onset、pitch、tempo 各自承担不同目标。
3. 算法参数必须靠数据回放调优。
4. 一个小项目也需要监控误检率与漏检率。

## 额外补充（可跳过）

- 音频特征工程里，采样率变化和窗口长度是最容易让实验不可复现的两个参数。
- `aubiotempo` 可以给你 baseline BPM，但生产里通常要二次滤波。
- 对实时推理而言，帧长和重叠率直接决定误检/漏检平衡。
- 你可以先用命令行验证思路，再把参数迁移到 Python/C 接口。

```bash
# 快速 sanity check：确认音高和节拍结果同时可视化
aubiotrack -i input.wav -t > /tmp/track.csv
aubio onset -i input.wav -j --method hfc > /tmp/onset.txt
```

- 将两份输出叠在一起，你会看到“音乐结构变化”与“节拍结构变化”如何错位。
- 不要把一个工具当成通用真理源头，把它当成可解释的特征提取步骤。
- 音频系统的可用性，不在于能否提取全部指标，而在于能否稳定、可调、可观测。

## 补充小结（可跳过）

- 先做一版 baseline，再逐步调 `win_s`, `hop_s`, `threshold`，不要一次上来就追求完美参数。
- 如果你要做实时音频产品，先定义输入链路：采样率、缓冲、回放策略。
- 看到报错时优先检查预处理流程，尤其是静音裁剪与归一化。
- 对接模型前先确认特征稳定性：同一首歌不同设备要尽量一致。
