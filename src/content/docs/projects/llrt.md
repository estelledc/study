## 是什么

LLRT（Low Latency Runtime）是 AWS Lambda 方向的轻量 JavaScript 运行时，核心目标是**降低冷启动成本**。日常类比：你以前上车先要启一整车引擎系统（Node.js）才跑服务，LLRT 更像把关键零件预热到可直接启动的状态，先跑起来再说。

它基于 QuickJS，强调更小体积、更快启动、适配 serverless 生命周期。LLRT 并不是“替代 Node 一切功能”的方案，而是“在 Lambda 上先要快、成本更可控”的方案。

对于事件驱动系统，响应时间波动比平均延迟更关键；LLRT 的价值主要在于减少冷启动抖动，把“请求刚到来”的第一印象时间压低。

## 为什么重要

你若把所有 lambda 迁都押在 Node.js 标准运行时，可能会出现：

- 大量业务触发一次性执行、但启动开销高
- 同样内存下并发上升，实例更容易被频繁回收
- 成本曲线不可解释，优化点看起来都在应用层

LLRT 的角色在于把“基础运行时成本”作为优化对象，而不是盯着业务代码局部改法。

## 核心要点

1. **QuickJS 引擎 + Rust 层封装**：更轻量的执行模型，配合 Lambda 自定义运行时流程。

```bash
# 示例：在 Lambda 中用 LLRT handler
export const handler = async (event) => {
  const payload = JSON.parse(event.body || '{}');
  return {
    statusCode: 200,
    body: JSON.stringify({ ok: true, route: payload.route || 'default' }),
  };
};
```

2. **三种接入模式**：自定义 runtime、Lambda layer、容器镜像。
3. **关注兼容边界**：LLRT 不是 Node 完全替代，Node-only API 需要查兼容矩阵再接入。

## 实践案例

### 案例 1：冷启动敏感接口先行改造

场景：一个 API 每分钟仅调用少量，但业务要求 99 分位低抖动。

```text
- 把关键路径 handler 改为 LLRT 兼容入口
- 保持事件结构不变
- 对外暴露仅 HTTP 返回的常规签名
- 通过基准测试记录 p50/p95 冷启动差异
```

收益：平均启动时长下降明显，尾延迟更稳定。

### 案例 2：Layer 方式复用 runtime

```text
- 在构建时下载 llrt bootstrap
- 打包公共依赖到同一层
- 多个函数共享同层镜像，减少重复安装体积
```

优点是升级 runtime 可集中管理，缺点是版本同步要比每函数自带更严格。

### 案例 3：容器镜像部署示例

```dockerfile
FROM --platform=arm64 busybox
WORKDIR /var/task/
COPY app.mjs ./
RUN chmod +x /usr/bin/llrt
CMD ["llrt", "app.mjs"]
```

容器路径下仍是 Lambda 生命周期控制，只是部署形态更可控。

## 踩过的坑

1. **兼容性预期差异**：某些 Node 内置 API 行为并不等价，先读兼容矩阵再上线。
2. **打包体积管理**：把过多外部依赖一起打进去会抹掉“轻量”优势。
3. **指标盲点**：只盯平均延迟会忽略冷启动尾部，建议观察 p95/p99。
4. **文档与实践差异**：示例里可运行路径和真实账号权限常有差异，需在 staging 先过一轮。

## 适用 vs 不适用场景

**适用**：
- Lambda 冷启动明显影响体验
- 函数短小、触发频率不高但要求稳定响应
- 成本敏感服务，愿意牺牲部分 Node 完整性换取启动优势

**不适用**：
- 重度依赖不兼容 API 的代码库
- 不希望维护额外 runtime 兼容与构建步骤
- 需要完整 Node 生态功能的场景

## 历史小故事（可跳过）

- **早期**：QuickJS 在轻量运行时方向成熟，LLRT 借此切入 serverless。
- **上线期**：与 AWS custom runtime 对齐，形成可落地的冷启动优化策略。
- **迭代阶段**：加入 layer/container 等接入方式，降低改造成本。
- **生态阶段**：围绕测试与兼容矩阵补齐文档边界。
- **当前**：在评估与真实使用中，更多团队把 LLRT 作为“按场景替换”选项。

## 学到什么

- serverless 的性能不只看函数体，runtime 选型影响尾延迟。
- 轻量运行时意味着更严格工程流程，但收益是更稳定。 
- 能否量化 p95/p99，决定你是否值得切换。
- 兼容边界是底线，不能把“快”当成盲目替代。

补充说明（治理与发布）

LLRT 在服务端场景好用，但很多团队会跳过“非功能性”验证，这通常是最大坑：

- 先量延迟，再看兼容率，不要只优化一个维度
- 先在压测环境确认异常日志格式稳定，再放大并发
- 确认所有依赖是否明确打包到 runtime 可见目录

## 落地检查清单

1. 冷启动测试：首次调用与第二十次调用对比
2. 内存曲线：并发上升后是否出现异常波动
3. API 覆盖：Node-only 调用是否存在回退路径
4. 兼容失败处理：错误码归类后才能排查根因
5. 部署回滚：发布失败时是否能一键切回原 runtime

LLRT 的实践不是“改一个运行时就结束”，而是建立一套可持续的回归机制：

- 同步版本
- 持续基准
- 兼容扫描
- 按环境分段发布

如果只做第一步，不看第二步，收益会很快被噪声吞掉。

总结性建议是：先在最关键 2~3 个接口验证，确认收益后再逐步扩展。
