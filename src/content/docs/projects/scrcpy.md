## 是什么

scrcpy 是一个把 Android 手机画面和声音投到电脑上，并允许用键盘鼠标控制手机的开源工具。日常类比：像把手机临时接成一台"电脑外接屏"，但这块屏还能反向把鼠标、键盘动作送回手机。

最小使用方式通常只有一行：

```bash
scrcpy
```

前提是手机打开 USB 调试，电脑能通过 `adb devices` 看到它。scrcpy 会把一个临时服务推到手机端，让手机把画面编码成视频流，再通过 USB 或 TCP/IP 发给电脑端显示。

它的关键特点不是"功能特别花"，而是轻：不用 root，不需要在手机上长期安装 App，跨 Linux / Windows / macOS，启动后只显示设备屏幕本身。

## 为什么重要

不用 scrcpy，下面这些事会变得很笨重：

- 想给别人演示手机操作，只能举着真机或用慢吞吞的模拟器。
- 想录一段 Android 操作视频，还要额外找录屏 App，再导出文件。
- 想在电脑上输入长文本到手机，得在小屏幕上慢慢点。
- 想调试真机应用，又希望看到真实设备表现，模拟器和截图都不够直接。

## 核心要点

1. **实时投屏像水管送水**：手机端负责"取水"和压缩，也就是把屏幕画面编码成 H.264 / H.265 / AV1 这类视频流；电脑端负责"接水"和显示。水管越顺，延迟越低；分辨率、帧率、码率越高，水压越大。

2. **控制输入像远程按按钮**：电脑鼠标键盘不会魔法般进入手机，而是被 scrcpy 转成 Android 能理解的输入事件。遇到某些设备限制时，可以改用 UHID / AOA 这类更像"物理键盘鼠标"的模式。

3. **功能组合像积木**：投屏、录制、摄像头、虚拟显示、无线连接、只控不看都不是独立 App，而是命令行选项的组合。学会几个核心选项，就能拼出自己的工作流。

## 实践案例

### 案例 1：高质量低延迟投屏给演示或调试用

官方 README 给的常见命令是：

```bash
scrcpy --video-codec=h265 --max-size=1920 --max-fps=60 --no-audio --keyboard=uhid
scrcpy --video-codec=h265 -m1920 --max-fps=60 --no-audio -K
```

**逐部分解释**：

- `--video-codec=h265`：让手机用 H.265 编码，通常同等码率下画质更好。
- `--max-size=1920` / `-m1920`：限制画面最大边，避免 2K / 4K 手机把电脑和 USB 带宽压满。
- `--max-fps=60`：把最高帧率压到 60fps，演示和大多数调试已经够用。
- `--no-audio`：不转发声音，减少一个可能出问题的通道。
- `--keyboard=uhid` / `-K`：把键盘模拟得更像真实外接键盘，解决部分输入兼容问题。

这个案例适合会议演示、真机调试、录屏前预览。它不是追求"所有参数拉满"，而是在画质、延迟、稳定性之间取一个稳妥点。

### 案例 2：把手机摄像头录成电脑上的 MP4 文件

官方 README 里的相机录制示例是：

```bash
scrcpy --video-source=camera --video-codec=h265 --camera-size=1920x1080 --record=file.mp4
```

**逐部分解释**：

- `--video-source=camera`：视频源不再是屏幕，而是设备摄像头；官方文档说明这需要 Android 12+。
- `--camera-size=1920x1080`：请求 1080p 相机画面；不同设备实际支持的尺寸可能不同。
- `--video-codec=h265`：用 H.265 记录更高压缩效率的视频。
- `--record=file.mp4`：电脑端把收到的音视频流封装成 MP4 文件。

录制文档还提醒：`--record` 捕获的时间戳来自设备端，所以网络或 USB 抖动不会像"录电脑窗口"那样污染最终文件。想只录不看窗口时，可以加 `--no-playback` 或 `--no-window`。

### 案例 3：无线连接或虚拟显示，把手机当一台轻量远程环境

无线连接的官方快捷方式是：

```bash
scrcpy --tcpip
scrcpy --tcpip=192.168.1.1:5555
```

虚拟显示的官方示例是：

```bash
scrcpy --new-display=1920x1080 --start-app=org.videolan.vlc
```

**逐部分解释**：

- `--tcpip`：先通过 USB 找到手机，再让 adb 切到网络连接；电脑和手机必须在同一网络。
- `--tcpip=192.168.1.1:5555`：当设备已经开放 adb 网络端口时，直接指定地址。
- `--new-display=1920x1080`：在 Android 上创建一个新的虚拟显示，不直接占用主屏内容。
- `--start-app=org.videolan.vlc`：在这个虚拟显示里启动指定 App，否则有些设备上可能什么也不显示。

这个案例适合临时摆脱数据线，或者把某个 App 放到单独窗口里跑。代价是无线链路更容易受网络质量影响，虚拟显示也依赖 Android 设备对多显示的支持。

## 踩过的坑

1. **`adb devices` 看不到手机**：scrcpy 依赖 adb 初始化连接，USB 调试、授权弹窗、驱动或数据线任何一环出问题都会失败。

2. **多台设备同时连接会报错**：官方 FAQ 要求用 `-s` 指定序列号，或用 `-d` / `-e` 选择 USB / TCP/IP 设备。

3. **鼠标键盘不能控制某些手机**：部分设备需要额外打开"USB debugging (Security Settings)"这类允许模拟输入的开发者选项，并重启后才生效。

4. **音频不是所有 Android 都能转发**：官方文档说明音频转发主要支持 Android 11+；Android 10 或更早版本会自动变成只有视频。

## 适用 vs 不适用场景

**适用**：

- 真机 App 调试，需要在电脑大屏上看手机动作。
- 会议演示、教学录屏、Bug 复现视频，需要快速投屏和录制。
- 想把电脑键盘鼠标临时当成手机输入设备。
- 想用命令组合控制画质、帧率、录制格式，而不是依赖固定 GUI。

**不适用**：

- 想完全不经过调试授权地远程控制陌生手机；scrcpy 不是远控木马。
- 想长期云端托管 Android 桌面；它更偏本机调试和近场投屏。
- 想保证所有 App 音频都能被抓；部分 Android 版本和 App 策略会限制音频捕获。
- 想替代 Android 模拟器做多机自动化测试；它能辅助观察，但不是测试调度平台。

## 历史小故事（可跳过）

- **2018 年**：Romain Vimont 发布 scrcpy，核心目标是低延迟、轻量、无需在设备上安装常驻 App。
- **早期定位**：它来自真机操作的痛点：投屏工具常常慢、重、带账号或广告，而开发者只想快速看到设备屏幕。
- **后来演进**：项目逐步加入无线连接、录制、音频转发、摄像头、虚拟显示、OTG、V4L2 webcam 等能力。
- **社区规模**：到 2026 年前后，GitHub star 已经约 113k，说明它从开发者工具扩散成了通用 Android 投屏工具。
- **维护特点**：官方 README 特别提醒只从官方仓库获取项目，说明社区流行后也出现了第三方下载混淆风险。

## 学到什么

1. **投屏本质是编码流**：看似是"手机屏幕搬到电脑"，工程上其实是手机编码、电脑解码、输入事件回传。
2. **低延迟来自取舍**：默认少缓冲能快，但网络抖动时可能不稳；加缓冲更顺，却会更慢。
3. **命令行工具的强大来自组合**：`--record`、`--no-playback`、`--tcpip`、`--new-display` 各自简单，组合后覆盖很多真实场景。
4. **真机能力不能被工具抹平**：摄像头尺寸、编码器、音频源、USB 驱动都受设备和系统版本影响。
