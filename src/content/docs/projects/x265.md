## 是什么

**x265** 是 **HEVC/H.265** 的开源编码实现，由 MulticoreWare 主导、VideoLAN 生态分发——在相同主观画质下通常比 [[x264]] **省 30–50% 码率**，代价是编码 CPU 更重、参数更复杂。

日常类比：[[x264]] 像成熟燃油车——加油站到处有。[[x265]] 像混动 SUV——同样路程更省油（比特），但买车保养（编码时间）更贵，且老设备可能不认（解码支持需确认）。

[[ffmpeg]] 调用：

```bash
ffmpeg -i in.mp4 -c:v libx265 -crf 28 -preset medium -tag:v hvc1 out.mp4
```

`-tag:v hvc1` 提高 Apple 设备硬解兼容性；CRF 28 约对标 x264 CRF 23 观感。

## 为什么重要

不理解 x265，下面这些事讲不清：

- 为什么 4K/HDR 点播更倾向 HEVC——带宽账单驱动
- 为什么 [[handbrake]] 提供 x265 预设但默认仍常是 x264——兼容 vs 效率权衡
- 为什么同画质对比必须固定 SSIM/VMAF 或盲测——只看码率数字会骗人
- 为什么 AV1 起来后 x265 仍是 HEVC 开源参考实现

## 核心要点

1. **更大 CTU 与更多分区模式**：HEVC 允许 64×64 编码单元，复杂纹理下 RD 搜索空间更大。

2. **SAO/Deblock 等环路滤波**：提升主观质量，也增加编码决策维度。

3. **多线程与 WPP**：帧内 wavefront 并行，服务器批转码可吃满多核。

4. **10-bit 主线**：HDR 内容常用 Main10 profile；8-bit 源上 10-bit 编码可减 banding。

5. **API 与 x264 不同**：调参不要照搬习惯，需重读 x265 CLI 文档。

## 实践案例

### 案例 1：4K 片源压流媒体

```bash
ffmpeg -i master_4k.mov -vf scale=1920:1080 -c:v libx265 \
  -crf 26 -preset slow -pix_fmt yuv420p10le out_1080p.mkv
```

先缩 1080p 再 HEVC，适合带宽受限分发；10-bit 减轻渐变区域色带。

## 踩过的坑

1. **没打 hvc1 tag**——部分 Apple 硬解失败，软解耗电飙升。

2. **CRF 数值不能照搬 x264**——HEVC CRF 刻度不同，盲目同数字会过糊或过锐。

3. **preset ultrafast 体积反增**——搜索太粗，RD 失效。

4. **GPL/商业授权**——x265 有 GPL 与商业许可双轨，产品集成要法务看清。

## 适用 vs 不适用场景

**适用**：
- 4K/高帧率归档与点播
- 带宽敏感但解码端支持 HEVC
- 研究 HEVC 工具集与率控

**不适用**：
- 要最大设备兼容（仍用 [[x264]]）
- 实时互动直播低延迟（CPU 过重）
- 专利敏感且可换 AV1/VP9

## 历史小故事（可跳过）

- **2013**：x265 项目发布，承接 HEVC 开源空白
- **2010s**：4K 普及推动 Netflix 等大规模 HEVC 部署
- **2020s**：AV1 竞争加剧，x265 仍是 HEVC 开源首选
- **现状**：VideoLAN 文档站维护；与 [[dav1d]] 解码侧同生态

## 学到什么

1. **新一代编码标准主要赢效率，输复杂度和编码算力**
2. **标签与 profile 是分发兼容性的一部分**
3. **对比实验要固定质量度量，不是固定 CRF 数字**
4. **训练管线解码**：[[decord]] 经 FFmpeg 解 HEVC，硬解决定吞吐
5. **代际升级要算总成本**：省 CDN 带宽是否覆盖转码集群 CPU

## 与同类对比

| 编码 | 相对 x264 码率 | 解码端 | 编码 CPU | 专利 |
|---|---|---|---|---|
| **x265** | 省 30–50% | 较普及 | 高 | HEVC 池 |
| [[x264]] | 基线 | 最广 | 中 | AVC 池 |
| [[libvpx]] VP9 | 类似 HEVC | 浏览器好 | 高 | 免版税 |
| SVT-AV1 | 更省 | 需新硬解 | 很高 | AOM 免版税 |

HEVC 是「带宽账单单换_codec_」的典型：CDN 费降了，转码集群 CPU 账单升了。
