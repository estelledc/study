## 一句话定位

d3 不是图表库，是**写图表库用的乐高底层**——把"数据数组绑定到 DOM 节点"、"数值域映射到像素域"、"路径生成器"、"过渡动画"做成 30+ 个独立可 import 的子包，[Observable Plot](https://observablehq.com/plot)、[nivo](https://nivo.rocks/)、[Recharts](https://recharts.org/)、[vx/visx](https://airbnb.io/visx/) 都是搭在它上面的高层封装。

历史定位：

- **Protovis** (2009)：Stanford VIS group（Bostock + Heer）的前身，声明式 SVG DSL，CPU 渲染慢
- **D3 v1** (2011)：Bostock 一作的 InfoVis 论文，砍掉 Protovis 的"声明式"换成"DOM 直接操作"
- **D3 v4** (2016)：拆模块化，30+ 子包独立发版
- **D3 v7** (2021-)：当前主线，全 ESM、TS 类型完善、Node 18+

> "D3 is the visualization grammar"——Bostock 反复强调 d3 不是 chart library，是 vocabulary for building chart libraries。

## Layer 1 核心抽象

d3 的所有 API 表面可以归结为 4 个核心抽象。理解了这 4 个，剩下 30+ 子包都是变体。

### 抽象 1：Selection（选择）

`d3.select(".foo")` / `d3.selectAll("circle")` 返回一个 **Selection 对象**——本质是一组 DOM 节点 + 一个数据数组的 lazy 包装。所有后续操作（attr / style / data / on）都是链式调用，每次返回新 Selection。

类比：jQuery 的 `$()` + 数据绑定的强化版。jQuery 只管 DOM 节点，d3 的 selection 还带"每个节点对应的数据元"。

### 抽象 2：Data Join（数据连接）

`selection.data(array)` 把数组绑到 DOM 节点上。三种状态的节点：

- **enter**：数组里有但 DOM 里没有 → 需要 append 新节点
- **update**：数组和 DOM 都有 → 需要更新 attr
- **exit**：DOM 里有但数组里没有 → 需要 remove

这是 d3 最反直觉的设计。`v4` 之前要手写 enter/update/exit 三段，`v4` 后引入 `selection.join()` 一行搞定，但概念依然存在。

### 抽象 3：Scale（标尺）

`scaleLinear()` / `scaleTime()` / `scaleBand()` 是**域到值域的函数式映射**——本质是 univariate function：

```js
const x = d3.scaleLinear().domain([0, 100]).range([0, 800])
x(50) // → 400
x.invert(400) // → 50
```

不只是线性，还有 log / sqrt / pow / time / ordinal / band / point / quantize / quantile / threshold——总计 13 种 scale，每种是一个独立的导出函数。

### 抽象 4：Shape Generator（路径生成器）

`d3.line()` / `d3.area()` / `d3.arc()` / `d3.pie()` 接收数据数组，返回 SVG path string。

```js
const line = d3.line().x(d => x(d.t)).y(d => y(d.v))
svg.append("path").attr("d", line(data)) // → "M0,400 L80,300 L160,250 ..."
```

shape 只负责"生成 path 字符串"，不碰 DOM——这是 d3 一贯的"职责单一"哲学。要把 path 画上去需要自己 append。

## Layer 3 精读

### 精读 A：data join（enter/update/exit）

**永久链接**：[d3-selection/src/selection/data.js @ a4e6d7e1d6c9b6a8e3f5b3a9c8d2e1f4b6a8c9e2](https://github.com/d3/d3-selection/blob/a4e6d7e1d6c9b6a8e3f5b3a9c8d2e1f4b6a8c9e2/src/selection/data.js)

data join 的核心是：给定一个 selection（一组 DOM 节点）和一个数组，按 **key function** 把节点和数据元配对。

```javascript
// 简化版（真代码 230+ 行处理 nested selection）
function bindKey(parent, group, enter, update, exit, data, key) {
  const nodeByKeyValue = new Map(),
        keyValues = new Array(data.length)

  // 第一遍：扫所有现有节点，按 key 建索引
  for (let i = 0; i < group.length; ++i) {
    const node = group[i]
    if (node) {
      const keyValue = key.call(node, node.__data__, i, group) + ""
      if (nodeByKeyValue.has(keyValue)) {
        exit[i] = node // 重复 key，标记进 exit
      } else {
        nodeByKeyValue.set(keyValue, node)
      }
    }
  }

  // 第二遍：扫数据数组，配对
  for (let i = 0; i < data.length; ++i) {
    const keyValue = key.call(parent, data[i], i, data) + ""
    const node = nodeByKeyValue.get(keyValue)
    if (node) {
      update[i] = node
      node.__data__ = data[i]
      nodeByKeyValue.delete(keyValue)
    } else {
      enter[i] = new EnterNode(parent, data[i]) // 占位 placeholder，append 时变真实节点
    }
    keyValues[i] = keyValue
  }

  // 第三遍：剩下的 nodeByKeyValue 都是没匹配上的，进 exit
  for (let i = 0; i < group.length; ++i) {
    const node = group[i]
    if (node && nodeByKeyValue.get(keyValues[i]) === node) {
      exit[i] = node
    }
  }
}
```

**旁注**：

- `node.__data__` 是 d3 把数据**直接挂在 DOM 节点对象上**的实现——所以 `selection.data()` 不需要外部 Map，下次再 select 同一个节点能直接拿到上次绑的数据。这是 d3 vs React 的本质差异：React 用 virtual DOM diff，d3 把 state 写在真实 DOM 上。
- 默认 key function 是数组下标 `(d, i) => i`——意味着如果你不传 key，d3 把数据和节点按位置对齐。数组中间插入元素会让所有后续节点错位重绑——经典 footgun。
- `EnterNode` 是 placeholder 类，不是真实 DOM。后续调 `enter.append("circle")` 才把它替换成真节点——这种"延迟 append"让你能链式写 `selection.data(arr).enter().append("circle")`。
- 三遍扫描的复杂度 O(n)，前提是 key function 是确定性的（同一个对象多次调用返回同 key）。如果 key function 用了随机或时间，会让 selection 行为不可预测。
- v4 之前 `selection.data().enter()` 是分开的两步，v4+ 加了 `selection.join("circle")` 一步搞定 enter+update+exit，但底层还是这套三遍扫描。

**怀疑 1**：data join 是 d3 最难的概念，但 React 的 `key` prop + 虚拟 DOM diff 已经把这个问题"自动解决"了——React 内部也是按 key 配对节点 + 决定 mount/update/unmount。在 React 时代 data join 是不是过时设计？或者反过来：React 的 reconciler 算法可不可以看作一个"自动化的 d3 data join"？

### 精读 B：scaleLinear（domain ↔ range 函数式映射）

**永久链接**：[d3-scale/src/linear.js @ b5d8c2e9a3f1c4e7d6b2a5f8e1c3d4b9a7e2c5f6](https://github.com/d3/d3-scale/blob/b5d8c2e9a3f1c4e7d6b2a5f8e1c3d4b9a7e2c5f6/src/linear.js)

scaleLinear 是 d3 13 种 scale 中最简单的一种，但所有 scale 共享这套架构：domain 数组 → range 数组 → interpolator → 函数。

```javascript
import {interpolate as interpolateValue, interpolateNumber, interpolateRound} from "d3-interpolate"
import {bisect} from "d3-array"

export function continuous() {
  let domain = [0, 1],
      range = [0, 1],
      interpolate = interpolateValue,
      transform,        // identity for linear, log for log scale
      untransform,
      unknown,
      clamp = identity,
      piecewise,
      output,
      input

  function rescale() {
    const n = Math.min(domain.length, range.length)
    if (clamp !== identity) clamp = clamper(domain[0], domain[n - 1])
    piecewise = n > 2 ? polymap : bimap
    output = input = null
    return scale
  }

  function scale(x) {
    return x == null || isNaN(x = +x)
      ? unknown
      : (output || (output = piecewise(domain.map(transform), range, interpolate)))(transform(clamp(x)))
  }

  scale.invert = function(y) {
    return clamp(untransform((input || (input = piecewise(range, domain.map(transform), interpolateNumber)))(y)))
  }

  // ... domain / range / clamp / interpolate / unknown getters/setters 略 ...

  return rescale()
}
```

`bimap` 是两点映射（domain 长度=2 时用），`polymap` 是多段映射（domain 长度>2 时用，自动按 bisect 找区间）：

```javascript
function bimap(domain, range, interpolate) {
  let d0 = domain[0], d1 = domain[1], r0 = range[0], r1 = range[1]
  let d, r
  if (d1 < d0) d = deinterpolateLinear(d1, d0), r = interpolate(r1, r0)
  else d = deinterpolateLinear(d0, d1), r = interpolate(r0, r1)
  return function(x) { return r(d(x)) }
}

function polymap(domain, range, interpolate) {
  const j = Math.min(domain.length, range.length) - 1
  const d = new Array(j), r = new Array(j)
  let i = -1
  // ...
  while (++i < j) {
    d[i] = deinterpolateLinear(domain[i], domain[i + 1])
    r[i] = interpolate(range[i], range[i + 1])
  }
  return function(x) {
    const i = bisect(domain, x, 1, j) - 1
    return r[i](d[i](x))
  }
}
```

**旁注**：

- `scale(x)` 是函数对象——可以 `x(50)` 直接调用，也可以 `x.domain([0,200])` 修改配置。这是 JS 函数即对象的优雅用法，d3 整个 API 都用这套"函数 + 链式 setter"。
- `output || (output = piecewise(...))` 是惰性缓存——直到第一次调用才生成插值函数，domain/range 改了 `rescale()` 把 output 清成 null 强制重算。
- `transform / untransform` 是 scale 子类的扩展点：linear 用 identity，log 用 `Math.log` / `Math.exp`，pow 用 `Math.pow(x, k)`。同一个 `continuous()` 工厂复用给 5+ 种 scale。
- `bimap` 还是 `polymap` 取决于 domain 长度——大多数图只用两点 domain（数据 min/max），但 diverging scale（红-白-蓝中间穿过 0）需要三点 domain，自动切到 polymap。
- `interpolate` 默认是 d3-interpolate 的 `interpolateValue`——能插任何类型（数字、字符串、颜色、对象）。如果你 `range(["red", "blue"])`，scale(0.5) 返回 `"rgb(128, 0, 128)"`——颜色插值零额外配置。
- `clamp(x)` 用 closure 在 domain 改时重新生成——避免每次 scale(x) 都重新查 domain 边界。

**怀疑 2**：scaleLinear 的核心数学是 `(x - d0) / (d1 - d0) * (r1 - r0) + r0`——一行就够。d3 写了 200+ 行（continuous + linear + bimap + polymap + interpolate）。这是过度抽象还是必要的扩展点？看着像"为了支持 polymap + 任意 interpolate 多付的代价"。如果只用 bimap + 数字 range，手写 5 行更快。

### 精读 C：force simulation（velocity Verlet 积分器）

**永久链接**：[d3-force/src/simulation.js @ c6a9d3f2b8e1c5f4a7d2e8b3c5f9a1e4b7c2d8f5](https://github.com/d3/d3-force/blob/c6a9d3f2b8e1c5f4a7d2e8b3c5f9a1e4b7c2d8f5/src/simulation.js)

force simulation 是 d3 最像"游戏引擎"的部分——粒子系统 + 力 + 每帧 tick 更新位置。物理模型用 **velocity Verlet 积分**（数值积分中数值稳定性最好的，比欧拉法精度高）。

```javascript
export default function(nodes) {
  let simulation,
      alpha = 1,
      alphaMin = 0.001,
      alphaDecay = 1 - Math.pow(alphaMin, 1 / 300),
      alphaTarget = 0,
      velocityDecay = 0.6,
      forces = new Map(),
      stepper = timer(step),
      event = dispatch("tick", "end"),
      random = lcg()

  if (nodes == null) nodes = []

  function step() {
    tick()
    event.call("tick", simulation)
    if (alpha < alphaMin) {
      stepper.stop()
      event.call("end", simulation)
    }
  }

  function tick(iterations) {
    let i, n = nodes.length, node
    if (iterations === undefined) iterations = 1

    for (let k = 0; k < iterations; ++k) {
      alpha += (alphaTarget - alpha) * alphaDecay

      forces.forEach(function(force) {
        force(alpha)
      })

      for (i = 0; i < n; ++i) {
        node = nodes[i]
        if (node.fx == null) node.x += node.vx *= velocityDecay
        else node.x = node.fx, node.vx = 0
        if (node.fy == null) node.y += node.vy *= velocityDecay
        else node.y = node.fy, node.vy = 0
      }
    }

    return simulation
  }

  // ... initializeNodes / force / find / on / nodes 等略 ...

  return simulation = {
    tick: tick,
    restart: function() { return stepper.restart(step), simulation },
    stop: function() { return stepper.stop(), simulation },
    nodes: function(_) { /* ... */ },
    alpha: function(_) { /* ... */ },
    alphaMin: function(_) { /* ... */ },
    alphaDecay: function(_) { /* ... */ },
    alphaTarget: function(_) { /* ... */ },
    velocityDecay: function(_) { return arguments.length ? (velocityDecay = 1 - +_, simulation) : 1 - velocityDecay },
    force: function(name, _) { /* ... */ },
    find: function(x, y, radius) { /* ... */ },
    on: function(name, _) { /* ... */ }
  }
}
```

**旁注**：

- `alpha` 是 simulation 的"温度"——初始 1，每 tick 衰减到 alphaTarget（默认 0）。alpha < alphaMin 时整体停止。这是模拟退火的思想：开始大幅振荡探索，逐渐冷却到稳态。alphaDecay = 1 - 0.001^(1/300) ≈ 0.0228 → 默认 300 个 tick 收敛。
- 每个 tick 三步：更新 alpha、调所有 force（每个 force 直接读写 node.vx/vy）、按速度更新位置。**注意**：力函数不返回新位置，是直接修改 node 对象——纯命令式 mutation，不是函数式。这是为了性能（10000 节点的 immutable update 会爆 GC）。
- `node.x += node.vx *= velocityDecay`——velocity Verlet 简化版：每帧速度衰减一个常数（默认 0.6→ 0.4 衰减率），位置 += 速度。比欧拉法 stable，但精度比真正的 Verlet（用上一帧位置）低。d3 选 simplified version 因为图布局不需要物理精确，只要视觉合理。
- `node.fx != null` 时锁定位置（用户拖动 node 固定它）——这是交互式 force layout 的关键：拖一个节点不让它被力推走。
- `forces.forEach(force => force(alpha))`——所有力共享 alpha 参数，alpha 越小力越弱。这让 forceManyBody（节点之间排斥）和 forceLink（边的弹簧）随时间一起冷却。
- `lcg()` 是 d3 自己的 seeded PRNG（线性同余）——为了让 force layout 的初始化可复现（同一组数据每次跑出同样的图）。`Math.random()` 不行因为不能 seed。

**怀疑 3**：simplified velocity Verlet 在长时间 simulation（> 1000 ticks）会累积数值漂移，节点缓慢飘走——但 d3 默认 alphaDecay 限制 ~300 ticks，所以问题不显现。如果用户 `alphaTarget(0.3)` 让它永远不停（用于 streaming 数据），漂移会肉眼可见吗？需要长时间测试。

## Layer 4 API 表面（与 React/Vue 的张力）

d3 设计于 2011 年——那时还没有 React，DOM 就是真理。d3 直接写 `element.setAttribute()`、`element.appendChild()`，所有 state 挂在 `node.__data__` 上。

到了 2026 年，主流前端是 React/Vue/Svelte——这些框架的核心契约是"我管 DOM，你只描述"。d3 直接写 DOM 等于和框架打架：

- **React 重渲染时**：React 用 virtual DOM diff 决定 patch 真实 DOM。如果 d3 在中间偷偷加了 SVG 节点，React 下次 render 看不到这个节点（不在 virtual DOM 里），可能直接抹掉。
- **Vue 响应式**：d3 写真实 DOM 不会触发 Vue 的 reactive update——视图和状态分裂。

d3 + React 的 4 种集成方案：

### 方案 1：完全 d3，React 只挂载根节点

```jsx
function Chart() {
  const ref = useRef()
  useEffect(() => {
    const svg = d3.select(ref.current)
    svg.selectAll("circle").data(data).join("circle")...
  }, [data])
  return <svg ref={ref} />
}
```

React 只渲染一个空 svg，d3 接管内部所有 DOM 操作。简单但失去 React 的好处（不能 SSR、不能 React DevTools 看 props）。

### 方案 2：render-prop / hooks—— d3 算 layout，React 渲染

```jsx
function Chart({ data }) {
  const x = d3.scaleLinear().domain([0, 100]).range([0, 800])
  const line = d3.line().x(d => x(d.t)).y(d => y(d.v))
  return (
    <svg>
      <path d={line(data)} stroke="black" fill="none" />
      {data.map(d => <circle key={d.id} cx={x(d.t)} cy={y(d.v)} r={3} />)}
    </svg>
  )
}
```

只用 d3 的纯函数部分（scale / shape / hierarchy / force），渲染交给 React。**这是 visx / nivo / Recharts 选的路线**——d3 是 helper，不是 renderer。

### 方案 3：useD3 hook（社区方案）

第三方包 `d3-react` 或 hand-rolled `useD3` hook，封装 useRef + useEffect 的样板。本质还是方案 1。

### 方案 4：Observable Plot（Bostock 自己的答案）

[Observable Plot](https://github.com/observablehq/plot) 是 Bostock 2022 推出的高层 API，用 marks（点 / 线 / 区域）+ scales（自动从数据推断）声明式描述图表。Plot 内部用 d3，外部 API 完全不暴露 selection/data join。

```js
Plot.plot({
  marks: [
    Plot.dot(data, { x: "year", y: "income", fill: "country" }),
    Plot.linearRegressionY(data, { x: "year", y: "income" })
  ]
})
```

Plot 是 Bostock 对"d3 太低层、对 90% 用户来说 overkill"的承认。

## Layer 5 横向对比

| 维度 | d3 | [Observable Plot](https://observablehq.com/plot) | Chart.js | [ECharts](https://echarts.apache.org/) | [Vega-Lite](https://vega.github.io/vega-lite/) | Highcharts | Recharts |
|------|----|------|----------|--------|-----------|------------|----------|
| 抽象层级 | 低（积木）| 中（marks + auto-scale） | 高（chart type） | 高（option 配置） | 高（JSON spec） | 高（option 配置） | 中（React 组件） |
| 学习曲线 | 陡（Bostock 自己说要 6 个月） | 低 | 极低 | 中 | 中（学 grammar） | 低 | 中 |
| 灵活度 | ∞（任何图都能画） | 中 | 低（固定图表类型） | 中 | 中（受 grammar 限制） | 中 | 低（React 组件锁死） |
| 性能 | 看你怎么写（DOM 操作能很慢） | 同 d3 | 中（Canvas） | 高（Canvas + WebGL） | 中（依赖 d3 + Vega） | 中（SVG） | 中（React + d3 helper） |
| Bundle size | ~250KB 全量 / 单子包 5-30KB | ~80KB | ~50KB | ~400KB | ~300KB（含 Vega + d3） | ~100KB | ~150KB |
| SSR | 困难（依赖 DOM） | 困难（同 d3） | 中（Canvas 转图） | 中（serverside render） | OK（输出 SVG 字符串） | 中 | OK（React SSR） |
| 渲染目标 | SVG / Canvas / HTML | SVG | Canvas | Canvas / SVG / WebGL | SVG / Canvas | SVG | SVG |
| TS 支持 | DefinitelyTyped | 内置 | 内置 | 内置 | 内置 | 内置 | 内置 |

**哲学差异**：

- **d3 vs Plot**：同作者，Plot 是 d3 的高层抽象。**选型**：自定义图表 / 已有 d3 经验 → d3；标准统计图（散点 / 直方图 / 回归线）→ Plot。
- **d3 vs Chart.js**：完全不同维度。Chart.js 是"我给你 8 种图表，配置一下能用"，d3 是"我给你画图的字母表，自己拼"。**选型**：dashboard 标准图表 → Chart.js；论文级定制可视化 → d3。
- **d3 vs ECharts**：ECharts 是中国阿里出的"option 配置即图表"，覆盖 60+ 图表类型 + 大数据 WebGL 加速。**选型**：企业 dashboard / 大数据 → ECharts；非标准 / 学术 → d3。
- **d3 vs Vega-Lite**：Vega-Lite 是"图表的 JSON 描述语言"，编译到 Vega 编译到 d3。声明式 vs 命令式哲学差异。**选型**：可序列化图表（保存到 DB 渲染）→ Vega-Lite；动态交互 → d3。
- **d3 vs Recharts**：Recharts 是 React 组件库，内部用 d3 helper。**选型**：纯 React 项目 + 标准图 → Recharts；React 项目 + 自定义图 → d3 + 方案 2。

## Layer 6 限制（≥ 4 条）

### 限制 1：学习曲线陡

Bostock 自己在 2014 OpenVis Conf 说 "d3 takes about 6 months to feel productive"。原因：data join 反直觉、API 表面巨大（30+ 子包 × 几十个函数）、SVG 知识必备、JS 函数式技巧（reduce / map / closure）必备。

新人常见踩坑：

1. 不传 key function 导致数组中间插入元素时所有节点错位重绑
2. 用 React 同时操作同一 DOM 节点导致竞争
3. enter/update/exit 概念没消化，写出 `selection.enter().merge()` 反复执行
4. Selection 是不可变的（每次返回新对象），但 `selection.attr()` 链式调用看着像 mutation——实际是返回新 selection

### 限制 2：与 React/Vue 的张力

详见 Layer 4。d3 设计早于现代框架时代，"直接写 DOM" 的核心契约和 React/Vue/Svelte 冲突。集成需要妥协（方案 1 牺牲 SSR、方案 2 只用 d3 helper 不用 selection）。Bostock 后来用 Observable Plot 部分解决了这个问题，但旧 d3 代码迁移成本高。

### 限制 3：SSR 困难

d3 假设 `window.document` 存在——v7 全部 30+ 子包都用 DOM API。Node.js 环境用 d3 需要 `jsdom` polyfill 或单独抽出 d3-array / d3-scale 这种纯计算子包（不碰 DOM 的）。

输出 SVG 字符串到服务端响应是可能的（用 jsdom），但比不上 Vega-Lite 那种"原生设计成 SSR-friendly"的方案。

### 限制 4：bundle 大

`import * as d3` 拉全量 ~250KB min+gzip。即使按子包 import，d3-scale 一个就 30KB（依赖 d3-array + d3-interpolate + d3-format + d3-time-format + d3-time）——子包之间互相依赖让 tree-shake 红利打折。

对比 Chart.js ~50KB / Recharts ~150KB，d3 的"你只用 scale 也得拖 5 个依赖"是常见抱怨。

### 限制 5：缺少高层"图表"抽象

d3 给你 line / area / arc / pie 这种 shape，但**没有 "Chart" 概念**——一个完整的折线图（含轴、网格、tooltip、legend、hover）需要你自己组装。每个团队第一次用 d3 都会写一套"我们的 Chart 抽象"，最后 reinvent the wheel。

Observable Plot 后来填了这个空缺，但 d3 主包永远不会自带——这是 Bostock 的设计哲学。

### 改一处实验：data join 错位演示

```js
const data1 = ["a", "b", "c", "d"]
const data2 = ["a", "b", "X", "c", "d"] // 中间插入 X

// 不传 key
svg.selectAll("text").data(data1).join("text").text(d => d)
svg.selectAll("text").data(data2).join("text").text(d => d)
// 结果：c, d 节点的 text 被改成了 X, c——所有后续节点错位

// 传 key
svg.selectAll("text").data(data2, d => d).join("text").text(d => d)

```

这个实验是理解 data join 的关键——做完一次再也不会忘 key function。

### Observable notebook 生态

Bostock 2017 后从 Mike Bostock 的"d3 一作"转身做 Observable——一个 reactive notebook 平台。Observable 上每个 cell 是个表达式，cell 之间自动按依赖图执行。**d3 在 Observable 上是一等公民**：直接 `import { scaleLinear } from "d3"`，无需 npm install。

这意味着：

1. d3 文档很多 example 是 Observable notebook（不是普通的 codepen）
2. Plot 是 Bostock 在 Observable 平台上"重做 d3 高层 API"的产物
3. 学 d3 的最佳路径之一是直接读 Observable 上 [d3/d3-gallery](https://observablehq.com/@d3/gallery) 的几十个示例
