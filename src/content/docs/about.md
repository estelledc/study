## 我相信什么（这个站点的"宪法"）

### 1. 判断力 > 实现力

AI 把"写代码"这件事的成本拉到了地板。但"在多个看似都对的方案里识别出
真正合适的那个""把模糊的业务需求转化成精确的规格""系统出问题时定位到
根因"——这些不会变便宜。

未来工程师的护城河，是**品味和判断**。

→ 这个站点选项目时优先看：**它能不能锻炼判断力**。
单纯"功能强大、用户多"的项目不一定上榜；展示了清晰设计取舍的项目优先。

### 2. 抽象层级上移，但下钻能力变得更稀缺

今天的工程师不再手写汇编。未来的工程师可能不再逐行写业务代码——
你描述意图、约束、边界，AI 生成实现。但这**不意味着低层知识不重要**——
当问题出在抽象的缝隙里，能下钻到底层的人变得更稀缺、更值钱。

赛车手要懂变速箱，不是为了换挡，是为了听出变速箱要坏的声音。

→ 这个站点的笔记一定**精读心脏代码**。学一个项目不是学它的 API，
是学**它内部的状态机、数据流、设计 trade-off**——这些 AI 暂时还学不会。

### 3. 学习能力是唯一不衰减的能力

技术栈的半衰期在缩短。新范式（MCP、Server Components、Agent SDK）
出现的速度在加快。能**快速搭建心智模型、从类比中迁移知识、在不确定
中做出合理决策**的人，比"精通某一门语言十年"的人有适应力。

→ 这个站点 [《怎么消化一个 GitHub 项目》](/study/method/) 是先于内容
建立的——方法论比单篇笔记更重要。

### 4. 工程师正在变成"系统编辑"

当代码生成变得便宜，**审阅、批判、整合 AI 产出的能力**会比从零写出代码
更核心。这要求你有非常强的代码阅读能力 + 内化的"什么是好代码"标准。

→ 这个站点的每篇笔记都贴**真实源码 + 行号 + 旁注**——不是因为我想
显摆，是因为"看懂别人代码"才是你以后大部分时间在做的事。

### 5. 软技能正在变成工程能力

需求澄清、跨团队沟通、把技术权衡讲给非技术决策者听——和 AI 协作
**本质上是一种"精确表达意图"的能力**，跟写好文档、做好 code review、
说清需求是同一种底层肌肉。

→ 这个站点的每篇笔记都有"**为什么不是它的同类**"和"**读完你能做之前
做不了的事**"两段——前者是判断力的证据，后者是能力差的承诺。
都在练同一块肌肉：把模糊的看法落成精确的判断。

### 6. 验证和可靠性工程会成为显学

代码生成变便宜后，"如何确信这段代码真的对"的成本会反而凸显。
测试设计、形式化方法、可观测性、对抗性思维——这些会从小众走向主流。

→ 队列里专门留了一块"验证 + 可靠性"。它们今天看起来不性感，
但 18 个月后会变成区分线。

### 7. 好奇心 + 动手热情 > 科班训练

这条不必证明，是我的赌注。门槛降低意味着更多非传统背景的人能做出有
价值的东西，而真正推动事情往前走的，往往是那些**就是想搞清楚某件事
是怎么运作的人**。

→ 这个站点本身就是一份"非科班学习者也能读懂高水平开源项目"的尝试。
如果你看完笔记觉得"我也敢去读源码了"，这个站点就成功了。

## 关于挑选标准

每个进入队列的项目至少满足 4 条：

- [ ] 能锻炼"判断力"——展示了清晰的设计取舍，不是简单的功能堆叠
- [ ] 心脏代码可读——核心机制能在 1-2 个文件里读完，不是迷宫
- [ ] 有"为什么不是它的同类"的故事——值得对比，不和别人完全重叠
- [ ] 能在本机 30 分钟内跑通——可以亲手改一处看效果
- [ ] 与 AI 时代工程师的某一面有连接（底层 / 系统 / AI 协作 / 验证 / 写作 / 好奇心）

不取的项目：

- 同类已经选了一个还没明显更好——并列推荐是知识加法，不是判断力培养
- 维护停滞超过 1 年
- 只是 "trending 上有名" 但读不出独特设计取舍的项目

## 笔记的硬要求（每篇都要满足）

每篇笔记必须含：

1. **一句话定位**——一行讲清是什么，不堆术语，不用项目自己的话
2. **Why（为什么推荐）**——它解决了什么以前没人解决的问题，**为什么是它而
   不是它的同类**（明确点名竞品并说明取舍）
3. **仓库地形**——目录注释表 + "心脏文件"标识
4. **核心机制**——3-5 段，**贴真实代码 + GitHub 永久链接 + 旁注**。
   不贴源码的笔记不及格
5. **Hands-on**——30 分钟跑通的最小命令 + **至少 1 个亲手改一处的实验**
6. **横向对比**——同类项目的哲学差异，不只功能差异
7. **与你工作的连接**——能立刻迁移的部分 / 下个月能用的部分 / **不要用的部分**
8. **读完你能做什么之前做不了的事**——能力承诺，不是知识列表
9. **自检问题**——3-5 个具体的、有文件路径和行号的真问题
10. **延伸阅读路径**——心脏文件读完后下一步该读什么，按什么顺序

参见 [《怎么消化一个 GitHub 项目》](/study/method/) 里的 7 层方法论。

为什么这样做？

1. **这就是未来工程师的工作方式**——和 AI 协作，由人提供判断和品味。
   藏着这件事，反而会让站点失去"活样本"的价值

   的判断；这部分必须由我贡献，AI 替不了
3. **诚实让阅读体验更好**——你看到的不是"一个零基础学习者一个人手写 50 篇深度笔记"
   （这不可信），而是"一个零基础学习者用 AI 把自己的输入放大十倍"（可复制）

这个站点本身就是 **"Jason × Claude Code" 协作模式的一个活样本**。
你能从这里直接看到一种新工作方式怎么运转——也欢迎你来共建。
