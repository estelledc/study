# Human-AI Interaction: Designing Interfaces for Large Language Models

> 来源: https://arxiv.org/abs/2401.00037
> 学习笔记日期: 2026-06-13

## 一、从日常类比开始

想象你有一个超级聪明的助手。

这个助手什么都能回答，但它不是真正的"人"。它没有直觉，不会察言观色，你说"帮我改一下"它也不知道该改哪里。

传统的电脑程序像一本"菜谱"——步骤写死了，你按照步骤走就行。但 LLM 不一样，它像一本"百科全书"——你有问题它回答你，但你得**问对问题**才能得到好答案。

这篇论文的核心问题就是：**怎么设计一个"界面"，让人和 LLM 合作得更顺畅？**

---

## 二、核心概念：交互范式的演变

### 2.1 什么是 HCI（人机交互）？

HCI 研究的是人和计算机之间如何"沟通"。你可以把它理解为一个翻译层——人说的话太模糊，计算机需要的指令又太精确，HCI 就是在这个两者之间架桥。

```
传统软件：    人 → 按钮/菜单 → 电脑执行
LLM 软件：    人 → 自然语言 → LLM 理解 → 自然语言回复
```

### 2.2 三大交互范式

论文提出了 LLM 时代 HCI 的三种核心模式：

| 范式 | 一句话描述 | 类比 |
|------|-----------|------|
| **Chat** | 人和 LLM 像对话一样交互 | 问客服 |
| **Prompt Engineering** | 通过精心设计"提示词"获得好结果 | 写信求帮助 |
| **Agent / Autonomous** | LLM 自己决定怎么做，人只看结果 | 委托任务给助理 |

## 三、核心概念详解

### 3.1 Prompt 是新的"编程"语言

在传统编程里，你用 Python / JavaScript 告诉计算机做什么。在 LLM 时代，你用**自然语言**（通常叫 Prompt）告诉 LLM 做什么。

```python
# 传统编程方式
def greet(name):
    return f"Hello, {name}!"

print(greet("Jason"))
# 输出: Hello, Jason!
```

```python
# LLM 交互方式（伪代码，展示 Prompt 的结构）
prompt = """
你是一个翻译助手。
任务：将中文翻译成英文。
示例：
输入：你好世界
输出：Hello World

现在请翻译：
输入：今天天气真好
输出：
"""

# LLM 的回复：A beautiful day today
```

**关键理解**：你写的不是"代码指令"，而是"上下文+任务描述"。LLM 通过阅读整个上下文来推断该做什么。

### 3.2 Prompt Engineering 的核心原则

不是随便写几句话就行。好的 Prompt 有结构：

```python
# 差的 Prompt
"帮我写封邮件"

# 好的 Prompt（带结构）
"""
角色：你是一位商务邮件专家。
任务：写一封项目进度更新邮件给团队。
背景：项目比计划晚了两周，原因是服务器配置问题。
语气：专业但不过于正式，表达歉意并提出补救计划。
格式：
- 简短的开场白
- 当前状态（1-2 段）
- 原因说明
- 补救计划（用列表）
- 结束语
"""
```

### 3.3 Chain-of-Thought（思维链）

这是论文中一个非常重要的概念。LLM 有时候直接回答会出错，但如果让它"一步步想"，准确率会大幅提升。

```python
# 直接提问（容易出错）
prompt = "小明有5个苹果，吃了2个，又买了3倍于剩余数量的苹果，现在有几个？"
# LLM 可能回答：15 （错！正确答案是 14）

# 思维链方式（引导 LLM 展示推理过程）
prompt = """
小明有5个苹果，吃了2个，又买了3倍于剩余数量的苹果，现在有几个？
请一步步思考。
"""
# LLM 的回复：
# 1. 小明最开始有 5 个苹果
# 2. 吃了 2 个，剩下 5 - 2 = 3 个
# 3. 买了 3 倍于剩余数量的苹果，即 3 * 3 = 9 个
# 4. 现在有 3 + 9 = 12 个苹果
# 答案：12 个
```

**类比**：这就像考试时，只写答案和写出解题步骤的区别。给出解题步骤的 LLM 更不容易出错。

### 3.4 Few-Shot Learning（少样本学习）

给 LLM 看几个"例子"，它就能学会新任务。这比从头教它容易得多。

```python
# 零样本（Zero-shot）：不给任何例子
prompt = "请判断这句话的情感是正面还是负面：\n这道菜真难吃"
# LLM 需要完全靠自己判断

# 少样本（Few-shot）：给几个例子
prompt = """
判断情感（正面 / 负面）：

今天天气真好 —— 正面
这部电影太糟糕了 —— 负面
我收到了心仪的礼物 —— 正面

这道菜真难吃 ——
"""
# LLM 更容易判断为：负面
```

## 四、LLM HCI 设计的四大挑战

### 4.1 不确定性

传统软件的输入输出是确定的——你点"保存"，它就保存。但 LLM 每次回答可能不一样，即使是同一个问题。

```python
import os
from openai import OpenAI

client = OpenAI(api_key=os.getenv("OPENAI_API_KEY"))

# 同样的问题，可能得到不同的回答
def ask_llm(question, temperature=0.7):
    """temperature 越高，回答越随机"""
    response = client.chat.completions.create(
        model="gpt-3.5-turbo",
        messages=[{"role": "user", "content": question}],
        temperature=temperature,  # 0.0 = 完全确定, 1.0 = 完全随机
    )
    return response.choices[0].message.content

# 运行 5 次同一问题
for i in range(5):
    print(f"第 {i+1} 次: {ask_llm('今天天气怎么样？')}")
# 输出可能每次都不同
```

**设计启示**：界面需要让用户感知到不确定性。比如加个"重新生成"按钮，或者标注"AI 生成内容可能有误"。

### 4.2 可解释性

用户怎么知道 LLM 为什么这样回答？它有没有在胡说？

```python
# 要求 LLM 解释自己的推理过程
prompt = """
问题：为什么天空是蓝色的？

请先给出答案，然后在"思考过程"部分解释你是如何得出这个答案的。
"""

response = ask_llm(prompt)
# 期望的回复结构：
# 答案：因为瑞利散射...
#
# 思考过程：
# 1. 识别问题类型为物理学/光学
# 2. 回忆相关概念：太阳光由多种颜色组成，波长不同
# 3. 应用瑞利散射原理：短波长的蓝光比长波长的红光更容易被大气散射
# 4. 得出结论
```

**设计启示**：让 LLM 展示"思考过程"，用户更容易信任它。

### 4.3 可控性

用户如何确保 LLM 的输出是"对的"？能不能限制它的回答范围？

```python
# 使用结构化输出（JSON Schema）约束 LLM 的输出
prompt = """
请按照以下 JSON 格式回复，不要添加任何其他内容：
{
  "姓名": "string",
  "年龄": "number",
  "职业": "string"
}

从以下文本中提取信息：
"李明，今年 28 岁，是一名软件工程师。"
"""

# 期望输出（格式严格）：
# {
#   "姓名": "李明",
#   "年龄": 28,
#   "职业": "软件工程师"
# }
```

**设计启示**：在需要精确结果的场景（如数据提取），给 LLM 强格式约束。

### 4.4 信任校准

用户既不能完全信任 LLM（它会"胡说八道"），也不能完全不信任（那用它就是浪费时间）。

**设计启示**：
- 对高确定性任务（翻译、事实查询）→ 信任度可以高一些
- 对创造性任务（写故事、出主意）→ 提示用户这是"建议"而非"事实"

## 五、总结

这篇论文帮我们理清了一个关键认知转变：

```
传统软件设计：   界面 = 按钮、菜单、表单
LLM 时代设计：   界面 = 对话 + Prompt + 输出解释
```

核心要点回顾：

- **Prompt 是新的编程语言**：需要结构化和精心设计
- **思维链（CoT）能大幅提升准确率**：让 LLM"一步步想"
- **少样本学习很强大**：给几个例子比从零教有效得多
- **不确定性是 LLM 的固有特性**：界面要适配它
- **可解释性和可控性是信任的基础**：让用户看到"为什么"和"怎么来的"
- **信任校准是关键设计目标**：让用户知道什么时候该信、什么时候该疑
