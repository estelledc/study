# DRust：语言引导的分布式共享内存，细粒度、全透明、超高效

## 1 一个日常类比：快递仓库的分拣员

想象一个大型快递仓库有 10 个分拣员（服务器）。每个分拣员负责把包裹（数据）送到正确的货架（内存地址）。

**问题**：当 10 个分拣员要同时操作同一个包裹时，必须有人拿着"唯一钥匙"（所有权），其他人只能看不能动——这就是分布式共享内存（DSM）要解决的**内存一致性**问题。

**传统做法**：每次有人要修改包裹，必须先通知其他所有人"放下那个包裹"（缓存失效），等所有人都确认后，才能移动包裹。这个"通知-等待"的过程就是性能杀手。

**DRust 的洞察**：Rust 语言自带的"所有权模型"已经天然保证了**同一时间只有一个可变引用**（Single-Writer）。如果系统能利用这个信息，就不需要显式地发通知——只要把包裹换到另一个地址，旧地址上的"过期拷贝"自然失效。

## 2 核心概念

### 2.1 分布式共享内存（DSM）

DSM 的目标是让多台机器共享一块统一的内存空间，程序员看起来就像在单机上操作一样。但多台机器之间需要通过网络通信来保证数据一致性，这是性能瓶颈的根源。

### 2.2 Rust 所有权模型

Rust 的所有权模型包含四个核心不变式：

1. **单一所有者**：每个值有且仅有一个所有者
2. **安全借用**：引用必须从所有者借用，借用结束必须归还
3. **唯一写入者**：同一时刻最多一个可变引用（`&mut`），且不能与其他引用共存
4. **多个读取者**：多个不可变引用（`&`）可以同时存在

这四个不变式合起来就是 DSM 领域熟知的 **SWMR（Single-Writer-Multiple-Reader）性质**。

### 2.3 DRust 的核心思路

DRust 不做任何代码改写，直接把一个单机 Rust 程序变成分布式程序。它利用了 Rust 编译器已经保证的 SWMR 性质：

- **读操作**：从远程服务器拉一份副本到本地缓存，因为不会有并发写入，可以直接读
- **写操作**：通过可变借用把对象从全局堆"搬"到执行写入的服务器，地址一变，其他服务器的旧缓存自动失效——**不需要显式发送失效消息**

## 3 代码示例 1：从单机到分布式，零代码改动

下面是一个简单的累加器。在普通 Rust 中它运行在单机上，在 DRust 中**同样的代码**直接分布在多台服务器上执行。

```rust
// 完全不需要修改的 Rust 代码

pub struct Accumulator {
    pub val: Box<i32>,
}

impl Accumulator {
    pub fn add(&mut self, delta: &i32) -> i32 {
        *self.val += *delta;
        *self.val
    }
}

fn main() {
    // 在 DRust 中，Box 分配在全局堆上
    // 对象可能被分配在任意一台服务器上
    let val: Box<i32> = Box::new(5);
    let b: Box<i32> = Box::new(10);
    let mut a = Accumulator { val };

    // 在服务器 A 上同步执行：
    // DRust 自动把 a.val 和 b 拉到 A 的本地
    let local_add = a.add(&*b); // a.val == 15

    // 在服务器 B 上异步执行：
    // DRust 只发送指针（浅拷贝），实际数据在需要时才拉取
    let remote_add = thread::spawn(move || {
        a.add(&*b) // a.val == 25
    }).join();
}
```

关键理解：

- 第 11-12 行：`Box::new` 在普通 Rust 中分配在本地堆上，在 DRust 中分配在**全局分布式堆**上
- 第 17 行：`thread::spawn(move || ...)` 把所有权转移到新线程。在 DRust 中，如果目标服务器是远程节点，它只发送指针，实际数据按需拉取（call-by-reference）
- 整个程序不需要任何分布式相关 API，全部是标准 Rust

## 4 代码示例 2：DRust 的缓存一致性协议

DRust 通过修改指针的内部布局来实现零消息的缓存一致性。看这个核心逻辑：

```rust
// DRust 扩展了 Rust 指针的内部结构：
// 每个 Box 指针包含：
//   - 全局地址（64位）：对象在全局堆中的唯一地址
//   - 颜色位（16位）：记录版本号，用于优化本地写入
//   - 本地缓存地址（64位）：指向本地缓存中的副本

// 可变借用（&mut）的解引用逻辑 —— 写操作
fn deref_mut(m: &mut MutRef) -> u64 {
    // 检查对象是否在本地的堆分区内
    if !is_local(m.global_addr) {
        // 不是本地？把它从远程服务器搬到本地上！
        // 注意：是"移动"而非"复制"——对象全局地址会改变
        m.global_addr = move_object_to_local(clear_color(m.global_addr));
    }
    // 返回本地可写的地址
    clear_color(m.global_addr)
}

// 不可变借用（&）的解引用逻辑 —— 读操作
fn deref(r: &Ref, cache: &Cache) -> u64 {
    // 检查对象是否在本地的堆分区内
    if is_local(r.global_addr) {
        return clear_color(r.global_addr); // 直接读，不用拉
    }

    // 远程对象：从远程服务器拉一份缓存
    if r.local_copy_addr == Null {
        // 先查本地缓存哈希表，避免重复拉取
        if let Some((local_addr, ref_count)) = cache.get(r.global_addr) {
            r.local_copy_addr = local_addr;
            cache.update(r.global_addr, (local_addr, ref_count + 1));
        } else {
            // 缓存中没有，从远程拉一份
            r.local_copy_addr = copy_from_remote(clear_color(r.global_addr));
            cache.insert(r.global_addr, (r.local_copy_addr, 1));
        }
    }
    // 返回本地缓存中的地址
    r.local_copy_addr
}
```

关键理解：

- **写操作时**：`move_object_to_local()` 把对象搬到本地并改变全局地址。其他服务器上的旧缓存因为指向的是**过期地址**，自然失效，后续读取会自动从新地址拉取
- **读操作时**：检查本地缓存，如果有就直接读（零网络开销），没有才从远程拉
- **不需要显式失效消息**：全局地址的改变就是"失效信号"——旧地址的东西不能再用了

## 5 协议演进：三种状态流转

DRust 的缓存一致性协议可以用一个简单的状态机描述：

```
         可变借用(&mut)
    ┌───────────────────┐
    │                   │
    ▼                   │
   已修改 ──────────────┘
    │                   │
    │ 不可变借用(&)     │ 可变借用(&mut)
    ▼                   │
   已共享 ◄─────────────┘
    │
    │ 所有引用归还
    ▼
   已拥有（原始所有者）
```

- **已拥有（Owned）**：对象由所有者独占，可自由转换到其他状态
- **已修改（Modified）**：通过可变借用移动到新地址，全局地址变更
- **已共享（Shared）**：多个服务器缓存了只读副本，全局地址不变
- **无效（Invalid）**：所有权转移后，原位置变为无效

## 6 技术亮点总结

| 特性 | 传统 DSM（如 GAM、Grappa） | DRust |
|------|--------------------------|-------|
| 一致性实现 | 显式广播失效消息 | 利用 Rust 所有权语义，零失效消息 |
| 代码改动 | 需要改写 | 零改动，完全透明 |
| 细粒度 | 块级（block-level） | 对象级（object-level） |
| 写操作同步 | 多次网络往返 | 一次移动操作 |
| 读操作缓存 | 需要维护一致性 | 地址变更自动失效，安全缓存 |
| 吞吐量提升 | 基线 | 比 GAM 高 2.64 倍，比 Grappa 高 29.16 倍 |
| 单机开销 | - | 仅比单机慢 2.42% |

## 7 关键设计决策

**1. 分区全局地址空间（PGAS）**：每个服务器负责全局堆的一个地址分区，对象有唯一全局地址，任何服务器都能访问。

**2. 指针着色（Pointer Coloring）**：DRust 保留全局地址最高 16 位作为"颜色"（版本号）。本地写入时不移动对象，只改颜色版本号，后续读到不同颜色的指针就知道缓存过期了。这避免了频繁移动对象带来的开销。

**3. 分布式调度器**：DRust 有全局控制器收集各节点资源信息，自适应地将线程分配到最合适的服务器，实现负载均衡。

## 8 思考题

1. Rust 的所有权模型在编译期做静态检查，但 DRust 运行在分布式环境中，对象的物理位置变了。DRust 是如何保证编译期保证的安全性在分布式场景下仍然成立的？

2. 如果同一个对象被频繁地读和写交替访问（R-W-R-W...），DRust 的"移动-失效-再拉取"策略会带来什么性能影响？有没有可能的优化方向？
