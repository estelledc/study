## 这篇笔记在讲什么

Google 的 Jeffrey Dean 和 Sanjay Ghemawat 在 2004 年 OSDI 会议上发表了一篇 13 页的论文，讲了
一个叫 **MapReduce** 的系统。它做了一件事：**让你在几千台机器组成的集群上跑数据处理任务时，
只需要写两个小函数，剩下的分布式细节框架全部替你搞定。**

这篇笔记从零基础开始，用日常类比逐步拆解，目标是让你不看任何术语也能理解 MapReduce 的思想。

## 日常类比：工厂流水线

想象你在经营一个大型食品加工厂，有下面这个需求：

> "我们收到了 1000 箱苹果，每箱里有 100 个苹果，每个苹果都要检查有没有腐烂，
> 最后统计每个果园（产地）有多少个坏苹果。"

**没有 MapReduce 的做法**：你自己站在流水线前，拿起一个苹果看一次，记在笔记本上，
看完 10 万个苹果要站很久。

**有 MapReduce 的做法**：

1. 你雇 100 个工人，每人分 1000 个苹果（map 阶段）
2. 每个工人只看自己那 1000 个，在本地记："果园A 有 3 个坏苹果，果园B 有 1 个……"
3. 工人们把各自的结果交给 10 个组长（shuffle 阶段）—— 同一个果园的所有记录集中到同一个组长那里
4. 每个组长把自己手里的数字加起来，报给老板（reduce 阶段）

老板（你）做了什么？你只是设计了流程，雇了工人，设了组长。**你没有亲手看过任何一个苹果。**

MapReduce 就是这样：你写两个函数描述"检查苹果"和"汇总数字"，框架负责雇工人、分苹果、找组长、算总和。

## 核心概念

### 概念 1：Map — "逐个处理"

Map 是你写的第一个函数。它接收**一条输入**，输出**零条或多条中间结果**。

回到苹果的例子：
- 输入：一个苹果（来自果园A，状态：腐烂）
- Map 函数的行为：如果是腐烂的，输出 ("果园A", 1)

在编程里：

```python
# Map 函数：对每条记录做简单处理
def map(果园名称, 苹果状态列表):
    for 状态 in 苹果状态列表:
        if 状态 == "腐烂":
            yield (果园名称, 1)
```

关键点：每条记录的处理**互相独立**。100 个工人可以同时工作，谁也不等谁。

### 概念 2：Reduce — "按组汇总"

Reduce 是你写的第二个函数。它接收**一个键和这个键对应的所有值**，输出**汇总结果**。

继续苹果例子：
- 输入：果园A，所有值为 [1, 1, 0, 1, 0, ...]（100 个 0 和 1）
- Reduce 函数的行为：把所有 1 加起来，输出 ("果园A", 42)

```python
# Reduce 函数：把同一个键的所有值合并
def reduce(果园名称, 标记列表):
    return (果园名称, sum(标记列表))
```

### 概念 3：Shuffle — "自动分组"

这是框架干的活，不需要你写。Map 阶段结束后，框架自动把**相同键的记录**送到同一个 Reduce 任务：
- 所有 ("果园A", ?) 送到组长A
- 所有 ("果园B", ?) 送到组长B

你不需要写任何分组逻辑——框架用 `hash(果园名)` 自动完成。

### 概念 4：容错 — "死了一个工人怎么办"

集群有几千台机器，每天总会死几台。MapReduce 的办法很简单：

- Map 任务跑在 worker 机器上，结果存在本地磁盘
- 如果某个 worker 挂了，Master 检测到它"沉默超时"，**把它的任务重新派给另一台机器**
- 因为 Map 是纯函数（同样的输入 → 同样的输出），重跑结果完全一样
- Reduce 任务完成后才算"写入成功"，因为它写到了分布式文件系统（GFS）

注意：这跟数据库常见的"数据复制"思路不同。MapReduce 靠**重算**来容错，
因为它的函数没有副作用，重跑不会出问题。

### 概念 5：本地化 — "别搬数据，搬工人"

假设数据已经在 100 号机房了，你还要从 1000 公里外运来一台机器处理？太浪费带宽了。

MapReduce 的优化：**把计算送到数据身边**。Master 知道每份数据存在哪些机器上，
于是优先把 Map 任务派到存有数据的机器上。这叫**数据本地化**——
类比：让面粉厂的师傅就地做面，而不是把面粉搬到另一个城市再做。

### 概念 6：备份任务 — "最后冲刺，找两个工人同时干"

Reduce 阶段末尾，大部分任务都快完成了，但总有 1-2 个慢工。
如果干等它，整个作业就被卡住了。

MapReduce 的解决办法：**对最后一个阶段的每个任务，启动一个备份副本在另一台机器上同时跑**，
谁先完成谁算数。这叫做**推测执行（speculative execution）**。
类比：老板看最后两个快递快送完了，给每个快递同时叫个外卖骑手，谁先到算谁的。

## 代码示例

### 示例 1：词频统计（WordCount）—— 大数据的 "Hello World"

统计一个包含数亿页网站的语料库中，每个单词出现了多少次。

```python
# Map: 读一行，把每个单词吐出来，每个都标为 1
def map(文件路径, 文本内容):
    for 单词 in 文本内容.split():
        emit(单词, 1)

# Reduce: 同一个单词的所有 1 加起来
def reduce(单词, 数量列表):
    emit(单词, sum(数量列表))
```

数据流：
```
输入: 3 个文件，每个 5GB

Map 阶段（在每台机器上并行）:
  机器1 -> ("hello", 1) ("world", 1) ("hello", 1)
  机器2 -> ("hello", 1) ("mapreduce", 1)
  ...

Shuffle 阶段（框架自动）:
  组长A收到所有 hello -> [1, 1, 1, 1, ...]
  组长B收到所有 world  -> [1, 1, 1, ...]
  组长C收到所有 mapreduce -> [1, 1, ...]

Reduce 阶段:
  组长A -> ("hello", 15832947)
  组长B -> ("world", 9201833)
  组长C -> ("mapreduce", 42819)
```

### 示例 2：倒排索引 — 搜索引擎的核心

搜索引擎需要知道：每个词出现在哪些网页中？

```python
# Map: 对每个文档，列出它包含的每个词
def map(文档ID, 文档内容):
    for 单词 in 文档内容.split():
        emit(单词, 文档ID)

# Reduce: 收集每个词对应的所有文档ID
def reduce(单词, 文档ID列表):
    # 去重后排序，得到该词出现在哪些页面
    唯一文档 = set(文档ID列表)
    emit(单词, sorted(唯一文档))
```

输出结果（倒排表）：
```
"机器学习" -> [doc_003, doc_047, doc_128, doc_5512, ...]
"深度学习" -> [doc_047, doc_128, doc_2001, ...]
```

搜索时，用户输入"机器学习"，直接查表返回文档列表——这就是 Google 早期搜索的核心。

### 示例 3：分布式排序 —— 把 100 亿行数据排好序

排序看起来比前面两个复杂，但 MapReduce 照样能解决：

```python
# 两阶段 MapReduce 实现

# 第一阶段：分区排序
def map(行号, 一行数据):
    # 按行的首字符哈希决定去哪个区
    分区号 = hash(行数据[0]) % 1000
    emit(分区号, 一行数据)

def reduce(分区号, 行列表):
    return sorted(行列表)

# 第二阶段：合并 1000 个已排序的分区
def map(分区号, 已排序的块):
    for 行 in 已排序的块:
        emit(行, None)

def reduce(行, _):
    emit(行)  # 直接输出，因为输入已经是全局有序了
```

MapReduce 论文 Section 7 详细讨论了分布式排序，100 亿行排序在千节点集群上大约 200 秒——
这比传统单机数据库排序快了数个数量级。

## 架构一览

```
                    ┌─────────────┐
                    │    Master   │  管理节点（只有 1 个）
                    │  - 分配任务  │
                    │  - 监控状态  │
                    │  - 容错重算  │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
         ┌────▼───┐  ┌────▼───┐  ┌────▼───┐
         │ Worker1│  │ Worker2│  │ WorkerN│
         │ - N map│  │ N map  │  │ N map  │
         │ - N red│  │ - N red│  │ - N red│
         └────┬───┘  └────┬───┘  └────┬───┘
              │            │            │
         ┌────▼────────────▼────────────▼───┐
         │         GFS (分布式文件系统)      │
         │  存放输入数据 + Map 中间结果 + 最终输出 │
         └──────────────────────────────────┘
```

Master 是唯一的控制节点，它的代码大约 1000 行。它维护两张紧凑表：
- 已完成 Map 任务数 / 总 Map 任务数
- 已完成 Reduce 任务数 / 总 Reduce 任务数

一张 1800 节点集群的运行时表只占大约 58KB——简单到令人惊讶。

## MapReduce vs 传统数据库

| 场景 | 传统 SQL 数据库 | MapReduce |
|------|----------------|-----------|
| 数据量 | GB ~ TB | TB ~ PB |
| 机器规模 | 几十台 | 几千台 |
| 单条记录随机读 | 快（毫秒级） | 不支持 |
| 全表扫描 | 慢 | 快（线性扩展） |
| 容错方式 | 复制 + WAL | 重算 + GFS |
| 适合类型 | OLTP（在线交易） | OLAP（离线分析） |

一句话总结：MapReduce 不是为了替代数据库，而是为**数据库处理不了的数据量**而生的。

## 历史影响

- **2004 年 OSDI**：论文发表，首次公开了 Google 内部大规模数据处理的设计
- **2006 年**：Doug Cutting 用 Java 实现了开源版 Hadoop（包含 HDFS + MapReduce）
- **2008-2014**：Hadoop 成为大数据时代的标志性框架，催生 Spark、Storm、Flink 等一系列后辈
- **遗产**：虽然纯 MapReduce 逐渐被 Spark 取代，但它确立的"限制表达力换可扩展性"范式
  至今仍是分布式计算的基石。Spark、Flink、Beam 的核心思想都可以追溯到这篇论文。

## 读完这篇论文你该知道

1. **两个函数为什么够用**：限制用户只能写 map + reduce，换来的是自动并行、自动容错、自动扩展——
   这是一种"通过约束获得自由"的经典设计哲学。

2. **重算容错的优雅之处**：不需要复杂的复制协议（如 Raft），只要函数是纯的，
   挂了就重跑。简单但有前提——输入数据必须可靠存储（GFS）。

3. **Master 为什么能扛住**：1800 节点只需要 58KB 的状态。关键在于只跟踪**计数**，
   不跟踪每个数据块的分布细节。

4. **为什么后来被 Spark 取代**：MapReduce 每次迭代都要写盘，迭代算法（机器学习）效率极低。
   Spark 把中间结果留在内存，速度快了 10-100 倍。但 Spark 的 RDD 范式仍然建立在
   map + reduce 的思想之上。
