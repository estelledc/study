## 是什么

《Go To Statement Considered Harmful》提出一个最朴素但难做的观点：控制流不要靠“跳来跳去”表达复杂逻辑，应该靠能让人类跟程序状态同步的结构来写。  

类比一下：你在看导航时，是不是更容易跟着“先后顺序”走，而不是每 30 米就按一次随机重定向按钮。`goto` 就像把状态机拆成一堆隐藏入口和出口，虽然可执行，但阅读者很难在脑内复原过程。

论文里，Dijkstra 不是要废掉所有跳转，而是警告：在“高层语言”里，滥用 `goto` 会让代码的进程状态不透明，导致维护成本陡增，错误也更难定位。对于理解和验证，它是坏的默认选择。

## 为什么重要

理解这篇文章，能直接回答几件工程里的事：

- 为什么你看见一段长函数读起来像“线索丢失”？
- 为什么团队里总有人说“先这样写很快，但以后没人敢改”？
- 为什么结构化控制（if/while/for）比散落标签更易测试和追踪？
- 为什么程序员更容易在“能运行”与“容易维护”之间做错误权衡？

`goto` 的问题不是语法古怪，而是“人脑模型与执行状态模型脱节”。这正是它危险的地方。

## 核心要点

1. **可预测的程序状态顺序**  
   类比：流程图里的箭头如果总是从上到下走，读者可预测；随机跳转会把“我现在在第几步”这件事打碎。  
   `if/while/for` 这些结构把状态变化显式化，降低理解门槛。

2. **以结构代替标签控制**  
   类比：你不把整栋楼都做成“楼梯直达任意层”，而是按楼层和入口组织。  
   结构化控制不允许无限穿插的状态跳转，所以 bug 更容易被隔离。

3. **可验证性胜过灵活性幻觉**  
   代码“可运行”不是终点，验证和推理才是。  
   不可读的控制流让审查、测试和静态分析更贵，项目长期成本被放大。

## 实践案例

### 案例 1：把标签循环改成循环结构

```c
for (int i = 0; i < n; i++) {
  if (scores[i] < 0) {
    continue;
  }
  total += scores[i];
}
```

逐部分解释：

- `for` 和条件分支清楚表达“每轮状态如何前进”。
- 相比用 `goto skip` 跳到下一个循环，`continue` 对阅读者的认知模型更友好。
- 代码审查时能快速判断循环边界和错误路径。

### 案例 2：错误处理改成早返回

```python
def safe_divide(a, b):
    if b == 0:
        return None
    if a is None or b is None:
        return None
    return a / b
```

逐部分解释：

- 错误条件先处理，主路径集中，像把“异常路径”先剥离。
- 比起中间跳到 `error_label` 再汇合，读者对异常情况的理解更直接。
- 逻辑测试时覆盖路径更完整。

### 案例 3：用状态枚举替代跳转

```js
const state = { READY: 0, RUN: 1, DONE: 2 };
let s = state.READY;

if (s === state.READY) {
  s = state.RUN;
}
if (s === state.RUN) {
  // do work
  s = state.DONE;
}
```

逐部分解释：

- 状态机改成显式状态值，让“现在在哪一步”可见。
- 与 `goto labelX` 相比，执行路径与业务语义绑定更紧。
- 对接测试和文档时，可以把每一步写成独立断言。

## 踩过的坑

1. **把 `goto` 误判为“无害微优化”**：局部确实方便，但规模化会吞掉可读性。
2. **把低级语言习惯生搬到高级程序里**：在高级语言里更少受限的结构更值得优先。
3. **把错误处理藏到末尾标签**：异常路径不可见，等于让未来排障成本翻倍。
4. **过度追求一致行数**：有时更短代码并不更可理解。

## 适用 vs 不适用场景

**适用**：

- 列表处理、状态机、流程计算这类有清晰阶段的核心逻辑。
- 新人团队、多人维护模块：可读性远比炫技更重要。
- 需要被静态分析/形式化审计的关键路径代码。
- 教育和代码评审中建立“可读性第一”的团队规范。

**不适用**：

- 一些底层汇编或特定场景下才有价值的极致优化路径。
- 某些受限嵌入式历史代码，为了兼容成本短期难以大改。
- 你只是在临时脚本里做一次性拼接，不要求长期维护。
- 团队有充分验证并明确限制了跳转范围的极端历史约定。

## 历史小故事（可跳过）

- 1968 年，这篇文章成为结构化程序设计运动的核心引用之一。
- 它推动了“以结构化控制替代无约束跳转”的课程共识。
- 之后的结构化分析与程序验证工具，很多都围绕“可追踪状态路径”展开。
- 在《go to》争论之后，编译器和语言生态更强调“可读语义”而非“自由跳转”。

## 学到什么

1. 可控的执行路径比“能随便跳”更重要。
2. 隐藏控制流最容易制造长期技术债，尤其在多人协作里成本爆炸式增长。
3. 结构化语句不是形式主义，而是工程可验证性的基础设施。
4. 一个代码库是否“可读”，常常比局部效率对项目寿命更关键。
