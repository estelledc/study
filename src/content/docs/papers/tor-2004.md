## 是什么

Tor 是一个**低延迟的匿名通信系统**——你的流量经过三个中继节点（relay），每一跳都剥掉一层加密，最后到达目标网站，途中没有任何单个节点能同时知道"谁在发"和"发给谁"。日常类比：你想寄一封匿名信，于是把信装进三层信封，最外层写着中间人 A 的地址，A 拆一层看到中间人 B 的地址转寄，B 再拆一层看到中间人 C 的地址转寄，C 拆掉最后一层取出信投递。每个中间人只知道自己的前后一站，没人能把寄信人和收信人对上。

这篇论文是 2004 年 USENIX Security 的工业级设计文档，在美国海军研究实验室 1990 年代第一代洋葱路由的基础上做了大幅改进：加入前向安全、目录服务器、拥塞控制、隐藏服务等。截至 2020 年代，Tor 网络仍有约 7000 个志愿中继节点在运行，是目前互联网上部署最广的匿名通信基础设施。

## 为什么重要

不理解 Tor，下面这些事都没法解释：

- 为什么 `.onion` 域名能让服务端也匿名——不是只保护客户端，服务器 IP 同样隐藏
- 为什么端到端加密（如 HTTPS）保护内容，但不保护"谁和谁在通信"这个元数据——Tor 专门解决的就是元数据泄漏
- 为什么匿名网络选择"低延迟 + 非完美安全"而不是"高延迟 + 理论完美"——这是实际部署的工程取舍
- 为什么全球监控者（能看到所有流量的对手）仍然能通过流量关联攻击破解 Tor——论文自己承认了这个边界

## 核心要点

论文的关键贡献可以拆成**四块**：

1. **伸缩式电路构建（telescoping circuit construction）**：第一代洋葱路由一次性把整条路径的密钥打包成一个"洋葱"交给第一跳。Tor 改成逐跳协商——先跟节点 A 建 TLS 连接并协商密钥，再通过 A 向 B 发 EXTEND 命令协商第二层密钥，再通过 A→B 向 C 协商第三层。好处：每一跳独立做 Diffie-Hellman 密钥交换，任何单个节点被攻破都不影响其他层的安全。

2. **前向安全（perfect forward secrecy）**：每条电路用临时密钥，电路关闭后密钥销毁。即使将来中继的长期私钥泄漏，过去录到的加密流量也解不开。类比：你每次通话用一次性密码本，通话结束撕碎，事后录音也没法破译。

3. **目录服务器 + 出口策略**：一组受信任的目录服务器维护"哪些中继在线、带宽多少、出口策略是什么"的签名列表。出口策略让每个中继自己声明"我愿意转发到哪些端口和 IP 段"，避免所有流量都压在少数出口节点上。

4. **隐藏服务（hidden services）**：通过引入点（introduction point）和会合点（rendezvous point）让服务器端也匿名。服务器不直接暴露 IP，而是在 Tor 网络内设一个"约会地点"，客户端和服务器各自建一条电路到那里碰头。

通信最小单位是**单元（cell）**，固定 512 字节。分两类：控制单元（control cell）在相邻节点间传递，包括 CREATE / CREATED / DESTROY / PADDING；中继单元（relay cell）沿整条电路传递，携带用户数据，常见命令有 RELAY DATA（应用数据）、RELAY EXTEND（扩展电路到下一跳）、RELAY BEGIN（打开 TCP 连接）、RELAY SENDME（流量控制，每收 100 个数据单元回一个 SENDME，类似 TCP 窗口）。

相比第一代洋葱路由（1996），Tor 的改进包括：密钥协商从一次性打包改为逐跳 DH 交换；加入前向安全（临时密钥用完销毁）；引入 SENDME 拥塞控制；多目录服务器取代中心化配置；出口策略让节点自由声明转发范围；隐藏服务从理论设计落地为可部署方案。

## 实践案例

### 案例 1：一条 Tor 电路的建立过程

假设你要访问 `example.com`，Tor 客户端（叫 OP，onion proxy）做这些事：

1. 从目录服务器下载当前中继列表，随机选三个节点 Guard→Middle→Exit
2. OP 与 Guard 建 TLS 连接，发 CREATE 单元（cell），双方做 Diffie-Hellman 密钥交换，得到密钥 K1
3. OP 通过已加密的连接向 Guard 发 RELAY EXTEND 单元，Guard 代替 OP 向 Middle 发 CREATE，双方协商密钥 K2
4. OP 再通过 Guard→Middle 发 RELAY EXTEND 到 Exit，协商密钥 K3
5. 现在 OP 拥有 K1、K2、K3 三把钥匙。发数据时先用 K3 加密，再用 K2 加密，再用 K1 加密——三层洋葱
6. Guard 用 K1 解开最外层转给 Middle，Middle 用 K2 解开转给 Exit，Exit 用 K3 解开拿到明文发给 `example.com`

关键：Guard 知道你是谁但不知道你要去哪；Exit 知道目的地但不知道你是谁；Middle 两头都不知道。

### 案例 2：隐藏服务的"约会"流程

假设 Bob 运行一个 `.onion` 网站：

1. Bob 选几个 Tor 节点作为"引入点"（introduction point），建好电路，把引入点地址签名后发布到目录
2. 客户端 Alice 查到这些引入点，自己选一个"会合点"（rendezvous point），建一条电路到会合点，带上一个一次性令牌
3. Alice 通过引入点把会合点地址和令牌转交给 Bob（Alice 不直接联系 Bob）
4. Bob 收到后，自己建一条电路到同一个会合点，出示令牌
5. 会合点把 Alice 和 Bob 的两条电路接在一起，双方开始通信

全程没有人知道 Bob 的真实 IP。Alice 的 IP 同样被 Tor 电路保护。

### 案例 3：出口策略的工程意义

一个大学机房的 Tor 中继管理员可以设置出口策略为 `reject *:*`（不做出口节点，只做中间中继），或 `accept *:80,443 reject *:*`（只允许 HTTP/HTTPS 流量出去）。这样：

- 大学不会因为有人通过该节点发垃圾邮件而收到投诉
- 节点仍然为 Tor 网络贡献带宽
- 论文指出，出口策略极大地降低了志愿者运营节点的法律风险，是 Tor 能大规模部署的关键设计

## 踩过的坑

1. **Tor 不防全局观察者**：论文明确说 Tor 的威胁模型只防"非全局"被动观察者。如果一个对手能同时监控 Tor 电路的入口和出口流量，通过时间关联（timing correlation）——看到"入口这边发了 500 字节，50ms 后出口那边出来 500 字节"——就能把两端关联起来。这是设计上的取舍，不是 bug。

2. **DNS 泄漏是常见失误**：如果应用程序自己做 DNS 查询而不是通过 Tor 的 SOCKS 接口，DNS 请求走的是明文网络，ISP 能看到你要访问什么域名。论文建议应用层必须走 SOCKS4A/SOCKS5 把域名也交给 Tor 解析。

3. **出口节点能看到明文**：Tor 只保护你到出口节点的通信匿名性，出口节点到目标服务器那一段如果不是 HTTPS，出口节点能看到完整内容。很多人误以为 Tor 等于端到端加密，实际上它只保护身份不保护内容——内容保护要靠 HTTPS/TLS。

4. **低延迟 vs 高延迟的取舍**：mix 网络（如 Mixminion）通过延迟、重排序和 dummy 流量抵抗流量分析，安全性更强。Tor 选择低延迟是为了支持交互式应用（网页浏览、即时通讯），代价就是更容易被流量关联攻击。论文认为这是务实的选择：没人用的完美系统不如很多人用的够好系统。

## 适用 vs 不适用场景

**适用**：

- 保护上网者的身份不被目标网站和本地网络知道——记者、活动人士、隐私敏感用户
- 隐藏服务场景：服务器不想暴露 IP（如匿名举报平台、SecureDrop）
- 绕过区域性网络封锁——目录服务器和网桥（bridge）机制让 Tor 在审查环境中仍可接入
- 安全研究和渗透测试中隐藏来源 IP

**不适用**：

- 抵抗全局被动观察者（如控制大量 ISP 的国家级对手）——Tor 自己承认这个边界
- 需要高匿名性的邮件通信——应选择高延迟的 mix 网络（Mixminion）
- 高带宽应用（视频流、大文件下载）——三跳中继 + 志愿节点带宽有限，体验差
- BitTorrent 等 P2P 协议——大量长连接会占用稀缺的 Tor 带宽，且协议本身容易泄漏真实 IP

## 历史小故事（可跳过）

1990 年代中期，美国海军研究实验室（NRL）的 Paul Syverson、Michael Reed 和 David Goldschlag 提出第一代洋葱路由。核心动机不是"帮平民匿名上网"，而是"让情报分析师在公开互联网上搜集信息时不暴露身份"。但他们很快意识到一个悖论：如果只有情报机构在用，那"使用洋葱路由"本身就是一个暴露身份的标签。匿名系统只有在大量普通人也用的时候才有效——这就是为什么 Tor 后来被开源并鼓励民间使用。

2002 年，Roger Dingledine 和 Nick Mathewson 加入项目，从零重写了代码，解决了第一代系统的单点故障、无前向安全、不支持 TCP 流等问题。2004 年论文发表时，Tor 网络约有 30 多个志愿节点。到 2024 年，这个数字超过 7000。

论文标题中的"second-generation"就是在强调这不是学术原型，而是对第一代系统的工程重写。三位作者中，Syverson 来自海军研究实验室（第一代的发明人），Dingledine 和 Mathewson 来自 MIT，负责代码实现。这种"学术发明 + 工程重写"的模式在系统安全领域很常见。

## 威胁模型与安全边界

论文对 Tor 的安全保证画了清晰的边界：

**能防的**：本地网络监控者（ISP、公司防火墙）只能看到你在用 Tor，不知道你访问什么；目标网站只能看到出口节点 IP，不知道真实用户是谁；单个被攻破的中继节点无法同时得到来源和目的地。

**不能防的**：全局被动观察者通过流量时间关联可以破解匿名性；主动攻击者控制入口和出口节点（Sybil 攻击）可以关联流量；应用层信息泄漏（浏览器指纹、Cookie、JavaScript 获取真实 IP）不在 Tor 保护范围内。

论文采取的设计哲学是：不追求信息论意义上的完美匿名，而是在计算安全假设下提供"足够好"的匿名性，让系统实际可用。这与 mix 网络追求理论完美但延迟高、用户少形成对比。

值得注意的是，Tor 的匿名集（anonymity set）取决于同一时刻有多少用户在用。如果全网只有 10 个人在用 Tor，观察者只需从 10 个人里猜；如果有 200 万人在用，猜对的概率骤降。这就是为什么 Tor 项目一直鼓励普通人也来用——每多一个用户，所有人的匿名性都变强。论文把这称为"匿名爱公司"（anonymity loves company）。

Guard 节点（入口守卫）的选择也值得关注。如果客户端每次随机选入口节点，长期来看总会碰上恶意节点。Tor 的解法是让每个客户端固定使用少量 Guard 节点一段时间，降低随机碰上坏入口的概率。这是概率论在安全设计中的直接应用。

## 学到什么

1. **匿名 ≠ 加密**：加密保护内容，匿名保护身份（谁在和谁通信）。Tor 解决的是后者，前者需要 TLS/HTTPS 配合。
2. **逐跳协商比一次性打包安全得多**：伸缩式电路构建让每个中继独立协商密钥，攻破一个不影响全局——最小权限原则在密码学中的体现。
3. **安全系统的部署需要工程妥协**：出口策略、目录服务器、低延迟设计，每个都是"理论纯度"让步于"实际可用"的例子。用户基数本身就是匿名性的来源——只有你一个人用的匿名系统毫无意义。
4. **威胁模型要画边界**：论文明确说"我们不防全局观察者"，这种诚实比号称"绝对安全"更有价值。设计安全系统的第一步是说清楚"防谁、不防谁"。
5. **匿名集越大越安全**：Tor 的匿名性来源于人群效应。这个洞见适用于所有隐私系统——VPN、匿名邮件、混币服务，用的人越多，单个用户越难被挑出来。
