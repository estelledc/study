## 是什么

IPFS（**InterPlanetary File System**）是一种**用文件内容当地址**的分布式文件系统。

日常类比：图书馆里找书，传统 Web（HTTP）是"去 3 楼东侧 第 5 排第 7 本"——书一搬走你就找不到了；IPFS 是"我要那本第一句写着'呼叫我以实玛利'的书"——只要图书馆里还有任何一本符合，就能给你。

技术定义：你把一个文件丢进 IPFS，系统先算它的 SHA-256，得到一个像 `QmXoy...` 的字符串（叫 **CID**，Content Identifier）。以后你只要拿这个 CID，全网任何节点都能验证并返回同一份内容。

## 为什么重要

不理解 IPFS，下面这些事都没法解释：

- 为什么 NFT 的图片地址长得像 `ipfs://QmXoy...` 而不是 `https://`
- 为什么 Filecoin、Arweave、Web3 存储这些东西能"去中心化"
- 为什么 libp2p 这套网络栈被 Polkadot / Ethereum 2 / Filecoin 都拿去用
- 为什么 Git 内部和 IPFS 用的几乎是同一种数据结构（Merkle DAG）

## 核心要点

IPFS 把三件事拼在一起：

1. **CID（内容寻址）**：文件 → SHA-256 → 拼上版本和编码标记（multihash + multicodec）→ CID。同样内容永远同样 CID；改一个 byte CID 就完全变。这意味着**地址即证据**——拿到数据后本地算一次哈希就能确认没被篡改，根本不用信任发数据的人。

2. **Merkle DAG（默克尔有向无环图）**：大文件切成 256KB 一块，每块算哈希，再把这些哈希组织成一棵树，根的哈希代表整个文件。改任何一块只影响它到根的那条路径，其他块完全复用——天然去重 + 版本。Git 的 commit/tree/blob 就是这个思路；区块链的"区块链"也是这个思路。

3. **libp2p（可换零件的 P2P 网络栈）**：传输层（TCP/QUIC/WebRTC）+ 加密（Noise/TLS）+ 多路复用（yamux/mplex）+ 路由（Kademlia DHT）+ 节点发现（mDNS/bootstrap）+ 块交换（BitSwap），每层都可替换。后来 libp2p 独立成项目，被一票区块链拿去用。

三件事合起来：你存一个 CID 的请求，先用 DHT 找到"谁有这个 CID"，再用 BitSwap 把块拉回来，本地按 Merkle DAG 拼装并验证哈希。一切对错都靠哈希自证，不依赖任何中心权威。

## 实践案例

### 案例 1：同样的文件只存一份

两个人各自上传一份一模一样的文件，看 CID 是否相同：

```bash
echo "hello ipfs" > a.txt
cp a.txt b.txt
ipfs add a.txt   # 输出 CID，例如 QmYwAPJ...
ipfs add b.txt   # 同一个 CID —— 内容一样，地址就一样
ipfs cat QmYwAPJ...   # 本地按 CID 取回，并校验哈希
```

逐部分解释：

- `ipfs add`：切块 → 算哈希 → 拼成 Merkle DAG → 得到根 CID。
- 两份文件字节相同，所以根 CID 相同；网络里只需存一份块。
- 若改 `b.txt` 一个字节再 `ipfs add`，根 CID 变了，但未改动的块仍可复用。

### 案例 2：HTTP 链坏了，IPFS 靠 pin 还在

```bash
ipfs add photo.jpg          # 得到 CID
ipfs pin add <CID>          # 告诉本节点：别被 GC 清掉
ipfs cat <CID> > out.jpg    # 任意仍 pin 着的节点都能取回
# 对比：https://my-blog.com/photo.jpg 在站点倒闭后直接 404
```

逐部分解释：

- HTTP 地址绑在服务器位置；服务器没了，链接就死。
- IPFS 地址绑在内容本身；只要还有节点 pin 住该 CID，就能取到。
- 这也是 NFT 元数据常放 `ipfs://...` 的原因——铸造平台关了，内容仍可被 pin 住。

### 案例 3：DHT 怎么找到块

类比：你在陌生人电话簿里找"谁有这本书"——每次问更接近目标号码的人，跳几跳就找到持有者。

你向自己的节点要某个 CID 时：

1. 把 CID 当成 256-bit 数字，在 Kademlia DHT 里朝"XOR 距离更近"的节点跳
2. 每跳问一次："你认识更接近这个 ID 的节点吗"
3. 约 O(log N) 步后拿到 provider 列表（声明自己有这个 CID 的节点）
4. 直连 provider，用 BitSwap 拉块，本地再验哈希

冷启动可能要几秒，热路径常是毫秒——所以真实用户大多走 `cloudflare-ipfs.com` 这类网关。需要"名字不变、内容可变"时再用 IPNS（公钥签名指向最新 CID），但解析会更慢。

## 踩过的坑

1. **不 pin 就被 GC**：节点默认会跑垃圾回收清掉不被 pin 的块。你以为"上传到 IPFS 就永久保存了"——错。要么自己节点 pin 住，要么花钱用 Pinata/Filecoin。

2. **第一次访问慢**：DHT 查找 + 跨洋拉块在秒级。所以 90% 真实流量经过中心化网关（ipfs.io / cloudflare-ipfs.com）——"去中心化"实际退化成"多了一层 CDN"。

3. **没有内置激励**：白皮书有意把激励层留给后续的 Filecoin。裸 IPFS 节点没人付钱凭什么帮你存——这也是 Filecoin 出现的原因。

4. **私密内容必须先加密**：内容寻址等于"任何人拿到 CID 就能下载"。想私密就必须先加密再上传，CID 只发给授权者。

5. **可变内容很尴尬**：CID 内容变了就变 CID。想让"我的博客主页"这种地址持续指向最新版，要用 IPNS（公钥签名指向当前 CID），但 IPNS 性能比 CID 差一截。

6. **大量小文件性能差**：每个文件都要单独走 DHT 找 provider。1000 个 1KB 小文件比 1 个 1MB 大文件慢得多。常见解法是先打成 tar/CAR 包，把 1000 个文件变 1 个 CID。

7. **DHT 投毒和女巫攻击**：开放 DHT 谁都能加入，恶意节点声称"我有这个 CID"再返回垃圾——客户端能靠哈希验证识破，但浪费了一次往返。生产部署常用私有 DHT 或 bootstrap 节点白名单。

## 适用 vs 不适用场景

**适用**：

- 不可变的大文件分发（NFT 元数据、软件镜像、数据集）
- 需要审查抵抗的内容（维基百科镜像应对封锁）
- Git 风格的版本化存储（每次改动留下完整历史）
- 多节点共享 + 自动去重的场景

**不适用**：

- 私有数据（除非自己加密，且加密后丢了去重红利）
- 需要 SLA 和强一致性的关键业务（IPFS 没保证可用性）
- 频繁更新的小文件（IPNS 同步慢，每次都要广播新 CID）
- 数据库类工作负载（IPFS 是文件系统不是数据库，没有索引和事务）

## 学到什么

1. **寻址方式的范式转移**：location-addressed（你在哪）→ content-addressed（你是什么）。这一换，缓存、去重、防篡改、抗审查全部白送
2. **Merkle DAG 是去中心化系统的瑞士军刀**：Git / IPFS / 区块链 / 证书透明度日志全用它
3. **网络栈分层就是工程红利**：libp2p 把传输/加密/复用/路由全做成可换零件，结果远超 IPFS 自身的影响
4. **去中心化的现实折扣**：真实用户走中心化网关；理论上人人是节点，实际上少数大节点扛起 90% 流量
