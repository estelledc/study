## 是什么

想象你住的小区有两个门——东门和南门。平时快递只走东门，南门留着以防东门堵了才绕路。
这篇论文提出的 CMT（Concurrent Multipath Transfer）方案说：两个门同时收快递，吞吐量能翻倍。

具体来说，SCTP 协议天生支持"多宿主"（multihoming）——一台主机可以绑定多个 IP 地址，
因而可拥有多条端到端路径。但标准 SCTP（RFC 4960）只选一条"主路径"发送新数据，
其余路径仅做故障切换的备份。CMT 打破这一限制：发送端把新数据块同时分散到所有可用路径上
并发传输，从而将多条路径的带宽聚合起来。

与后来的 MPTCP 不同，CMT 不需要修改 TCP 栈，它直接利用 SCTP 已有的多宿主地址协商机制，
只需在发送端增加调度和拥塞控制逻辑即可工作。

## 为什么重要

- 这是传输层多路径并发传输最系统化的早期研究（2006 年），比 MPTCP（2011 年 RFC 6824）早五年。
  论文提出的三大算法至今是多路径拥塞控制的参考起点。
- 论文揭示了"接收缓冲区阻塞"（receive buffer blocking）这一多路径传输的核心难题——
  后来 MPTCP、QUIC 多路径扩展都要面对同样的设计挑战。
- 证明了仅修改发送端逻辑就能利用现有 SCTP 的多宿主能力获得显著吞吐提升，
  无需中间网络设备配合，降低了部署门槛。
- 五种重传策略的系统性对比，为后续多路径协议的数据调度研究提供了实验方法论。

## 核心要点

论文发现朴素地在多条路径上并发传输会产生三个副作用，并一一提出修正算法：

**副作用 1：不必要的快速重传。**
不同路径 RTT 不同，后发的数据块可能先到达接收端，导致接收端产生"gap"并发出重复 SACK。
标准 SCTP 看到 3 个重复 SACK 就触发快速重传，但这些"丢失"其实只是乱序。
解决方案——Split Fast Retransmit（SFR）：发送端为每条路径独立维护 TSN 计数，
只有来自同一路径的 SACK 报告的 gap 才计入该路径的重复确认计数器，避免跨路径的误判。

**副作用 2：拥塞窗口增长过慢。**
标准 SCTP 收到一个 SACK 只为主路径增加 cwnd。在 CMT 中数据分布在多条路径上，
但 SACK 是整体确认的，导致各路径 cwnd 增长比单路径时慢很多。
解决方案——Cwnd Update for CMT（CUC）：根据 SACK 中确认的字节数，
按比例分摊给各路径独立更新 cwnd，让每条路径像单独运行一样正常增长拥塞窗口。

**副作用 3：确认流量暴增。**
接收端收到乱序数据块时按规范必须立即发送 SACK，多路径下乱序成为常态，
SACK 数量急剧上升，反向路径拥塞。
解决方案——Delayed Ack for CMT（DAC）：接收端在乱序数据到达时不立即发送 SACK，
而是等待一小段时间（类似 TCP 延迟确认），减少不必要的 SACK 风暴。

**五种重传策略：**
当数据块确实丢失需要重传时，发到哪条路径？论文评估了五种策略：
RTX-SAME（发回原路径）、RTX-ASAP（发到最先可用的路径）、RTX-CWND（发到 cwnd 最大的路径）、
RTX-SSTHRESH（发到 ssthresh 最大的路径）、RTX-LOSSRATE（发到丢包率最低的路径）。
结论：发到另一条可用路径（而非原路径）通常更优，因为原路径可能正在经历拥塞。

## 实践案例

**电信信令网络：** SCTP 最初就为电话信令（SS7 over IP）设计，运营商核心网节点通常双归属，
CMT 可让信令在主备链路上同时跑，提升信令通道吞吐并缩短故障切换时间。

**移动终端多接入聚合：** 手机同时连接 WiFi 和 4G 时拥有两个 IP 地址。
CMT 的思路直接启发了后来 Apple 在 iOS 7+ 上部署 MPTCP 实现 WiFi/蜂窝无缝切换与带宽聚合。

**数据中心多路径：** 胖树拓扑下服务器有多网卡多 IP。CMT 在传输层实现负载均衡，
与网络层的 ECMP（等价多路径路由）互补——ECMP 按流分配路径，CMT 在单个连接内按包分配。

**视频流传输：** 大文件或高码率视频传输中，聚合两条路径带宽可以减少缓冲时间。
但需要注意乱序重组延迟对实时播放的影响——对于点播场景（允许预缓冲）收益明显，
对直播场景需要配合播放缓冲策略。

**文件备份与灾备同步：** 在异地灾备场景中，两个数据中心之间往往有多条物理链路。
CMT 可以聚合这些链路带宽加速大量数据的同步传输，同时任一链路故障不中断传输。

## 踩过的坑

**接收缓冲区阻塞（rbuf blocking）：**
这是多路径传输中最致命的问题。快路径的数据到了但无法交付给应用层，
因为慢路径上的前序数据还没到（SCTP 保序交付要求）。接收缓冲区被占满后发送端被通告窗口卡住，
整个传输停顿。论文在后续工作（2007）中提出 CMT-PF（Potentially-Failed destination state）：
如果某路径连续超时，将其标记为"可能失败"暂停向其发送新数据，从而释放接收缓冲区空间。

**路径不对称时性能退化：**
如果两条路径带宽差异极大（如 100Mbps vs 1Mbps），朴素的轮询调度会把大量数据堆积在慢路径上，
慢路径数据迟迟不到导致快路径的数据在接收缓冲区排队无法交付。
后续研究引入加权调度——按路径 cwnd 或 RTT 比例分配数据量。

**SCTP 部署限制：**
NAT/防火墙普遍不支持 SCTP 协议号（132），多宿主地址在 NAT 后无法正确映射。
这也是 SCTP 至今未能在公网大规模部署的主要原因，CMT 更多停留在学术和电信专网场景。

## 适用 vs 不适用场景

**适用：**
主机确实拥有多条独立端到端路径（不共享瓶颈链路）的场景——电信信令双归属、
数据中心多网卡服务器、移动设备 WiFi+蜂窝双栈。需要高吞吐或快速故障切换时收益最大。

**不适用：**
路径共享瓶颈链路时（如两个 IP 走同一根上行线缆），并发传输无法提升带宽反而增加乱序开销。
对延迟极敏感的实时交互场景（如在线游戏、VoIP），乱序重组等待时间引入不可接受的抖动。
公网 NAT 穿越困难的环境中 SCTP 本身部署受限，不如直接使用 MPTCP 或 QUIC 多路径扩展。
单条路径已经能满足带宽需求的轻量级请求（如 DNS 查询、REST API 调用），
引入多路径带来的复杂性远大于收益。

## 历史小故事

SCTP 最初（2000 年 RFC 2960）是为了解决电信信令从 SS7 向 IP 迁移的可靠性问题而设计的，
多宿主只是容灾手段。Iyengar 在 University of Delaware 的博士论文（2006）中首次系统地提出
"何不把备用路径也用起来同时传数据？"这个看似显而易见的想法。
但实现起来远比想象中复杂——三个副作用算法 + 接收缓冲区阻塞问题耗费了数年研究。
这条路线后来影响了 IETF 的 MPTCP 工作组，许多设计决策（如子流独立拥塞控制）
都能追溯到 CMT 的经验教训。

## 学到什么

- 多路径传输不是"把数据拆开发到多条路"这么简单。核心难点在于：
  拥塞控制要按路径独立、重传判断不能跨路径混淆、接收端重组要能容忍大范围乱序。
- "按路径独立管理状态"是多路径传输设计的第一原则——后来 MPTCP 的子流（subflow）概念
  与 CMT 的路径独立 cwnd 管理一脉相承。
- 传输层设计取舍是"吞吐 vs 延迟 vs 公平性"三角：
  CMT 牺牲了一定延迟（乱序重组等待）换取吞吐提升，适合批量传输但不适合交互式应用。
- 协议的部署能力（deployability）和协议本身的设计优劣同样重要——
  SCTP 技术上优于 TCP 的多个方面，却因 NAT 穿越问题至今无法在公网普及。

## 延伸阅读

- RFC 4960：SCTP 协议标准，理解多宿主基础机制、四次握手和心跳探测
- RFC 6824 / RFC 8684：MPTCP 标准，CMT 思想在 TCP 生态中的工程化演进，
  对比 CMT 可以看到哪些设计决策被保留、哪些被改进
- Iyengar et al. 2007："Performance Implications of a Bounded Receive Buffer in CMT"——
  深入分析接收缓冲区阻塞问题及 CMT-PF 方案
- IETF draft-tuexen-tsvwg-sctp-multipath：SCTP 多路径扩展的标准化草案，
  将 SFR/CUC/DAC 三算法写入协议规范
- Wischik et al. 2011："Design, Implementation and Evaluation of Congestion Control for MPTCP"——
  MPTCP 拥塞控制设计，可与 CMT 的 CUC 算法对照理解演进
