## 是什么

Snap 是 Google 内部用来**把主机网络功能从 Linux 内核里搬出来、放到一个用户态微内核进程**里运行的系统。它接管了虚拟机和容器的所有网络流量——虚拟交换、流量整形、加密、负载均衡——全都在用户态完成，内核只负责收发最底层的数据包。

日常类比：传统 Linux 网络栈像一栋大楼的物业公司，所有事情——门禁、水电、快递分拣——都由物业一个部门管。某天物业太慢了，Google 决定把快递分拣这件最忙的事外包给一个独立公司（Snap 进程），这个公司有自己的员工和流程，大楼只负责开门让快递车进出。好处是：独立公司可以随时升级流程、不用等大楼停电检修（内核升级），而且多栋楼可以共用同一套分拣系统。

类比的边界：Snap 并不是完全独立于大楼——它仍然依赖 Linux 内核做最底层的包收发和硬件驱动。它"独立"的是上层网络功能（虚拟交换、加密等），不是整个网络栈。

Snap 不是学术原型——它跑在 Google 全球几百万台服务器上，处理的网络流量覆盖了 Google 所有云服务（包括 GCE 虚拟机的 VPC 网络）。论文说"我们的经验覆盖了三代硬件和五年演进"，这是一篇工业系统论文。

和同年的学术工作 [[shenango-2019]]（MIT 的微秒级调度器）不同，Snap 关注的不是单机极致延迟，而是**百万台规模下的可运维性**：怎么在不中断服务的前提下每周给网络栈上新功能。

## 为什么重要

不理解 Snap，下面这些事都说不清：

- 为什么 Google Cloud 能做到热升级网络功能而不重启虚拟机——Snap 是独立用户态进程，重启它不影响内核和 VM
- 为什么 [[arrakis-2014]] 和 [[ix-2014]] 提出的"绕过内核"思路没有在大规模生产环境里原封不动落地——Snap 说明纯 kernel-bypass 在多租户场景下太难管理，需要一个可控的中间层
- 为什么 AWS Nitro 选择用专用硬件做网络卸载，而 Google 选择用软件（Snap）——两条路线的 trade-off 在这篇论文里有详细讨论
- 为什么 [[dpdk]] 在单机高性能场景好用，但在几百万台异构服务器上就不够——Snap 需要更强的资源隔离和热升级能力
- 为什么云厂商越来越多地把网络栈搬出内核——Snap 论文用五年运营数据证明了这条路可行
- 为什么 [[linux-kernel]] 的通用网络栈在云网络场景下越来越力不从心——Snap 量化了内核栈的每包开销和升级代价

## 核心要点

Snap 的设计可以拆成三块来理解：

**1. 用户态微内核架构**

Snap 以一个普通的 Linux 用户态进程运行，但它的内部结构像一个微内核：有自己的调度器、内存管理、IPC 机制，各个网络功能（防火墙、虚拟交换、加密）以"引擎"（engine）的形式插入，像微内核里的服务模块。类比：一个"操作系统里的操作系统"——它跑在 Linux 上，但内部按微内核的方式组织。

每个引擎是一段独立的网络处理逻辑，有自己的输入队列和输出队列。引擎之间通过内部通道连接——流量进来后沿着引擎链条流过，每个引擎做一件事（比如第一个引擎做包解析，第二个做 ACL 过滤，第三个做加密），最后送出去。这和 Unix 管道的哲学很像：每个程序做一件事、通过管道串起来。

为什么不直接用 [[dpdk]] + 应用自己写网络功能？因为 DPDK 给你的是原始的收发包能力，上面的虚拟交换、加密、整形全要自己从头写。Snap 在 DPDK 之上提供了一个完整的引擎框架，让团队可以像写插件一样添加网络功能，不用碰底层收发包代码。

**2. 三种 CPU 调度模型**

这是论文最有价值的贡献之一。Snap 支持三种运行模式：

- 专核（dedicating cores）——像 [[dpdk]] 一样独占 CPU 轮询，延迟最低但浪费 CPU。适合延迟敏感型负载，比如金融交易网关。在这种模式下 Snap 的线程永远不会被 Linux 调度器打断，但代价是这些核心即使没有流量也不能被应用使用。
- 共享调度（spreading）——利用 Linux CFS 与应用共享 CPU，省资源但延迟抖动大。适合吞吐型负载，比如大文件传输。CFS 不知道 Snap 线程的延迟需求，可能把它调度到队尾等好几毫秒。
- 自适应轮询（adaptive polling）——引擎在空闲时让出 CPU、忙时立刻唤醒，在延迟和 CPU 利用率之间动态平衡。这是 Google 最终在大多数场景下使用的模式。具体做法是：Snap 线程在没有包可处理时调用 `epoll_wait` 让出 CPU；一旦有包到达，立刻切回忙等模式连续处理，直到队列再次为空。

Google 发现大多数场景用自适应轮询就够了，只有少数极端低延迟场景才需要专核。这个结论背后的直觉是：大部分云负载的网络流量是突发的——90% 的时间包率很低，10% 的时间包率很高。自适应轮询在低包率时省 CPU，高包率时自动切成忙等，两头都兼顾。

**3. 可热升级、可独立演进**

传统内核网络模块升级需要滚动重启整台机器。Snap 是用户态进程，升级时先启动新版 Snap、迁移状态、切换流量，旧进程退出——整个过程对上层 VM 和容器透明。Google 每周都在给 Snap 上线新功能，这在内核里不可想象。

热升级的具体流程：新版 Snap 进程启动后，通过 Unix 域套接字从旧进程接收所有连接状态（包括每条流的序列号、加密密钥等），然后原子地切换数据路径，旧进程平滑退出。整个过程在几百毫秒内完成，用户看不到中断。

这个能力对于安全补丁尤其重要。内核漏洞修复通常需要滚动重启几百万台机器，可能要几周。Snap 的漏洞修复可以在几小时内全球推送完毕，因为每台机器上只需要重启一个用户态进程。

## 实践案例

Google Cloud 的 VPC 网络是 Snap 最大的用户。每台 GCE 宿主机上跑一个 Snap 进程，所有虚拟机的网络流量都经过它。Snap 在这个场景下做了至少五件事：虚拟交换（决定包发给哪台 VM）、带宽整形（限制每台 VM 的带宽配额）、IPsec 加密（跨数据中心的流量加密）、负载均衡（Maglev 的软件实现）、遥测（每条流的延迟和丢包统计）。

这些功能在传统架构里分散在内核模块、iptables 规则、独立守护进程里，调试和升级都是噩梦。比如改一条 iptables 规则可能影响到毫不相关的另一条规则；改一个内核网络模块要等几个月的内核发布周期。Snap 把它们统一到一个进程里，所有引擎共享同一套监控和调试基础设施，复杂度大幅降低。

论文里有两个值得记住的真实数据。第一，Snap 从"所有引擎共享一组线程"改成"每个引擎独立线程池"之后，P99 延迟下降了 50%。这个改动在内核模块里几乎不可能做——你没法给内核网络栈的某个子模块单独分配线程。第二，在 Snap 的自适应调度模式下，CPU 利用率比专核模式降低了 30-40%，而 P99 延迟只增加了约 10 微秒。对于绝大多数云负载来说，这个 trade-off 非常划算——省下来的 CPU 可以卖给租户。

## 踩过的坑

### 1. 用户态不等于自动隔离

很多人以为"搬到用户态"就天然安全了。但 Snap 跑在 root 权限下，一旦崩溃会影响整台宿主机所有 VM 的网络。Google 花了大量工程投入在 Snap 的容错上——包括心跳检测、自动重启、状态快照。教训：用户态只是让你能热升级，不会自动给你隔离。想要真正的故障域隔离，还得靠进程级看门狗和快速恢复机制。

### 2. 共享调度的尾延迟陷阱

早期 Snap 用 Linux CFS 共享 CPU。CFS 对延迟敏感任务不友好——一旦 Snap 的线程被调度走，几毫秒后才能回来，导致网络延迟出现尾部尖刺。论文里记录的 P99 和 P99.9 延迟差距达到 10 倍以上。解法是自适应轮询：在流量高峰期切成忙等（busy-poll），空闲时主动让出。这个教训和 [[shenango-2019]] 的发现一致：通用 OS 调度器不适合微秒级网络处理。

### 3. NUMA 感知不能事后补

Snap 最初没仔细处理 NUMA（非一致性内存架构）。结果发现引擎在 NUMA 远端内存上分配缓冲区时，延迟翻倍。后来强制每个引擎绑定到特定 NUMA 节点、在本地内存上分配所有数据结构。教训：在多路服务器上，任何高性能数据路径都必须从第一天就考虑 NUMA 拓扑。这不是优化，是正确性问题——远端内存访问的延迟差异足以让性能指标完全变形。

### 4. 微内核的"消息传递"开销

Snap 内部引擎之间用消息传递（类似微内核 IPC）。在每秒百万包场景下，消息传递本身成了性能热点。Google 后来引入了共享内存通道来减少拷贝，并做了零拷贝优化。教训："微内核架构"在教科书上很优雅，但在每秒几百万次操作的数据路径上，IPC 开销必须极端优化——这和 [[l4-1995]] 微内核几十年来的经验完全一致。

## 适用 vs 不适用场景

**适用**：

- 大规模多租户云网络——需要热升级、资源隔离、灵活部署
- 网络功能演进速度快——每周上线新功能，不能等内核发布周期
- CPU 资源不是最紧缺资源——Snap 的自适应调度在 CPU 充足时表现最好
- 需要对网络功能做细粒度遥测和调试——所有引擎在同一个进程里，可以统一 profiling

**不适用**：

- 单机极致低延迟——不如 [[dpdk]] 专核轮询，Snap 的自适应调度毕竟有模式切换开销
- 硬件卸载已到位的场景——SmartNIC / DPU 可以在硬件里做 Snap 做的事，CPU 开销更低
- 小规模部署——Snap 的运维复杂度（热升级框架、状态迁移、监控）只有在几万台以上规模时才值得
- 固定功能不需要频繁更新的网络设备——交换机 / 路由器固件不需要每周热升级

## 历史小故事（可跳过）

Google 内部网络栈的演进分三代。第一代是改 Linux 内核模块——快但升级痛苦，每次改动要走内核发布流程，周期以月计。第二代是在内核里加了一个"可插拔模块框架"——比改内核好一点，但仍然受内核 API 稳定性约束，而且 bug 照样会 panic 整台机器。第三代就是 Snap——彻底搬到用户态。

论文的一个关键洞察是：内核网络栈的性能瓶颈只是问题的一半，真正让 Google 下决心搬出来的是"迭代速度"——用户态进程可以像普通应用一样用 CI/CD 流水线发布，不用等内核窗口。这和微服务从单体里拆出来的逻辑一模一样：不是因为性能，是因为组织和发布效率。

Snap 这个名字没有特别的缩写含义——论文里也没解释。但在 Google 内部，它被定位为"可以 snap in"（卡入）任何宿主机的网络引擎，像乐高积木一样插拔。

有趣的是，Snap 论文发表在 SOSP 2019，和 [[shenango-2019]]（NSDI 2019）同年。两篇论文从不同角度回答了同一个问题："内核网络栈太慢了怎么办？"Shenango 说"造一个更好的调度器"，Snap 说"把整个栈搬出来、按需分配资源"。两者并不矛盾——Shenango 的调度思想完全可以用在 Snap 内部。

Snap 的设计也和 [[xen-2003]] 有一段渊源。Xen 证明了硬件虚拟化可以把多个操作系统安全地跑在一台物理机上。Snap 把这个思路缩小了一个级别：它不虚拟化整个操作系统，而是虚拟化网络栈——让每台机器上的多个 VM/容器共享一个高效的网络处理引擎，同时通过进程边界维持隔离。

## 学到什么

1. **"搬出内核"不只是为了性能**——Snap 论文反复强调，迭代速度和可维护性才是搬到用户态的首要动机。性能可以通过专核或硬件卸载补回来，但内核模块的发布流程和调试难度是结构性问题。

2. **资源调度是用户态网络栈最难的部分**——三种调度模型（专核 / 共享 / 自适应）反映的是 CPU 利用率 vs 延迟的根本矛盾。没有万能解，只有场景适配。

3. **微内核架构在工业界是可行的**——Snap 证明了"把大系统拆成小服务、用消息传递通信"这个微内核思路能在生产环境跑五年以上，前提是你愿意在 IPC 性能上持续投入。

4. **热升级能力是云基础设施的刚需**——任何不能热升级的基础组件，最终都会成为整个系统的发布瓶颈。

5. **工业论文的价值在数据**——Snap 论文最有说服力的不是架构图，而是几百万台服务器上的生产数据。学术原型能证明"这个思路行得通"，工业论文能告诉你"行得通到什么程度、在哪里会出问题"。

6. **控制面和数据面分离是一个反复出现的设计模式**——从 [[arrakis-2014]] 到 SDN 到 Kubernetes 到 Snap，"谁做决策"和"谁搬数据"分开管理这个思路在系统设计里无处不在。理解了这个模式，后面遇到类似架构就能快速定位。
