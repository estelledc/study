# Junction：让内核旁路在云端变得实用

## 一、日常类比：高速公路收费站

想象一下你每天开车上下班。正常的网络通信就像走有收费站的普通公路——每个数据包经过操作系统内核时，都要像过收费站一样排队、检查、缴费（做权限检查、内存复制、上下文切换）。内核就像一个非常谨慎但动作缓慢的收费站管理员。

内核旁路（Kernel Bypass）的做法是直接修一条不经过收费站的专用高速通道——应用程序直接与网卡对话，跳过了内核这个"收费站"。这确实快了很多，但问题来了：

- **专用车道只能一辆车走**：以前的内核旁路系统要求每个应用程序独占核心和内存，就像一条高速路只能给一辆车用，其他车全得堵在后面。
- **每辆车都要重新买特殊汽车**：以前的系统要求改写应用程序的代码才能上这条高速路。

Junction 论文要解决的问题是：**能不能修一条多车道的高速路，让成千上万辆不同的车（应用程序）都能走，而且不需要它们换车？**

他们的答案是：能。这就是 Junction。

## 二、核心概念

### 1. 内核旁路（Kernel Bypass）是什么

传统网络通信中，数据从网卡到应用程序需要经过这条路径：

```
网卡 → 内核网络栈 → 内存复制 → 应用程序
```

内核旁路则是：

```
网卡 → 应用程序（直接）
```

跳过了中间两个环节，性能提升可达一个数量级（10 倍）。

### 2. 云端面临的三大矛盾

在数据中心里，运营商面临三个互相冲突的需求：

| 需求 | 传统内核旁路的问题 |
|------|-------------------|
| **高密度**（一台机器跑很多实例） | 需要独占核心和内存，最多支持几百个实例 |
| **兼容性**（不用改写现有程序） | 要求改写应用程序代码 |
| **安全性**（多租户隔离） | 没有内置隔离，依赖虚拟机（带来额外开销） |

Junction 说：这三个矛盾不是必须妥协的，我能同时满足。

### 3. Junction 的架构：三层设计

```
┌─────────────────────────────────────────┐
│         Host Kernel (Linux)              │  ← 只保留最基本的资源分配
│  ┌───────────┐  ┌─────────────────────┐  │
│  │ Scheduler │  │  Instance 0 (kProc)  │  ← 每个实例是一个 Linux 进程
│  │ (调度器)   │  │  ┌───────────────┐  │
│  │ 核心分配  │  │  │ Junction Kernel│  │  ← 库式内核，和应用共享地址空间
│  │ 核心分配  │  │  │ + 应用程序      │  │
│  └───────────┘  │  └───────────────┘  │
│  ┌───────────┐  │  ┌───────────────┐  │
│  │  ...      │  │  │ Instance 1     │  │
│  └───────────┘  └─────────────────────┘
└─────────────────────────────────────────┘
         ↓
   ┌─────────────┐
   │  NIC (网卡)   │  ← 提供事件队列、共享缓冲池等高级功能
   └─────────────┘
```

### 4. 三大设计贡献

**（1）高密度：让一台机器跑数千个实例**

- **共享缓冲队列**：传统做法是每个核心独立分配网卡接收缓冲区。Junction 用一个共享缓冲池，所有核心共享，减少了大量浪费的内存。
- **多包缓冲**：一个缓冲区可以装多个数据包（最小 256 字节而非 1500 字节），减少碎片。
- **NIC 事件队列**：网卡在空闲队列收到包时，自动向调度器发通知，不用调度器轮询所有队列。
- **定时器轮**：用高精度定时器（16 微秒）只关注最近要超时的实例，跳过那些"很闲"的。

**（2）兼容性：跑未修改的 Linux 二进制程序**

- **系统调用转函数调用**：Junction 修改了 glibc，让系统调用变成普通的函数调用。寄存器不用保存恢复，还能用向量化指令。
- **共享地址空间**：Junction 内核和应用在同一个进程内，不需要复制参数，不需要缓解瞬态执行攻击。
- **实现 126 个系统调用**：覆盖了 Go、Java、Python、Node.js 等运行时所需的全部接口。

**（3）安全性：减少攻击面**

- 只允许 11 个系统调用进入内核（详见下方代码示例）。
- 每个实例是一个 Linux 进程，用 seccomp 过滤其他所有调用。
- 通过 NIC 硬件隔离实现租户间内存隔离。

## 三、代码示例

### 示例 1：Junction 需要的系统调用（仅 11 个）

传统 Linux 进程每秒可能发出几百个系统调用。Junction 把它们限制到只有 11 个：

```c
// Junction 允许的系统调用清单（共 11 个）

// --- CPU 调度（1 个） ---
yield_core();   // 交出核心：告诉调度器"我需要核心"或"我可以用完还给你"

// --- 内存管理（5 个） ---
mmap();         // 分配内存或映射文件
munmap();       // 释放内存
madvise();      // 给内核内存管理建议（比如"这部分内存不用了"）
mprotect();     // 修改内存保护权限
mremap();       // 调整已有内存映射

// --- 加载二进制文件（3 个） ---
open();         // 以只读方式打开 jail 目录中的文件
close();        // 关闭文件描述符
pread64();      // 从文件的指定偏移读取数据（多线程安全）

// --- 辅助（2 个） ---
write();        // 仅用于调试日志输出到 stdout
exit_group();   // 进程退出（只调用一次）
```

**关键对比**：gVisor 允许 64 个系统调用，gVisor 的 Gofer 组件允许 57 个。Junction 只保留 11 个，攻击面缩小了 3.2 到 7.6 倍。

### 示例 2：共享缓冲队列的伪代码

传统内核旁路系统中，每个核心独立管理接收缓冲区：

```c
// --- 传统做法：每个核心独立的缓冲池（浪费大） ---
for each core i:
    pool[i] = allocate_buffers(NUM_BUFFERS)  // 每个核心一份
    for each queue j assigned to core i:
        post_buffers(pool[i], queue[j])       // 每个队列都要预分配

// 问题：如果有 N 个实例 × 16 个核心 = 16N 份缓冲池，大量浪费
```

Junction 的做法，用共享缓冲池：

```c
// --- Junction 做法：一个实例一个共享缓冲池 ---
shared_buffer_pool = allocate_buffers(TOTAL)       // 整个实例共享

// 每个接收队列只持有一个"引用"，实际数据在共享池中
for each queue j:
    queue[j].ref = &shared_buffer_pool

// 当数据包到达时，NIC 直接从共享池中取缓冲区
// 用 per-core 引用计数器协调回收（无共享缓存行，无锁竞争）
```

### 示例 3：系统调用从"陷阱"到"函数调用"的转换

传统系统调用（慢）：

```c
// 传统方式：用户态 → 内核态（约 100-500 纳秒的上下文切换开销）
void* buf = malloc(1024);    // 触发 syscall → 陷入内核 → 执行 → 返回
// 寄存器保存 → 权限检查 → TLB 切换 → 执行 mmap → 返回
```

Junction 方式（快）：

```c
// Junction 方式：函数调用（约 10 纳秒，减少约 50 倍）
// glibc 被修改，调用 mmap() 时跳转到 Junction 内核的函数
// 通过 trampoline page（地址随机化防 ASLR 攻击）找到 Junction

// 伪代码展示修改后的 glibc 行为：
void* mmap(void* addr, size_t len, int prot, int flags, int fd, off_t offset) {
    // 不再陷入内核，直接调用 Junction 的用户态实现
    return junction_mmap(addr, len, prot, flags, fd, offset);
    // 优势：
    // 1. 不需要保存/恢复寄存器（FP 和向量寄存器可以随意使用）
    // 2. 不需要缓解瞬态执行攻击（因为应用和内核共享地址空间）
    // 3. 参数不用复制（直接访问，不需要拷贝到内核空间）
}
```

### 示例 4：核心调度器的工作流程

```c
// 调度器（运行在专用核心上）的调度循环：

while true:
    // 1. 检查事件队列（NIC 通知：哪个实例的队列收到了包）
    event = poll_nic_event_queue()

    // 2. 检查定时器轮（哪些实例的定时器到期了）
    timed_out = check_timer_wheel()

    // 3. 将可用核心分配给需要的实例
    for instance in event ∪ timed_out:
        core = allocate_idle_core()
        kthread = pick_idle_kthread(instance)
        pin_kthread_to_core(kthread, core)

    // 4. 向运行超时的线程发送 UIPI 中断（用户态 IPI）
    for instance in running_over_quota:
        send_user_ipi(instance.core, instance.kthread)
```

## 四、性能数据总结

Junction 在 NSDI '24 论文中展示了这些结果（基于 Intel Xeon 6354、200GbE 网卡）：

**吞吐量对比（比原生 Linux）：**

| 应用 | 吞吐量提升 | CPU 节省 |
|------|-----------|---------|
| nginx (C) | 1.62× | 19% |
| Node.js | 3.69× | 65% |
| Python | 2.5× | 约 50% |
| Go | 2.8× | 约 55% |
| Rocket (Rust) | 2.0× | 约 40% |
| Tomcat (Java) | 2.2× | 约 45% |

**密度对比：**

| 系统 | 最大实例数（128GB 内存） | P99 延迟 |
|------|------------------------|---------|
| Linux | 不限（很低密度） | ~12ms |
| Firecracker | ~500 | ~1ms |
| Caladan | ~200 | ~200µs |
| **Junction** | **3,500** | **~350µs** |

**关键数字**：Junction 支持 19–62 倍于现有内核旁路系统的实例数，性能持平甚至更好。

## 五、为什么这个工作重要

1. **填补了空白**：之前的内核旁路系统（eRPC、Demikernel、Caladan）虽然快，但都需要改写代码、只支持少数几个应用。Junction 是第一个能在不改动代码的前提下，同时满足高密度和兼容性的方案。

2. **让内核旁路从"实验室"走向"生产"**：Go、Java、Python、Node.js 这些现代语言的运行时以前在内核旁路中完全跑不了。Junction 打破了这个限制。

3. **安全设计是云原生的**：不是"先追求性能，安全后面再说"，而是从第一天就通过最小化系统调用、seccomp 过滤、NIC 硬件隔离来确保安全。

## 六、个人思考

学习这篇论文后，我最大的感受是：**Junction 的核心思想不是发明了什么全新的硬件，而是聪明地"组合使用"现有硬件的能力**。

- 用 NIC 的事件队列替代轮询 → 省 CPU
- 用共享缓冲池替代独立缓冲池 → 省内存
- 用 WRFSBASE 指令替代系统调用 → 省上下文切换
- 用定时器轮替代全量检查 → 省调度延迟

这就像一个人用现有工具拼出了一个高效的厨房——不需要买新设备，关键是知道怎么组合。

另一个有趣的点是 Junction 团队（MIT CSAIL + Azure Research）选择了"最小化内核接口"而非"完全替换内核"的策略。这意味着他们可以直接复用 Linux 现有的调试和监控工具——这是一种务实的工程选择。
