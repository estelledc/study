## 是什么

UC Berkeley CS186（Introduction to Database Systems）是 UC Berkeley 本科数据库系统课程，由 Prof. Alvin Cheung 讲授。课程从 SQL 出发，自底向上拆解数据库管理系统的内部实现：存储层（磁盘/缓冲区/文件）、索引层（B+ 树）、查询执行与优化器、事务与并发控制（2PL）、崩溃恢复（ARIES）、分布式事务（2PC/Paxos）、NoSQL 系统（MongoDB/MapReduce/Spark）。本笔记覆盖其核心阅读主题——DBMS 分层架构、B+ 树索引、两阶段锁（2PL）和 ARIES 恢复算法。

**日常类比**：数据库就像一个超级图书馆。SQL 是你写的借书条（"给我 2024 年出版的科幻小说，按作者排序"）。查询优化器是聪明的图书管理员——它决定是先去科幻区翻书架（全表扫描），还是先查索引卡片（B+ 树索引）。事务管理器是门禁系统——两个人不能同时修改同一本书的借阅状态（2PL 锁）。恢复系统是监控录像——就算图书馆突然停电，录像（WAL 日志）能回放所有操作，把借阅记录恢复到停电前一秒（ARIES）。

## 为什么重要

数据库是几乎所有应用的底层依赖。理解其内部原理有三层价值：

1. **写出高效 SQL**：知道 B+ 树结构才能理解为什么 `WHERE id = 123` 快而 `WHERE name LIKE '%abc'` 慢；知道优化器工作原理才能看懂 EXPLAIN 输出。
2. **系统设计直觉**：DBMS 的分层架构、WAL 日志、MVCC 等设计模式被分布式系统、消息队列、存储引擎广泛复用。
3. **面试与工程判断**：索引选择、事务隔离级别、连接池配置等日常决策都扎根于数据库内部原理。

课程推荐教材：Ramakrishnan & Gehrke（"Cow Book"）和 Silberschatz, Korth & Sudarshan（"Korth"）。

## 核心要点

### 1. DBMS 分层架构

数据库管理系统从顶到底分为两大层，每层有明确职责：

```
客户端 / 应用层
│
├─ 查询处理器 (Query Processor)
│  ├─ 解析器 (Parser)：SQL 文本 → AST（抽象语法树）
│  ├─ 重写器 (Rewriter)：视图展开、谓词下推、常量折叠
│  ├─ 优化器 (Optimizer)：逻辑计划 → 代价最低的物理计划
│  └─ 执行器 (Executor)：火山模型，open() → next() → close()
│
├─ 存储管理器 (Storage Manager)
│  ├─ 访问方法 (Access Methods)：堆文件、B+ 树索引、哈希索引
│  ├─ 缓冲区管理器 (Buffer Manager)：内存中缓存磁盘页
│  └─ 磁盘空间管理器 (Disk Space Manager)：页面分配与回收
│
├─ 事务管理器 (Transaction Manager)
│  ├─ 锁管理器 (Lock Manager)：共享锁 / 排他锁
│  └─ 日志管理器 (Log Manager)：WAL 写入与恢复
│
└─ 持久化存储 (Disk)
```

**火山模型（Iterator Model）**：执行器中每个算子（扫描、过滤、连接、排序）都实现 `open()` / `next()` / `close()` 三个接口。`next()` 是拉取模式——上层算子调用下层算子的 `next()`，元组自底向上流动。好处是流水线化（无需物化中间结果），坏处是函数调用开销大（每次一行）。MySQL、PostgreSQL、SQLite 都用这套模型。

### 2. B+ 树索引

B+ 树是数据库最核心的索引结构。它是一棵自平衡搜索树，所有实际数据（或数据指针）只存储在**叶子节点**中，内部节点只存路由键和子节点指针。

**结构与特性**：

```
         [内部节点: 键 30 | 70]
        /          |           \
   [叶子]        [叶子]         [叶子]
   键 10-30 →   键 30-70 →    键 70-100
   (链表相连)    (链表相连)     (链表相连)
```

- 叶子节点之间用**双向链表**连接 → 范围查询（`BETWEEN`、`>`、`<`）只需找到起点然后沿链表扫描
- 每个节点恰好对应**一个磁盘页**（通常 4KB-16KB）→ 访问一个节点 = 一次磁盘 I/O
- 高扇出（fanout）：一个节点可容纳数百个键 → 树很矮（3-4 层可索引数亿行）
- 插入导致节点满时**分裂**，删除导致节点太空时**合并或重分配**（实际中常延迟合并）

**为什么不用二叉搜索树？** 二叉搜索树每个节点只有一个键，磁盘 I/O 按页读取，一次读 16KB 却只用一个键是巨大浪费。B+ 树每个节点塞满一页，一次 I/O 读几百个键。

**实际规模估算**（4KB 页，键 4 字节 + 指针 8 字节）：
- 高度 2 → ~111,000 行
- 高度 3 → ~37,000,000 行
- 高度 4 → ~312,000,000 行

顶层 1-2 级通常常驻内存（缓冲池），因此大多数查找只需 **1-2 次磁盘 I/O**。

### 3. 两阶段锁（2PL）

两阶段锁是一种**悲观并发控制**协议，保证多个事务同时运行时数据一致性（可串行化）。

**两种锁**：

| 锁类型 | 用途 | 兼容性 |
|--------|------|--------|
| 共享锁 (S) | 读数据 | 多个事务可同时持有 |
| 排他锁 (X) | 写/修改数据 | 仅一个事务可持有，阻塞所有其他访问 |

**两条规则构成"两阶段"**：

1. **增长阶段**：事务只获取锁，不释放任何锁
2. **收缩阶段**：事务只释放锁，不获取任何新锁

一旦释放了第一个锁，就进入收缩阶段，再也不能获取新锁。

```
增长阶段              收缩阶段
[LOCK A]              [UNLOCK A]
[LOCK B]              [UNLOCK B]
[LOCK C] ─── 分界点 ──→ [UNLOCK C]
  只获取                 只释放
```

**严格两阶段锁（Strict 2PL）**：事务持有的所有排他锁（写锁）直到提交或中止才释放。这防止了级联回滚——事务 B 永远读不到事务 A 未提交的修改，因为 A 的写锁还握着。

**死锁问题**：2PL 会导致死锁——T1 等 T2 的锁，T2 等 T1 的锁，循环等待。数据库用死锁检测（定期扫描等待图找环，杀掉一个受害者）或死锁预防（Wait-Die、Wound-Wait 策略）解决。

**日常类比**：你和室友共用一个冰箱。贴上"使用中"便签就是获取排他锁——别人不能同时往冰箱放东西。按清单一次拿完所有食材（增长阶段），关冰箱门（释放最后一个锁 = 收缩），在此之前不能重新打开冰箱拿漏掉的东西。

### 4. ARIES 崩溃恢复算法

ARIES（Algorithm for Recovery and Isolation Exploiting Semantics）是 IBM 研究员 C. Mohan 提出的数据库崩溃恢复算法，被 IBM Db2、Microsoft SQL Server 等商业数据库采用。它基于三条原则：

**原则一：预写日志（Write-Ahead Logging, WAL）**
任何数据页的修改在写入磁盘之前，对应的日志记录必须先刷到稳定存储（磁盘）上。日志是 append-only 的顺序写，远快于数据页的随机写。

**原则二：重复历史（Repeating History）**
崩溃后恢复时，ARIES 把日志中所有操作——包括未提交事务的操作——全部重放一遍，把数据库恢复到崩溃前一瞬间的精确状态。之后再撤销未提交事务。

**原则三：撤销期间记日志（Logging During Undo）**
连撤销操作本身也写日志，用特殊的补偿日志记录（Compensation Log Record, CLR）。这样如果恢复过程中又崩溃了，重启动时不会重复撤销已经撤销过的操作。

**三段恢复过程**：

| 阶段 | 方向 | 做什么 |
|------|------|--------|
| 分析 (Analysis) | 正向扫描 | 从最近检查点开始扫描日志，确定哪些事务活跃、哪些页是脏的 |
| 重做 (Redo) | 正向扫描 | 重放所有日志记录，数据库恢复到崩溃前状态 |
| 撤销 (Undo) | 反向扫描 | 从日志末尾往回，回滚所有未提交事务 |

**日常类比**：你和朋友合作编辑一份文档，"修订模式"开着（WAL日志）。每半小时拍一张快照——列出谁在编辑哪段（检查点）。如果电脑突然蓝屏：
1. 分析：从最近快照看谁在编辑、哪些段落有未保存修改
2. 重做：把快照后所有修订重放一遍，文档回到蓝屏前的样子
3. 撤销：把没点"保存"（未提交）的人改的内容回退掉

## 实践案例

### 案例 1：B+ 树范围查询的威力

```sql
-- 假设 users 表在 age 列上有 B+ 树索引
-- B+ 树叶子节点按 age 排序且双向链表相连

-- 点查询：从根遍历到叶子，3-4 次 I/O
SELECT * FROM users WHERE age = 25;

-- 范围查询：找到起始叶子，沿链表顺序扫描
-- 25 到 35 之间的所有行都在连续的叶子节点中
SELECT * FROM users WHERE age BETWEEN 25 AND 35;

-- 排序也能利用索引顺序，避免额外排序
SELECT * FROM users WHERE age > 18 ORDER BY age;
-- 如果走 age 索引，叶子节点已经按 age 排好序，无需额外排序
```

**要点**：B+ 树的范围查询能力来自叶子节点的链表连接。B 树（非 B+ 树）数据分散在内部节点，做范围查询需要反复回溯，效率低得多。这也是几乎所有现代数据库都使用 B+ 树变体的核心原因。

### 案例 2：事务隔离与 2PL 行为模拟

虽然现代数据库多用 MVCC（多版本并发控制）而非纯 2PL，理解 2PL 的行为能帮你看懂隔离级别的本质。

```sql
-- 场景：银行转账，T1 从 A 转 100 给 B
-- 隔离级别：SERIALIZABLE（相当于 Strict 2PL）

-- 事务 T1
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 'A';  -- 获取 A 的排他锁
UPDATE accounts SET balance = balance + 100 WHERE id = 'B';  -- 获取 B 的排他锁
COMMIT;  -- 释放所有排他锁
-- 在 COMMIT 之前，T1 持有的所有写锁不释放
-- 因此其他事务读不到 T1 未提交的修改（防止脏读）

-- 事务 T2（与 T1 并发）
BEGIN;
SELECT balance FROM accounts WHERE id = 'A';
-- 如果 T1 还没提交，T2 在 Strict 2PL 下会阻塞等待 A 的排他锁释放
-- 但在 READ COMMITTED 级别下（非 Strict 2PL），T2 可能读到旧版本
COMMIT;
```

**要点**：`SERIALIZABLE` 隔离级别相当于 Strict 2PL——读写锁都持有到事务结束。`READ COMMITTED` 只对写加锁到事务结束，读锁即时释放（读已提交数据），因此一个事务内两次读同一行可能看到不同值（不可重复读）。理解锁的持有时间 = 理解每个隔离级别到底在防什么异常。

### 案例 3：WAL 日志的数据结构（概念示意）

```python
# ARIES 日志记录的概念结构
# 实际实现远比这复杂，但核心字段如下

class LogRecord:
    def __init__(self, lsn, prev_lsn, txn_id, rec_type, page_id, 
                 offset, before_image, after_image):
        self.lsn = lsn              # 日志序列号，单调递增
        self.prev_lsn = prev_lsn    # 同一事务的上一条日志的 LSN
        self.txn_id = txn_id        # 事务 ID
        self.rec_type = rec_type    # UPDATE / COMMIT / ABORT / CLR / CHECKPOINT
        self.page_id = page_id      # 被修改的数据页 ID
        self.offset = offset        # 页内偏移
        self.before_image = before_image  # 修改前的字节（用于 Undo）
        self.after_image = after_image    # 修改后的字节（用于 Redo）


# 事务执行 UPDATE 时的日志流程
def execute_update(txn_id, page_id, offset, old_val, new_val, log_mgr):
    # Step 1: 先写日志（WAL 原则）
    lsn = log_mgr.next_lsn()
    record = LogRecord(
        lsn=lsn,
        prev_lsn=log_mgr.get_last_lsn(txn_id),
        txn_id=txn_id,
        rec_type="UPDATE",
        page_id=page_id,
        offset=offset,
        before_image=old_val,
        after_image=new_val
    )
    log_mgr.append(record)    # 追加到日志文件（顺序写，快）
    log_mgr.flush()           # 刷到磁盘（WAL 的核心保证）
    
    # Step 2: 日志落盘后才能修改数据页（可以延迟写）
    buffer_pool.update_page(page_id, offset, new_val)
    page = buffer_pool.get_page(page_id)
    page.lsn = lsn  # 数据页记录最后修改它的 LSN


# 崩溃恢复的三个阶段（简化逻辑）
def recovery(log_file, checkpoint):
    # Phase 1: Analysis — 正向扫描
    active_txns = set()        # 活跃事务集合
    dirty_pages = {}           # 脏页表：page_id → 最早修改它的 LSN
    
    for record in log_file.scan_from(checkpoint):
        if record.rec_type == "CHECKPOINT":
            active_txns = set(record.active_txns)
            dirty_pages = record.dirty_pages
        elif record.rec_type == "UPDATE":
            active_txns.add(record.txn_id)
            if record.page_id not in dirty_pages:
                dirty_pages[record.page_id] = record.lsn
        elif record.rec_type in ("COMMIT", "ABORT"):
            active_txns.discard(record.txn_id)
    
    # Phase 2: Redo — 正向重放所有修改
    for record in log_file.scan_from(checkpoint):
        if record.rec_type == "UPDATE" or record.rec_type == "CLR":
            page = buffer_pool.get_page(record.page_id)
            if page is None or page.lsn < record.lsn:
                # 页不在内存或其 LSN 早于这条日志 → 需要重做
                buffer_pool.update_page(record.page_id, 
                                        record.offset, 
                                        record.after_image)
    
    # Phase 3: Undo — 反向撤销未提交事务
    for record in reversed_log(log_file):
        if record.txn_id in active_txns and record.rec_type == "UPDATE":
            # 写 CLR（撤销期间记日志，防二次崩溃）
            clr = LogRecord(
                lsn=log_mgr.next_lsn(),
                prev_lsn=record.lsn,
                txn_id=record.txn_id,
                rec_type="CLR",
                page_id=record.page_id,
                offset=record.offset,
                before_image=record.after_image,
                after_image=record.before_image
            )
            log_mgr.append(clr)
            # 执行撤销：写入 before_image
            buffer_pool.update_page(record.page_id, 
                                    record.offset, 
                                    record.before_image)
    
    # 所有活跃事务已撤销完成，数据库处于一致状态
```

**要点**：ARIES 的精妙之处在于它只用 `after_image` 支持 Redo、用 `before_image` 支持 Undo，Clean Page 判断用 `page.lsn < record.lsn` 避免重复工作（幂等性）。CLR 保证崩溃-恢复-崩溃-重新恢复也能正确收敛。

## 踩过的坑

1. **B+ 树不等于二叉树**：B+ 树每个节点是一个磁盘页（4KB-16KB），包含几十到几百个键。把它想象成"多叉树"而非二叉树，才能真正理解为什么树高 3-4 层就能处理亿级数据。

2. **B+ 树的范围查询优势来自叶子链表**：很多人知道"B+ 树适合范围查询"但不知道为什么——答案在于叶子节点之间的双向链表。这个设计在 B 树（非 Plus）中不存在。

3. **2PL 和 MVCC 不是一回事**：现代数据库（PostgreSQL, MySQL InnoDB）多用 MVCC 而非纯 2PL。但理解 2PL 才能理解隔离级别（SERIALIZABLE 约等于 Strict 2PL 的行为），也才能看懂为什么 Snapshot Isolation 下会有写偏斜（Write Skew）异常。

4. **ARIES 不是唯一恢复方案**：SQLite 不实现 ARIES（太复杂），而是用更简单的"回滚日志"（rollback journal）或 WAL mode。理解 ARIES 是理解"完整解决方案"，而实际实现会做权衡取舍。

5. **不要用 `SELECT *` 测试索引效果**：`SELECT *` 可能需要回表（从索引找到行指针再读堆文件取完整行）。如果查询只涉及索引列（覆盖索引），不需要回表，性能差异巨大。用 EXPLAIN 看执行计划而不是凭感觉。

6. **事务隔离级别的默认值各有不同**：PostgreSQL 默认 READ COMMITTED，MySQL InnoDB 默认 REPEATABLE READ，Oracle 默认 READ COMMITTED。不要假定所有数据库行为一致。

7. **WAL 的代价是写入放大**：每次数据修改至少写两次（一次日志、一次数据页）。但日志是顺序写（快），数据页可以延迟合并写，整体吞吐反而提升。这就是为什么即使有写入放大，WAL 也让数据库更快，不更慢。

## 历史小故事

B+ 树的历史可以追溯到 1970 年 Rudolf Bayer 和 Edward McCreight 在波音研究实验室发明的 B 树（"B" 的含义至今有争议——Balance？Bayer？Boeing？）。B+ 树是 B 树的变体，将数据全部移到叶子层并加链表连接，1970 年代后期在 IBM System R 中首次用于数据库索引。至今 50 多年，B+ 树仍是关系型数据库索引的事实标准——这种设计经得起时间考验。

ARIES 的故事同样传奇。1980 年代末，IBM Almaden 研究中心的 C. Mohan 和同事在 System R 和 DB2 的实践中发现，当时流行的"影子页"恢复方案性能太差。Mohan 提出基于 WAL + 重复历史的方案，用三段恢复（分析-重做-撤销）和 CLR 解决崩溃中再崩溃的难题。1992 年论文发表后，ARIES 迅速成为工业标准，IBM Db2、Microsoft SQL Server 都基于它。Aries 这个名字也暗含双关——算法缩写恰好对应十二星座中的白羊座，寓意"重生"（数据库从崩溃中重生）。

## 学到什么

完成 CS186 的学习后，你应该能回答：

1. **为什么 `WHERE id = 123` 很快但 `WHERE name LIKE '%abc'` 很慢？** 前者走 B+ 树精确查找（O(log n)），后者无法利用有序索引必须全表扫描。
2. **数据库崩溃后为什么数据不丢？** WAL 日志保证所有修改先写日志再写数据页；崩溃后 ARIES 三阶段恢复重放日志到一致状态。
3. **两个事务同时改同一行会怎样？** 靠锁管理器：先到的事务获取排他锁（X lock），后来的事务阻塞等待；Strict 2PL 下锁在事务提交时释放。
4. **优化器怎么决定用哪个索引？** 基于统计信息（表行数、列分布直方图）估算每种执行计划的 I/O 成本，选代价最小的。
5. **NoSQL 解决了什么问题？** 关系型数据库的 ACID 约束和 schema 强制执行在分布式场景下有扩展瓶颈；NoSQL 牺牲部分一致性换取水平扩展和灵活数据模型。
