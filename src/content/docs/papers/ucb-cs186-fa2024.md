## 是什么

这是 UCB CS186 2024 秋季数据库课程的阅读路线速记：围绕数据库内部结构，把“会用 SQL 的人”带到“会做数据库设计决策”的层面。日常类比：你先会开车（SQL），再会看导航系统（查询器）和发动机模型（存储）——课程读表帮助你知道什么时候该“加档位”、什么时候该“降速”避免抖动。

我会按这条路线把题目拆成五段：磁盘、索引、执行、并发、恢复。

## 为什么重要

数据库不只是 API；同一个接口下，不同内核实现会导致完全不同的性能和一致性行为。

读这个列表能快速校准工程决策：

- 什么时候 B+Tree 已足够，什么时候要倒向 LSM
- 事务到底在哪个层面打破了“读到旧数据”的幻想
- 为什么索引、日志、buffer manager 的交界最容易出事故

## 核心要点

### 1）存储结构

课程核心是把页、段、日志看成一个闭环系统：

- Page 为最小管理单位，决定了读放大
- WAL 和 redo/undo 决定崩溃后恢复边界
- 空间回收不是纯“删文件”，而是和 GC/分裂策略耦合

### 2）B+Tree 与索引

B+Tree 常作为默认起点，是因为顺序可见、范围查询友好。

- 叶子节点有链表增强 range scan
- 内部节点分裂/合并关系到 insert 性能
- 二级索引带来 tuple locator 问题，需要回表策略

### 3）并发控制

课程里 Aries / 2PL / MVCC 的比较很关键。

- 2PL：简单理解安全，但 hold 锁范围大时吞吐会掉
- S2PL/可串行化：边界清晰，代价也清晰
- MVCC：读写分离更强，但垃圾回收和可见性检查成本高

### 4）恢复机制

ARIES 风格强调“先分析日志再重做”。

- 重放阶段恢复幂等性关键
- 检查点（checkpoint）不能只当检查点，还要绑定 LSN 管理
- 任何“只靠内存索引”的快照在 crash 时都是假快照

### 5）执行与优化

从解析计划到物理执行，优化器不会替你做“业务直觉”。

- 连接顺序影响指数级；
- 索引选择不止选有无，需要选代价模型
- 统计信息老化会造成明显错误路径

## 实践案例

### 案例 1：课程中的小账本系统

假设单机账本有大量区间查询：

- 纯主键索引会让范围扫描慢。
- 改用联合索引可把扫描变成顺序读。
- checkpoint 时间不及时会让 crash 后恢复时间暴增。

### 案例 2：电商购物车高并发更新

一个商品数量更新场景：

- 若只靠行锁且事务太长，热点商品可能“队列化”。
- 读多写少时 MVCC 平滑，但长事务会造成版本雪崩。
- 课程建议把关键路径拆事务，减少持锁窗口。

### 案例 3：日志驱动的批量导入

导入大量记录时：

- 批量 WAL + 小事务可控。
- 恢复测试要跑“中断+重启”而不是“跑完才测”。
- checkpoint 与 flush 策略直接影响恢复时间 SLO。

## 踩过的坑

1. **把 B+Tree 当万能结构**：范围场景不一定适合 LSM。
2. **小看恢复测试**：如果只做功能测试，不做 crash-replay，很难发现系统层漏洞。
3. **把并发当锁策略问题**：实际问题更多是版本可见性和隔离级别。
4. **误用索引**：索引很贵，查询快不等于整体系统快。
5. **计划器盲信**：优化器基于统计信息，不等于业务真值。

## 适用 vs 不适用场景

适用：

- 学习数据库 internals 的入门到中级过渡阶段
- 想从 SQL 工程师切换到 storage / query planner 方向
- 需要重建一套自研数据库课程或内部技术雷达

不适用：

- 只想快速上手一个托管数据库（应直接看官方文档）
- 完全不关心事务、一致性和恢复的产品项目
- 要求分钟级上线的轻应用（先用托管服务）

## 历史小故事（可跳过）

- 传统课程里，B+Tree + 2PL 一直是主轴；2020 后 MVCC、LSM、向量索引等成为补充。
- CS186 把课程和工程作业强绑定，常被当作“做过一门完整数据库内核课就像看了《红宝书》浓缩版”。
- 现在的云数据库实现经常把课程里每一层都切出来给工程团队做可观测化。

## 学到什么

1. 数据库的关键瓶颈常在“边界”而不是“算法”：日志/缓冲/并发边界。
2. 阅读 list 的价值在于形成统一语言：同事讲 B+Tree Split，团队听得到。 
3. 并发与恢复不分家，都是对“时间”建模的系统问题。
4. 性能不是单点调优，必须在代价模型和数据分布上同步。
5. 好的数据库能力来自“读得懂结构图”而不是“会写更多 SQL”。
