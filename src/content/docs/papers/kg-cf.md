## 是什么

KG-CF 是一个**用 LLM 的推理能力过滤冗余上下文，从而提升知识图谱补全（KGC）排名的框架**。日常类比：你想判断"张三是李四的同事"这一事实是否成立，翻了一遍公司通讯录找到 10 条关联路径。但其中 7 条跟"同事关系"不沾边——比如"张三点过的外卖店也是李四常去的"，这就是冗余上下文。KG-CF 做的事就是：**先让 LLM 筛掉废话路径，再用轻量模型对剩余路径打分排序**。

之前的知识图谱补全方法大致分两条路：

- **传统 embedding 方法**（TransE、RotatE、TuckER 等）：把实体和关系压缩成向量，用向量运算预测缺失的边。优点是快，缺点是语义理解弱——"苹果公司"和"苹果水果"在向量空间里可能很近。
- **LLM 直接做 KGC**：把路径转成文本喂给 LLM，让 LLM 判断三元组成立不成立。问题是 LLM 按 token 顺序生成数字（如 `0.732`），累加误差大，而且路径里太多无关信息会干扰 LLM 判断。

KG-CF 的核心洞察：**LLM 擅长理解语义但不擅长输出精确数字**。所以不要求 LLM 直接打分，而是让 LLM 做它擅长的事——判断"这条路径跟这个三元组有没有关系"（二元是/否），然后把 LLM 的判断能力蒸馏给一个轻量的 LSTM 分类器，最后用 BERT 做实际打分。

## 为什么重要

不理解 KG-CF，下面这些事没法解释：

- 为什么 2024-2025 年 LLM + KGC 的工作越来越多，但 LLM 很少被用来直接做排名的 numeric score——都在绕道用 LLM 做辅助判断
- 为什么知识图谱路径推理中，"冗余信息"是最被低估的性能杀手——路径越长，噪声越淹没有用信号
- 为什么蒸馏（distillation）在 KGC 场景里不是锦上添花，而是生存必需——LLM 推理所有路径的开销是指数级的，不蒸馏没法工业化
- 为什么"LLM 当老师，小模型当打分器"的 pipeline 模式在 2025 年成为 KGC 主流范式

## 零基础准备

在理解 KG-CF 之前，先搞懂几个基本概念：

### 知识图谱（Knowledge Graph）
一种图数据库，存储的是**三元组**：(头实体, 关系, 尾实体)。例如：
- (北京, 是首都, 中国)
- (Lakers, 拥有球员, LeBron James)
- (LeBron James, 效力于, Lakers) —— 和前一条是反向关系

### 知识图谱补全（Knowledge Graph Completion, KGC）
知识图谱永远是"缺的"——真实世界中存在但还没录入的关系。KGC 的任务：给定头实体和关系，预测尾实体。例如：
- 问：(Lakers, 拥有球员, ?) → 答：LeBron James
- 本质是对所有候选尾实体**排序**，排最前面的是最可能的答案。

### 路径（Path）
在图中，从头实体到候选尾实体经过的一串边。例如判断"北京是中国的首都"时：
- 有用路径：(北京, 位于, 华北平原)(华北平原, 所属国家, 中国)
- 无用路径：(北京, 方言, 北京话)(北京话, 也流行于, 天津)(天津, 港口, 渤海海运) —— 跟"首都"没关系

### 上下文过滤（Context Filtering）
把无用路径丢掉，只保留对判断有帮助的路径——这就是 KG-CF 的核心创新。

## 核心要点

KG-CF 是三阶段流水线：

### 阶段 1：LLM 路径标注（Path Labeling）

**目的**：让 LLM 给每条路径打标签——相关（1）还是无关（0）。

怎么做：对查询 (头实体 h, 查询关系 r_q, ?) 和候选尾实体 t，先用 BFS 在图里找出从 h 到 t 的所有简单路径。每条路径加上查询三元组本身，拼成一段文本，喂给 LLM。LLM 输出 0 或 1。

关键设置：
- 实体名和关系名都被"文本化"——比如 `inv(plays_for)` 翻转成 "plays for"，让 LLM 读到的就是自然语言
- 训练时只用一小部分路径交给 LLM 标注（全量太贵），产生训练集 D_sc
- LLM 在这里的角色是"标注员"，不是"打分器"——它只管说 yes/no，不管精确分数

### 阶段 2：LSTM 序列分类器蒸馏（Context Filtering）

**目的**：用 LLM 标注的数据训练一个轻量 LSTM，让它学会 LLM 的过滤品味，然后批量处理所有路径。

为什么必须蒸馏：一个三元组可能有几十条路径，整个数据集有几十万三元组。全让 LLM 过一遍，API 费用和延迟都扛不住。

LSTM 架构：
- 输入：路径中每一步的 (实体, 关系, 下一个实体) 拼接向量，每一步额外拼上**查询关系 r_q**（让模型始终知道"我们在问什么关系"）
- 关键设计：同类别实体共享 embedding（比如所有人名共用一个），而不是一个实体一个 embedding。这强制 LSTM 学习**抽象的模式匹配**而非死记实体 ID——类比：学"同事关系"的图模式（A 在 X 公司 → X 公司员工含 B → 同事），不记"张三在腾讯"的具体事实
- 输出：sigmoid 后的 0-1 概率，二元交叉熵训练

过滤逻辑：
- 正例三元组：只保留 LSTM 预测为"相关"（label=1）的路径
- 负例三元组：只保留 LSTM 预测为"不相关"（label=0）的路径

过滤后产生的干净数据集 D_plm 进入第三阶段。

### 阶段 3：BERT 路径打分与三元组排名（PLM Scoring）

**目的**：用过滤后的干净路径训练 BERT，让它给每条路径打分，最后用最高路径分作为三元组的得分。

输入格式：`[CLS] 补全三元组的文本 [SEP] 路径的文本 [SEP]`

BERT 输出一个 logit → sigmoid → [0,1] 之间的分数。训练目标仍是二元交叉熵，但标签变成了**三元组在 KG 中是否真实存在**。

三元组最终得分：
```
score(e_t) = max{ BERT的路径分数 | 所有从 h 到 e_t 的过滤后路径 }
```
取最高分那条路径的分数——因为只要有一条强证据路径，三元组就可能成立（类比：一个人只要"有一份工位记录在腾讯"就可以推断是腾讯员工）。

排名：对所有候选尾实体 e_t 按 score 降序排列，最高的排第一。

为什么不让 BERT 直接打分而要用 max：路径推理天然是"一条路通就行"——你不需要每条路径都相关，只要找到一条强证据就够了。

## 实验结果

### 三个基准数据集

| 数据集 | 规模 | 特点 |
|--------|------|------|
| WN18RR | ~41K 实体, 11 种关系 | WordNet 词关系，长尾分布 |
| FB15K-237 | ~15K 实体, 237 种关系 | Freebase，关系种类多 |
| NELL-995 | ~75K 实体, 200 种关系 | NELL 自动抽取，实体名长文本（KG-CF 最受益于此） |

两个设置：
- **Transductive（传导）**：训练和测试的实体集合一致
- **Inductive（归纳）**：训练时没见过测试时的实体——更接近真实场景

### Transductive 结果（Hits@1 / MRR）

| 方法 | WN18RR | FB15K-237 | NELL-995 |
|------|--------|-----------|----------|
| RuleN | 64.6 / 67.1 | 60.2 / 67.5 | 63.6 / 73.7 |
| TuckER | 60.0 / 64.6 | 61.5 / 68.2 | 72.9 / 80.0 |
| BERTRL | 66.3 / 68.7 | 61.9 / 69.6 | 68.6 / 78.2 |
| **KG-CF** | **67.5 / 70.3** | **62.3 / 70.9** | **73.1 / 82.0** |

### Inductive 结果（Hits@1 / MRR）

| 方法 | WN18RR | FB15K-237 | NELL-995 |
|------|--------|-----------|----------|
| GRAIL | 76.9 / 79.9 | 39.0 / 46.9 | 55.4 / 67.5 |
| BERTRL | 75.3 / 79.5 | 54.1 / 60.6 | 71.7 / 81.0 |
| **KG-CF** | **78.5 / 80.9** | **51.2 / 58.3** | **79.5 / 86.6** |

三个数据集两个设定全都最优。NELL-995 上的优势最明显——因为 NELL 的实体名是长文本描述，LLM 能靠语义理解判断路径相关性，这正是传统 embedding 方法的短板。

### 消融实验（Ablation Study）

在 WN18RR 上做了三项消融：

| 消融 | 操作 | 效果 | 解释 |
|------|------|------|------|
| -pf | 去除正例路径过滤 | 小幅下降 | 不过滤正例路径就混入噪声 |
| -nf | 去除负例路径过滤 | 降幅比 -pf 小 | KG 本身不完整，有些"负例"其实是未标注的正例 |
| -te | 实体名替换为匿名 ID | **大幅下降** | 实体文本描述是 LLM 推理的关键，纯拓扑不够 |

### 路径长度分析

- 固定测试长度 3，训练用更长路径（4, 5）：**有帮助**——训练时见更多上下文模式，测试时也能受益
- 训练和测试路径长度匹配：路径越长，效果越好，但**计算量指数增长**——实际应用需要在性能和开销之间折中

## 实践案例

### 案例 1：KG-CF 如何判断一个公司收购案

问：(ByteDance, acquired, ?)，候选：Musical.ly, Cisco, 茅台

KG 里已有的相关信息（采样 3 条路径）：

```
P1: (ByteDance, founded_in, 北京)(北京, is_in, 中国)(Musical.ly, launched_in, 上海)(上海, is_in, 中国)
    → 两者都是中国互联网公司 → 相关 ✓

P2: (ByteDance, CEO_is, 张一鸣)(张一鸣, graduate_of, Nankai)(Cisco, HQ_in, San Jose)
    → 跟收购毫无关系 → 无关 ✗

P3: (ByteDance, product_has, TikTok)(TikTok, acquired_via, Musical.ly)(Musical.ly, rebranded_to, TikTok)
    → 直接描述收购 → 高度相关 ✓
```

KG-CF 的处理流程：
1. LLM 标注：P1=1, P2=0, P3=1
2. LSTM 学会这个过滤模式，对全网路径过滤，P2 被丢弃
3. BERT 对 P1, P3 打分，其中 P3 得分最高 → 取 max → Musical.ly 排名第一

### 案例 2：泛化能力——为什么 LSTM 用共享 embedding

同一个关系模式可能出现在完全不同的实体之间。例如判断"师生关系"：

- 模式：(A, graduate_from, X)(X, has_professor, B) → 推测 (A, student_of, B)
- 这个模式对 "张三-清华-李教授" 和 "Alice-MIT-Bob" 都成立

传统方法可能学会了"张三→清华→李教授"是师生，但遇到"Alice→MIT→Bob"就懵了。KG-CF 的 LSTM 因为实体共享 embedding，学会的是"graduate_from + has_professor → student_of"这个**图模式**，跟具体身份无关。

### 案例 3：路径长度对效果的实际影响

用 WN18RR 数据：
- 训练路径长度 2，测试长度 3：Hit@1 = 66.2
- 训练路径长度 3，测试长度 3：Hit@1 = 67.5
- 训练路径长度 4，测试长度 3：Hit@1 = **68.0**

训练时用更长的路径，在短路径测试时还有提升——说明长路径训练让模型学会了更细颗粒度的上下文判断，即使测试路径短也能用上这个能力。

但实际部署中的权衡：
- 路径长度从 3 扩到 4：候选路径数膨胀 5-10x → LSTM 过滤开销 5-10x → BERT 打分候选数也多 5-10x
- 如果 BERT 推理一亿条路径需要 1 小时，五亿条就要 5 小时——production 场景可能不可接受

### 案例 4：对比传统 KGC 方法——什么时候 KG-CF 优势最大

| 场景 | 推荐方法 | 原因 |
|------|----------|------|
| 实体名含丰富语义（如 NELL） | KG-CF | LLM 能用文本理解，embedding 方法抓不住 |
| 关系种类少且规则性强（如 WN18RR） | KG-CF 略优，差距小 | 规则推理 LLM 也有帮助但 feature 不多 |
| 实体只有 ID 没有文本 | 传统 embedding | LLM 没有文本可读，优势归零 |
| 延迟要求 < 10ms | 传统 embedding | KG-CF 三阶段 pipeline 太慢 |
| 新实体每天大量涌入（inductive）| KG-CF | 共享 embedding + 图模式泛化 = 零样本推理 |

## 踩过的坑

1. **LLM 标注成本是隐藏瓶颈**：论文只在部分路径上跑 LLM 标注（sampling），但如果路径分布高度偏斜（某些实体路径多、某些少），小样本无法覆盖全貌。实际工程中需要做**分层采样**——确保每种关系类型都有足够标注样本。

2. **蒸馏有质量损失**：LSTM 的过滤准确率永远低于 LLM——LLM 标注准确率 ~90%，LSTM 蒸馏后 ~82-85%。这 5-8% 的差距会导致一些有用路径被误杀。但好在 KG-CF 用 max score——只要有一条关键路径幸存，三元组就不会被误判。

3. **BERT 打分时，路径数越多、推理越慢**：过滤确实减少了路径数，但如果某些实体之间有数百条路径（常见于 NELL 这种密集图），即使过滤完仍有几十条，BERT 推理仍慢。论文没有给出极端路径数下的工程优化。

4. **实体文本化是双刃剑**：NELL 的实体描述是自动抽取的，有时含噪音（"Apple"在 NELL 里可能被抽成水果相关的文本），LLM 标注时也会被误导。论文没有讨论如何处理文本噪音。

5. **Inductive 场景中 FB15K-237 提升有限**：KG-CF 在 FB15K-237 的 inductive 设置上只比 BERTRL 好一点点（Hits@1: 51.2 vs 54.1），甚至不如某些场景下的 GRAIL。原因是 FB15K-237 的关系种类多但实体文本描述相对短，LLM 的语义优势发挥不出来。

6. **多跳推理的深度限制**：KG-CF 理论上支持任意长度路径，但从实验结果看，路径长度超过 4 以后提升极小且开销爆炸，实践中基本限定在 3-4 跳。

## 适用 vs 不适用场景

适用：

- 实体名含丰富自然语言描述的知识图谱（如 NELL、ConceptNet）
- 需要高精度排名的 KGC 场景（如知识问答、推荐系统的召回层）
- 已有 LLM 接口但不想花大钱每条路径都调——蒸馏策略很务实
- 有新实体持续加入的场景（inductive KGC）——共享 embedding 提供零样本能力
- 研究和复现——代码开源，框架结构清晰

不适用：

- 实体只有 ID 没有文本——LLM 无文本可读，直接退化到纯拓扑方法
- 延迟敏感的线上服务——三阶段 pipeline 延迟 > 100ms，除非全部预计算
- 大规模生产级 KG（亿级实体）——stage 3 BERT 推理所有候选不够快
- 预算极度有限——LLM 标注 + LSTM 训练 + BERT 训练三条链都要 GPU
- 关系推理极其复杂的场景（如逻辑约束推理）——KG-CF 基于路径，不支持 ¬, ∨, ∧ 等逻辑运算

## 历史小故事

- 2013：TransE 提出——KGC 进入 embedding 时代，向量加法表示关系平移
- 2016-2018：RotatE、TuckER、DistMult 等——embedding 方法的百花齐放期
- 2019：KG-BERT —— 第一次把预训练语言模型用于 KGC，但只做三元组分类
- 2021：BERTRL —— 把路径推理引入 KGC，用 BERT 对路径打分然后用 max 聚合，这是 KG-CF 的直接前身
- 2023-2024：LLM 爆发，一堆论文尝试让 LLM 直接做 KGC 的 numeric scoring，但效果不稳定
- 2024.12：KG-CF 在 IEEE BigData 发表——提出"LLM 不直接打分而是过滤上下文"的新范式，成为 LLM + KGC 方向的代表工作

## 学到什么

- LLM 在 structured data 任务中的最佳角色往往是"标注员/过滤器"，不是"最终打分器"——让 LLM 做它擅长的语义判断（yes/no），把数值对比交给专门的模型
- 蒸馏不是"省钱的妥协"，而是 KG 规模下唯一的 scalable 方案——全量 LLM 推理所有路径在百万级 KG 上不可行
- 共享 embedding + 抽象图模式是 inductive KGC 的关键——让模型泛化到没见过的实体
- 上下文过滤（context filtering）这个思路不限于 KG——任何用 LLM 处理结构化数据、面临噪声路径/候选的场景都可以参考
- 文本描述的质量决定了 LLM + KGC 方法的上限——NELL 效果好的本质原因不是算法，是数据本身有长文本
- Max-pooling 作为路径聚合函数简单但有效——一条强证据 > 十条弱证据
