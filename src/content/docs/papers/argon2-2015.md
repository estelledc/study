## 是什么

Argon2 是 2015 年 Password Hashing Competition（PHC）的获胜算法——一种**故意设计成吃内存的密码哈希函数**。日常类比：

> 把「猜密码」想成在仓库里找一把钥匙。
> - MD5 / SHA-256 直接哈希：像把钥匙编号刻在门牌上——GPU 可以同时试几百万块门牌，几乎不占场地。
> - PBKDF2：规定你必须在跑步机上原地跑 10 万圈才能试一次——CPU 会累，但攻击者买一万台跑步机也能并行，几乎不需要仓库。
> - bcrypt：每人要占一小块固定工位，稍好一点，但现代 GPU 仍能把工位缩得很小。
> - Argon2：规定每次尝试必须租下整整 64 MiB 到 2 GiB 的仓库，并在里面按规则搬货、搅拌（多轮读写大块内存）。攻击者若把仓库缩成「小货架」省租金，搅拌规则会逼他反复跑远路。

一句话：Argon2 的目标不是「算得慢」，而是让并行暴力破解**同时吃满时间和内存带宽**——专用 ASIC / GPU 很难在不大买内存的前提下把成本压下去。

算法内部用 BLAKE2b 做可变长哈希，用压缩函数 G（1024 字节进、1024 字节出）搅拌数据。互联网标准形态是 IETF RFC 9106（2021 年），对应算法版本 1.3（版本字节 `0x13`）。默认应选混合变体 Argon2id。

## 为什么重要

不理解 Argon2，现代「存密码」实践会停留在过时方案：

- 为什么 libsodium、PHP 7.2+、Ruby 2.5+ 都把 Argon2id 作为 PBKDF2 之外的第一推荐——因为它同时防 GPU 并行和 ASIC 暴力
- 为什么 OWASP 密码存储备忘录取代 bcrypt/scrypt 时优先 Argon2id——内存成本 + 侧信道抗性两者兼顾
- 为什么用户口令不能直接用 HKDF 扩密钥——口令熵低、易被字典攻击，必须先过 Argon2 这类慢哈希提熵
- 为什么 2010 年代 GPU 农场让 bcrypt、PBKDF2 的「迭代次数」防御迅速贬值——只烧时间不烧内存，攻击者买多块显卡就行

## 核心要点

Argon2 的核心设计剥开只有三个概念：

**1. 旋钮 m / t / p——三种可调成本**

m 是内存（KiB），t 是轮数（passes），p 是并行 lane 数。类比：你要在仓库里搬货——m 决定了仓库多大，t 决定了你要搬几圈，p 决定了你能同时开几条搬运通道。攻击者想省钱就得缩小 m，但缩小后 t 会变得极慢（时间-内存权衡），不划算。

RFC 9106 给两种抄作业参数：首选 Argon2id, t=1, p=4, m=2 GiB；低内存就用 Argon2id, t=3, p=4, m=64 MiB。调参原则是先拉 m 再拉 t。

**2. 三种变体——数据依赖 vs 数据独立**

| 变体 | 内存访问怎么定 | 擅长 | 弱点 |
|------|--------------|------|------|
| Argon2d | 下一块读哪由当前块内容决定 | 抗时间-内存权衡最强 | 访问模式泄露给旁路计时攻击 |
| Argon2i | 地址只由索引算出来 | 抗侧信道，适合登录服务 | 抗 TMTO 弱，要多做 passes |
| Argon2id | 前半段像 i，后半段像 d | 兼顾两者，默认推荐 | 实现略复杂 |

多租户登录服务上，Argon2d 可能通过缓存计时泄露密码信息，所以 RFC 9106 要求任何合规实现 MUST 支持 Argon2id。

**3. 内存填充 = 矩阵搅拌**

内部分配 p 条 lane 乘以 q 列的 1024 字节块矩阵 B[i][j]，按 slice 顺序用压缩函数 G 填充。每个新块依赖前一块和另一个「参考块」——Argon2d 用当前块内容选参考块的位置，Argon2i 用索引公式算。t > 1 时重复多轮并与旧块 XOR。最后一列 XOR 成摘要，用 BLAKE2b 输出 tag。

核心洞察：块大（1 KiB）、总量可达 GiB 级，迫使实现真的去 touch RAM，而不是只在 CPU 缓存里打转。

## 实践案例

### 案例 1：用 Python 哈希和验证密码

```python
# pip install argon2-cffi
from argon2 import PasswordHasher
from argon2.low_level import Type, hash_secret_raw
import os

# 高层 API：默认就是 Argon2id
ph = PasswordHasher(
    time_cost=3,        # t
    memory_cost=65536,  # m，单位 KiB → 64 MiB
    parallelism=4,      # p
    hash_len=32,
    salt_len=16,
)

password = "correct horse battery staple"
encoded = ph.hash(password)
# 输出形如: $argon2id$v=19$m=65536,t=3,p=4$<salt_b64>$<hash_b64>

ph.verify(encoded, password)   # 成功则无异常
# ph.verify(encoded, "wrong")  # 抛出 VerifyMismatchError

# 低层 API：自己管 salt，输出原始 tag（适合 KDF 场景）
salt = os.urandom(16)
tag = hash_secret_raw(
    secret=password.encode(),
    salt=salt,
    time_cost=3,
    memory_cost=65536,
    parallelism=4,
    hash_len=32,
    type=Type.ID,
)
# tag 为 32 字节，可再喂给 HKDF 按上下文切分密钥
```

### 案例 2：用 Node.js 实现登录验证（RFC 9106 首选参数）

```javascript
import { randomBytes, argon2, timingSafeEqual } from "node:crypto";
import { promisify } from "node:util";

const argon2Async = promisify(argon2);

async function hashPassword(password) {
  const salt = randomBytes(16);
  const tag = await argon2Async("argon2id", {
    message: Buffer.from(password, "utf8"),
    nonce: salt,
    parallelism: 4,
    tagLength: 32,
    memory: 1 << 21,   // 2 GiB（KiB 单位）
    passes: 1,
    secret: Buffer.alloc(0),
    associated: Buffer.alloc(0),
  });
  return { salt, tag }; // 入库时保存 salt + tag
}

async function verifyPassword(password, salt, expectedTag) {
  const tag = await argon2Async("argon2id", {
    message: Buffer.from(password, "utf8"),
    nonce: salt,
    parallelism: 4,
    tagLength: 32,
    memory: 1 << 21,
    passes: 1,
    secret: Buffer.alloc(0),
    associated: Buffer.alloc(0),
  });
  return timingSafeEqual(tag, expectedTag);
}
```

关键细节：验证时用 `timingSafeEqual` 做常量时间比较，防止攻击者通过测量比较耗时逐字节猜出哈希值。

### 案例 3：移动端用 libsodium 的 crypto_pwhash

```c
#include <sodium.h>

#define OPSLIMIT  crypto_pwhash_OPSLIMIT_MODERATE
#define MEMLIMIT  crypto_pwhash_MEMLIMIT_MODERATE  // 或显式 64MB

unsigned char hash[crypto_pwhash_BYTES_MAX];
unsigned char salt[crypto_pwhash_SALTBYTES];

randombytes_buf(salt, sizeof salt);

if (crypto_pwhash(hash, sizeof hash,
                  password, password_len,
                  salt,
                  OPSLIMIT, MEMLIMIT,
                  crypto_pwhash_ALG_ARGON2ID13) != 0) {
    // 内存不足——移动设备上可能要降级参数
}
```

算法标识 `ARGON2ID13` 即 Argon2id v1.3，与 RFC 9106 一致。Ente、Bitwarden 等客户端的密码管理器都用这套 API 做主密码拉伸。移动设备上 m 过大时可能返回 -1，应降级到 SECOND RECOMMENDED（64 MiB, t=3）或排队到服务端算。

## 踩过的坑

1. **「Argon2 比 SHA-256 安全」——用途不同**：SHA-256 是快哈希，Argon2 是故意慢的口令拉伸。前者用来校验文件完整性，后者用来存密码，不能互相替代。

2. **「内存越大越好，t 永远是 1」——要在可接受登录延迟内平衡**：服务端设 2 GiB 时每次登录 0.5 秒，并发 100 个登录就会吃光 200 GiB 内存。移动端 2 GiB 更不现实——必须按部署环境选参数。

3. **多租户服务器上用 Argon2d——可能泄露访问模式**：如果攻击者和正常用户共享同一物理 CPU 核心，Argon2d 的数据依赖访问模式可通过缓存计时泄露密码信息。共享主机上优先 Argon2id。

4. **验证时用普通字符串比较而非常量时间比较**：`tag === expectedTag` 会在第一个不同字节处提前退出，攻击者可以通过测量耗时逐字节猜哈希——必须用 `timingSafeEqual` 或 `sodium_memcmp`。

## 适用 vs 不适用场景

**适用**：

- 用户密码存储（注册时哈希、登录时验证）——这是 Argon2 的主场
- 从低熵口令派生加密密钥（磁盘加密、文件加密的 KDF 步骤）
- 加密货币工作量证明（用 Argon2d，抗 ASIC 的内存成本）
- 需要现代标准且能承受 0.5-3 秒延迟的任何口令拉伸场景

**不适用**：

- 需要微秒级响应的场景——Argon2 故意慢，不适合做请求签名哈希
- 已持有高熵随机密钥时的派生——用 HKDF 就行，不需要烧内存
- 嵌入式设备内存不足 64 MiB——可用但参数需极端压缩，安全边际很低
- 纯在线猜测防御——Argon2 帮不上忙，限速、MFA、账号锁定才是主力

## 历史小故事（可跳过）

- **2010 年代初**：GPU 农场和 FPGA 集群让 bcrypt、PBKDF2 的迭代次数防御迅速贬值。scrypt 率先提出「内存成本」概念（RFC 7914），证明让攻击者同时烧内存比只烧时间更有效。

- **2013-2015 年**：密码哈希社区举办公开的 Password Hashing Competition（PHC），类似 AES 和 SHA-3 的竞赛模式。Argon2 由卢森堡大学团队提交，在 24 个候选算法中经过两年透明评审胜出。

- **2015 年**：Argon2 获 PHC 冠军，论文发布在 password-hashing.net。设计上有意提供三种变体（d/i/id），让部署者按威胁模型选择——这是区别于 scrypt 的关键优势。

- **2021 年**：IETF 发布 RFC 9106，将 Argon2 标准化为互联网协议。规定任何合规实现 MUST 支持 Argon2id，并给出 FIRST/SECOND RECOMMENDED 两组参数，结束了多年来「不知道参数设多少」的混乱。

- 此后：libsodium 内置支持、PHP 7.2+ 的 `password_hash` 加入 `PASSWORD_ARGON2ID`、Node.js 15+ 的 `crypto.argon2`、Ruby 2.5+ 的 `has_secure_password`——主流生态全部跟进。

## 学到什么

1. **安全 = 成本经济学**：Argon2 的设计哲学不是让破解「不可能」，而是让破解「不划算」。内存成本让 GPU 的并行优势大打折扣——8 GiB 显卡面对 2 GiB 的 Argon2 只能同时算 4 个，而不是 PBKDF2 的百万个。

2. **没有免费的午餐——三种变体的取舍**：Argon2d 抗 TMTO 最强但有侧信道风险，Argon2i 安全但要多耗轮数，Argon2id 折中但实现略复杂。选型依赖威胁模型——这是安全工程的核心思维。

3. **标准化之前的混乱**：RFC 9106 出来之前，每个人都在猜参数——有人设 m=8 MiB（太低）、有人设 t=100（不看 m 只看 t 没意义）。标准化不是「定个最优值」，而是「给不知道的人一个不会错的起点」。

4. **弱口令仍会输**：Argon2 抬高破解成本，但不能把 `123456` 变安全。它配合的是用户教育 + MFA + 限速组成的纵深防御，不是单点银弹。
