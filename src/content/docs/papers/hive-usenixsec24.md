# HIVE：用硬件来保护 eBPF

## 一、先搞明白：eBPF 是什么？

想象你住在一栋公寓楼里。大楼管理员（Linux 内核）负责管理所有设施。
正常情况下，住户不能随便拆墙改管道——但 eBPF 给了住户一种"安全的改装权限"。

**eBPF（extended Berkeley Packet Filter）** 是 Linux 的一个特性，允许用户态程序
把小段代码（BPF program）加载到内核里运行。它广泛用于：

- 网络数据包过滤与转发
- 系统性能监控与追踪（tracing）
- 安全检测与入侵防御

听起来很棒，但有个根本问题：**加载到内核的代码，内核怎么相信它是安全的？**

## 二、当前方案的问题：eBPF Verifier

目前 Linux 用 **eBPF Verifier（验证器）** 来解决这个问题。

Verifier 的工作方式类似"代码审查"：在 BPF 程序真正运行之前，
它会对程序做 **静态分析（静态 = 不运行就分析）**，检查程序是否会：

1. 越界访问内存（Out-of-Bounds, OOB）
2. 泄露内核信息（Pointer Leakage）
3. 导致内核崩溃（Crash / DoS）
4. 进行 Spectre 等侧信道攻击

**问题出在哪？**

- **安全方面**：过去十年里，eBPF 验证器的漏洞（CVE）中有 90% 以上
  来自完整路径分析（full-path analysis）。36/60 个 eBPF 的 CVE 都源于验证器本身。
  验证器太复杂，代码量巨大，不可能做到完美。

- **复杂性方面**：有些合法的 BPF 程序因为验证器的分析能力有限，
  居然无法通过验证。开发者不得不写"迎合验证器的代码"（verifier-oriented programming），
  比如用掩码操作来降低验证复杂度。

**类比**：就像机场安检，安检员（Verifier）要看每个旅客的每一样东西，
判断是否危险。安检员太忙、判断有误，有时让危险分子混过去，
有时又把无辜旅客拦下。

HIVE 的思路完全不同：**不再靠"审查"，而是靠"隔离"**。

## 三、HIVE 的核心思想：把 eBPF 当成"内核级应用"来隔离

HIVE 的关键洞察是：**BPF 程序本质上是一种新的内核模式应用程序。**

类比用户态程序：用户程序运行在用户态（EL0），不能访问内核内存，
因为内核内存页被设为"特权页"。用户程序崩溃了，不会直接搞垮整个系统。

HIVE 想对 BPF 程序做同样的事——虽然 BPF 程序运行在内核态（EL1），
但通过硬件辅助的隔离，让它像用户程序一样"受限"。

## 四、三大硬件特性

HIVE 利用了 AArch64 架构的三个硬件特性：

### 4.1 非特权加载/存储指令（LSU, Load/Store Unprivileged）

ARM 架构中，`ldtr` 和 `sttr` 是一对特殊的加载/存储指令。
无论它们在哪种特权级别执行，都**被当作用户态（EL0）权限**来处理。

```
普通加载: ldr x0, [x1, #8]    ← 权限取决于当前运行级别
非特权加载: ldtr w0, [x1, #8] ← 始终当作 EL0 权限，无法访问特权页
```

HIVE 把所有 BPF 代码编译为使用 `ldtr`/`sttr` 指令。
内核内存是特权页（P-Page），BPF 内存是非特权页（U-Page），
所以 BPF 程序根本**碰不到**内核内存。

### 4.2 指针认证（PA, Pointer Authentication）

AArch64 的 PA 特性提供两条指令：

- `pacda` — 给指针"签名"（加上认证码）
- `autda` — 验证指针"签名"是否合法

```
pacda x0, x6   ← 把 x0 指针用 x6 作为"钥匙"进行签名
autda x0, x6   ← 验证 x0 的签名是否正确，不正确则触发异常
```

签名后的指针如果被恶意修改（篡改地址），认证就会失败，
触发异常，HIVE 借此捕获并阻止非法访问。

### 4.3 E0PD（EL0 禁用访问）

E0PD 是一个 CPU 寄存器位，控制用户态能否访问某些内存。
HIVE 用它来防止 BPF 程序和用户态程序互相访问对方的内存。

## 五、BPF 指针分类：HIVE 的精妙设计

BPF 程序中有 18 种指针类型，HIVE 把它们分为两大类：

### 5.1 包容型指针（Inclusive Type Pointers）

指向 **BPF 自身的数据对象**，比如栈、map 值、数据包缓冲区等。
这些指针可以安全地泄露，因为它们指向的是 BPF 自己的内存空间。

**HIVE 的处理方式**：把所有 BPF 数据放到独立的"BPF 地址空间"，
映射为非特权页。包容型指针只要被限制在这个空间内就安全了。

### 5.2 独占型指针（Exclusive Type Pointers）

指向 **内核对象**，比如网络 socket、TCP socket、上下文对象等。
这些指针绝对不能泄露，否则就暴露了内核内存布局。

**HIVE 的处理方式**：
1. 用 PA 给指针签名
2. 首次访问时，通过异常探测（exception-based probing）来识别目标
3. 通过白名单校验是否允许访问
4. 通过后，把代码"热补丁"为正常的加载/存储指令

## 六、代码示例

### 示例 1：LSU 指令 — 非特权内存访问

```asm
# BPF 程序尝试读取数据（HIVE 编译后）

# 原始 BPF 指令（伪代码）：
#   R1 = *(u32 *)(R1 + 8)    ← 从 sk_buff 结构读取 len 字段

# HIVE 编译后的汇编（使用了 ldtr 非特权加载）：
mov   x5, x0                # x5 = R0 (保存原始指针)
ldtr  w1, x5, #8            # 非特权加载：读取 [x5 + 8] 的 32 位值
                              # ⚠️ 如果 x5 指向内核特权页 → 硬件触发异常！
```

**解释**：普通 `ldr` 指令如果地址指向内核页，能正常读取。
但 `ldtr` 被当作用户态权限，访问特权页会触发异常。
BPF 程序因此被阻止访问内核内存。

### 示例 2：指针认证 — 独占型指针的安全访问

```asm
# HIVE 在 BPF 程序入口处的指针认证设置

# 1. 加载执行计数器到 x24
mov   x5, 8
mov   x6, &exe_cnt
ldadd x5, x24, x6           # x24 = exe_cnt (全局递增计数器)

# 2. 计算修饰符 = exe_cnt OR ctx_type
orrx  x6, x24, #ctx_type    # ctx_type = 类型标识（3位常量）
                              # x6 = exe_cnt | ctx_type ← 作为 PA 的修饰符

# 3. 对上下文指针签名
pacda x0, x6                # x0 = context_ptr，用 x6 签名
                              # 签名后的指针只有用同一个 x6 才能验证通过
```

```asm
# 后续访问上下文对象时：

# 1. 尝试用 ldtr 读取（HIVE 统一用非特权指令）
ldtr  w1, x0, #8            # 尝试读取 context->field[8]

# 2. 如果 x0 是合法的独占型指针（已签名且指向允许的白名单对象）
#    HIVE 的异常处理器会捕获这次"非法"访问：

# 3. 异常处理程序（HIVE 注入）：
#    a) 用 autda 认证 x0 指针
#    b) 从认证结果中提取类型信息
#    c) 检查该类型是否白名单允许访问
#    d) 如果允许 → 把这条 ldtr 替换为正常的 ldr 指令
#    e) 如果禁止 → 卸载（uninstall）整个 BPF 程序
```

```asm
# 替换后的正常指令（后续执行不再触发异常）：
ldr   w1, x0, #8            # 正常的特权加载，可以读取内核对象
```

**解释**：第一次访问时，`ldtr` 触发异常 → HIVE 的异常处理器
介入检查 → 验证通过后"热补丁"为正常 `ldr` → 后续访问直接走正常路径。
这比静态验证更灵活，也能支持更复杂的 BPF 程序。

### 示例 3：描述符表 — 指针混淆

```asm
# 使用描述符表来隐藏真实指针地址

# 1. 对描述符做掩码操作，防止越界
and   x14, x14, #0xFFFF      # 只取低 16 位

# 2. 从类型对应的描述符表中查找真实指针
ldr   x15, [x23, x14, lsl #3]  # x23 = 表基址, x15 = 真实内核指针

# 3. 用正常指令访问内核对象
ldr   w1, x15, #8            # 访问内核对象字段
```

**解释**：BPF 程序拿到的不是真实指针，而是表中的索引（描述符）。
真实指针存在内核侧的表中，程序无法直接获取。
这防止了指针泄露，也防止了 Spectre 攻击。

## 七、HIVE 的整体架构

```
用户空间（EL0）
  ─── 用户程序：只能访问 U-Page

内核空间（EL1）
  ├── 内核代码：LS 指令（普通加载/存储）
  ├── 内核数据：P-Page（特权页，BPF 碰不到）
  │
  ├── BPF 代码：LSU 指令（非特权加载/存储）
  ├── BPF 空间（1TB 独立地址空间）：U-Page
  │    ├── BPF 栈
  │    ├── BPF Map
  │    └── 数据包缓冲区
  │
  └── Shadow BPF 空间（1TB）：供内核侧 helper 函数访问
```

关键设计：
- **BPF 空间**：BPF 数据的独立地址空间，所有内存为 U-Page
- **Shadow BPF 空间**：内核 helper 函数通过它间接访问 BPF 数据
- **E0PD 切换**：内核调用 BPF 程序时，切换 E0PD 位以控制访问权限

## 八、安全保证

HIVE 的目标是替代验证器的"完整路径分析"，同时保持相同的安全保证：

| 安全目标 | HIVE 如何实现 |
|---------|-------------|
| 内存安全 | BPF 空间独立 + 非特权指令，BPF 碰不到内核内存 |
| 防止信息泄露 | 独立地址空间 + 描述符表隐藏真实指针 |
| 防止 DoS | 异常捕获 + 执行时间监控 + 自动卸载 |
| 防止 Spectre | LSU 指令受 CSV3 补丁保护，描述符表用掩码 |

## 九、性能表现

在 Apple M1（支持 PA 特性）上的实验结果：

- **微基准测试**：LSU 指令与普通指令速度几乎一样（~0.34-0.50 周期）
- **指针认证**：pacda/autda 约 7 周期，首次探测约 60K 周期（仅一次）
- **eBPF 整体开销**：相比原生 eBPF，HIVE 的平均开销约 **3.3x-4.2x**
- **但**：HIVE 能跑通那些验证器拒绝的复杂程序

| 应用场景 | 原生 eBPF 开销 | HIVE 开销（对比无 BPF） |
|---------|--------------|---------------------|
| Apache + Tracee | 76.6% CPU | 76.6% CPU |
| Nginx + BCC | 95.5% CPU | 103.3% CPU |
| Redis + eBPF | 55.9% CPU | 60.5% CPU |

对于 I/O 加速场景，HIVE 甚至能跑通验证器拒绝的程序，
在小型 I/O 操作时加速高达 181%。

## 十、总结

HIVE 的核心贡献可以概括为一句话：

> **把 eBPF 从"需要静态审查的代码"转变为"可以用硬件隔离运行的内核应用"。**

它做对了三件事：
1. **用 LSU 指令做内存隔离** — 简单、高效、硬件保障
2. **用指针认证保护内核指针** — 动态、灵活、支持复杂程序
3. **用指针分类减少检查负担** — 包容型放 BPF 空间，独占型走认证流程

这篇论文展示了一个重要的范式转变：与其让验证器越来越复杂，
不如让硬件来承担安全保证的工作。

## 思考题

1. HIVE 依赖于 AArch64 的 PA 和 E0PD 特性。如果要在 x86 上实现类似方案，
   应该利用 x86 的哪些硬件特性？

2. HIVE 的"异常探测"机制在首次访问时开销很大（60K 周期），
   但只执行一次。这种"懒加载"式的安全检查在其他场景中有类似应用吗？

3. 如果内核本身存在漏洞（不在 HIVE 的信任范围内），HIVE 还能提供什么保护？
