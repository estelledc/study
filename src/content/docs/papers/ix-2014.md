## 是什么

IX 是一个**数据面操作系统**：它把 Linux 内核里负责网络处理的部分抽出来，放进一个独立的受保护"数据面"，让这个数据面专门、高速地处理网络包，同时保持普通 OS 的隔离保护能力。

日常类比：把厨房和前厅彻底隔开。Linux 是前厅经理（控制面），负责分配桌子、统筹资源；IX 数据面是厨房（数据面），专注做菜、绝不被前厅打扰。菜出来的速度比共用一个走廊时快几倍，同时客人还是被好好地隔离在各自桌子。

IX 的实现依赖两件硬件：**Intel VT-x 虚拟化扩展**（CPU 里内置的"虚拟机模式"，让一个进程可以像真正的操作系统一样直接控制页表和中断，不需要经过 Linux 转发）和**多队列 NIC**（每个核独占一条网卡队列，消除跨核同步）。

背景知识：操作系统中"特权级"分层——ring 0 是操作系统内核的最高权限（可以控制 CPU、内存、设备），ring 3 是普通程序只有有限权限（不能直接摸硬件）。Linux 完整跑在 VMX root ring 0（最高层控制者），数据面跑在 VMX non-root ring 0（独立的内核态空间，和 Linux 互不干扰），应用跑在 ring 3——三方完全隔离。

数据面本身是一个精简的 library OS（39K SLOC）：41% 来自 DPDK 网卡驱动，26% 来自 lwIP TCP/IP 栈，15% 来自 Dune 库，其余约 7K 行是新代码。它不支持交换（swap），全程只用 2MB 大页，内存分配全走 per-thread 内存池——一切设计都围绕"消除热路径的不确定性"展开。

## 为什么重要

不了解 IX，下面这些现象都说不清楚：

- 为什么传统 Linux 在数据中心里有时吞吐比不上用户态 bypass 栈（mTCP、DPDK），却又比它们安全
- 为什么 kernel bypass（DPDK/RDMA）能快但缺乏保护，而 IX 是最早把"快"和"安全"同时做到的系统
- 为什么"跑满一条 10 GbE 只需 3 个核而不是 8 个"——IX 的 run-to-completion + 自适应批处理是关键
- 为什么后来的 Shenango、Caladan、Shinjuku 都在 IX 的架构基础上做调度改进
- 为什么数据中心应用的尾延迟问题（p99/p999）在内核协议栈时代难以突破 100 µs，而 IX 将其压到 30-45 µs

## 核心要点

1. **控制面与数据面分离**：Linux 负责粗粒度资源分配（整核分配、大页内存、NIC 队列绑定）；数据面负责 TCP/IP 栈和应用逻辑，两者之间通过 VT-x 隔离，调度开销降到极低。类比：物流中心把"订单调度大厅"和"分拣传送带"物理隔开，传送带永远不等调度大厅开会。这种粗粒度分配避免了精细抢占带来的上下文切换，代价是核资源弹性较差（IX 后续工作 Shenango 专门解决这一点）。

2. **Run-to-completion + 有界自适应批处理**：每次轮询 NIC 队列时，把一批包从收包→TCP/IP 协议→应用逻辑→发包全部跑完再处理下一批（run to completion）。批大小上限固定（默认 B=64），只在拥塞时才批量处理，空闲时立即执行。类比：外卖员拿到一批订单，先把这批全部送完再接新单，而不是每接一单就出发一次——既减少出门次数（摊销 system call 开销），又不让早到的订单等太久。

3. **零拷贝 API + flow-consistent 同步消除**：IX 的 API 把收到的包的内存缓冲区直接只读映射给应用，没有 copy；NIC 用 RSS（网卡内置的流量哈希分发功能）把同一 TCP 连接的包总是哈希到同一核，每个弹性线程独享一组 TCP 流，无需锁或缓存同步。类比：快递柜每个格子贴上固定收件人标签，快递员只往对应格子放，收件人也只开自己的格子——不需要门卫协调，互不干扰。IX 的 API 设计遵循"可交换性规则"：两个线程的操作之间没有共享状态，可以并发执行而无需协调。ARP 表等少量共享结构用 RCU 方案保护——RCU 是一种"多人同时读无需等待、只有写的时候才需要协调"的并发控制机制，读路径完全无开销。

## 实践案例

### 案例 1：高吞吐低延迟键值缓存（memcached）

将 memcached 从 libevent/Linux 移植到 IX 的 libix 事件库（替换 event loop API，不动业务逻辑）。以下为 API 迁移的概念示意——IX 是需要在支持 VT-x 的服务器上专门编译部署的研究系统，下面的代码展示的是「改哪里」而非「直接粘贴运行」：

```c
// Linux / libevent 原版
event_set(&ev, fd, EV_READ, on_read_cb, NULL);
event_add(&ev, NULL);

// IX libix 等价调用（libix 提供相似事件模型，接口对齐 libevent）
ixev_ctx_init(&ctx);
ixev_set_handler(&ctx, IXEVIN, on_read_cb);
// 两者语义相同：注册"有数据可读时回调 on_read_cb"，业务逻辑不动
```

移植后：Linux 内核态 CPU 占比从 ∼75% 降至 <10%；在 500 µs p99 SLA 下，吞吐提升 2.8×（ETC 工作负载）到 3.6×（USR 小包工作负载）。关键原因：IX 消除了每个包的内核进出开销，应用逻辑直接处理 NIC 缓冲区里的数据，缓存局部性大幅提升。

值得注意的是，IX 使用了 6 个核而非 Linux 的 8 个核就达到更高吞吐——多余的核可以还给其他服务，体现了 IX 资源效率的优势。ETC 负载提升幅度（2.8×）低于 USR（3.6×），原因是 ETC 含更多写操作，memcached 内部的应用锁争用成为新的瓶颈——网络栈已不再是短板。

### 案例 2：微秒级延迟 RPC 服务（NetPIPE ping-pong 基准）

测量 IX 端到端延迟：两台 IX 服务器互 ping，64B 消息单向延迟 5.7 µs；对比 Linux 的 24 µs、mTCP 的更高延迟（mTCP 激进批处理以吞吐换延迟）：

```
系统        单向延迟   达到 5 Gbps 所需消息大小
Linux-Linux   24 µs         385 KB
mTCP-mTCP     更高           更大
IX-IX         5.7 µs         20 KB
```

IX 延迟低的核心：polling 代替中断（无唤醒等待），run-to-completion（无中间缓冲），零拷贝（不搬数据）。

横向参考：普通 HTTP/1.1 单次 localhost 请求约 50-200 µs（含内核 TCP 栈），gRPC 首次调用含序列化约 200-500 µs，而 IX 端到端 5.7 µs 约等于一次 L3 cache miss 的时间。这意味着对于 fan-out 型微服务（一次用户请求触发几百个下游 RPC），把每个 RPC 延迟从 50 µs 压到 5 µs，可将整体 p99 从百毫秒级降至个位数毫秒级。

### 案例 3：多核线速扩展（4×10 GbE）

IX 将单 socket 服务器通过 NIC bonding 扩展到 4×10 GbE（约 40 Gbps）：

```
配置          吞吐 (n=1, s=64B)    TCP 连接/秒
IX 10GbE       ~1.5M msg/s         ~0.95M conn/s
IX 4×10GbE     ~3.5M msg/s         ~3.8M conn/s（线性扩展）
Linux 10GbE    ~0.43M msg/s        受限于 CPU 同步
```

实现路径：RSS 把流哈希到不同 NIC 队列，每个弹性线程独占一条队列，4 核就跑 4 条 10 GbE，无跨核协议栈同步。相比之下 mTCP 需要全 8 核才能跑满 1×10 GbE。

IX 在 4×10 GbE 配置下使用单个处理器 socket（8 核/16 超线程）实现线速，证明了数据面架构在单 NUMA 节点内已完全消除了原有的扩展瓶颈。这对需要在单机部署多个服务副本、以低延迟转发内部 RPC 的场景特别有价值。

## 踩过的坑

1. **无 IOMMU 导致物理地址暴露**：原型把含物理地址的 DMA descriptor ring 直接映射进数据面，若数据面代码有漏洞则可绕过内存隔离。后续 Arrakis 用 SR-IOV + IOMMU 弥补了这个洞；IX 论文承认这是"对数据面代码的信任"。

2. **弹性线程不能阻塞**：弹性线程同时承担协议栈和应用逻辑，一旦应用代码阻塞（例如等文件 I/O），整批 TCP 流都会停止处理。正确做法是把耗时操作扔给 background thread，但这要求移植时仔细梳理应用的阻塞调用点。

3. **超 10 万并发连接时 L3 cache 溢出**：25 万并发连接时，每个报文 L3 cache miss 从 1.4 次暴增至 25 次，吞吐骤降约 53%。根因是 lwIP 的 TCP PCB 数据结构过大，working set 超出 L3 容量。论文指出需要紧凑化 PCB 数据结构，但原型未做。

4. **POSIX 不兼容，移植成本不可忽视**：IX 抛弃 POSIX socket API，应用必须用 libix 重写事件循环；libix 提供了 libevent 兼容层，但零拷贝路径需要额外手动适配，对已有大型服务的迁移成本较高。

## 适用 vs 不适用场景

**适用**：

- 数据中心内对 p99 尾延迟有严格 SLA 的服务（RPC、键值缓存、软件负载均衡）
- 需要同时满足高包率（>10 Mpps）和多租户隔离（不能信任应用代码）的场景
- 核数固定、负载相对稳定、可以独占整核给网络的场景
- 拥有多队列 NIC 且支持 Intel VT-x 的 x86-64 服务器硬件环境

**不适用**：

- 需要频繁、细粒度抢占的通用计算工作负载（IX 的弹性线程模型为网络 I/O 优化，不适合 CPU 密集）
- 应用代码有大量阻塞调用且难以重构（弹性线程模型要求非阻塞）
- 需要 IOMMU 级别硬件隔离、或 SR-IOV 多租户 NIC 虚拟化的生产环境（原型未支持）
- 负载高度突发、需要毫秒级核心动态增减的场景（应考虑 [[shenango-2019]] 等后续系统）

## 历史小故事（可跳过）

- **2012 年**：Adam Belay 在 OSDI 发表 Dune——用 VT-x 让用户态程序安全访问页表、ring 0 特权。IX 在两年后直接使用 Dune 作为数据面内核的基础。
- **2014 年 OSDI**：IX 与 Arrakis 同台发表。两者思路相近：都用硬件虚拟化把 I/O 数据面从 OS 剥离；差别在于 IX 保留完整 Linux 做控制面（兼容性优先），Arrakis 用 Barrelfish 做控制面（学术探索性更强）。
- **2014-2017 年**：DPDK 在工业界大规模铺开，与 IX 殊途同归：都绕过内核网络栈，但 DPDK 完全放弃保护，IX 维持三向隔离。
- **2019 年以后**：Shenango（NSDI 2019）、Caladan、Shinjuku 等系统在 IX 架构上引入核心资源动态调度，解决 IX 静态核分配下的资源利用率问题，是 IX 影响力延续的直接体现。
- **现状**：IX 的数据面设计思路已被各大云厂商内部系统吸收；DPDK + 用户态 TCP 栈成为业界主流，但完整的三向隔离保护方案（保留 Linux 控制面）仍是 IX 的独特贡献。

## 学到什么

1. **保护和性能不必二选一**：IX 打破了"kernel bypass 快但不安全"的常识——VT-x 三向隔离的开销约等于一次 L3 cache miss，在高包率下可以完全摊销。
2. **Run-to-completion + 自适应批处理**是微秒级延迟的关键配方：不是"批量越多越好"，而是"只在拥塞时才批，批大小有上界"，两个约束缺一不可。
3. **API 决定架构上限**：IX 放弃 POSIX、换用 zero-copy + 显式流控的 API，才使零拷贝在协议栈全程可行；兼容层（libix/libevent）让大多数应用可以低成本移植。
4. **每核独占资源是多核扩展的秘诀**：flow-consistent hashing + 每核独享队列/内存池，使数据面在常路径上完全无锁、无 coherence traffic——这个设计原则在后续 Shenango、Caladan 中被继承并推广。
5. **硬件特性即架构支点**：IX 的整个设计以"VT-x + 多队列 NIC"为前提；没有这两项硬件，同样的软件设计无法实现。这提示我们：突破性系统往往是在新硬件刚普及时找准了最佳利用姿势的人。
