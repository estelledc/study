## 一、为什么需要向量数据库？

### 日常类比：图书馆的进化

想象你有一个图书馆：

- **传统数据库**就像一个用卡片记录的目录系统。你想找书，必须输入精确的书名、作者或 ISBN。如果你只记得"那本封面是蓝色的、讲的是人工智能的书"，卡片目录就找不到。

- **向量数据库**则像一个"懂内容"的图书管理员。每本书都被转化为一组数字（向量），这组数字编码了书的内容。你想找相似的书，管理员会计算向量之间的距离，找到内容最接近的书——即使书名完全不同。

这就是向量数据库的核心价值：**它不只是"精确匹配"，而是"语义相似"**。

### 技术背景

现代 AI 系统（大语言模型、推荐系统、图像识别）都需要把文本、图片、音频等非结构化数据转化为高维向量（几十到几千维的数字数组）。这些向量捕捉了数据的"语义特征"。向量数据库就是专门用来高效存储和检索这些高维向量的系统。

传统数据库（如 MySQL）处理结构化数据很擅长，但面对高维向量的相似性搜索时，效率极低。

## 二、核心概念

### 1. 向量（Vector）

向量就是一个数字数组。比如一段文字经过 AI 模型处理后，可能被转化为一个 768 维的向量：

```
[0.12, -0.34, 0.56, 0.01, ..., -0.22]  # 共 768 个数字
```

每个数字代表文本在某个"语义维度"上的特征值。语义相近的文本，其向量在空间中距离更近。

### 2. 相似度搜索（Similarity Search）

向量数据库的核心功能是"找到和查询向量最相似的几个向量"。常用的距离度量方式：

- **欧氏距离**：两点之间的直线距离，距离越小越相似
- **余弦相似度**：衡量两个向量方向的夹角，夹角越小越相似

### 3. 索引（Index）

如果没有索引，每次搜索都要遍历所有向量（暴力搜索），数据量大时极慢。索引就像书的目录，帮你快速定位到可能相似的向量区域。

论文将索引技术分为两大类：

| 类别 | 精确搜索 (NNS) | 近似搜索 (ANNS) |
|------|---------------|----------------|
| 树形方法 | KD-Tree, Ball-Tree, R-Tree, M-Tree | — |
| 哈希方法 | — | 局部敏感哈希 (LSH) |
| 图方法 | — | HNSW (Hierarchical Navigable Small World) |
| 量化方法 | — | 标量/乘积量化 |

### 4. 存储技术

论文从四个维度分析向量数据库的存储策略：

- **分片（Sharding）**：把数据分散到多台机器上。包括范围分片、哈希分片、地理分片
- **分区（Partitioning）**：在单台机器内把数据分组。包括范围分区、列表分区、K-Means 分区
- **缓存（Caching）**：把高频访问的向量暂存在内存中。包括 FIFO、LRU、MRU、LFU 等淘汰策略
- **复制（Replication）**：把数据复制到多个节点以提高可用性。包括主从复制、多主复制、无主复制

## 三、搜索算法详解

### 精确最近邻搜索 (NNS)

暴力搜索：遍历所有向量，计算距离，取最近的。时间复杂度 O(n)，保证结果精确，但数据量大时太慢。

树形方法：用空间划分缩小搜索范围。

- **KD-Tree**：在低维空间表现好，但在高维空间中会遇到"维度灾难"——所有点都差不多远，树的分隔效果失效
- **Ball-Tree**：用超球体划分空间，比 KD-Tree 更适合高维数据，但构建成本更高
- **R-Tree**：用最小包围矩形组织空间数据，适合地理空间查询
- **M-Tree**：支持动态插入和删除，但维护开销大

### 近似最近邻搜索 (ANNS)

在实际应用中，我们通常不需要 100% 精确的结果——只要"足够接近"就行。ANNS 通过牺牲少量精度，换取巨大的速度提升。

#### 哈希方法：局部敏感哈希（LSH）

核心思想：把相似的向量映射到同一个"桶"里。传统哈希函数让相同的输入产生相同的输出，而 LSH 让"相似的输入"以高概率产生"相似的输出"。

公式表达：

```
Pr[h(p) = h(q)] = f(d(p, q))
```

意思是：p 和 q 被映射到同一哈希值的概率，是它们距离 d(p,q) 的函数。距离越小，碰撞概率越高。

#### 图方法：HNSW（层级导航小世界图）

这是目前最流行的向量搜索算法，被 Pinecone、Milvus 等主流数据库采用。

把每个向量看作图中的一个节点，节点之间用边连接。搜索时从顶层开始，沿着边快速"导航"到最近的区域，逐层细化，直到找到最接近的向量。

优势：搜索速度快，召回率高，支持动态更新。

#### 量化方法

把高维向量压缩为更短的编码，减少存储空间和计算量。

- **标量量化（SQ）**：把每个浮点数压缩为更少的比特
- **乘积量化（PQ）**：把向量分成多个子空间，每个子空间用码本近似

## 四、主流向量数据库对比

论文测试了多个开源向量数据库的性能：

| 数据库 | 类型 | 特点 |
|--------|------|------|
| FAISS | 库 | Facebook 开发，高性能索引库，适合研究场景 |
| Milvus | 原生系统 | 分布式架构，功能丰富，社区活跃 |
| Pinecone | 云服务 | 全托管服务，开箱即用 |
| Weaviate | 原生系统 | 支持混合搜索（向量 + 关键词） |
| Vespa | 扩展系统 | 在现有数据库中增加向量能力 |

关键发现：没有"最好的"向量数据库，只有"最适合"的。选择取决于数据规模、延迟要求、预算和部署方式。

## 五、代码示例

### 示例 1：用 FAISS 构建向量索引

FAISS 是 Facebook 开源的高效向量搜索库，支持多种索引类型。

```python
import faiss
import numpy as np

# 1. 准备数据：假设我们有 1000 个 128 维的向量
dimension = 128
num_vectors = 1000
data = np.random.random((num_vectors, dimension)).astype('float32')

# 2. 创建一个 FAISS 索引
# Flat 索引 = 暴力搜索，保证精确但慢
index = faiss.IndexFlatL2(dimension)

# 3. 把向量加入索引
index.add(data)
print(f"索引中有 {index.ntotal} 个向量")

# 4. 创建一个查询向量
query = np.random.random((1, dimension)).astype('float32')

# 5. 搜索最接近的 5 个向量
k = 5
distances, indices = index.search(query, k)

print(f"查询结果:")
for i in range(k):
    print(f"  第 {i+1} 近: 向量 ID={indices[0][i]}, 距离={distances[0][i]:.4f}")
```

**说明**：
- `IndexFlatL2` 使用欧氏距离做暴力搜索
- `search()` 返回两个数组：距离和对应的向量 ID
- 数据量小时够用，但数据量大了需要换更高级的索引

### 示例 2：用 HNSW 索引加速搜索

HNSW 是近似搜索，速度远快于暴力搜索。

```python
import faiss
import numpy as np

# 1. 准备更大的数据集
dimension = 128
num_vectors = 100000
data = np.random.random((num_vectors, dimension)).astype('float32')

# 2. 创建 HNSW 索引（乘积量化 + 图索引）
# 先用 PQ 把向量压缩为 64 维（子空间数 8，每个子空间码本大小 256）
quantizer = faiss.IndexFlatL2(dimension)
index = faiss.IndexPQ(quantizer, dimension, 8, 8)  # m=8 子空间, log2(M)=8

# 3. 训练量化器（学习向量分布）
index.train(data)

# 4. 加入向量
index.add(data)

# 5. 查询
query = np.random.random((1, dimension)).astype('float32')
k = 10
distances, indices = index.search(query, k)

print(f"HNSW 搜索 Top {k} 结果:")
for i in range(k):
    print(f"  第 {i+1} 近: 向量 ID={indices[0][i]}, 距离={distances[0][i]:.4f}")
```

**说明**：
- `IndexPQ` 使用乘积量化压缩向量，大幅节省内存
- 训练阶段让索引学习数据的分布规律
- 搜索时用的是压缩后的编码，速度远超暴力搜索
- 代价是搜索结果有少量误差（近似）

## 六、向量数据库与 LLM 的协同

论文重点讨论了向量数据库和大语言模型（LLM）的双向赋能：

### VDB 为 LLM 解决问题

LLM 有四个痛点，VDB 都能帮忙：

1. **幻觉问题**：LLM 可能编造答案。通过 VDB 检索真实数据作为参考，可以降低幻觉
2. **知识过时**：LLM 训练数据有截止日期。VDB 可以存储最新知识，实现动态更新
3. **成本高**：重新训练 LLM 很贵。用 VDB 外挂知识库更高效
4. **记忆缺失**：LLM 没有长期记忆。VDB 存储用户历史，实现个性化

### LLM 为 VDB 解决问题

- 用 LLM 理解查询意图，改进搜索相关性
- 用 LLM 生成更高质量的向量表示
- 用 LLM 自动优化索引参数

### 典型架构：RAG（检索增强生成）

```
用户问题 → LLM 编码为向量 → VDB 检索相似文档 → 文档喂给 LLM → LLM 生成回答
```

这就是目前最流行的 AI 应用架构之一。

## 七、未来挑战

论文总结了向量数据库面临的五个核心挑战：

1. **高维向量索引和搜索**：维度越高，"距离"的区分度越低（维度灾难），需要新的索引策略
2. **异构向量数据类型**：文本、图像、音频的向量混存，需要统一的搜索框架
3. **分布式并行处理**：海量数据下的跨节点高效搜索仍是难题
4. **新兴场景集成**：自动驾驶、元宇宙等新场景对 VDB 提出独特需求
5. **数据安全与隐私保护**：向量本身可能泄露训练数据，需要加密和差分隐私技术

## 八、学习小结

这篇 survey 的核心框架可以总结为：

```
向量数据库 = 存储 × 搜索

存储层面：分片、分区、缓存、复制
搜索层面：精确搜索（树形）→ 近似搜索（哈希/图/量化）

应用场景：RAG、推荐系统、图像识别、语义搜索
```

作为学习者，建议的进阶路径：

1. 先用 FAISS 的 `IndexFlatL2` 体验暴力搜索
2. 换成 `IndexHNSWFlat` 感受近似搜索的速度提升
3. 尝试 Milvus 或 Weaviate，了解生产级向量数据库的完整功能
4. 实践中做一个 RAG 应用，体验 VDB + LLM 的组合威力
