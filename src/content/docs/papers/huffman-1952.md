# Huffman 编码（1952）：作业题写出来的 70 年标准

## 0. 一句话

> 给每个字符分配一个二进制码，频率越高码越短，并保证没有任何码字是另一个的前缀——平均码长就能逼近信息论的熵下界，并且这样构造出来的码是**最优**的。

---

## 1. 故事：从期末考试到行业标准

1951 年秋，MIT，一门信息论课。教授 Robert Fano 给学生出了选择题：

- 选项 A：参加期末考试
- 选项 B：写一篇论文，**证明 Shannon-Fano 编码是最优的**，或给出一个更好的编码

David A. Huffman 选了 B，理由很现实：他不想考试。

一开始他做不出来，几乎要放弃回去复习。在放弃前的最后一刻，他想到了**自底向上**而不是 Fano 的**自顶向下**——把两个最小频率合并成一个新节点，递归构造。

他不仅找到了一个比 Shannon-Fano 严格更好的方法，还**证明**了它在整数码长前提下是最优的。

这篇论文 1952 年发表，4 页正文，10 个公式。70 年后，你电脑里**每一个 zip 文件、每一张 PNG、每一段 MP3、每一次 HTTP gzip 响应**——都还在用它。

这是一个写代码的人需要消化的事实：**简单算法 + 正确证明，可以撑 70 年**。

## 2. 背景：Shannon 之前与 Shannon 之后

### 2.1 编码问题：为什么需要"压缩"

数据本质是符号序列。最朴素的编码方式是**定长编码**：n 个符号每个用 ⌈log₂(n)⌉ 位。

举例：26 个英文字母，每个用 5 位（实际 ASCII 用 8 位是为对齐字节）。

定长编码忽略了一个关键事实：**符号出现频率不一样**。在英文中，'e' 频率约 12%，'z' 约 0.07%。如果 'e' 和 'z' 都给 8 位，大量字节都"浪费"在传递高频字符上。

变长编码的直觉：高频字符用短码，低频字符用长码。但变长编码有一个**致命问题**——怎么解码？

举例：A=0, B=01, C=11。看到 "011"，到底是 "AB1" 还是 "0BC" 还是 "AC"？歧义。

解决办法：**前缀码（prefix code）**——任何码字都不能是另一个码字的前缀。这样从左到右扫描比特流，就能唯一切分出码字边界。

### 2.2 信息熵（Shannon, 1948）

**Definition 1（信息熵 / Shannon Entropy）**

对一个离散随机变量 X，取值集合 {x₁, ..., xₙ}，对应概率 {p₁, ..., pₙ}，X 的信息熵定义为：

```
H(X) = -Σᵢ pᵢ · log₂(pᵢ)
```

单位是 bit/symbol。它表示编码该变量所需的**理论最小平均比特数**。

直觉：log₂(1/p) 是"罕见程度"——一个 1/8 概率的事件，需要 3 bit 来描述（把"8 选 1"压到"确定"）。期望值 H(X) 就是平均不确定性。

举几个边界例子帮助消化：

- 均匀分布（n 个等概率符号）：H = log₂(n)，定长编码就是最优的
- 偏斜极端（一个符号概率 1，其他 0）：H = 0，根本不需要编码
- 二值偏斜（p=0.99, 1-p=0.01）：H ≈ 0.08 bit/symbol，但 Huffman 至少要给两个符号各 1 bit

### 2.3 Shannon-Fano 编码（1948-1949）

Shannon-Fano 是 Huffman 之前的最优尝试。算法：

1. 按频率降序排列符号
2. 把符号分成两组，使两组频率和**尽量接近**
3. 上半组分配前缀 0，下半组分配前缀 1
4. 对每组递归

**问题**：自顶向下的"尽量接近"不是真的最优。某些情况下分组会失败，平均码长比真正最优长一点。Shannon 自己在 1948 原始论文中就承认这种构造方式"在某些情况下不够紧"。

但他没给出更好的算法——这正是 Huffman 1951 年作业的起点。

## 3. Huffman 算法

### 3.1 关键定义

**Definition 2（前缀码 / Prefix Code）**

一组码字称为前缀码，当且仅当其中**任何一个码字都不是另一个码字的前缀**。前缀码可以唯一解码，无需分隔符。

例：{0, 10, 11} 是前缀码（0 不是 10 或 11 的前缀，10 不是 11 的前缀）。
反例：{0, 01, 11} 不是（0 是 01 的前缀，看到 "0..." 不知道在等啥）。

**Definition 3（最优性 / Minimum-Redundancy）**

对一个频率分布 {pᵢ}，定义平均码长：

```
L = Σᵢ pᵢ · lᵢ
```

其中 lᵢ 是符号 i 的码长。在所有可能的前缀码中，使 L 最小的码称为**最优前缀码**——Huffman 在原论文里称为 **minimum-redundancy code**。

### 3.2 算法（贪心，自底向上）

输入：n 个符号及其频率 (s₁, p₁), ..., (sₙ, pₙ)
输出：n 个二进制码字

1. 把所有符号当作单节点的"树"，节点权重等于频率
2. 取出权重**最小的两个**节点，合并为一个新节点，新节点权重 = 两子节点权重之和
3. 把新节点放回集合
4. 重复 2-3，直到只剩一个节点（即根）
5. 从根开始遍历：左走标 0、右走标 1，到叶节点的路径就是该叶对应符号的码字

整个算法只有合并与堆操作，C 实现可以压在 50 行以内。

### 3.3 算法正确性的几个定理

**Theorem 1（Huffman 算法的最优性）**

Huffman 算法构造出的前缀码是最优前缀码——即不存在任何其他前缀码的平均码长 L 严格更小。

**证明思路**（归纳）：

（1）**引理**：在最优前缀码中，频率最低的两个符号必然是兄弟节点（在码树中处在最深的同一层）。

证明：假设不是，可以通过交换让它们成为兄弟，平均码长不增（可能减少）——所以原来不是最优的，矛盾。

（2）**归纳**：n=2 时显然成立（两个码字 {0, 1}）。假设 n-1 时成立。对 n 个符号：

- 把频率最小的两个 a, b 合并为超符号 ab，频率 = pₐ + p_b
- 用 n-1 时的最优算法处理（含超符号）
- 还原 a, b 时：把超符号 ab 的码字 c 分别延长为 c0 和 c1

由引理，原最优码中 a, b 也必为兄弟，所以这种构造**不会比原最优差**。归纳得证。∎

**Theorem 2（Kraft 不等式 / Kraft's Inequality, 1949）**

对任何前缀码，码长 {l₁, ..., lₙ} 必满足：

```
Σᵢ 2^(-lᵢ) ≤ 1
```

反过来：给定满足 Kraft 不等式的码长集合，**存在**一个前缀码以这些数为码长。

直觉：在二叉树上，每个深度 lᵢ 的叶子节点"占据"了 2^(-lᵢ) 比例的"码空间"（满二叉树）。所有叶子占据的不能超过 1。

**Theorem 3（Shannon 第一定理 / 无噪声编码定理 / 1948）**

对任何离散无记忆源 X，存在唯一可解码的编码，使得平均码长 L 满足：

```
H(X) ≤ L < H(X) + 1
```

下界 H(X) 来自信息熵的定义；上界 H(X) + 1 是因为**码长必须是整数**——这就是 Huffman 编码的天花板。

**Theorem 4（Huffman 的具体上界）**

Huffman 编码的平均码长 L 满足：

```
H(X) ≤ L_huffman ≤ H(X) + 1
```

且当所有 pᵢ 都是 2 的负整数次幂（即 pᵢ = 2^(-kᵢ)）时，L = H(X)，达到熵下界。

**Theorem 5（块编码降低冗余 / Block Coding）**

把 n 个符号"打包"成长度为 k 的"块"再编码。新的"超符号"有 nᵏ 个，频率分布是原分布的 k 次卷积。对超符号做 Huffman：

```
H(X) ≤ L_block / k < H(X) + 1/k
```

随着 k → ∞，L_block / k → H(X)。

**代价**：码表大小 O(nᵏ) 爆炸。256 个字节符号、k=2，码表已经 65536 项；k=3 就 1600 万。这是一条**理论上漂亮、工程上窒息**的路。

## 4. 图解：构造一棵 Huffman 树

考虑 6 个符号 A, B, C, D, E, F，频率分别为 5, 9, 12, 13, 16, 45（总和 100）。

![Huffman 树构造](/papers/huffman-1952/01-prefix-tree.webp)

**步骤详解**：

| 步骤 | 操作 | 当前森林（按权重排序） |
|------|------|------------------------|
| 0    | 初始化         | A:5, B:9, C:12, D:13, E:16, F:45 |
| 1    | 合并 A+B → 14  | C:12, D:13, 14, E:16, F:45 |
| 2    | 合并 C+D → 25  | 14, E:16, 25, F:45 |
| 3    | 合并 14+E → 30 | 25, 30, F:45 |
| 4    | 合并 25+30 → 55| 55, F:45 |
| 5    | 合并 F+55 → 100| 100（根） |

**最终码表**：

| 符号 | 频率 | 码字 | 码长 |
|------|------|------|------|
| F    | 45   | 0    | 1    |
| C    | 12   | 100  | 3    |
| D    | 13   | 101  | 3    |
| E    | 16   | 111  | 3    |
| A    | 5    | 1100 | 4    |
| B    | 9    | 1101 | 4    |

**平均码长**：

```
L = 0.45·1 + 0.12·3 + 0.13·3 + 0.16·3 + 0.05·4 + 0.09·4
  = 0.45 + 0.36 + 0.39 + 0.48 + 0.20 + 0.36
  = 2.24 bit/symbol
```

**信息熵**：

```
H = -[0.45·log₂(0.45) + 0.12·log₂(0.12) + 0.13·log₂(0.13)
    + 0.16·log₂(0.16) + 0.05·log₂(0.05) + 0.09·log₂(0.09)]
  ≈ 2.20 bit/symbol
```

差距：L - H ≈ 0.04 bit/symbol，约 1.8% 冗余。这是**整数 bit 的代价**。

注意：Huffman 树**不唯一**——同频率符号合并顺序不同会得到不同树，但平均码长 L 相同。

## 5. 实现：从理论到字节流

### 5.1 数据结构与复杂度

朴素实现：每次扫描数组找最小两个 → O(n²)

优化：用**最小堆（priority queue / min-heap）**

- 建堆：O(n)
- 每次合并：取两次最小（O(log n)）+ 插入一次（O(log n)）
- 共 n-1 次合并：**O(n log n)**

更进一步：如果频率已排序，用**两个队列**（一个原始队列、一个合并队列）轮流取最小，可以做到 **O(n)**。这是 Van Leeuwen 1976 的观察。

### 5.2 Canonical Huffman Code（规范化 Huffman）

直接传 Huffman 树代价高（每个非叶节点 2 个指针 + 叶节点的符号 ID）。工程上几乎都用 **Canonical Huffman**：

- **只传每个符号的码长**（一个 byte 即可，对 256 字符的字母表共 256 字节）
- 解码端按规则重建码字：长度短的在前，相同长度按符号字典序

zlib / DEFLATE / Brotli 全都用这个方案。这一步把"传一棵树"压成了"传一个长度数组"。

### 5.3 GitHub 实现参考

**zlib（DEFLATE 标准实现）**

Huffman 编码核心在 `trees.c`，包含静态/动态两种模式的码表构造与位流输出：

https://github.com/madler/zlib/blob/51b7f2abdade71cd9bb0e7a373ef2610ec6f9daf/trees.c

zlib 的 DEFLATE 流头部有 2 bit 标志：00=stored、01=static、10=dynamic、11=reserved。
- **静态 Huffman** 用 RFC 1951 写死的码表，不传码表，对小文件友好（节省码表开销）
- **动态 Huffman** 自带码表，再用第二棵 Huffman 树压缩"码长序列"——是 Huffman 套 Huffman

**facebook/zstd（现代 Huffman + ANS 混合）**

zstd 把 Huffman 用于 literals 编码，FSE（tANS 的一种）用于 sequences 编码：

https://github.com/facebook/zstd/blob/794ea1b0afca0f020f4e57b6732332231fb23c70/lib/common/huf.h

zstd 的 Huffman 实现叫 `huf`，用了 weighted decoding table 和 unrolled decoding loop 做性能优化，比 zlib 快约 4-5 倍。

**google/snappy（极速压缩，反例）**

snappy **不用** Huffman——只用 LZ77 + 简单字节标签。设计目标是 throughput，不是 ratio：

https://github.com/google/snappy/blob/27f34a580be4a3becf5f8c0cba13433f53c21337/snappy.cc

snappy 牺牲 5-10% 压缩率换 3 倍速度。这是一个反例：**Huffman 不是免费的**——表查询、位移、对齐都有开销。

### 5.4 解码（agent 视角的难点）

编码端简单（查表）；解码端要**根据流式位输入找到匹配码字**。两种主流方案：

- **位逐个读**：从根开始走树，直到叶子。每符号 O(maxlen) 次比较，慢
- **查表加速**：建一张 2^k 项的表（k 是最大码长），一次查表得到符号 + 已消费 bit。zlib 用 k=9 的双层查表

写解码器时最容易踩的坑是 **bit endianness**：DEFLATE 的码字是 LSB-first，写错一个方向整流就读不出来。

## 6. 怀疑：Huffman 在 2026 年还配做"基础"吗？

### 怀疑 1：整数 bit 是 Huffman 的天花板

Huffman 给每个符号分配整数 bit 的码长。但信息论的下界 H(X) 可以是分数。

举例：两个符号 A, B 的概率分别为 0.9 和 0.1。

```
H(X) = -[0.9·log₂(0.9) + 0.1·log₂(0.1)] ≈ 0.469 bit/symbol
```

但 Huffman 必须给 A 和 B 各 1 bit，即 L = 1 bit/symbol。**冗余 113%**！

解决办法：

- **Block Huffman**：把 k 个符号合成一个超符号编码。k=4 时 L/k ≈ 0.67，k=8 时 ≈ 0.55。但码表大小 O(2^k) 爆炸
- **Arithmetic Coding（算术编码，1976-1979）**：把整个消息编码成 [0, 1) 区间内一个数，每个符号缩小区间，不需要整数 bit 边界。理论上接近 H(X)，无 +1 损失
- **ANS / Asymmetric Numeral Systems（Duda, 2009）**：算术编码的高速变体，速度接近 Huffman、压缩率接近算术编码

**怀疑点**：Huffman 的"+1 上界"在低熵分布下是灾难。Huffman 之所以能撑到 2026，靠的是**速度**——表查询每符号约 1ns。但 ANS 把速度差距追平了，Huffman 的"速度护城河"正在消失。

### 怀疑 2：现代压缩里 Huffman 不是核心

打开任何现代压缩算法（gzip / zstd / Brotli / xz / LZ4），Huffman 都只是流水线**最后一环**。真正的核心是：

1. **LZ77/LZ78 字典匹配**：找到重复字符串，用 (offset, length) 编码——这一步压缩了 60-80% 的体积
2. **上下文建模（context modeling）**：根据前 N 个字符预测下一个字符的概率分布（Brotli 用了 6 阶上下文）
3. **熵编码（Huffman / ANS / Arithmetic）**：把上下文模型预测出的概率分布编码成比特流

Huffman 在第 3 步——**且可被替换**。zstd 的 sequences（offset+length 对）就是 ANS 编码的；只有 literals 还用 Huffman。Brotli 也只在 literals 用 Huffman。

**怀疑点**：把 Huffman 从压缩教学的"中心"摆到"配角"，可能更接近现代工程现实。教科书一上来就讲 Huffman，可能让初学者误以为压缩 = 熵编码。事实上压缩 = LZ77 + 上下文 + 熵编码，前两步贡献了 80% 的压缩比。这个调整需要 5-10 年的教材重写。

### 怀疑 3：DEFLATE 30 年没换，何时被 ANS 取代？

DEFLATE = LZ77 + Huffman。RFC 1951 定稿于 1996 年。从那以后：

- zip / gzip / PNG / HTTP gzip / WebSocket compression / Git pack 全都还在用 DEFLATE
- 2015 年 Brotli（LZ77 + 二阶上下文 + Huffman），CR 比 gzip 高 20%，但只在 HTTP/CSS/JS 场景普及
- 2016 年 zstd（LZ77 + ANS），Facebook/Linux/CloudFlare 都用，**但未取代 gzip 在 HTTP 中的地位**

为什么换不掉？

- **生态锁定**：浏览器、服务器、中间件、CDN 全都内建 gzip。换 ANS 需要协议升级
- **够用就行**：HTTP gzip 平均压缩 70%，多 5% 性价比不高
- **风险厌恶**：换协议是 breaking change，没人想做第一个吃螃蟹的

预测：**HTTP/3 之后**才有可能强推。但 HTTP/3 已经定稿（QUIC + HPACK），gzip 还在。下一个机会窗口是 HTTP/4，时间未知——可能再等 10 年。

**怀疑点**：Huffman 的"事实标准"地位是路径依赖，不是技术优越。生态比算法重要——这对学算法的人是反直觉的。

### 怀疑 4：静态 vs 动态 Huffman 的码表开销

教科书讲 Huffman 默认是"静态 Huffman"——预先扫一遍数据统计频率，构造一棵树，然后编码。这意味着：

- **两遍扫描**：第一遍统计、第二遍编码。流式数据（streaming）做不到
- **必须传码表**：解码端需要码表才能解码。一棵 256 符号的 Huffman 树压缩成 canonical 形式后约 100-300 字节

对小文件这是**致命的**。100 字节的输入，码表就 100+ 字节，压缩后比原来还大。

DEFLATE 的工程妥协（来自 RFC 1951）：

- **静态 Huffman 模式**：用 RFC 1951 写死的码表，不传码表（对小文件友好）
- **动态 Huffman 模式**：传码表，再用第二棵 Huffman 树压缩码表的码长序列（对大文件友好）
- **stored 模式**：不压缩，直接存（对随机数据/已压缩数据友好——硬压会变大）

工程师用 2 bit 标志位选模式，这个 hack 撑了 30 年。

**怀疑点**：把 Huffman 当作一个"算法"教是不够的。它是一个**协议家族**，工程实现充满 hack 和 fallback。仅看 1952 原论文会错过这层复杂度，导致写出来的实现"在 benchmark 数据集上漂亮、在真实小文件上灾难"。

### 怀疑 5：自适应 Huffman 是不是被低估了？

自适应 Huffman（FGK 1973 / Vitter 1987 algorithm V）允许编码端和解码端**同步更新树**——每编码一个符号，两边都按新频率调整树。优点：

- **单遍扫描**，无需统计阶段
- **不传码表**，省开销
- **流式友好**，适合实时压缩

听起来完美？为什么 zlib 不用？

推测原因：

- 实现复杂度高（树的 sliding/swap 操作很容易写错）
- 速度比静态 Huffman 慢 2-3 倍（每符号都要更新树结构）
- 在 LZ77 已经做了大部分压缩的前提下，自适应 Huffman 的边际收益不明显

**怀疑点**：自适应 Huffman 在教科书里只占半页，但理论上是更优雅的设计。可能值得专门读 Vitter 1987 algorithm V 的原论文。也许它真正的舞台不是文件压缩，而是"流式协议头部压缩"——但那块现在也被 HPACK / QPACK 占了（用静态字典 + 静态 Huffman）。

## 7. 与其他熵编码的对比

| 方法 | 论文年份 | 平均码长 | 速度 | 实际应用 |
|------|----------|----------|------|----------|
| Shannon-Fano | 1948 | H + 2 上界 | 中 | 已淘汰 |
| Huffman | 1952 | H + 1 上界 | 快（表查询） | DEFLATE / JPEG / MP3 / HPACK |
| Arithmetic | 1976-79 | ≈ H | 慢 | JPEG2000 / H.264 CABAC |
| Range Coding | 1979 | ≈ H | 中 | RAR / LZMA |
| ANS (rANS / tANS) | 2009 | ≈ H | 快（追平 Huffman） | zstd / LZFSE / JPEG XL |

**结论**：ANS 在压缩率和速度上**同时**打败 Huffman。但生态惯性让 Huffman 在 HTTP / PNG / ZIP 等"老协议"中继续存活。这是技术演进的常见格局：新方法不仅要更好，还要好到值得**重写所有依赖**——这是个高得离谱的门槛。

## 8. 我学到了什么

### 8.1 算法层面

- Huffman 算法是**贪心 + 二叉树合并**，O(n log n)，C 实现 < 50 行
- 关键证明技巧：**交换论证（exchange argument）**——假设最优解不满足某性质，构造一个不变差的解使其满足，矛盾
- Huffman 树**不唯一**，但平均码长 L 唯一

### 8.2 理论层面

- 信息熵 H(X) = 编码该变量所需的**理论最小平均比特数**
- Huffman 在整数码长前提下达到 H + 1 上界，**最优**
- 当所有概率都是 2 的负整数次幂时，L = H，达到熵下界
- 块编码可以让 L/k → H，但代价是码表 O(nᵏ) 爆炸

### 8.3 工程层面

- 朴素 Huffman 不实用——必须配 Canonical Huffman + 多模式选择（static/dynamic/stored）
- 现代压缩 = LZ77 + 上下文建模 + 熵编码，Huffman 只在最后一步，**可替换**
- ANS 在 2009 年后逐步蚕食 Huffman 的地盘，但生态锁定让 DEFLATE 续命

### 8.4 论文写作启发

Huffman 这篇 4 页论文有几个写法值得抄：

1. **问题定义清晰**：第一节就把"什么是最优前缀码"用一行公式定义清楚，没废话
2. **构造性证明**：先给算法、再证明它的正确性（自底向上归纳），逻辑闭环
3. **承认前人**：明确 cite Shannon-Fano 是 baseline，自己做的是"小步改进"
4. **不掩盖缺陷**：明确指出 Theorem 4 的 +1 上界，不假装完美

**反例警告**：现代论文很多反着来——把简单事情写复杂、不做对照实验、隐藏失败模式。Huffman 这篇是反向参考——能用 4 页讲清楚的东西就不要写成 14 页。

### 8.5 给自己的提醒

- 信息熵 H 不是玄学，就是"平均不确定性"。多算几个例子（均匀 / 偏斜 / 极端）就有感觉
- 学 Huffman 的下一步**不是**学算术编码，而是学 LZ77——因为现代压缩主战场不在熵编码
- 学 Huffman 的更深一步**才**是 ANS / FSE，因为这是 Huffman 的现代继承者
- 写解码器先在 toy 数据上跑通，再上大数据——bit endianness 错一个方向你 debug 一周

## 10. 一句话总结（给未来的自己）

> Huffman 不是"压缩"，是"基于已知概率的熵编码"。压缩的难点 90% 在**怎么得到概率**（LZ77 + 上下文建模），剩下 10% 才是熵编码本身。Huffman 是入门门槛——但要真正理解现代压缩，需要把它放到 "LZ77 → 上下文建模 → 熵编码" 的整条流水线里看，并且知道它的继承者 ANS 已经在敲门。
