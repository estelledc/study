## 是什么

d-HNSW 想解决一个很现实的问题：  

向量数据库越来越大，一台机器装不下所有向量和索引，但业务又不想把查询延迟拉到“秒级”。  

论文给的答案是把“算”和“存”分开：

- 计算侧（Compute Pool）处理查询逻辑；  
- 存储侧（Memory Pool）保存索引和原始向量；  
- 通过 RDMA 做远程读写，减少 CPU 干预。  

你可以把它理解成一个“前台柜员 + 仓库”的系统，柜员拿着订单快速调库，而不是仓库自己跑全部业务逻辑。  

## 为什么重要

传统向量检索（尤其 HNSW）在单机上很快，但扩展到跨机房/跨卡时，最大瓶颈往往不是计算，而是“网络来回次数”。  

对于 HNSW 的 greedy 搜索，如果每跳都发 RDMA 请求，会出现大量小包往返。d-HNSW 的核心不是换算法，而是优化“请求次数”。  

这类思路很适合当下 LLM/RAG 语境：embedding 规模不断涨，检索频率高，延迟每毫秒都值钱。  

## 核心要点

论文给了三大机制，分别对应不同瓶颈：

### 1）Representative Index Caching（代表索引缓存）

先用一份轻量 meta-HNSW（抽样代表向量）作为“路标树”。  

- 作用：快速定位请求向量最可能落在哪些子图；  
- 好处：每次查询只拉取少量子图，不需要全量遍历。  

可类比：不是查图书馆整排书，而是先找索引卡片位置，再走到几格书架。  

### 2）RDMA 友好布局

把每个子 HNSW 块尽量连续放内存，并预留溢出区来放新插入向量，避免每次都追着散落地址。  

- 减少 RDMA round trip 次数；  
- 插入扩容时更容易维持块内连续访问；  
- 查找路径更易预读。  

### 3）Query-aware Batch Loading（查询感知批量加载）

查询通常会成批到来。论文的做法是：同一批次里“同样需要的子图”只取一次。  

- 去重 partition 加载；  
- 复用最近加载块；  
- 在保持 recall 的前提下降低带宽和延迟。  

## 实践案例

### 案例 1：延迟为何能降到 117×

在 SIFT1M top-10 的实验里，d-HNSW 相对 naive d-HNSW 提示了数量级差异。  

主要不是“算得更复杂”，而是减少了网络往返。  

### 案例 2：LD 模式的批处理价值

查询批次一大，重复子图越多，收益越明显。query-aware 合并会把同一子图共享一次后服务多个 query。  

对于多租户 embedding 检索，这个模式尤其实用。  

### 案例 3：插入场景下的连续写入优势

普通方法插入向量容易碎片化，后续查询拉取更多分散地址。  

布局里给每对子图共享溢出区让追加插入更“顺手”。  

### 案例 4：与现有基线的对比逻辑

论文把 d-HNSW 与 naive d-HNSW、无 doorbell 变体对比。  

结果上看：核心差别在“网络访存路径”，而非单独某个距离计算优化。  

## 踩过的坑

1. **把 HNSW 的跳表特性忽略了**：Greedy 路径不可预知，盲读会放大延迟。  
2. **以为单次查询就是单次请求**：批处理系统里应按 batch 拆解重复数据。  
3. **没为新插入留空间**：后续扩容时碎片会吞掉绝大多数收益。  
4. **RDMA 把 NIC 当无限资源**：doorbell size 不能无限放大，要平衡并发和尾延迟。  
5. **以为 recall 高就自然快**：d-HNSW 在保持 recall 的同时优化网络，速度提升才有意义。  
6. **超参数与 batch size 绑死**：EF/Search 或 sub-HNSW 选择要按场景调。  
7. **只看 SIFT，不看 GIST**：不同数据分布会放大（或掩盖）收益。  

## 适用和不适用

### 适用场景

- 向量规模接近单机内存上限；  
- 查询并发高，延迟预算紧（RAG/推荐/检索增强生成）；  
- 查询分布呈批次到达、可做队列合并；  
- 希望在不牺牲 recall 的前提下显著降内网流量。  

### 不适用场景

- 向量规模很小，单机 HNSW 即可；  
- 写入极低、索引变更非常频繁（缓存维护成本不划算）；  
- 基础设施没有 RDMA / 低延迟互联。  

## 历史小故事

论文把 d-HNSW 放在一个更大的趋势里：  

- HNSW 已经成熟，但传统假设仍是“内存与计算同机”；  
- Cloud/AI 架构开始把 compute 与 memory 分离；  
- 之后的系统像 CXL-based index、KV-store in-memory 分离都在补同一个问题：跨组件通信优化。  

这并不意味着“全盘替换”，而是把 HNSW 的强项保留，把瓶颈移到网络层解决。  

## 学到什么

1. 架构变了，算法常数项也变了：单机“快”的算法不一定是分布式快。  
2. RDMA 的收益来自减少往返次数，而不是单次传输更快。  
3. 工程上最值钱的不是花哨结构，而是可复用的缓存与排班策略。  
4. 任何高性能系统里，batch 的设计常常比单条最优更关键。
