# LLM 写代码——一篇综述的零基础学习笔记

> 论文：A Survey on Large Language Models for Code Generation
> 作者：Juyong Jiang, Fan Wang, Jiasi Shen, Sungju Kim, Sunghun Kim
> 发表于 TOSEM 2025，arXiv 2406.00515

---

## 一、这篇论文在讲什么

用一句话概括：

> 这篇论文把所有"让大模型写代码"的研究，按技术路线分成了十个类别，逐一讲清楚每种方法怎么做、做到什么程度、还有什么没解决。

你不用懂任何术语。我们先从日常类比开始。

---

## 二、日常类比：AI 就像一个新来的实习生

想象你开了一家公司，招了一个新实习生。他的能力有大有小，教他的方式也分好几种。

| 实习生的训练阶段 | 论文里的术语 | 通俗解释 |
|---|---|---|
| 看大量书、积累经验 | **预训练（Pre-training）** | 让模型先读遍互联网上的代码，形成"编程直觉" |
| 学完基础后，专门练写代码 | **持续预训练（Continual Pre-training）** | 在上面基础上，再用纯代码数据继续训练 |
| 你告诉它"帮我写个排序函数"，它就真的写了 | **指令微调（Instruction Tuning）** | 让模型学会理解人类指令，而不是只补全代码片段 |
| 写对了给糖、写错了打手板 | **反馈强化学习（RLHF / RL）** | 用测试用例或人类偏好，让模型自己改进 |

这篇论文的核心贡献，就是把过去几年围绕这四个阶段的所有研究，做了一个系统性的整理。

## 三、核心概念拆解

### 3.1 预训练：先让模型"识字"

想象一个从没写过代码的人，突然让他读 100 万行 Python、Java、C++ 代码。读完之后，他可能不会写完整程序，但他知道 `for` 循环长什么样、函数怎么定义、缩进是什么意思。这就是**预训练**。

论文里提到的重要预训练模型：

- **CodeT5**（2021）—— 编码器-解码器架构，既能理解代码也能生成代码
- **CodeGen**（2023）—— 专门做多轮程序合成的开源大模型
- **StarCoder**（2023）—— 读了 80+ 种编程语言的代码，开源免费
- **Code Llama**（2023）—— 在 Llama 2 基础上继续训练代码版本
- **DeepSeek-Coder**（2024）—— 突破闭源模型的限制

### 3.2 指令微调：从"补全"到"理解任务"

预训练出来的模型，最擅长的是**代码补全**——你给它半行代码，它帮你写完。但这还不够。

真正有用的是它能理解人类指令。比如你说"写一个计算斐波那契数列的函数"，它应该直接给你完整函数，而不是让你先写个开头。

这就是**指令微调**——用大量"问题 -> 代码"的数据对模型进行微调。

论文里重要的指令微调工作：

- **Code Alpaca**（2023）—— 用 LLaMA 做基础，加上代码指令数据微调
- **WizardCoder**（2023）—— 用"进化指令"（Evol-Instruct）自动构造高质量的训练数据
- **MagicCoder**（2023）—— 主张"源代码就是全部所需"，不需要额外数据

### 3.3 强化学习：写错了就改

模型写完代码之后，怎么知道对不对？有两种办法：

1. **跑测试用例**—— 如果所有测试都通过了，就给正向反馈
2. **人类打分**—— 让人类看两段代码，选更好的那个

论文里提到的相关方法：

- **CodeRL**（2022）—— 用深度强化学习，结合预训练模型和代码执行结果
- **RLTF**（2023）—— 用单元测试的反馈来做强化学习
- **StepCoder**（2024）—— 利用编译器反馈来改进代码生成

### 3.4 Prompt 工程：怎么问，决定答案好不好

同一个模型，你问得好，它答得好；问得差，它就答偏了。

论文讨论了多种 Prompt 策略：

- **思维链（Chain-of-Thought）**—— 让模型先写思路再写代码
- **自我调试（Self-Debugging）**—— 让模型自己找 bug 并修复
- **测试驱动生成（Test-Driven）**—— 先生成测试用例，再根据测试写代码

## 四、代码示例

### 示例 1：预训练模型的代码补全

预训练模型就像"填空高手"。你给它代码的一半，它帮你补完。

```python
# 你给模型这样的输入：
def fibonacci(n):
    if n <= 1:
        return n
    a, b = 0, 1
    for i in range(2, n + 1):

# 模型补全的输出：
        a, b = b, a + b
    return b
```

这就是 StarCoder、CodeGen 这类预训练模型最擅长做的事——看到一半代码，猜接下来的内容。

### 示例 2：指令微调后的代码生成

指令微调后的模型能直接理解人类指令，生成完整函数。

```python
# 人类指令：写一个函数，接收一个字符串列表，返回按长度排序后的结果

def sort_by_length(strings):
    return sorted(strings, key=len)
```

这就是 WizardCoder、Code Alpaca 这类模型能做到的事——你直接告诉它"做什么"，它给你"怎么做"的代码。

### 示例 3：强化学习驱动的自修复

模型写完代码后，用测试反馈来改进：

```python
# 第一轮生成（可能有 bug）：
def merge_sorted(arr1, arr2):
    result = []
    for x in arr1:
        result.append(x)
    for x in arr2:
        result.append(x)
    return result  # 忘了合并两个有序数组的逻辑

# 测试反馈：merge([1,3,5], [2,4,6]) = [1,3,5,2,4,6] 预期 = [1,2,3,4,5,6]
# 不通过！需要修复

# 自修复后：
def merge_sorted(arr1, arr2):
    result = []
    i, j = 0, 0
    while i < len(arr1) and j < len(arr2):
        if arr1[i] <= arr2[j]:
            result.append(arr1[i])
            i += 1
        else:
            result.append(arr2[j])
            j += 1
    result.extend(arr1[i:])
    result.extend(arr2[j:])
    return result
```

这就是 CodeRL、StepCoder 这类工作的核心思路——让模型在"测试-反馈-修复"的循环中自己变强。

## 五、论文的十大分类框架

论文把这整个领域分成了 10 大类，你可以把它想象成一张"地图"：

1. **预训练与基础模型** —— 怎么训练一个懂代码的 AI
2. **指令微调与参数高效微调** —— 怎么用少量数据让它学会按要求写代码
3. **反馈强化学习** —— 怎么用测试或人类反馈让它自我改进
4. **Prompt 改进** —— 怎么提问才能得到更好的代码
5. **高质量数据合成** —— 怎么自动生成高质量的训练数据
6. **仓库级与长上下文** —— 让它理解整个项目，而不只是单个函数
7. **检索增强（RAG）** —— 让它先查文档再写代码
8. **自主编程 Agent** —— 让它自己规划、写代码、调试、部署
9. **安全与伦理** —— 生成的代码会不会泄露隐私、有没有漏洞
10. **评估与基准测试** —— 怎么判断一个模型写代码写得好不好

## 六、重要的评测基准

论文重点介绍了以下几个代码生成评测标准：

- **HumanEval** —— 最经典的基准，164 个 Python 函数题
- **MBPP** —— 比 HumanEval 更大，包含 974 个简单编程题
- **BigCodeBench** —— 更贴近实际开发任务的基准
- **EvalPlus** —— HumanEval 的增强版，测试更严格
- **SWE-bench** —— 用真实 GitHub Issue 来评测，看模型能不能修 bug

这些基准就像"期末考试"，用来比较哪个模型写代码的能力最强。

## 七、论文发现的几个关键趋势

**趋势 1：开源模型正在追赶闭源模型**

StarCoder、Code Llama、DeepSeek-Coder 这些开源模型，在 HumanEval 上的得分已经接近甚至超过一些闭源模型。这意味着个人开发者也能用上很强的代码生成工具。

**趋势 2：从"写函数"到"修整个项目"**

早期的工作集中在单个函数级别的代码生成。最新的研究（如 RepoCoder、SWE-agent）已经能做到理解整个代码仓库、修复实际 Issue。

**趋势 3：多语言支持越来越重要**

模型不再只支持 Python，开始支持 Java、C++、JavaScript、Go 甚至 Verilog（硬件描述语言）。

**趋势 4：安全与隐私问题受到重视**

CodexLeaks 等工作发现，Copilot 可能无意中泄露训练数据中的敏感代码。论文呼吁更多研究关注代码生成的安全性。

---

## 八、学术与工业界的差距

论文特别指出了一个值得关注的现象：

学术界用的评测基准（如 HumanEval）题目相对简单，而工业界的实际需求要复杂得多。一个模型在 HumanEval 上得分 90%，不代表它能在真实项目里可靠地写代码。

论文呼吁：

- 设计更贴近实际的评测基准
- 研究模型在真实开发流程中的表现
- 关注代码的可维护性、可读性，而不仅是"能不能跑"

## 九、学习建议

如果你刚入门这个领域，建议按以下顺序学习：

1. 先了解 Transformer 的基本原理（"注意力机制"是什么）
2. 了解大语言模型的训练三阶段：预训练 -> 指令微调 -> 强化学习
3. 体验 GitHub Copilot 或 Claude Code 等实际工具
4. 在 HumanEval 上跑一跑，看看不同模型的得分
5. 然后回到这篇论文，对照十个分类去找具体的论文阅读
