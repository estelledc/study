## 是什么

Naiad 是一个把批处理和流式处理结合得很早的系统。

日常类比：你在写作业时，有的题是一次性提交（批处理），有的是边写边改（流式）。
Naiad 的目标是把这两种方式统一成同一个执行框架。

传统流计算往往按“持续读写”建模。
Naiad 的核心思想更像“数据本身携带时间戳和依赖”，
让系统知道“这条数据在不同时刻对结果会有怎样影响”，并能正确处理迭代。

## 为什么重要

- 实时指标和离线统计经常不能割裂：你既要秒级看结果，也要最终一致。
- 一些算法本身是迭代性的（比如图计算、推荐更新），流式系统不设计好会重算很多。
- 流系统如果不能回到“某个时间点状态”，问题定位会很痛。

Naiad 通过 timely dataflow 提供了一条统一路径。
这让“每来一条数据怎么更新答案”变得可控。

## 核心要点

1. **Timely dataflow 的时间维度**：每条记录携带逻辑时钟，不再只是按到达顺序处理。

- 类比：超市里排队不仅按顺序，也按就餐时间段分批。
- 时间帮助系统判断“同一轮次是否已收齐”。

2. **可迭代计算（iteration）**：Naiad 支持迭代循环，让某些算法不必拆成离线任务。

- 类比：你算出推荐后再喂给下一轮算子，像反复打磨。
- 要避免死循环，需要严格边界控制。

3. **统一批处理与流式语义**：同一套算子可处理历史批和实时增量。

- 类比：同一张课程表既支持周计划也支持今天临时加课。
- 工程价值是减少维护多个系统时的重复开发。

## 实践案例

### 案例 1：网站点击统计

```scala
// 伪代码
for event in events {
  byWindow(event.timestamp).count += 1
  emit(toWindowState)
}
```

- 你可以按分钟/小时窗口持续更新。
- 遇到迟到数据，通过时间戳规则回放即可。

### 案例 2：图计算迭代

```python
for step in range(max_iter):
    ranks = edges.join(scores).map(update)
    if converged(ranks, eps):
        break
```

- Naiad 的迭代模型使这类场景不用外部 orchestrator 持续重跑全量。
- 关键是边界条件和收敛条件。

### 案例 3：在线报警窗口

```bash
stream input | group-by key, tumbling=10s | join state_cache | emit alerts
```

- 流里实时触发报警，不必等到下一轮 batch。
- 错误率可控：每条输入都可追到时间窗口。

## 踩过的坑

1. **没理解时间语义**：只按 arrival order 设计时，迟到数据会导致多版本错判。
2. **迭代次数未设上限**：无收敛条件会把资源耗尽。
3. **状态清理缺失**：窗口过期后没收缩会导致内存持续膨胀。
4. **把调度和语义混淆**：执行计划稳定不代表结果语义一致。

## 适用 vs 不适用场景

**适用**：
- 高实时性+历史补算都需要的指标系统。
- 图计算、推荐更新、社交互动传播这类迭代工作流。
- 复杂依赖链，能从统一执行模型中受益的团队。

**不适用**：
- 纯离线 ETL，没有增量诉求。
- 延迟可容忍的简单报表。
- 团队缺少运维能力，无法维护状态和窗口配置。

## 历史小故事（可跳过）

- **2010s 前期**：批处理系统和流处理系统几乎隔离。
- **2013**：timely dataflow 带来统一语义的讨论。
- **中后期**：后续系统从 Naiad 学到“增量 + 迭代”组合的关键。
- **今天**：不少现代引擎在架构上可以看到这条思想。

## 学到什么

1. 统一数据语义比拼系统品牌更关键。
2. 时间是分布式系统里最容易被忽略又最容易出 bug 的一层。
3. 一切迭代计算都要有收敛条件。
4. 只有可追溯到状态版本的系统，才能长期稳定。

## 额外复盘（可跳过）

- 在没有显式时间语义的情况下，系统更容易把迟到消息当成新事实，导致窗口输出抖动。
- 如果你把迭代模型当成“无限循环”而非“有界收敛”，很容易把资源打满。
- 状态存储不是加速器，而是工程风险放大器；缺少清理机制会吞掉你所有优化收益。
- 你可以把 timely dataflow 看成“同一张图上画两种线”：一条是数据路径，另一条是时间路径。

```text
输入数据 -> 时间打标 -> 状态计算 -> 触发判断 -> 输出结果 -> 再迭代输入
```

这张链条告诉你：即使你写的是批处理语义，只要把时间关系建对了，也能拿到流式稳定性。

- 在工程实践中，常见顺序是先拿小数据量跑通时间线，再逐步放开并发与窗口长度。
- 记住一个经验：如果窗口和迭代一起失控，先查时间维度，不要先加机器。
- 复杂系统里最值钱的时间常识，不是吞吐，而是“状态能否解释自己为什么这样变化”。
