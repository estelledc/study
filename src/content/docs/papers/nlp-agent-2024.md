## 是什么

CoALA 是一篇把 LLM Agent 放回认知架构传统里理解的论文。

它的全名是 Cognitive Architectures for Language Agents。

作者不是发明某个单一新 agent，而是给当时已经爆炸增长的 agent 工作画一张地图。

这张地图把语言 agent 拆成三块：记忆、动作、决策。

记忆负责保存当前上下文、长期知识和过去经验。

动作负责让 agent 操作内部记忆、调用工具、和外部环境交互。

决策负责决定下一步做什么、什么时候调用模型、什么时候停下来。

如果说 ReAct 像一种具体打法，CoALA 更像分析打法的语法。

它关心的问题是：LLM 怎么从一个文本补全器，变成能持续行动的系统。

## 为什么重要

2023 年之后，agent 论文和框架增长很快。

很多系统都在说 planning、memory、tool use、reflection，但术语并不统一。

CoALA 的价值是把这些概念放进一个共同框架。

这样读不同论文时，就不用每次重新学习一套名字。

它也提醒读者：Agent 不是 LLM 本身，而是 LLM 加上一组控制结构。

这对工程实现很关键。

如果系统失败，不一定是模型不够强，也可能是记忆写错、动作空间太乱、决策循环没有退出条件。

CoALA 把这些失败点拆开，让调试更具体。

## 核心要点

第一，agent 需要工作记忆。

工作记忆就是当前任务里正在被使用的信息，比如目标、观察、草稿和中间结论。

第二，agent 需要长期记忆。

长期记忆可以是向量库、日志、知识库，也可以是模型参数里已经学到的知识。

第三，agent 需要动作空间。

动作不只包括调用外部 API，也包括检索记忆、写入记忆、总结状态。

第四，agent 需要决策过程。

决策过程决定什么时候思考、什么时候行动、什么时候观察反馈。

第五，LLM 调用只是这个循环中的一个组件。

把所有复杂性都压给一次 prompt，通常会得到脆弱系统。

## 实践案例

最简单的 CoALA 式 agent 可以这样想。

用户给一个任务：整理三篇论文的差异。

工作记忆保存任务目标、当前已读论文、尚未确认的问题。

长期记忆保存过去读论文的方法、术语解释和已有笔记。

动作空间包括搜索论文、打开 PDF、摘录段落、写摘要、更新表格。

决策模块每轮选择一个动作。

如果缺资料，就检索。

如果资料足够，就比较。

如果比较完成，就写结论。

这个例子说明，agent 的关键不是"模型一次回答得多聪明"。

关键是系统是否能把任务拆成可观察、可恢复、可继续的步骤。

## 踩过的坑

第一个坑是把 memory 当成向量库同义词。

向量库只是长期记忆的一种实现，不等于完整记忆系统。

第二个坑是动作空间太大。

工具越多，agent 越容易在不相关动作里迷路。

第三个坑是没有退出条件。

很多 agent 不是能力不够，而是循环一直继续，成本不断上升。

第四个坑是把 reflection 当成万能修复。

反思如果没有可验证反馈，可能只是把错误说得更圆。

第五个坑是没有区分内部动作和外部动作。

写 scratchpad 和调用支付 API 的风险完全不同，不该放在同一权限层。

## 适用

CoALA 适合用来读 agent 论文。

看到一个新系统时，可以问它的记忆、动作、决策分别是什么。

它也适合做 agent 架构评审。

如果某个模块说不清属于哪一类，就说明边界可能混乱。

它还适合指导产品里的 agent 功能设计。

比如客服 agent、代码 agent、研究 agent，都能按这三块拆解。

它不适合直接当实现手册。

论文提供的是概念框架，不是某个库的 API 文档。

真正落地还要处理权限、成本、可观测性和失败恢复。

## 历史小故事

CoALA 特意把 LLM Agent 接到更早的认知科学传统上。

早期 AI 里，研究者已经在做生产系统和认知架构。

生产系统用一组 if-then 规则描述智能行为。

Soar 等认知架构试图解释人类如何记忆、推理和行动。

LLM 出现后，很多新系统看起来很新。

但从 CoALA 视角看，它们常常是在用神经语言模型替换过去的符号规则。

旧问题并没有消失。

系统仍然要决定记什么、怎么行动、如何选择下一步。

这也是论文标题里 "Cognitive Architectures" 的含义。

## 学到什么

读 CoALA 最大的收获是把 agent 从 prompt 崇拜里拉出来。

一个 agent 的能力来自系统结构，而不只是底层模型。

记忆设计决定它能不能跨步骤工作。

动作设计决定它能不能改变外部世界。

决策设计决定它能不能稳定推进任务。

评估 agent 时，也应该分别评估这些层。

失败日志里要能看到每一步的状态、动作和原因。

这样才知道问题是检索错、工具错、规划错，还是模型生成错。
