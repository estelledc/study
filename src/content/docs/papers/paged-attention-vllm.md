## 是什么
PagedAttention 把 LLM KV Cache 当作“主机内存分页’来管理，显著降低碎片和换入换出抖动。
类比：像图书馆把整本书的整页借阅改成“按需翻到某一页再归还”，不是先把整本书搬进桌面。

典型入口通常是把上下游系统分成一个清晰的主链路：
- 依赖识别或资源发现
- 核心计算或转发
- 结果输出与可观测。


## 为什么重要
- 把 KV cache 长时间占满显存会直接导致服务降速或 OOM。
- 请求并发时，静态分配会放大空窗区和碎片。
- 固定 buffer 区方案在长上下文场景下非常吃力。
- 需要让 token 并发时“按实际页”分配显存。
- 把 swap/evict 策略从工程实现变成可推演系统行为。

当你只盯最终接口时，你经常会误判是“模型/库本身慢”，实际上瓶颈往往在接入层。

## 核心要点
1) 机制分解：
- 按 block 与 page 粒度管理 KV cache，替代大页整段管理。
- 引入可回收空闲页池，支持逐块交换与预取。
- 结合 scheduler 做热点页保留与淘汰，减少重建开销。
- 在引擎里把内存分配从“猜测式”变成“状态可观测”。

2) 典型收益：
- 可控性提升、吞吐上升、异常定位更快。

## 踩过的坑
1. 把 page 大小设得过大，导致碎片并没减少，反而让回收不平滑。
1. 没区分可重用与长尾热点页，出现抖动和反复交换。
1. 指标只看显存占用，忽略了重排后的请求拥塞和等待时间。
1. 把 scheduler 改动与 cache 策略耦合过深，导致排查困难。

## 适用 vs 不适用
**适用**：
- 推理服务希望在单机承载更多并发。
- 上下文长度可变且峰值变化大。
- 团队具备 KV 结构监控和追踪链路。
- 推理链路以延迟稳定性为主。

**不适用**：
- 极低延迟硬性场景且模型规模小，简单预分配更合适。
- 缺少显存分配观测体系，策略改动难以复现。
- 只做离线批处理任务。
- 服务不允许引入动态换页复杂性。

3. 指标加固：将耗时、错误、资源放入固定大盘。
4. 扩容验证：再验证不同负载窗口。
在社区讨论中，大家更关注“问题可复现性”与“回归成本”。


## 学到什么
1. 复杂系统先找一个稳定基线，再谈优化。
2. 参数不是“越大越好”，目标应是吞吐与稳定平衡。
3. 可观测指标比口头经验更有说服力。
4. 小步迭代能显著降低回归风险。

可以自测的三问：
- 现在的瓶颈到底在 CPU、IO 还是内存？
- 参数改动后是否会引入长尾延迟？
- 回归场景是否覆盖了异常输入？
