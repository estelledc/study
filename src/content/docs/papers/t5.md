## 一句话总结

T5 把所有 NLP 任务（翻译 / 分类 / 问答 / 总结）统一成「输入一段文本，输出一段
文本」这一个格式，再配上一个**标准 encoder-decoder Transformer**、一个叫
**span corruption** 的预训练目标、以及一个 750 GB 的清洗后 Common Crawl 语料
**C4**。论文用 24 张消融表把「架构 / 目标 / 数据 / 规模」四个维度都翻了个遍，
得出「encoder-decoder + span corruption + 大数据 + 大模型」这个组合在 transfer
learning 上最优。它既是 BERT 之后第二代 transfer learning 的高峰，也是
ChatGPT 时代「一切任务都用 prompt」这种思想的方法论祖先之一。

## 历史定位

T5 处在 BERT (2018, encoder-only) 和 GPT-3 (2020, decoder-only) 之间，是这条
时间线上不可省的一环：

- 2017: Transformer (Vaswani et al.)；原始 paper 用的就是 encoder-decoder（机器翻译）
- 2018-06: GPT-1 (Radford et al.)，decoder-only + 微调
- 2018-10: BERT (Devlin et al.)，encoder-only + MLM + 微调
- 2019-02: GPT-2 (Radford et al.)，decoder-only + 大规模、零样本生成开始有戏
- 2019-07: RoBERTa (Liu et al.)，BERT 的"训练配方"优化
- 2019-09: ALBERT (Lan et al.)，BERT 参数共享
- **2019-10: T5 / C4 (Raffel et al.) ← 本文**
- 2020-05: GPT-3 (Brown et al.)，175B / few-shot in-context learning
- 2021-10: mT5 (Xue et al.)，T5 多语言扩展（101 种语言）
- 2022-10: Flan-T5 (Chung et al.)，T5 + instruction tuning
- 2022-11: ChatGPT，prompt + RLHF 工业落地
- 2024-: decoder-only 一统，但翻译、总结、code 上 encoder-decoder 仍有 niche

T5 的特殊位置：它是**最后一篇还认真做架构 / 目标消融的论文**。GPT-3 之后大家
不再问"哪个架构好"，而是问"scaling 规律是什么"（Chinchilla / scaling laws），
架构成为常量。T5 的 ablation matrix 是后来无人再做、也无人再做得起的实验。

## Section 1: 动机 — 任务格式不统一是个工程灾难

写这篇笔记的第一性原理：先想清楚 T5 之前的世界长什么样。

2018-2019 的 NLP 工具箱里，每个任务都是一种独立的接口形状：

- **翻译**：seq2seq，输入 source 句、输出 target 句，用 BLEU 评估，一般用 encoder-decoder
- **分类（GLUE / CoLA / MNLI）**：BERT encoder + 一个线性 head，logits 输出，用 accuracy
- **抽取式问答（SQuAD）**：BERT + 两个 head（start 位置 / end 位置），span index 输出
- **总结（CNN/DailyMail）**：seq2seq，又是 encoder-decoder，但用 ROUGE
- **NER**：每个 token 一个分类 head（BIO 标注）
- **语义相似度（STS-B）**：回归 head，连续值

每个任务的：

1. 输入格式不同（单句 / 句对 / 文档）
2. 输出格式不同（label / span / 文本 / 数字）
3. loss 函数不同（cross-entropy / span loss / regression）
4. 评估指标不同（accuracy / F1 / BLEU / ROUGE / Pearson）
5. 网络头部不同（每个任务都要训新 head）

这意味着：

- **代码工程量爆炸**：同一个 BERT backbone 上挂十几个 task-specific head
- **多任务学习困难**：不同任务的 loss 量级不同，加权超参全靠玄学
- **transfer learning 不公平**：BERT pretrain → 每个任务 finetune，预训练时学到
  的"通用语言能力"在不同任务的 head 形状下表现不一致
- **实验难做对照**：换一个架构（比如想试 prefix LM），所有 head 都得重写

T5 的提问：**能不能用一个架构 + 一个 loss + 一种输入输出格式覆盖一切？**
如果可以，预训练 / 微调 / 多任务 / 评估 / 工程化这五件事就都解耦了。

> 怀疑：这个动机其实带了一个很强的假设——**模型容量足够大时，统一格式不会
> 比 task-specific 设计差**。在 T5 时代（≤ 11B）这个假设勉强成立，但在 BERT-base
> 110M 的小模型上，task-specific head 几乎一定胜过 text-to-text。Raffel 是不是
> 默认了"reader 在意的就是 large 模型"，把 statement 写得过于普适？

## Definition 1: text-to-text

```
f(input_text) -> output_text
```

input 和 output 都是一串自然语言 token。模型不知道当前是哪个任务，靠的是
input 里的**任务前缀**和**输入内容本身**。

例：

- input  : `"translate English to German: That is good."`
- output : `"Das ist gut."`

- input  : `"cola sentence: The course is jumping well."`
- output : `"not acceptable"`

- input  : `"question: Where was Obama born? context: ... Honolulu, Hawaii ..."`
- output : `"Honolulu, Hawaii"`

- input  : `"summarize: <article ~1000 tokens>"`
- output : `"<summary ~50 tokens>"`

注意三件事：

1. **分类任务的 label 也是文本**："not acceptable" / "acceptable" 是字符串，
   不是 logits。loss 是字符级 / token 级的 cross-entropy，跟翻译完全一样。
2. **回归任务被离散化**：STS-B 的 0-5 相似度被分桶成 21 个 0.2 间隔的字符串
   ("0.0", "0.2", ..., "5.0")。论文 §3.1.1 承认这是工程妥协。
3. **多任务训练时所有任务的 loss 同质**：都是 next-token cross-entropy，没有
   不同 head 的权重平衡问题。

## Definition 2: prefix prompting

任务前缀（task prefix）是写在 input 最前面的一小段固定字符串，作用是告诉模型
"现在请你执行哪个任务"。

例：

- `"translate English to German: "`
- `"cola sentence: "`
- `"summarize: "`
- `"stsb sentence1: ... sentence2: ..."`

这个前缀**不是特殊 token**，就是普通的英文字符串，会被 SentencePiece tokenizer
切成普通 subword。模型在预训练时见过大量自然语言里类似 "translate" / "summarize"
的词，finetune 时只需要学到「看到这个固定串，应当输出对应格式」。

> 怀疑：T5 的 prefix 设计是 ad-hoc 的——`"translate English to German:"` vs
> `"translate to German:"` 哪个好？论文没系统消融。这种"早期 prompt engineering"
> 跟后来 Flan / Alpaca 的 instruction tuning 几乎是同一件事，但 T5 的前缀短而
> 模板化，instruction tuning 的指令长而自然语言化。这两条路线是不是只是同一思想
> 在不同模型规模下的两个表现？

## Definition 3: span corruption

预训练目标。给一段连续文本（比如来自 C4 的网页段落），随机选若干**连续 span**
（平均长度 3 个 token，总覆盖率 15%）替换成唯一的 sentinel token
`<extra_id_0>`、`<extra_id_1>` 等，让 decoder 把被遮的 span 内容生成出来。

例（论文 Figure 2）：

```
原文:    Thank you for inviting me to your party last week.
mask:    Thank you <X> me to your party <Y> week.

decoder
target:  <X> for inviting <Y> last <Z>
```

注意几个细节：

1. encoder 看到的是带 sentinel 的不完整文本（**双向注意力**，每个位置看全句）
2. decoder 看到的 target 也是 sentinel 序列加上各 span 的内容
3. **不是逐 token mask**（BERT 风格）：是连续多 token 一起 mask，更接近"补段落"
4. `<Z>` 是终止符，告诉 decoder "spans 都生成完了"

为什么连续 span 比 i.i.d. mask 好？论文 §3.3 给的直觉：i.i.d. mask 大部分情况
下被遮的是冗余信息（"the dog is on the [MASK]" 里 [MASK] 几乎可以从上下文猜出），
而连续 span 强迫模型学**长距离依赖**和**篇章生成能力**。这一点在 T5 的消融
（§3.3 表 7）里得到了直接验证：span corruption 在所有 transfer 任务上都优于
BERT-style i.i.d. masking。

> 怀疑：span corruption 在 T5 的对比里赢了，但 BERT-style 在 BERT 上赢了，
> 这是不是因为「encoder-decoder + 连续 span」和「encoder-only + 离散 mask」是
> 各自架构最适配的 objective，而非 span corruption 本身一定是天然最强？后来
> UL2 (Tay 2022) 提出 mixture-of-denoisers，混合多种 corruption 进一步提升，
> 印证了"没有银弹 objective"这件事。

## Section 3.1: text-to-text 范式的工程实现

把上面的 Definition 1-3 落到具体工程上，T5 的训练流水线大概长这样：

**阶段 1: 预训练（unsupervised）**

- 数据：C4，750 GB 清洗后的英文网页文本
- 输入：连续 ~512 token 的网页文本片段
- 目标：span corruption（15% mask rate，平均 span 长度 3）
- 模型：encoder-decoder Transformer（标准结构 + 一些细节调整，见 §2.2）
- 训练步数：T5-Base 默认 524288 步，batch 128，约 34B token

**阶段 2: 有监督多任务微调**

- 数据：GLUE / SuperGLUE / SQuAD / WMT 翻译 / CNN-DM 总结的混合
- 输入格式：`"<task prefix>: <task input>"`
- 输出格式：`"<task output as text>"`
- loss：next-token cross-entropy（与预训练同质）
- 多任务采样策略：examples-proportional mixing，按数据集大小采样
  （§3.5.2 / 表 11 还对比了 equal mixing / temperature-scaled mixing）

**阶段 3: 评估**

- 输入：测试集 input 加 task prefix
- 输出：模型 generate 的 text，按任务专属 metric 计算（accuracy / BLEU / ROUGE / F1）

整个流水线的优雅之处：**只有一种 loss**，**只有一个模型**，**只有一种数据格式**。
预训练代码、微调代码、评估代码差异极小。

## Section 3.2: 架构选择 — encoder-decoder 为何能赢

T5 论文 §4.1 做了一个非常昂贵的 ablation：**固定参数量预算**，把同样的 P 参数
分配到三种架构上，看哪个 transfer learning 最好。

三种架构：

1. **encoder-decoder**（T5 默认）：encoder 双向 + decoder 因果 + cross-attention
2. **prefix LM**（decoder-only with bidirectional prefix）：input 部分双向，
   output 部分因果。Raffel 等人称之为"折中方案"
3. **standard LM**（GPT-style decoder-only）：全因果，input 和 output 都从左到右

参数预算固定为同一个数（约 220M，T5-Base 规模），但分配方式不同：encoder-decoder
有一半参数在 encoder、一半在 decoder；prefix LM 和 standard LM 全部参数都在
"decoder"（实际是 attention 模式不同）。

论文 Table 2 的结果（GLUE / CNN-DM / SQuAD / WMT 14 等任务平均）：

| 架构              | 平均得分（相对 baseline） |
|-------------------|--------------------------|
| Encoder-Decoder   | 最高                     |
| Prefix LM         | 略低                     |
| Standard LM       | 最低                     |

直觉解释：

1. encoder 双向看全输入，对**理解类任务**（分类 / QA）天然友好
2. decoder 因果生成，对**生成类任务**（翻译 / 总结）天然友好
3. 一个模型同时具备两种偏置 → transfer learning 通吃
4. cross-attention 允许 decoder 在生成每个 token 时**重新看一遍输入**，类似人类
   "翻译时反复对照原文"

但这里有个关键 caveat：**这是在 P=220M 规模下的结论**。

> 怀疑：T5 §4.1 的消融在 ≤ 11B 范围内成立，但 GPT-3 (2020, 175B) 是 standard LM，
> 在 zero / few-shot 上吊打 T5-11B。这是不是说明：(a) scaling 改变了架构最优解，
> 或者 (b) "transfer learning"（finetune）和 "few-shot in-context learning" 是
> 两套不同的能力，T5 的消融只覆盖前者？我倾向 (b)——T5 的 finetune 范式假设有
> 标注数据，但 GPT-3 范式不需要。两条路线的胜负要看下游需求。

## Section 3.3: 训练目标的消融（论文 §3.3 表 7）

T5 把训练目标也系统消融了一遍。被对比的目标：

| 目标                    | 描述                                            |
|------------------------|------------------------------------------------|
| Prefix LM              | input 双向，output 因果，预测下一个 token        |
| BERT-style MLM         | i.i.d. 15% mask，预测被遮的 token              |
| Deshuffling            | 把句子里的 token 随机打乱，模型要还原原始顺序     |
| MASS-style             | 一段连续 span 被 mask，decoder 生成被遮 span     |
| **Span corruption**    | 多个连续 span 被 sentinel 替换，decoder 生成全部 |
| i.i.d. noise (replace) | i.i.d. mask 但用 sentinel 替换                  |

结论（论文 §3.3 表 7）：

- **span corruption** 在 GLUE / SuperGLUE / SQuAD / WMT 全部任务上平均最高
- BERT-style MLM 在分类任务上接近 span corruption，但在生成任务上明显弱
- Prefix LM 中规中矩，没有亮点
- Deshuffling 全面最差（对应"句序还原"过于人工，不像下游任务）

T5 默认配置最终选定：

- **mean span length = 3**
- **corruption rate = 15%**
- **目标是生成所有 spans（不是一次生成一个）**

> 怀疑：T5 的 span corruption 在它自己的架构下赢了，但 ELECTRA (Clark 2020) 的
> replaced token detection 在更小预算下达到 BERT 同等性能。如果 T5 把 ELECTRA-style
> 也加进消融表，结论会不会变？这是 T5 论文的一个明显遗漏——表 7 只对比了"生成式"
> 目标，没有包含"判别式"目标。

## Section 3.4: C4 数据集 — 第一个 750 GB 级清洗语料

C4 = Colossal Clean Crawled Corpus。基础数据是 Common Crawl 2019-04 的一个月
快照（约 6.1 TB 原始 HTML），清洗后剩 **745 GB / ~156B token**。

清洗规则（论文 §3.4 / 表 8 旁边那段叙述）：

1. **只保留以句号 / 问号 / 感叹号 / 引号结尾的"完整句子"** —— 过滤导航 / 列表 / 代码片段
2. **去掉短于 5 句的页面**
3. **去脏话（List of Dirty, Naughty, Obscene and Otherwise Bad Words）** —— 论文
   用了一个公开列表，命中即整页丢弃
4. **去重**：完全相同的三句话窗口只保留一份
5. **语言检测**：langdetect 库判定为英语 ≥ 99% 概率才保留
6. **去 boilerplate**：重复模式（"Lorem ipsum" / "Javascript is required" 等）
7. **去 placeholder**：含 `{` 或 `}` 的页面（怀疑是 JS 模板）

最终结果：

- **745 GB / ~156B token**（vs BERT 用的 BookCorpus + Wikipedia 合计 16 GB）
- 清洗 ratio 大约 90%（5.5 TB 被丢掉，0.7 TB 保留）

为什么这个数据集这么重要：

1. **量级跨越**：从 16 GB 到 745 GB，46 倍。模型能 scale 到更大。
2. **公开发布**：Google 把 C4 放在 TensorFlow Datasets 上，所有人都能复现 T5
3. **后续标准**：mT5 用 mC4（多语言版），LLaMA / Falcon / 大量开源模型的语料配方
   都包含 C4

> 怀疑：C4 的"去脏话"列表是英文硬编码的。GeoDiversity 论文（Dodge et al. 2021）
> 指出这个过滤会显著删除非裔英语方言（AAVE）和 LGBTQ+ 语料。这意味着 T5 在
> 边缘化群体的语言上系统性欠学。这是数据清洗"道德"vs"科学"边界的经典案例——
> 看似中立的过滤规则实际有强 sociolinguistic 偏差。

## Section 4: 模型规模 — 5 个尺寸覆盖 60M 到 11B

T5 训练了五个规模的模型（论文 §3.7 表 1）：

| 名称        | 参数  | 层数 (enc+dec) | 隐藏 d_model | FFN d_ff | heads | d_kv |
|-------------|-------|---------------|--------------|----------|-------|------|
| T5-Small    | 60M   | 6 + 6         | 512          | 2048     | 8     | 64   |
| T5-Base     | 220M  | 12 + 12       | 768          | 3072     | 12    | 64   |
| T5-Large    | 770M  | 24 + 24       | 1024         | 4096     | 16    | 64   |
| T5-3B       | 3B    | 24 + 24       | 1024         | 16384    | 32    | 128  |
| T5-11B      | 11B   | 24 + 24       | 1024         | 65536    | 128   | 128  |

注意几件事：

1. **3B 和 11B 不再加深，只加宽 FFN**：论文 §3.7 解释"加深 layer 收益递减，
   加宽 FFN 在 TPU 上更高效"
2. **encoder 和 decoder 总是同样深**：没有不对称设计
3. **head 数随规模增加 K/V 维度也会变化**：T5-3B / 11B 的 d_kv = 128（vs Small/Base/Large 的 64）

训练成本（论文 §3.7 末段 / 隐式信息）：

- T5-11B 在 1024 张 TPU v3 上训了大概 1 万小时（粗略估算 ~30 万 TPU 小时）
- 推理：T5-11B 在单卡 TPU v3-8 上勉强能跑，需要切分

> 怀疑：T5-11B 在 2019 是巨无霸，但 2024 年 LLaMA-3 8B 在大多数任务上更强。
> 这意味着 T5 路线在"模型小"的市场仍可能有优势——encoder-decoder 在翻译 /
> 总结上的小模型 SOTA 仍然由 mT5 / Flan-T5 把持。decoder-only 真的全面取代了
> encoder-decoder 吗？还是只在"对话 / 通用 chat"那个 niche 里取代了？

## Section 5: 大规模消融实验（24 张表）

T5 论文最有价值的部分其实不是 T5 本身，而是这套消融方法论。论文 §3 一共有
24 张消融表，覆盖：

| 维度          | 消融的内容                                                          | 论文表号  |
|---------------|--------------------------------------------------------------------|-----------|
| 架构          | encoder-decoder / prefix LM / standard LM                          | 表 2-3    |
| 训练目标      | 6 种 corruption / mask 策略                                         | 表 7      |
| corruption 率 | 10% / 15% / 25% / 50%                                              | 表 5      |
| span 长度     | 2 / 3 / 5 / 10 token                                               | 表 6      |
| 数据集        | C4 / C4-clean / RealNews / WebText / Wikipedia                     | 表 8      |
| 数据规模      | full / 1/4 / 1/16 / 1/64 / 1/256                                   | 表 9      |
| finetune 策略 | adapter / prefix tuning / 全参数 finetune                          | 表 10     |
| 多任务采样    | equal / examples-proportional / temperature-scaled                  | 表 11     |
| 训练步数      | $2^{16}$ / $2^{18}$ / $2^{20}$ / $2^{21}$ 步                       | 表 12     |
| 模型规模      | Small / Base / Large / 3B / 11B                                    | 表 13     |

每张表的消融都是**控制变量**：固定其它配置，只改一个维度，跑完 finetune，
报告下游任务的平均得分。

这是工业界资源才能完成的实验。Brown et al. 2020 (GPT-3) 几乎不做消融——
他们说"训一次 175B 模型已经太贵"，只能"信经验"。T5 是最后一个还能做这种
ablation matrix 的论文。

主要可重复使用的发现：

1. **架构**：encoder-decoder ≥ prefix LM > standard LM（在 ≤ 11B 规模下）
2. **目标**：span corruption ≥ BERT-style MLM > prefix LM > deshuffling
3. **数据**：scaling 一直有效，没看到 saturation；C4 ≥ Wikipedia + BookCorpus
4. **finetune**：full finetune > adapter > prefix tuning（在 T5 时代）
5. **多任务**：multi-task 训练 + finetune 比单任务略好，且节省总训练量
6. **模型规模**：scaling 一直有效，T5-11B 仍未饱和

> 怀疑：T5 的消融用的是"固定预算 + 单变量"的方法学，假设各维度近似独立。但
> 后来 Chinchilla (Hoffmann 2022) 发现"模型大小 vs 数据量"这两个维度严重耦合
> （compute-optimal 关系）。如果 T5 重做一遍消融，把这种耦合纳入，结论会变吗？
> 至少 T5-11B 用的是 1T token，按 Chinchilla 的 20 token / param 法则其实
> token 量略不够，对应的"最优配置"应该在更大数据上 retrain。

## Section 6: 评估结果

T5-11B（论文最后一个表）在多个 benchmark 上的成绩：

| Benchmark           | T5-11B  | 之前 SOTA              | 提升        |
|---------------------|---------|------------------------|-------------|
| GLUE (avg)          | 89.7    | 88.4 (RoBERTa)         | +1.3        |
| SuperGLUE (avg)     | 89.3    | 84.6 (RoBERTa+iter)   | +4.7        |
| SQuAD v1.1 (F1)     | 95.6    | 94.8 (XLNet)           | +0.8        |
| CNN/DM (ROUGE-L)    | 22.1    | 21.6 (BART)            | +0.5        |
| WMT En-De (BLEU)    | 32.1    | 33.8 (Edunov 2018, 大数据) | -1.7    |
| WMT En-Fr (BLEU)    | 43.4    | 45.6 (Edunov 2018)     | -2.2        |

T5-11B 全面接管了**理解类任务**（GLUE / SuperGLUE / SQuAD），但在**机器翻译**
上略输给专门的翻译系统——后者用了更多回译（back-translation）数据增强。

值得注意的几件事：

1. **SuperGLUE 是 T5 论文的标志性突破**：从 84.6 跳到 89.3，幅度大到改变
   benchmark 进度条（GLUE 后 SuperGLUE 是更难一档的 benchmark，T5 几乎让它饱和）
2. **多任务联合 finetune 比单任务略好**：T5 在多任务版本上跑出大部分 SOTA
3. **翻译略弱**：暗示 text-to-text 范式在严格的"领域已经独立优化几十年"的任务上
   不是天然最优——这给后人一个提示，**统一格式不等于通杀**

## Section 7: 衍生工作 — text-to-text 思想的回潮

T5 之后这条线的演化：

**直接 T5 衍生：**

- **mT5**（Xue 2021）：T5 多语言扩展，101 种语言，词表 250K
- **byT5**（Xue 2022）：直接在 byte 级训练，无 SentencePiece，对噪声更鲁棒
- **Flan-T5**（Chung 2022）：T5 + instruction tuning（1800+ 任务），small 模型
  也能 zero-shot
- **UL2**（Tay 2022）：mixture-of-denoisers，把 span corruption / prefix LM /
  其它 objective 混在一起预训练
- **PaLM 2**（Anil 2023）：UL2 风格目标 + decoder-only，部分实现 T5 思想 +
  GPT 架构

**思想被吸收（不直接是 T5 后代但用了 text-to-text 范式）：**

- **GPT-3 / ChatGPT**：所有任务都通过 prompt 转成文本输入文本输出，本质就是
  text-to-text。区别是 GPT 不再有"task prefix"概念，prompt 里直接用自然语言
  描述任务（"Translate the following English to German: ..."）
- **InstructGPT / Constitutional AI**：instruction tuning 是 prefix prompting 的
  自然演化——把短模板换成长指令
- **代码 LLM（Codex / StarCoder）**：text-to-text 框架下，input 是函数签名 / 注释，
  output 是函数体

**T5 仍然占优的领域：**

1. **翻译 / 总结的小模型 SOTA**：Flan-T5-base (220M) 在 X-Sum 等总结任务上仍优于
   同尺寸的 decoder-only 模型
2. **可控生成**：encoder 双向理解 + decoder 生成的解耦，让 prompt-based 控制更稳定
3. **检索增强生成（RAG）的 reader 端**：cross-attention 天然适合"输入大段 context、
   输出短答案"的任务，FiD（Izacard 2021）就是 T5 + retrieved passages

## 限制与遗留问题

按"承认论文不完美"的精神列下来：

1. **推理慢于 decoder-only**：encoder-decoder 在生成第一个 token 之前要先跑一遍
   完整 encoder forward。decoder-only 没这一步。在生产推理服务上，T5-11B 的
   p99 延迟比同尺寸 decoder-only 模型高 1.5-2 倍。

2. **训练成本巨大**：T5-11B 训练 ~30 万 TPU 小时，按 2019 价位约百万美元级。
   这把"做架构 ablation"的门槛抬到只有 Google / DeepMind / OpenAI 能跨进去。
   论文之后再没人做过同等规模的架构消融。

3. **C4 清洗有偏**：去脏话列表是英文硬编码 + 启发式规则，会过度过滤非主流方言
   和性少数语料。"道德过滤"本身引入了语料偏差，论文未做 fairness 评估。

4. **prefix 设计 ad-hoc**：`"translate English to German:"` 是经验选定的，
   论文没系统比较 prefix 措辞（"translate to German:" / "english to german:" / ...）
   对下游性能的影响。后来的 prompt engineering 研究填了这个坑。

5. **与 instruction tuning 时代的差距**：T5 的 prefix 是短模板，Flan / Alpaca 的
   instruction 是自然语言。T5 原始范式在 zero-shot 任务上明显弱于 instruction-tuned
   模型。Flan-T5 (2022) 才补上这一课。

6. **STSB 回归被离散化**：把连续值分成 21 个桶强行变文本，虽然能跑，但理论上
   损失了精度。在严格回归任务上 text-to-text 范式有先天弱点。

7. **多语言能力弱**：T5 是英文 only。要做多语言必须等 mT5（论文发布两年后），
   而 mT5 又不能直接共享 T5 权重——这意味着 T5 路线在"一个模型支持所有语言"
   这个工程目标上需要重新训练。

## 我从 T5 学到了什么

零基础学习者视角下，三件事最重要：

**1. "统一接口"是软件工程的胜利**

T5 教给我的最深的一件事跟模型本身无关：当你面对一堆形状各异的子问题时，
**先想能不能把它们的接口形状统一**，然后再上模型。NLP 之前每个任务一个 head，
是在"接口形状"层面没收敛。T5 把所有任务收敛到 `text -> text`，整个工具链
（数据 / 训练 / 评估）的复杂度立刻降一个数量级。这跟 Unix "everything is a file"
是同一种思维——把异构复杂度藏到接口下面。

**2. 消融才是论文的真正贡献**

T5 这篇论文的"模型"（encoder-decoder + span corruption + C4）单看每一项都不是
最早的、最 novel 的。它的贡献是**用 24 张表把整个设计空间扫了一遍**，留下了
后人可以直接引用的实验结论。下次我读 paper，要分清"作者的 novel claim"和
"作者真正贡献的工程数据"——后者往往更有长期价值。

**3. 任何"最佳实践"都要问"在什么 scale 下"**

T5 的"encoder-decoder 最佳"结论在 ≤ 11B 成立，GPT-3 的 175B decoder-only 又把
故事改写一次。这告诉我：技术结论几乎都是 **scale-dependent** 的。读到任何
"X 比 Y 好"，第一反应应该是问"在什么参数规模 / 数据规模 / 任务类型下？"
