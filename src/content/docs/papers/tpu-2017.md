# TPU 学习笔记 — Google 的张量处理单元

## 一、从日常类比开始

假设你在一家大超市里结账。

**传统方式（CPU）：** 一个收银员，什么都会做——扫码、称重、查价格、收现金、刷卡、找零、装袋。什么都行，但慢。

**并行方式（GPU）：** 开了 100 个收银台，每个台子都跟那个传统收银员差不多。适合"有很多类似的小任务"同时做。

**TPU 的方式：** 直接修了一条传送带。商品放上去，自动扫、自动算、自动出小票。不能搬大件（不支持任意计算），但只要你扫的是标准商品（矩阵乘法），速度是传统方式的几十倍。

TPU 就是 Google 自研的"神经网计算传送带"。它不做通用计算，只专攻一件事：**深度学习中的密集矩阵乘法**。

## 二、核心概念

### 2.1 为什么需要 TPU？

深度学习的核心运算只有两种：

1. **矩阵乘法（MatMul）**：两个大矩阵相乘
2. **加法（Add）**：加上偏置项，再做激活函数

这两种运算占了神经网络 90% 以上的计算量。既然 90% 的工作都是一样的，为什么要用"什么都能干但什么都慢"的通用 CPU 呢？

**关键洞察：** 专门设计电路做矩阵乘法，效率可以提升一个数量级。

### 2.2 TPU 的心脏：Matrix Multiply Unit (MMU)

TPU 的核心是一个巨大的 **MAC（乘加）阵列**：

- 65,536 个 8-bit MAC 单元，排成 256 x 256 的网格
- 每个 MAC 单元每纳秒做一件事：`a * b + c`（乘一个数，再加一个数）
- 峰值吞吐：92 TOPS（每秒 92 万亿次运算）
- 所有 MAC 单元**同步工作**，没有分支，没有跳转——确定性极强

想象一个 256 x 256 的足球场，每个座位上坐着一个人。每秒钟，每个人同时做 `a × b + c` 的运算，所有人动作完全一致。这就是**同步数据流（SDF）模型**。

### 2.3 片上内存：28 MiB 的"工作台"

- TPU 自带 28 MiB 的 **on-chip SRAM**（不是传统 GPU 的显存）
- 这些内存是**软件管理的**——程序员需要手动把数据搬进搬出
- 没有缓存（cache），没有预取（prefetch）
- 好处：带宽极高，延迟极低，功耗极低

### 2.4 确定性执行模型

CPU/GPU 用了很多"花招"来提高平均吞吐量：

| CPU/GPU 优化 | TPU 的态度 |
|---|---|
| 缓存（Cache） | 没有，直接删了 |
| 乱序执行（Out-of-order） | 没有 |
| 多线程超线程 | 没有 |
| 分支预测 | 没有 |
| 预取（Prefetch） | 由软件显式控制 |

TPU 的逻辑很简单：深度学习的模型是固定的、确定的。既然我知道接下来要做什么，就不需要那些"猜"的花招。**确定性 = 可预测的延迟 = 适合 99 分位响应时间要求。**

### 2.5 性能数据（vs  contemporaries）

- 比同期的 Intel Haswell CPU 快约 **15x - 30x**
- 比同期的 Nvidia K80 GPU 快约 **15x - 30x**
- 能效比（TOPS/Watt）高出 CPU **200x**，GPU **30x - 80x**
- 尽管有些应用利用率很低，但整体优势巨大

## 三、核心概念详解

### 3.1 矩阵乘法：深度学习的基石

在神经网络中，每一层都在做同一件事：

```
输出 = 激活(输入 * 权重 + 偏置)
```

这就是一个矩阵乘法（输入 × 权重），加一个偏置向量，再套一个激活函数（如 ReLU）。

**直观理解：** 如果你有 1000 个输入特征，要映射到 500 个隐藏单元，你需要一个 1000 x 500 的权重矩阵。每个隐藏单元都要做 1000 次乘法 + 1000 次加法。这就是 TPU 的强项——它同时做 65,536 组这样的乘加运算。

### 3.2 软件管理的内存（Software-Managed SRAM）

TPU 的内存不像 CPU 缓存那样自动管理。你需要自己把数据从主存"搬"到 TPU 的片上 SRAM，算完后再"搬"回来。这听起来麻烦，但换来的是：

- 确定性：没有缓存命中/未命中的随机延迟
- 低功耗：SRAM 比 DRAM/显存省电得多
- 高带宽：片上内存和计算单元在同一芯片上

### 3.3 数据流架构（Dataflow Architecture）

TPU 是**同步数据流（Synchronous Dataflow）**模型：

1. 所有 MAC 单元在同一时钟信号下工作
2. 数据一旦进入 TPU，就按照预定的路径流动
3. 没有条件分支，没有循环（由宿主 CPU 控制循环）

这就像一个装配工厂：零件从一端进，经过固定的加工步骤，从另一端出。

## 四、代码示例

### 4.1 TensorFlow 中的 TPU 训练（高级 API）

在 TensorFlow 中，使用 TPU 几乎不需要写底层代码——框架自动处理了 TPU 的编程细节：

```python
import tensorflow as tf

# 1. 找到 TPU 集群
resolver = tf.distribute.cluster_resolver.TPUClusterResolver(
    tpu='my-tpu-endpoint'
)
tf.config.experimental_connect_to_cluster(resolver)
tf.tpu.experimental.initialize_tpu_system(resolver)

# 2. 创建 TPU 分布式策略
tpu_strategy = tf.distribute.TPUStrategy(resolver)

# 3. 定义模型（和普通的 Keras 模型一样！）
with tpu_strategy.scope():
    model = tf.keras.Sequential([
        tf.keras.layers.Dense(256, activation='relu', input_shape=(784,)),
        tf.keras.layers.Dense(128, activation='relu'),
        tf.keras.layers.Dense(10, activation='softmax')
    ])

# 4. 编译并训练（和往常一样！）
model.compile(
    optimizer='adam',
    loss='sparse_categorical_crossentropy',
    metrics=['accuracy']
)

# 5. 训练——TPU 自动接管
model.fit(train_dataset, epochs=10)
```

**关键点：** 在第 3 步的 `with tpu_strategy.scope():` 块中，所有在 TPU 上创建的变量和运算都会被分发到 TPU 的 MAC 阵列上执行。前面的数据加载和后面的评估可能在 CPU/GPU 上运行，框架会自动处理数据搬运。

### 4.2 手动管理 TPU 内存的示意（概念级）

TPU 的内存是软件管理的，下面用伪代码示意数据搬运的过程：

```python
import tensorflow as tf

# 模拟：从主机内存（HBM）把权重和数据"拉"到 TPU 片上 SRAM
# 实际中由 XLA 编译器自动优化，这里仅作概念演示

def tpu_matmul_concept(A, B):
    """
    A: 输入矩阵 [batch, 256]  -- 在主机 HBM 中
    B: 权重矩阵 [256, 256]    -- 在主机 HBM 中
    返回: 结果 [batch, 256]    -- 由 TPU 计算后送回 HBM
    """
    # 第一步：把数据搬到 TPU 的片上 SRAM
    # 在真实 TPU 上，这由 XLA 编译器生成的 DMA 指令完成
    a_on_chip = tf.tpu.rewrite(A)   # DMA: HBM -> SRAM
    b_on_chip = tf.tpu.rewrite(B)   # DMA: HBM -> SRAM

    # 第二步：在 TPU 的 256x256 MAC 阵列上做矩阵乘法
    # 65,536 个 MAC 单元同时工作，同步数据流模型
    result_on_chip = tf.tpu.matmul(a_on_chip, b_on_chip)

    # 第三步：把结果搬回主机 HBM
    result = tf.tpu.rewrite(result_on_chip)  # DMA: SRAM -> HBM

    return result

# 一个更贴近实际的例子：完整的前向传播
def tpu_forward_pass(inputs, weights, biases):
    # 全连接层：output = relu(inputs @ weights + biases)
    matmul_result = tf.matmul(inputs, weights)    # TPU 矩阵乘法
    add_result = tf.add(matmul_result, biases)     # TPU 向量加法
    output = tf.nn.relu(add_result)                # TPU 激活函数
    return output
```

**重点：** 在真实使用中，你不需要写 `tf.tpu.rewrite()` —— XLA（Accelerated Linear Algebra）编译器会自动分析你的计算图，决定哪些数据该搬到 SRAM，何时搬，以及 MAC 阵列的最优调度方式。

### 4.3 从 CPU 视角看 TPU 的编程模型

类比 CPU 上写 C 程序：

```c
// CPU 上：你写循环，编译器/硬件决定如何优化
for (int i = 0; i < N; i++) {
    for (int j = 0; j < N; j++) {
        C[i][j] += A[i][k] * B[k][j];  // 编译器可以自动向量化、用缓存、乱序执行
    }
}
```

在 TPU 上，你写 TensorFlow 代码，XLA 编译成**数据流图**，TPU 硬件按照预定的调度执行：

```
[输入数据] --> [DMA搬运到SRAM] --> [MAC阵列计算] --> [DMA搬回HBM] --> [输出结果]
```

没有循环、没有分支、没有条件。一切都是流水线式的。

## 五、TPU 的架构总结

```
                    +--------------------------+
                    |      TPU Chip            |
                    |                          |
  主机 HBM <--DMA--> |  28 MiB 片上 SRAM       | <--->  256x256 MAC 阵列
                    |  (软件管理, 无缓存)       |      (65,536 个 8-bit MAC)
                    |                          |
                    |  Control Logic (SDF)     |
                    +--------------------------+
                            |
                    +-------+-------+
                    |  Host CPU     |
                    | (TensorFlow)  |
                    |  负责循环、   |
                    |  数据预处理、 |
                    |  控制逻辑     |
                    +---------------+
```

## 六、为什么 TPU 这么"简单"却这么强？

回顾一下：

1. **砍掉所有"花招"：** 没有缓存、没有乱序执行、没有分支预测
2. **专注于矩阵乘法：** 65,536 个 MAC 单元组成的 256 x 256 阵列
3. **软件管理内存：** 程序员（通过 XLA）显式控制数据搬运
4. **确定性执行：**  predictable latency，适合生产环境

这就好比：
- CPU = 一辆跑车，什么路都能跑，但赛车场上不一定最快
- GPU = 一辆多座巴士，能拉很多人，但每人的体验不一样
- TPU = 一条高铁轨道，只能跑固定线路，但那条线路上的速度无人能敌

## 七、一句话总结

**TPU 的核心设计哲学：既然深度学习的计算模式是高度规律且确定的，那就造一个专门做这件事的"传送带"，砍掉所有通用计算需要的花招，换来几十倍的性能提升和能效优势。**
