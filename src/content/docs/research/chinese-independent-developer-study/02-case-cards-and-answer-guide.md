# 真实案例与答案检查

> 目的：把“看懂架构”升级为“能根据证据分诊真实状态”。
>
> 所有案例只使用公开 GitHub 证据，不推断私有 Routine 内部行为。

## 案例 1：普通 PR 成功进入主版面

**上游锚点：**

- PR：[1c7/chinese-independent-developer#1133](https://github.com/1c7/chinese-independent-developer/pull/1133)
- merge commit：`58185a2de07cd2aea247b06fb689f08f8555d884`
- 时间：2026-07-16
- 证据等级：E3 外部结果

### 输入

贡献者提交“记账tracker”，PR 只修改 `README.md`：

```text
+3 行 / -0 行
```

### 输出

PR 在约 6 分钟后以 squash 方式合并，主版面新增：

- 制作者与公开链接。
- 产品 URL。
- 一句话功能描述。

### 这能证明什么

- 贡献者 PR 路径可以产生一个干净的单提交结果。
- 条目格式与主版面 schema 一致。
- merge commit 保留 PR 和贡献者的公开关联。

### 不能证明什么

- 公开数据不能证明是 Routine、维护者还是其他自动化执行了 merge。
- 没有公开评论，不能证明“感谢评论 + 去签名回读”规则在本 PR 执行。
- 没有测试结果，不能证明 URL 唯一、分类一定正确或站点链接可访问。

### 初学者应该学到

**政策符合不等于执行来源已证明。** 结果看起来符合 Skill，不代表一定是 Skill 执行的。

## 案例 2：新增后仍需要分类纠正

**上游锚点：**

- commit：[`30c6e09a`](https://github.com/1c7/chinese-independent-developer/commit/30c6e09a30edadb82ea841138c6436c606659ef0)
- 时间：2026-07-17
- 证据等级：E3 外部结果

### 输入状态

7 个与具体游戏相关的规划或计算工具最初出现在主版面，包括：

- 宝可梦 Nuzlocke 指南。
- 原神伤害计算器。
- 星穹铁道跃迁计算器。
- Minecraft 附魔与像素画工具。

### 修正动作

维护提交做了两件事：

1. 把 7 个条目从主 `README.md` 移到 `pages/README-Game.md`。
2. 调整 IPTV Player 描述，把产品类型和主要功能提前。

### 为什么值得研究

当前 Skill 已经写明“任何游戏类产品进入游戏版面”，但真实新增结果仍需要后续整理。可能原因有多种：

- 投稿本身分类不清。
- PR 路径原样合并，没有走通用格式化流程。
- Agent 对“游戏工具是否算游戏”判断不一致。
- 规则是在问题出现后才补强。

公开证据不能单独确定是哪一种。

### 初学者应该学到

**自然语言分类需要反例集。** 只有规则文本，没有历史错例和期望输出，Agent 很难稳定处理边界案例。

## 案例 3：定时器正常，外部认证失败

**上游锚点：**

- Action：[run 29545709871](https://github.com/1c7/chinese-independent-developer/actions/runs/29545709871)
- head：`30c6e09a30edadb82ea841138c6436c606659ef0`
- 时间：2026-07-17 00:46 UTC
- 证据等级：E3 外部结果

### 观察

```text
schedule 触发
runner 启动
curl 调用 Routine API
API 返回 authentication_error
workflow 检测到 type:error
退出码 1
```

### 正确分诊

| 层 | 状态 |
|---|---|
| GitHub schedule | 正常 |
| runner 与 shell | 正常 |
| HTTP 请求发出 | 正常 |
| Routine API 认证 | 失败 |
| Skill 加载 | 未开始 |
| 评论/Issue/PR 扫描 | 未开始 |
| README 写入 | 未开始 |
| 感谢评论 | 未开始 |

### 错误分诊

- “Skill 分类逻辑坏了”。
- “GitHub Actions 没有定时触发”。
- “README 写入冲突”。
- “旧 Python 脚本测试没通过”。

这些假设都位于失败点之后或另一条链上，没有当前证据支持。

### 初学者应该学到

**先找最后一个成功边界和第一个失败边界。** 不要从最终没产出直接猜最深层代码故障。

## 三个案例放在一起

| 案例 | 输入 | 最后成功边界 | 主要问题 |
|---|---|---|---|
| PR #1133 | 贡献者 PR | merge commit | 结果成功，但执行者与完整 Skill 步骤未证明 |
| commit `30c6e09a` | 已新增条目 | Git commit | 分类和文案需要补偿性整理 |
| Action `29545709871` | 定时触发 | HTTP 请求发出 | 外部 token 认证失败 |

它们分别对应：

```text
成功路径的证据边界
业务规则的校正闭环
外部依赖的失败分诊
```

## 应用题

先独立回答，再看后面的检查点。

### 题 1

Action 绿色，但 30 分钟后 README 没变化。列出至少四种互不等价的解释。

### 题 2

一个贡献者 PR 把“需要命令行安装的 CLI 工具”加到主版面。最小正确处理是什么？为什么不能只留言让贡献者重做？

### 题 3

数据探针发现同一 URL 在主版面和程序员版面各出现一次。能否自动删除主版面那条？

### 题 4

Skill 用完整 URL 精确匹配。如果两个投稿分别是：

```text
https://example.com
https://example.com/?utm_source=github
```

当前规则会怎样？你会怎样改进而不误伤？

### 题 5

两个 Routine 会话同时从同一个 `master` 开始，把不同条目插入当天日期区块。即使每次单文件写入都完整，为什么仍可能冲突或丢更新？

### 题 6

旧 Python 脚本 4 个单测通过，能否把它声明为当前 Skill 的灾备？还缺哪些证明？

## 答案检查点

### 题 1

至少应区分：

- 本轮没有新输入，Routine 正常提前退出。
- 触发请求被接受，但下游 Routine 失败。
- URL 被判定为已存在。
- 输入被判为拒绝类。
- Git push 或 merge 失败。
- README 已更新，但观察的是旧缓存或错误分支。

只看绿色触发 Action 无法选择其中一个。

### 题 2

检查 PR 内容后，把条目移动到程序员版面，再按贡献者 PR 路径合并并保留归属。当前 Skill 把冲突处理和分类责任放在维护自动化侧，不应把普通整理成本转嫁给投稿者。

### 题 3

不能。先确认：

- 两条是否描述同一产品形态。
- 是否正在做版面迁移。
- 一个 URL 是否同时提供网页产品和开发者工具。
- 历史链接和当前链接是否需要保留。

重复探针提供候选，不提供删除授权。

### 题 4

精确字符串匹配会把它们当成不同 URL。改进可以先做 URL canonicalization：

- 主机名转小写。
- 规范尾部 `/`。
- 只移除明确允许忽略的 tracking 参数。
- 保留影响资源身份的 query。
- 对 canonical 后冲突仍只报警，不自动删除。

### 题 5

因为这是跨读写周期的并发问题：

```text
A 读取旧 master
B 读取旧 master
A 写入并 push
B 基于旧版本写入并 push
```

Git 可能拒绝 B，也可能在自动冲突处理不严谨时丢掉一方内容。需要乐观锁、重新拉取、冲突合并或串行队列，不是“文件不会半写”就能解决。

### 题 6

不能。还缺：

- 当前三类输入的等价覆盖。
- 三版面分类。
- URL 幂等规则。
- PR squash merge 与贡献者归属。
- 评论签名清理和回读。
- 当前 Skill 的离线案例。
- 真实 GitHub 对象的非破坏性集成测试。

## 类别验收

- [x] 唯一项目与 upstream 已确认。
- [x] 项目定位、主链、取舍和限制已有源码依据。
- [x] 锚点项目有输入到输出追踪。
- [x] 当前 Skill 与旧 Python 方案完成对比。
- [x] 零基础导读已提供术语、类比和学习路线。
- [x] 4 个本地单测通过，Markdown 数据探针可复现。
- [x] 6 个应用题带答案检查点。
- [x] 当前 upstream 增量和 Routine 失败已重新核验。

剩余风险：未运行真实 Routine，不接触仓库 secret，不对外部产品链接做全量可用性检查。
