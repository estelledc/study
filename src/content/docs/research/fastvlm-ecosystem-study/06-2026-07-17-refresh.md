## 3. 唯一漂移：MLX Swift LM

GitHub compare：

```text
ahead=1
behind=0
files=3
```

提交：

```text
Gemma 4: aspect-preserving image resize with per-image dynamic token counts;
fix multi-image requests
```

关键变化：

1. 不再把所有图片固定 resize 到 800×800；
2. 按输入长宽比和 `max_soft_tokens` 计算目标尺寸；
3. 宽高都对齐 `pooling_kernel_size * patch_size`；
4. pooled token 数由每张图真实 patch grid 动态计算；
5. 多图先 pad 到共同 canvas，再按 `frames` 切回真实尺寸；
6. prompt 中第 i 个 image placeholder 展开成第 i 张图的 soft token 数；
7. image 数量与 placeholder 数量不匹配时 fail closed；
8. 新增 12 组普通、极端和相机尺寸测试。

这不改变 FastVLM 主结论，但修正了一个常见误区：

> 一个 VLM 的 visual token count 不一定是模型级常量；它可能随图片长宽比、
> patch size、pooling kernel、max token budget 和多图数量动态变化。

本轮只做 upstream E1 diff，没有在 `FETCH_HEAD` 构建 Swift package 或运行新增测试。

## 4. 四种预算路线

最小实验固定：

```text
image = 768 x 768
text tokens = 64
decoder layers = 24
```

### 普通 Patch ViT

```text
(768 / 16) * (768 / 16) = 2304 visual tokens
```

视觉塔和 LLM 都面对长视觉序列。

### 层次化输出

用 32 像素等效输出 stride 表示逐级下采样：

```text
(768 / 32) * (768 / 32) = 576 output tokens
```

它是结构 proxy，不是 FastViTHD 精确 FLOPs。

### Projector Pool

视觉塔仍处理 2304 patch，projector 再压到 144：

```text
vision work proxy = 2304
LLM visual tokens = 144
```

它降低 prefill，不抹去已经发生的视觉编码成本。

### Decoder Prune

LLM 前 8 层处理 576 visual token，后 16 层只保留 128：

```text
vision work proxy = 576
early LLM cost remains
late LLM sequence shrinks
```

FastV 类方法不减少视觉塔和 prune layer 之前的成本。

```bash
cd explorations/research/fastvlm-ecosystem-study/labs
PYTHONDONTWRITEBYTECODE=1 python3 edge_vlm_budget.py
```

结果：

```text
patch_vit: vision_tokens=2304 llm_initial=2304 llm_late=2304 pair_proxy=134578176
hierarchical: vision_tokens=576 llm_initial=576 llm_late=576 pair_proxy=9830400
projector_pool: vision_tokens=2304 llm_initial=144 llm_late=144 pair_proxy=1038336
decoder_prune: vision_tokens=576 llm_initial=576 llm_late=128 pair_proxy=3866624
measurement: ttft_ms=74.0 decode_tps=50.0
camera: next=frame-3 dropped=2
```

`pair_proxy` 是每层序列长度平方的教学 proxy，只用于解释方向。它忽略：

- hidden size；
- attention head；
- MLP；
- kernel fusion；
- memory bandwidth；
- Core ML/MLX execution unit；
- quantization；
- dynamic shape overhead。

不能把 proxy 数字写成 FLOPs、毫秒或能耗。

## 6. Measurement Contract

真实设备记录必须包含：

```text
run_id
model_id + model_commit
device + OS
image_id + prompt_id
cold_start
thermal_state
max_new_tokens + temperature
peak_memory_mb
```

TTFT 分阶段：

```text
image_decode
preprocess
vision
projector
prefill
first_decode
```

实验中的 74ms 是合成数据，用来验证求和和字段门，不是任何 Apple 设备或模型的
实测结果。

比较函数会拒绝：

- 冷启动与热启动混比；
- max output 不同；
- prompt/image 不同；
- thermal state 不同；
- 缺 device/OS/commit；
- 缺任一 TTFT stage。

## 7. 连续相机背压

`LatestFrameBuffer` 只保存最新帧：

```text
push frame-1
push frame-2 -> drop frame-1
push frame-3 -> drop frame-2
pop -> frame-3
```

这与 FastVLM App 的 `bufferingNewest(1)` 同一原则：

> 对实时视觉，处理过期帧通常比丢帧更糟。

但 one-slot buffer 只解决队列陈旧，不解决采样策略、场景去重、温升和取消。

## 10. 稳定结论

1. 端侧 VLM 优化要说明成本在哪一步减少，不能只报最终 token 数。
2. Projector 压缩只减少 LLM 输入，视觉塔成本已经支付。
3. Decoder pruning 只减少后续层，不减少视觉塔和前 K 层。
4. 图片长宽比和多图请求会让 token count 动态变化。
5. TTFT 必须拆分 image/preprocess/vision/projector/prefill/first decode。
6. 冷热启动、thermal state 和 output contract 不一致时不能比较。
7. 持续相机需要背压、取消和热稳定性，单图 benchmark 不够。
8. 静态 proxy 用于架构思考，真机结论必须来自真实设备测量。

## 11. 仍未证明

- FastVLM 权重在本机或 iPhone 的推理。
- Core ML 视觉塔实际落到 CPU/GPU/ANE 的比例。
- 21 个项目的模型下载、构建和 benchmark。
- OCR、计数、小物体和空间定位在压缩下的质量。
- 10 分钟连续相机的温升、功耗、帧率和降频。
- Gemma 4 新 resize 实现在 Swift package 中的本地测试结果。
