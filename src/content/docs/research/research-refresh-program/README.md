# 第三方源码深度研究重做计划

> 启动日期：2026-07-17
>
> 目标：重新核验 `explorations/research` 的全部正式研究项目，并把每个类别收束成零基础实习生能看懂、能动手、能自测的学习包。
>
> 参考结构：[`estelledc/rhino-bird-2026`](https://github.com/estelledc/rhino-bird-2026)，本轮参考快照为 `9008f225d6918986f2c1a2c8827ca975b180cf25`。

## 结论先行

这不是“把现有 147 篇类别专题再扩写一遍”。本轮要把研究完成度从“有材料”改成可审计的五件事：

1. 每个正式项目都能映射到 canonical upstream、固定提交和至少一个类别。
2. 每个项目都有源码支持的定位、主链、关键取舍、限制和推荐入口。
3. 每个类别都有从直觉到技术定义的学习路线，而不只是横向摘要。
4. 每个类别至少有一个最小动手实验，或明确记录无法运行的环境 blocker。
5. 结论、测试、真实运行和上游结果分级标注，不能把静态阅读写成实测。

## 从犀牛鸟项目复用什么

参考项目最有价值的不是文档数量，而是下面这条交付链：

```text
Landscape
  -> Track deep dive
  -> Project deep dive
  -> Beginner guide
  -> Hands-on lab
  -> Case / Issue / contribution evidence
  -> Blocking quality gates
```

本轮直接复用五个原则：

1. **三层研究结构**：先画领域地图，再解释类别级机制，最后落到单项目源码。
2. **分级学习路径**：读者知道前置知识、建议顺序、预计投入和完成标准。
3. **最小实验**：至少跑通一条可观察的输入到输出链，而不是只给安装命令。
4. **案例证据**：案例必须有上游锚点、定位过程、修复前后或方案前后对照。
5. **数字门禁**：项目数、章节数、案例数和实际文件保持一致，不能手工漂移。

不复用“篇数越多越好”的隐含假设。类别完成度由证据和实践闭环决定，不由字数决定。

## 每个类别的最终合同

现有文件可以继续复用，不强制改名；但最终必须能映射到以下八类工件：

| 工件 | 必须回答的问题 |
|---|---|
| 最终接班页 | 30 分钟内应该先读什么、记住什么、下一步做什么？ |
| 领域全景 | 这个类别解决什么问题，生态分几层，边界在哪里？ |
| 逐项目深析 | 每个项目的定位、输入输出、主控制流、关键源码、取舍和限制是什么？ |
| 横向比较 | 为什么不同项目选择不同方案，实习生应该如何选？ |
| 零基础导读 | 前置知识、术语、类比、学习顺序和掌握等级是什么？ |
| 最小实验 | 用什么命令产生什么可观察结果，失败时如何分诊？ |
| 案例与自测 | 能否把机制用于新场景，而不是复述原文？ |
| 来源与快照 | upstream、fork、commit、许可证和验证边界是否可复查？ |

### 单项目最低内容

每个唯一项目至少包含：

1. 一句话定位和日常类比。
2. 输入、输出与 3-8 步主控制流。
3. 技术栈、入口文件和 2-5 个关键源码锚点。
4. 一个关键设计选择，以及它解决的问题和付出的代价。
5. 项目最强项、明显限制和不适用场景。
6. 当前证据等级和未验证声明。
7. 零基础读者的第一个阅读任务或动手任务。

只有类别锚点项目要求追到核心函数和关键测试；索引、数据集、模板仓等按其真实角色分析，不强行套用 Agent loop。

## 证据等级

| 等级 | 名称 | 可写成什么 |
|---|---|---|
| E0 | 项目自述 | “README 声称……” |
| E1 | 静态源码 | “固定提交中的类型、调用链或测试表明……” |
| E2 | 本地验证 | “命令在当前环境运行并得到……” |
| E3 | 外部结果 | “真实 PR、Issue、部署或用户流程显示……” |

规则：

- E0 不能升级成“系统已经实现”。
- E1 不能升级成“当前机器可运行”。
- E2 必须记录命令、结果和未覆盖风险。
- E3 必须给出可复查的上游锚点。

## 类别完成门

一个类别只有同时满足以下条件才标记为完成：

- [ ] 正式成员清单与 canonical upstream 已去重。
- [ ] 每个唯一项目达到单项目最低内容。
- [ ] 锚点项目至少完成一条端到端源码追踪。
- [ ] 横向比较覆盖机制、代价和适用边界。
- [ ] 零基础路线从类比进入术语，并提供前置知识。
- [ ] 至少一个 E2 最小实验通过，或 blocker 有复现证据和替代练习。
- [ ] 至少三个应用型自测题带答案检查点。
- [ ] 数字、内部链接、快照和验证命令通过门禁。

“已有 README”“已有逐项目章节”或“文档很长”都不能单独通过完成门。

## 有界推进合同

长期目标由多个有限 Cycle 组成，主线程持续负责审计和整合。

### Program

- `status`：active
- `scope`：14 类、201 个唯一上游项目
- `activated_by`：用户 2026-07-17 `/goal`
- `review_after`：每完成 1 个类别，或连续 3 个 writer cycle
- `external_outcome`：仓库中存在可学习、可运行、可复查的类别学习包
- `superseded_by`：暂无

### Writer Cycle

- `objective`：只关闭一个明确的类别缺口。
- `scope`：最多 3 个项目深析，或 1 个类别级实验/接班页。
- `acceptance_checks`：相关命令 + `make lint`；跨层变更再跑 `make check`。
- `budget`：单次不超过 60 分钟，不扩展到新的外部主题。
- `external_outcome`：覆盖矩阵至少一个单元格产生证据等级提升。
- `stop_conditions`：
  - 连续 3 个 cycle 没有覆盖或证据等级变化；
  - 出现超过 1 个工作日未跟进的外部 blocker；
  - 需要付费 API、GPU、账号或发布权限，且没有本地替代验证；
  - 当前类别尚未验收时，不开启第二个类别写入。

阶段性最低验证：

```bash
make lint
python3 scripts/explorations/restore-projects.py --audit
git diff --check
```

全部类别完成后的最终验证：

```bash
python3 src/content/docs/research/research-refresh-program/completion_audit.py \
  --check-worktrees
make check
git diff --check
```

确定性覆盖审计检查类别成员数、项目锚点、实验、自测工件和正式源码工作树，设计参考
`rhino-bird-2026/scripts/check-stats.rb` 与 `audit-case-quality.py`。
