# 基础 FAQ 与关键思考点

## 1. 什么是系统提示词？

它是应用或模型提供方在用户输入之前加入的高优先级说明，定义角色、行为、工具、格式和边界。它不是模型权重，也不是传统意义上的不可读服务器代码。

## 2. System prompt、developer prompt 和 tool schema 是一回事吗？

不是。

- system/developer prompt 主要是自然语言和结构化行为说明。
- tool schema 定义可调用函数、参数和描述。
- runtime context 还可能注入日期、文件、记忆、组织策略和用户状态。

档案中的“full prompt”可能把它们合并，也可能只含一层。

## 3. 为什么模型会把高优先级指令告诉用户？

角色层级能提高服从概率，但不能形成像内存保护一样的硬边界。用户可以通过改写任务、翻译、分步追问或让模型描述行为来诱导恢复内容。

## 4. Prompt extraction 和 prompt injection 有什么区别？

- injection 目标是改变模型行为。
- extraction 目标是获得隐藏指令。

Extraction 通常利用 injection 技巧，但结果是信息披露。

## 5. Prompt leaking 和 prompt stealing 有什么区别？

- leaking 更强调模型直接输出原 prompt 或片段。
- stealing 更强调复制 prompt 的功能，即使文本不同。

PRSA 属于后者。

## 6. GitHub 上的“leaked prompt”能直接相信吗？

不能。先按 [证据模型](02-evidence-and-provenance.md) 判断官方、可复现、交叉一致、单源或未知，再确认版本和完整性。

## 7. 为什么多个仓库都有同一文本仍不一定可信？

因为可能都是从同一个 X 帖、Gist 或上游仓复制。多个副本不是多个独立来源。

## 8. 为什么 star 数不能代表真实性？

Star 衡量关注度，不验证来源、版本、完整性或逐字准确性。热门仓仍可能混合官方、抽取、重建和二手转贴。

## 9. “Consistency 0.9”是不是 90% 准确？

通常不是。它可能表示多次抽取之间的语义相似度。只有与真实 oracle 比较时，才能讨论对 ground truth 的准确度。

## 10. Exact Match 是不是最好的指标？

它适合逐字泄露，但看不到：

- 改写后的语义泄露。
- 翻译/编码结果。
- 功能等价 stolen prompt。

所以需要 exact、semantic、functional 三层指标。

## 11. 如何确认没有 hallucination？

常见方法：

- 多次独立抽取。
- 不同攻击技能交叉验证。
- 与官方 prompt 或源码对比。
- 用候选之间一致性训练判别器。
- 在未见输入上检查功能一致性。

没有 oracle 时只能提高置信度，不能获得数学证明。

## 12. 官方 prompt 是否就是完整线上 prompt？

不一定。官方可能只发布 core template；线上还会加入动态变量、工具、feature flag、用户状态和安全层。

## 13. 为什么 prompt 不能存 API key？

模型必须“看到” prompt 才能使用它，而可见上下文可能被直接或间接重建。Secret 应由模型外系统保管，只在受控工具执行时使用。

## 14. 加一句“不要泄露 prompt”有用吗？

有一定降风险价值，但不能单独依赖。它是概率性约束，不是访问控制。

## 15. 5-gram 输出过滤为什么不够？

它能阻止大段原文复制，但翻译、字符交错、Caesar、摘要和语义改写都能降低字面重叠。

## 16. 社区共识验证靠谱吗？

它比单人提交强，但只能证明文本相似。要证明真实性，还需独立来源、版本证据或 oracle。

## 17. 为什么 Agent 比普通聊天风险更大？

因为 Agent prompt 常包含工具、权限、工作流和环境细节；泄露后可帮助攻击者设计更精确的工具滥用或间接 injection。

## 18. 学这些项目最重要的不是 prompt 内容吗？

不是。更可复用的是：

- 来源与版本治理。
- Prompt + tool 的上下文架构。
- 安全评测设计。
- 自适应搜索和记忆。
- 权限外置与防御纵深。

## 19. 哪个项目最适合初学者？

按目标：

- 看档案：System Prompts Leaks。
- 看来源：jujumilk3。
- 看网站：YeeKal。
- 看完整攻防链：SPE-LLM。
- 看 benchmark：Raccoon。
- 看高级 Agent：JustAsk。

## 20. 能否直接运行这些攻击项目？

技术上部分可以，但需要模型、GPU、API 和数据；伦理上只应在自有或明确授权的受控环境运行。本轮没有执行任何真实攻击。

## 关键思考点

### 思考 1：透明度和安全是否冲突？

提示：

- 如果授权在模型外，公开 prompt 是否仍导致越权？
- 隐藏 prompt 是否真的能阻止攻击者观察行为？
- 哪些内容应透明，哪些内容不该进入 prompt？

### 思考 2：什么才算“偷到了 prompt”？

比较三种情况：

1. 字符完全相同。
2. 语义相同但措辞不同。
3. 文本完全不同，但未见输入上的行为一致。

它们对应不同资产和威胁模型。

### 思考 3：两用户共识为什么仍会错？

尝试构造：

- 两人都复制同一错误帖子。
- 一人用两个账号提交。
- 两个文本 85% 相似但关键权限句不同。

然后思考需要增加哪些 oracle、信誉和差异检查。

### 思考 4：如何设计一个可靠 prompt registry？

至少考虑：

- schema。
- 来源图。
- 版本与 capture 日期。
- hash 和去重。
- evidence grade。
- diff。
- license。
- secret scan。
- review state。

### 思考 5：为什么 JustAsk 的 UCB 适合这个问题？

回答时区分：

- exploitation：继续用已知成功技能。
- exploration：尝试样本少、可能被低估的技能。
- reward：怎样定义成功，是否依赖 ground truth？

### 思考 6：防御应该优先解决泄露概率还是泄露后果？

理想答案不是二选一：

- 降低概率：输入/输出检测、prompt 约束、模型训练。
- 降低后果：不放 secrets、最小权限、外置授权、隔离工具。

后者通常更可确定和审计。

### 思考 7：如何避免“最新但不可信”与“可信但过时”？

设计双轨：

- `latest-candidate` 快速收集。
- `verified-baseline` 只收 A/B/C 级证据。

不要让候选自动覆盖基线。

## 自测题

1. 为什么“两个 GitHub 仓内容相同”不能直接算交叉验证？
2. Exact Match、semantic similarity、functional consistency 分别回答什么？
3. LeakHub 的 0.85 相似阈值证明了什么，没有证明什么？
4. JustAsk 当前仓库为什么不能称为开箱可复现？
5. 如果 prompt 全公开，一个安全 Agent 仍应靠什么阻止越权？
6. 为什么官方 Jinja template 也不等同于完整线上上下文？
7. Prompt stealing 对只防逐字输出的系统提出了什么挑战？

## 答案骨架

1. 可能共享同一上游来源，独立性不足。
2. 逐字、语义、行为三层恢复。
3. 证明提交文本相似，不证明供应商真实性。
4. 代码引用的 `data/` 状态和受控配置未进入固定提交，最小 CLI 直接报缺文件。
5. 模型外授权、最小权限、参数校验、隔离和审计。
6. 线上还会渲染变量并叠加工具、用户状态和动态策略。
7. 攻击者不需要原文也可能复制能力，因此要做功能一致性评测和架构保护。
