## 3. 八个漂移仓的影响

### 3.1 Hermes：Memory 不能拖死主循环

122 个提交里，与本类别直接相关的变化包括：

- 外部 `MemoryProvider.prefetch()` 增加默认 8 秒边界；
- 同一 provider 的上次 prefetch 卡住时，下一轮跳过而不是继续堆线程；
- Honcho 增加 first-turn wait、context/dialectic cadence、query rewrite 配置；
- memory query rewrite 从具体 provider 抽成通用模块；
- SessionDB 遇到重复 title 时保留所有 session，只清理旧 alias；
- API callback verifier 放到 event loop 外，并在异常时 fail closed；
- webhook 的非 ASCII 签名不再触发 `compare_digest` 崩溃；
- Cron fallback API key 走统一 resolver。

这组变化支持一个关键判断：

> Memory 是外部依赖，不应拥有 Agent turn 的活性；失败可以降级，污染不能静默写入。

### 3.2 Mnemosyne：写入审批成为显式入口

增量加入 `mnemosyne_apply_pending`：

```text
remember/batch
  -> pending/memory
  -> human or policy review
  -> apply_pending
  -> BEAM write path
```

同时修复：

- ONNX runtime 显式线程数，避免受限容器 affinity 噪声；
- embedding API 429/5xx 加 jitter 的有界重试；
- polyphonic recall 从结果 content 计算相似度；
- Hermes integration 暴露 pending write tool。

它让“Memory admission gate”从研究建议变成可见接口，但真实权限、并发审批和恶意
pending payload 仍需 E2E。

### 3.3 OpenClaw：长期运行首先是 ownership 问题

本轮复用 LambChat 类的有界审计结果：

- durable channel ingress；
- claim lease、owner identity 和 stale recovery；
- per-lane serialization；
- adoption、retry、dead-letter 和 stall watchdog；
- workspace/hook 状态迁入 SQLite；
- transcript、compaction 和 subagent settle 修复。

这些机制不是“更聪明”，而是避免消息丢失、重复执行和错误恢复。它仍是个人 Agent
runtime，不能因 durable state 就写成组织级 SaaS。

### 3.4 nanobot 与 Lethe：核心结论未改变

- nanobot 只有 Web UI folder picker，Python turn state machine 未命中。
- Lethe 只有 repo 迁移和 browser-daemon error 呈现，Actor snapshot/长期任务主链未变。

### 3.5 ZeroClaw：channel turn 进入统一 lifecycle

增量让 channel-originated turn 也发出成对的 Agent start/end 事件，并用 drop-safe guard
覆盖提前退出。另有：

- per-channel debounce；
- WhatsApp/WeChat persisted identity；
- webhook 无 secret 时拒绝启动；
- capability-safe quickstart 默认值；
- Semgrep/CodeQL 门禁。

可观测事件是恢复和审计的前提，但事件存在不等于 side effect exactly-once。

### 3.6 IronClaw：预算与认证都走共享合同

增量把：

- OAuth fake/durable 实现放到同一 conformance suite；
- model cost 移到共享 crate；
- unknown paid model 使用非零保守成本；
- cache read、cache creation 和 output 分项计费；
- per-run usage/cost 投影到 WebChat。

这比把未知成本静默记为零更安全。成本表仍是软件快照，不能替代 provider 账单。

### 3.7 Letta Code：subagent 也需要持久 identity

增量补充：

- subagent `conversation_id` 进入 state snapshot；
- stream-json 尾部截断被显式识别，避免静默丢结果；
- runtime start 保存 Skill source selection；
- app server 与 websocket 路径使用同一 Skill 来源；
-慢文件下载有 timeout。

这说明“子 Agent 返回了一段文本”不是完整合同，还需要 identity、完整流和父子状态关联。

## 6. 三张失败卡

### 卡 1：进程重启后任务从头做

根因不是模型忘记，而是“下一步”只存在 prompt/transcript。

最小修复合同：

```text
task_id + status + next_step + lease + append-only events
```

### 卡 2：工具输出直接写入长期 Memory

工具文本可能过时、恶意或属于另一个用户。正确路径是：

```text
candidate + source
  -> verify / dedupe / permission
  -> accept or reject
```

### 卡 3：新 Skill 单次成功后覆盖旧版

一次成功可能是随机波动。需要 baseline、held-out case、minimum gain、snapshot 和
rollback。没有这些证据，只能叫“自动修改”，不能叫“持续改进”。

## 7. 稳定结论

1. 长期任务需要 durable next step，不应只靠 transcript。
2. 同一任务同一时刻只有一个 live lease owner。
3. Memory 写入比检索更危险，必须有 source 和 admission。
4. Skill 优化必须保留 snapshot，并能 promote/revert/rollback。
5. Scheduler 触发不等于 job 成功，恢复和 delivery 要分开。
6. Background subagent 脱离当前 turn 不等于跨进程 durable。
7. Prompt cache 会影响能力热更新策略，不能随意重建系统前缀。
8. “Self-improving”必须说明更新对象、metric、验证集和回滚。

## 8. 仍未证明

- 真实模型 API 下的 Hermes 完整 Agent loop。
- Gateway、多 channel 和 Cron 的进程崩溃恢复 E2E。
- Honcho/Mnemosyne 等外部 Memory 的真实延迟、权限和一致性。
- 任一 Skill optimizer 在 held-out 真实任务上长期提升。
- Docker/WASM/remote sandbox 对恶意代码的完整隔离。
- 22 个项目的全套测试和统一 benchmark。
