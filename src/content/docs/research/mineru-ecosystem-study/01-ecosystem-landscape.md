# 生态与发展现状

## 先建立直觉

传统 PDF 文本提取像“把书里的字符逐个抄出来”，文档解析则像“重新排版整本书”。后者不仅要认识字，还要回答：

- 哪些字属于同一行、同一段和同一章节？
- 两栏内容应该先读左边还是右边？
- 表格的合并单元格、行列关系是什么？
- 公式应该输出 Unicode、图片还是 LaTeX？
- 页眉页脚要不要保留？
- 跨页表格和断开的段落如何合并？
- 每段内容在原页的坐标是什么，能否用于可追溯引用？

因此，“PDF 转 Markdown”不是一个算法，而是一条由文件解析、计算机视觉、OCR、语言模型、几何后处理和工程调度共同组成的流水线。

## 领域边界

以下概念经常被混在一起：

| 概念 | 输入 | 主要输出 | 重点 |
|---|---|---|---|
| 原生文本提取 | 数字 PDF | 字符、坐标 | 读取 PDF 内部文本对象 |
| OCR | 页面图像 | 文本、文字框 | 从像素识字 |
| 版面分析 | 页面图像 | 区域类别、边界框 | 判断标题、正文、表格、图片等 |
| 文档解析 | PDF/Office/图像 | 结构化文档 | 组合文本、版面、表格、公式和阅读顺序 |
| 结构化字段抽取 | 文档 + 目标 schema | JSON 字段 | 找出发票号、金额、合同甲方等业务字段 |
| 文档 ETL | 多格式文档 | 语义元素、chunk、metadata | 为检索、向量库和 Agent 准备数据 |

MinerU、Docling、Marker 属于文档解析；Unstructured 更偏文档 ETL；MarkItDown 更偏轻量格式转换；字段抽取模型不等同于文档解析器。

## 完整技术栈

一条成熟主链通常包含九层：

1. **输入与安全检查**：识别文件类型、页数、损坏页、加密和大小限制。
2. **原生内容读取**：从 PDF/Office 内部结构提取已有文本、样式、图片和元数据。
3. **页面渲染与预处理**：控制 DPI、方向校正、去畸变、裁剪和分辨率。
4. **版面检测**：识别正文、标题、表格、公式、图片、页眉页脚等区域。
5. **区域内容识别**：文本 OCR、表格结构、公式 LaTeX、图表理解。
6. **阅读顺序与关系恢复**：排序区域、合并行段、建立标题层级。
7. **文档级后处理**：跨页表格、断段、页眉页脚、重复内容和置信度处理。
8. **中间表示与导出**：统一 JSON/对象树，再渲染 Markdown、HTML、chunks。
9. **运行时与评测**：批处理、GPU 调度、失败重试、指标、可视化和回归集。

很多项目只覆盖其中几层。判断项目是否“可替代 MinerU”，不能只看它是否输出 Markdown。

## 四条主路线

### 路线 A：原生解析与规则

代表：MarkItDown、OpenParse 的 PDFMiner/PyMuPDF 路径、Unstructured `fast`。

机制：

- 直接读取 PDF 字符、字体和坐标。
- 用几何规则恢复行、段、标题和简单表格。
- 对 Office 文件读取 XML/文档对象模型，不先转成图像。

优点：

- CPU 即可运行，速度快，结果可解释。
- 不会凭空生成页面中不存在的文字。
- 对“本来就有干净文本层”的文档性价比高。

代价：

- 扫描件没有文字层时失效。
- 复杂版面、乱码字体、跨页关系和无边框表格较弱。
- 规则多后容易产生边界条件堆积。

### 路线 B：多专家 Pipeline

代表：MinerU `pipeline`、PaddleOCR PP-StructureV3、Marker、PDF-Extract-Kit。

机制：

- 版面检测模型先分区。
- OCR、公式、表格等专家模型各自处理擅长区域。
- 规则和小模型负责阅读顺序与文档级合并。

优点：

- 每个错误可定位到具体阶段。
- 可以按业务替换一个子模型。
- 专家模型在常规文档上通常稳定、高效。

代价：

- 模型多、坐标系多、配置复杂。
- 上游区域漏检会让下游完全看不到内容。
- 跨模块误差会累积，后处理代码容易变重。

### 路线 C：端到端专用 VLM

代表：dots.ocr、DeepSeek-OCR、olmOCR、GLM-OCR 模型、Dolphin、MinerU2.5、MinerU-Diffusion。

机制：

- 把页面当成图像输入视觉语言模型。
- 用 prompt 指定输出 Markdown、JSON、HTML 表格或定位框。
- 一个模型同时学习识字、布局、阅读顺序和格式生成。

优点：

- 对扫描、手写、复杂背景和长尾版式更有适应性。
- 减少多个专家模型之间的坐标和格式对齐。
- 可以用同一模型通过 prompt 切换任务。

代价：

- GPU、显存和吞吐成本更高。
- 输出可能不满足 schema、重复、截断或出现幻觉。
- 错误定位更难，通常需要约束解码、重试和结果校验。

### 路线 D：Hybrid 混合系统

代表：MinerU `hybrid`、GLM-OCR 自托管管线、Marker `--use_llm`、OpenDataLoader hybrid。

机制：

- 原生文本或轻量模型处理确定性部分。
- VLM 只处理复杂区域、OCR 页面或需要全局理解的步骤。
- 通过置信度、页面分类、layout 结果或 effort 档位决定计算量。

优点：

- 把 VLM 用在最值钱的区域。
- 保留原生文本准确性，同时改善复杂表格和长尾页面。
- 能提供速度/准确率档位。

代价：

- 路由条件和两套结果融合本身很复杂。
- “谁负责最终真相”需要明确，否则重复或冲突。
- 评测必须覆盖路由错误，而不只是单模型准确率。

## 2024-2026 的演进

### 2024：专家模型流水线成熟

- MinerU、PDF-Extract-Kit 把版面、OCR、公式、表格组合成完整解析流程。
- Docling 用统一 `DoclingDocument` 把多格式输入和多种输出连接起来。
- OmniDocBench 开始提供文本、表格、公式、阅读顺序的统一细粒度评价。

这一阶段的主要问题是“如何把多个模型拼成可用工具”。

### 2025：专用文档 VLM 快速出现

- olmOCR 强调大规模 PDF 线性化、训练数据生产和可验证规则评测。
- dots.ocr 用单 VLM 同时输出版面 JSON 和内容。
- MonkeyOCR、Dolphin、OCRFlux 分别探索结构-识别-关系、异构锚点和跨页合并。
- Pipeline 不再是唯一主流，评测开始比较专家流水线与端到端 VLM。

这一阶段的主要问题是“一个模型能否替代复杂流水线”。

### 2026：Hybrid、推理效率和服务化成为重点

- MinerU 3.x 增加 native Office、hybrid effort、异步任务 API、router、长文档窗口和流式落盘。
- MinerU-Diffusion 探索块级并行扩散解码，目标是降低自回归 OCR 的串行瓶颈。
- PaddleOCR、GLM-OCR、Docling 都在加强 SDK、服务端和可组合管线，而不只发布模型。
- OmniDocBench 主分支在 2026-04-30 更新到 v1.7；README 的综合榜单仍明确标注为 `v1.6_full`，说明评测代码、数据版本和榜单结果必须分别记录。

这一阶段的问题已变成“如何稳定、可扩展、低成本地运行文档解析基础设施”。

## 当前生态图

| 能力层 | 代表项目 | 本轮观察 |
|---|---|---|
| 全栈 Hybrid | MinerU | 三后端汇入共同 middle JSON，服务层最完整 |
| 统一文档对象 | Docling | format backend + pipeline + `DoclingDocument` |
| 可插拔处理链 | Marker | Provider → Builder → Processor → Renderer |
| OCR/专家模型平台 | PaddleOCR | Python 层主要包装 PaddleX 管线，模型与部署生态广 |
| 大规模 VLM 线性化 | olmOCR | 页面级请求、work queue、S3、vLLM、规则 benchmark |
| 单 VLM 多任务 | dots.ocr | prompt 决定 layout/OCR/grounding，工程层较薄 |
| 结构关系三元组 | MonkeyOCR | VLM 结果 + Magic-PDF 风格几何后处理 |
| OCR token 压缩 | DeepSeek-OCR | 重点在视觉 token 压缩和推理适配 |
| 有界流式 Pipeline | GLM-OCR | PageLoader → Layout → OCR，bounded queue 提供背压 |
| 两阶段异构提示 | Dolphin | 页面布局先行，再按元素类型并行解码 |
| 跨页 VLM 合并 | OCRFlux | 页面解析、跨页元素匹配、跨页表格合并三阶段 |
| 轻量格式转换 | MarkItDown | 文件识别 + 优先级 converter + plugin |
| 文档 ETL | Unstructured | 多格式 partition → typed elements → chunking |
| 语义节点构建 | OpenParse | text/table ingest → processing steps → Node |
| 评测基础设施 | OmniDocBench、olmOCR-bench | 组件指标与可验证规则互补 |
| 专家子模型 | DocLayout-YOLO、UniMERNet | 版面与公式能力可独立训练/替换 |
| 新解码研究 | MinerU-Diffusion | 非自回归块级去噪，工程成熟度仍低于 MinerU |

## 评测现状

OmniDocBench 把端到端质量拆成：

- 文本编辑距离。
- 表格 TEDS / TEDS-S。
- 公式 CDM。
- 阅读顺序编辑距离。
- 按文档类型、语言、版式和视觉退化进一步切片。

本地固定的 OmniDocBench README：

- 主分支标为 v1.7。
- 综合表标题仍是 `v1.6_full`。
- 该表记录 MinerU2.5-Pro 为 95.75、MinerU Pipeline 为 86.47。

这些数字只能说明“在该版本 benchmark 和官方推理配置下的公开结果”。不能直接推出：

- 在自己的合同、发票、论文或扫描档案上同样领先。
- 推理成本、吞吐、显存和稳定性最好。
- 许可证和部署约束适合生产。
- 不同项目自己发布的 benchmark 可以互相拼表比较。

olmOCR-bench 的思路不同：它把“某段文字必须出现”“顺序必须正确”“表格/数学格式必须满足规则”写成 7000 多个可验证测试，并对多次生成做统计。它更接近行为测试，OmniDocBench 更接近组件级标注与相似度评分，两者互补。

## 发展趋势

1. **中间表示会比 Markdown 更重要。** Markdown 只是展示格式；坐标、层级、表格单元格、置信度和来源关系必须保存在结构化 IR 中。
2. **页面级识别将转向文档级推理。** 跨页表格、标题层级、脚注和断段无法只看单页解决。
3. **Hybrid 会长期存在。** 原生文本便宜且可靠，VLM 擅长复杂视觉，两者没有必要互相完全替代。
4. **推理运行时成为产品壁垒。** 有界队列、窗口、缓存、重试、服务健康和多 GPU 路由决定生产可用性。
5. **评测从平均分走向失败类型。** 业务更关心“表格不能错一格”“页码不能进正文”“引用必须能回到 bbox”。
6. **许可证分层更复杂。** 代码、模型权重、数据集和服务条款可能分别使用不同协议。

## 仍未解决的核心问题

- 扫描、手写、旋转文字和低资源语言仍是长尾。
- 复杂无边框表格、跨页表格和表内图片仍不稳定。
- VLM 输出 schema 约束、重复和截断需要额外防线。
- 大型文档的内存、吞吐、失败恢复和增量重跑缺少统一标准。
- 很多 benchmark 忽略真实业务的字段完整性、可追溯引用和成本。
- “文档解析”与“业务字段抽取”仍需两层系统协作。

## 关键思考点

1. 如果原生 PDF 已有完整文本层，为什么还要把整页送给 VLM？
2. Pipeline 中最危险的是模型准确率低，还是上游漏检导致下游完全失去输入？
3. 一个中间表示至少应保存哪些字段，才能同时支持 Markdown、RAG chunk 和原页引用？
4. 跨页合并应该由规则、分类模型还是生成模型负责？怎样证明没有误合并？
5. 业务验收应该按“页面平均分”还是按“关键字段零错误”设计？
