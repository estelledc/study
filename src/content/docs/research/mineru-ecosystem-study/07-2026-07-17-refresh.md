## 3. 为什么不直接跑 MinerU 全链

固定提交的唯一公开 e2e test：

```text
tests/unittest/test_e2e.py
```

它会同时运行：

- layout；
- OCR；
- formula；
- table；
- pipeline model；
- txt/ocr 两套路径。

项目的 `test` extra 又依赖 `mineru[core]`，会安装 Torch、Transformers、
ONNX Runtime、Gradio 和 pipeline/VLM 组件，还需要模型权重。本机没有预下载权重，
也没有为本轮分配 GPU 与模型下载预算。

因此本轮边界是：

- MinerU 主链保持 E1 源码追踪；
- E2 选择不需要模型的原生 PDF 路线；
- 不把“依赖能安装”冒充“模型能推理”；
- 不把 CPU 小 PDF 结果外推到扫描、公式、表格或长文档。

## 4. MarkItDown 真实测试

一次性环境：

```text
/tmp/document-parser-venv-20260717
Python 3.11
```

从 pinned 本地源码安装 MarkItDown PDF extra 和 OpenParse，不修改源码。MarkItDown
运行：

```bash
PYTHONDONTWRITEBYTECODE=1 \
/tmp/document-parser-venv-20260717/bin/python -m pytest -q \
  packages/markitdown/tests/test_pdf_memory.py \
  packages/markitdown/tests/test_pdf_tables.py \
  packages/markitdown/tests/test_pdf_masterformat.py
```

结果：

```text

```

覆盖：

- 每页资源关闭，避免内存随页数线性增长；
- 普通文本 PDF 回退 pdfminer；
- form/table 页面按页走 pdfplumber；
- 单次打开 PDF；
- table、borderless table 和 master format。

两个 skip 是可选 fixture/dependency 条件，不是失败。

## 5. OpenParse 官方测试入口失败卡

执行：

```bash
PYTHONDONTWRITEBYTECODE=1 \
/tmp/document-parser-venv-20260717/bin/python -m pytest -q \
  src/tests/test_schemas.py \
  src/tests/processing/test_steps.py \
  src/tests/text/pdf_miner/test_core.py
```

在 collection/session start 阶段退出：

```text
Exit: Pytest must be run from the project root directory
```

但命令实际 cwd 已是仓库根。根因在 `src/tests/conftest.py`：

```python
expected_dir = Path(__file__).resolve().parent / "src"
```

这会得到不存在的：

```text
<repo>/src/tests/src
```

因此：

- 这是 test harness contract failure；
- 测试体没有执行；
- 不是 OpenParse 解析失败；
- 本轮不修改第三方源码绕过 gate。

另外，上游 LFS 额度已耗尽，官方部分 PDF fixture 无法完整恢复。两项约束都应在
复现报告中保留。

## 6. 同一 PDF 对照

输入：

```text
markitdown/packages/markitdown/tests/test_files/test.pdf
PDF 1.4
1 page
92,971 bytes
数字文本 PDF
```

它是 MarkItDown pinned commit 中的真实 fixture，正文来自一篇多 Agent 论文。

运行：

```bash
cd src/content/docs/research/mineru-ecosystem-study/labs
PYTHONDONTWRITEBYTECODE=1 \
/tmp/document-parser-venv-20260717/bin/python document_parser_lab.py
```

输出：

```text
markitdown: phrases=4/4 order=pass chars=5194 units=1 source_boxes=0
openparse: phrases=4/4 order=pass chars=5255 units=3 source_boxes=3
```

共同 gate：

- `Introduction` 存在；
- `Large language models (LLMs)` 存在；
- `Customizable and conversable agents` 存在；
- `Conversation programming` 存在；
- 四个短语按预期顺序出现。

差异：

| 解析器 | 产品合同 | 本轮可见输出 |
|---|---|---|
| MarkItDown | 轻量格式转 Markdown | 单一 Markdown 文本 |
| OpenParse | 面向检索的 Node | 3 个 Node，保留 element/bbox |

字符数差异不能直接解释为质量差异。OpenParse 的 `<br>`、节点拼接和 MarkItDown 的
换行策略都会改变长度。

运行时还出现 `Object <id> not defined` / `Overwriting cache` PDF 警告，但两套 API
都返回结果并通过内容 gate。正确记录是“输入 PDF 有交叉引用警告且降级成功”，不是
静默删除日志，也不是据此判定输出完全正确。

## 8. 指标解释

### 必含短语

证明关键文本存在，不证明：

- 没有重复；
- 没有多余幻觉；
- 句子内部完全正确；
- 表格、公式和层级正确。

### 阅读顺序

本实验只检查 4 个锚点的相对顺序。它不能发现锚点之间的段落错序。

### Unit 与 source box

证明输出保留了结构/定位入口，不证明 bbox 精确，也不证明 Node 切分适合 RAG。

### 字符数

只用于观察格式差异，不进入通过/失败 gate。

## 9. 稳定结论

1. 文档解析验收必须拆成 text、order、structure、provenance 和 operational quality。
2. 同一输入、同一 expected invariants 比跨榜单拼分更可信。
3. Markdown 适合人读，不足以独自承担调试和可点击引用。
4. 原生数字 PDF 是成本 baseline，不应一开始就调用 GPU VLM。
5. parser warning、test harness failure、dependency blocker 和 output quality 是四类问题。
6. OpenParse Node 的 bbox 是额外能力，不代表其文本必然优于 MarkItDown。
7. MinerU full pipeline 需要模型/GPU 的事实必须显式保留，不能用轻量 E2 代替。

- 扫描 PDF、复杂表格、公式、双栏、跨页和坏页。
- bbox 与人工标注的几何误差。
- p50/p95 延迟、峰值 RAM/VRAM 和 pages/s。
- 许可证是否符合具体生产业务。
