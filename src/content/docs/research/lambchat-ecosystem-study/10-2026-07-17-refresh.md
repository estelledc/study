## 3. 六个漂移仓的影响

### 3.1 DeepAgents：基础 harness 没有变

23 个提交主要落在 `libs/code`：

- plugin marketplace、搜索和 reload；
- MCP OAuth 非阻塞刷新；
- TUI、debug console 和 model selector；
- QuickJS subagent interrupt 传播。

`libs/deepagents/deepagents/` 没有文件变化，`libs/deepagents` 只更新依赖元数据。因此
LambChat 对 Filesystem、Skills、Subagent、Memory 等 middleware 的基础判断仍成立。

### 3.2 Dify：图生成从一次大调用变成并行节点构建

`WorkflowGenerator` 现在采用：

```text
planner
  -> bounded parallel node builders
  -> deterministic graph assembly
  -> structural validation
```

它用线程池限制并发，每个节点只生成局部 config；一个 fragment 失败会取消未开始任务，
整个 graph fail closed。另一个增量是在 tenant 创建后异步安装默认 plugin，并等待安装
状态后配置默认 model。

这强化了两个结论：

1. Workflow-first 产品把“生成图”和“执行图”分成不同控制流。
2. tenant 初始化也是可失败的后台任务，不能等同于创建一条数据库记录。

### 3.3 LibreChat：实时流需要原子序号和可恢复状态

PR `#14313` 将每个 delta 的：

```text
INCR sequence
PUBLISH event
```

合并成一次 Redis Lua 调用，既减少一次 round trip，也让序号分配与发布处于同一 Redis
操作。`getResumeState()` 并行读取 content、run steps 和 pending steers；in-flight
steer 还新增了取消与 settled 防回填逻辑。

这不是“用了 SSE 就能恢复”，而是：

- run/job state 要持久；
- event 要有顺序；
- pending steer 要有稳定 ID；
- live 与 resume contract 要能重新投影同一状态。

### 3.4 OpenClaw：本地 Agent runtime 也需要 durable ingress

360 个提交、2340 个变化文件不能逐条展开。本轮只审计 gateway、agent、channel、
sandbox、workspace 和 daemon 路径，看到的主要变化是：

- durable channel ingress drain；
- claim lease、owner identity、stale recovery；
- per-lane serialization；
- adoption、deferred、retry、dead-letter 和 stall watchdog；
- workspace setup state 从文件迁入 SQLite；
- control-plane write rate limit；
- sandbox env value size bound；
- transcript、compaction、subagent settle 和 gateway OOM 防护。

这些变化表明单机/个人 Agent 也会遇到与队列系统相同的 ownership 和 recovery 问题。
但 OpenClaw 仍不是组织级多租户 SaaS；durable ingress 不能替代 tenant root、组织预算
或不互信用户的执行隔离。

### 3.5 MCP Gateway Registry：先把控制面和数据面量对

新增 `target_kind` allowlist，将流量区分为：

- `a2a_agent`
- `virtual_mcp_server`
- `mcp_server`
- `control_plane`
- `unknown`

`/api/*`、static 和 OAuth 路径不会被误算成 MCP 数据面。这是观测正确性增强，没有改变
Registry 当前的凭证、授权和部署边界。

### 3.6 Loomcycle：sandbox sidecar 已落地，但租户身份仍待补

新 builder sidecar 通过 HTTP MCP 暴露：

```text
sandbox_open / exec / write / read / close / list
```

每个 session container 默认：

- network none；
- read-only rootfs；
- non-root；
- drop all capabilities；
- tmpfs workspace；
- CPU、memory、PID、TTL 和 output limits。

这是从“runtime substrate 文档”走向真实执行面的重要增量。但源码和文档明确写明：

```text
P1 = one shared bearer -> one principal
P2 = attested per-tenant identity, deferred
```

所以当前 sidecar 能证明 session/container 隔离机制，不能证明多个不互信 tenant 的身份
隔离已经完成。

## 5. 三张失败卡

### 卡 1：客户端断线

错误处理：

```text
SSE disconnect -> cancel run
```

后果：移动网络抖动会改变业务执行状态。

正确合同：

```text
disconnect -> remove subscriber
run -> independent background lifecycle
reconnect -> replay by event ID / sequence
```

### 卡 2：event 已持久化，publish 失败

不能宣称“两份都写了，所以是事务”。数据库和 Redis/SSE 之间没有天然原子事务。

本轮实验采用：

```text
durable event commit
  -> best-effort live publish
  -> publish failure is visible
  -> reconnect replays durable event
```

更复杂系统可以加入 transactional outbox，但仍需明确谁是权威源。

### 卡 3：有 checkpoint，但邮件重复

checkpoint 只证明 graph state 到达某一步。若邮件已发送、receipt 尚未记录就崩溃，
恢复后仍可能重复发送。

必须另存：

```text
operation_id
status = pending / succeeded / failed / uncertain
provider_receipt
```

并在重试前查询外部权威状态。

## 6. 稳定结论

1. **Product、run control、harness、governance 和 execution 是五层。**
2. **连接生命周期不能拥有 run 生命周期。**
3. **event log 是历史源，SSE 是可丢失的实时投影。**
4. **tenant ID 必须进入 session、run、workspace、event、quota 和 credential key。**
5. **quota 必须在 tool call 时再次执行并 fail closed。**
6. **checkpoint 与 side-effect receipt 是两类证据。**
7. **sandbox 隔离和 tenant 身份隔离必须分别证明。**
8. **MCP 统一连接协议，不自动提供 policy、quota、credential 或 audit。**

## 7. 仍未证明

- LambChat 使用真实 MongoDB、Redis、PostgreSQL 的进程崩溃恢复。
- arq 多 Worker 下 run lease 和 event 顺序。
- E2B/Daytona/CubeSandbox 真容器的路径、网络和清理隔离。
- 真实 MCP credential rotation 与日志泄漏检查。
- 浏览器、移动端和桌面端的断网重连 E2E。
- 14 个项目的全套测试和生产部署。
- Loomcycle builder sidecar 的 Podman/gVisor 端到端运行。

这些边界不会被 88 个单元测试或 7 个教学实验升级为已验证。
