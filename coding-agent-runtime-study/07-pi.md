# 07. Pi：最小内核与激进扩展

## 一句话定位

Pi 像一套透明的乐高底板：核心只保证模型、消息、工具和事件循环能正确配合；MCP、子 Agent、权限弹窗、计划模式和工作流由扩展或外部系统选择性搭建。

固定快照：`earendil-works/pi@c6d83715`。

## Monorepo 分层

| package | 职责 | 类比 |
|---|---|---|
| `pi-ai` | 多 provider 模型、消息、流事件、认证 | 通用通信协议 |
| `pi-agent-core` | Agent state、Agent loop、tool interface | 小型发动机 |
| `pi-coding-agent` | CLI、session、tools、extensions、TUI mode | 完整车辆 |
| `pi-tui` | 终端渲染库 | 仪表盘 |
| `orchestrator` | 实验性编排 | 外挂调度器 |

最大的学习价值是分层清楚：可以单独读 `pi-agent-core`，不必先理解整个 TUI。

## CLI 入口

`packages/coding-agent/src/cli.ts` 只有约 20 行：

1. 设置 process title 与环境；
2. 配置 HTTP dispatcher；
3. 调 `main(args)`。

真正的 composition root 是 `main.ts`：

- 解析参数；
- 选择 interactive / print / JSON / RPC；
- 处理 project trust；
- 打开或创建 session；
- 解析 model；
- 创建 AgentSession services/runtime；
- 最后启动对应 mode。

模式分流见 `main.ts:796-844`。

这个入口设计强调：

> CLI 只把输入转换成 SDK/runtime 配置，核心能力应能被其他程序直接调用。

## 三层运行对象

### Agent loop

纯函数式控制流，负责：

- message → model stream；
- tool call；
- tool result；
- steering；
- follow-up；
- event emission。

### Agent

有状态包装器，负责：

- model、tools、system prompt；
- active run；
- pending message queue；
- stream event 转成当前 state；
- abort；
- prompt/continue。

### AgentSession

coding-agent 产品层，负责：

- session persistence；
- model/auth；
- prompt template 与 Skill；
- extension lifecycle；
- compaction；
- retry；
- branching；
- tool selection；
- TUI/RPC 事件。

不要把这三层叫成同一个“Agent”。它们分别是算法、运行实例和产品会话。

## Agent loop 双层循环

源码：`packages/agent/src/agent-loop.ts:155-275`。

```text
outer:
  inner:
    inject steering
    stream model
    execute tools
    prepare next turn
    optional stop

  get follow-up
  if none: agent_end
```

### 为什么是两层

一层 while 只能回答“模型还要不要调用工具”。Pi 还要表达：

- steering：在当前 run 内尽快纠偏；
- follow-up：当前 run settle 后再做。

把它们混成一个 queue，用户在长任务中说“别改 A，改 B”可能要等任务完全结束才生效；反过来，把所有 follow-up 都当 steering，会意外改变正在进行的任务。

## 模型上下文转换

`streamAssistantResponse()`：

```text
AgentMessage[]
  → transformContext
  → convertToLlm
  → { systemPrompt, messages, tools }
  → streamFn(model, context)
  → Agent events
```

源码：`agent-loop.ts:281-373`。

分成两个变换层的原因：

- `transformContext` 仍处理 Pi 自己的 AgentMessage，例如压缩、裁剪；
- `convertToLlm` 处理 provider 可接受的通用 Message；
- provider adapter 再把通用 Message 转成 Anthropic/OpenAI/Google 请求。

这避免 provider-specific shape 污染 Agent loop。

## 流式状态

模型流事件包括：

- start；
- text start/delta/end；
- thinking start/delta/end；
- toolcall start/delta/end；
- done/error。

Pi 在 `start` 时把 partial message 放入 context，delta 时替换尾项，done 时替换为 final message。

这种做法同时满足：

- UI 实时更新；
- history 中没有一串重复 delta；
- tool call 参数在最终消息中完整落盘。

## 工具执行

### 并发选择

默认受全局 `toolExecution` 控制；只要批次中一个工具声明 `executionMode: "sequential"`，整批串行。

源码：`agent-loop.ts:413-428`。

适合声明 sequential 的工具：

- 修改共享文件；
- 改全局 cwd；
- 依赖上一调用副作用；
- 不能并发访问的外部系统。

### 执行前

`prepareToolCall()`：

1. 按名字查工具；
2. 可选 `prepareArguments`；
3. schema validation；
4. `beforeToolCall` hook；
5. 检查 abort；
6. 返回 prepared 或 immediate error。

源码：`agent-loop.ts:602-666`。

`beforeToolCall` 可以实现自定义 permission gate，但 Pi 不把它包装成内置统一政策产品。

### 执行后

`afterToolCall` 可以修改：

- content；
- details；
- terminate；
- isError。

源码：`agent-loop.ts:711-755`。

这允许扩展：

- 截断输出；
- 加 metadata；
- 把特定结果标记为终止批次；
- 规范化 provider/tool 差异。

### 截断响应保护

若 assistant response 的 stop reason 是 `length`，Pi 不执行其中工具调用。即使 JSON 恰好能 parse，也可能缺少模型本来还要生成的参数。

源码：`agent-loop.ts:207-214`。

这是比 schema validation 更早的一层完整性检查。

## AgentSession 的 prompt pipeline

源码：`packages/coding-agent/src/core/agent-session.ts:1093-1253`。

一次 prompt 依次经过：

1. extension command；
2. extension input event；
3. Skill command 与 prompt template expansion；
4. streaming 时选择 steer/follow-up；
5. flush pending bash messages；
6. 验证 model；
7. 验证 auth；
8. pre-send compaction；
9. 构造 user message；
10. 注入 next-turn messages；
11. `before_agent_start` extension；
12. 应用本 turn system prompt override；
13. `_runAgentPrompt()`。

这说明最小 Agent loop 外面仍需要产品级输入编译器。

## Provider 抽象

`pi-ai` 支持多个原生 API：

- Anthropic Messages；
- OpenAI Responses；
- OpenAI Chat Completions；
- OpenAI Codex；
- Google Generative AI / Vertex；
- Bedrock Converse；
- Mistral Conversations；
- 其他 OpenAI-compatible provider。

provider 目录拆成：

- model catalog；
- stream function；
- API-specific message transform；
- auth/OAuth；
- compatibility flags。

### 统一接口的边界

统一不代表抹平所有差异。Model metadata 仍要表达：

- provider；
- API 类型；
- context window；
- thinking；
- images；
- caching；
- headers；
- transport；
- compatibility quirks。

因此 `pi-ai` 更像“带类型的适配层”，不是把所有 provider 强行伪装成同一个最小 HTTP endpoint。

## Session tree

Pi 的会话持久化是最适合教学的实现之一。

### JSONL

每行一个 entry：

- session header；
- message；
- model change；
- thinking level change；
- compaction；
- branch summary；
- custom extension state；
- custom message；
- label；
- session info。

源码：`packages/coding-agent/src/core/session-manager.ts:30-149`。

### Tree

除 header 外，每个 entry 有：

- `id`
- `parentId`

当前 leaf 指向 active branch。这样一个 JSONL 文件可以同时保存多个分支。

### 两种 extension state

| 类型 | 是否进入 LLM context | 用途 |
|---|---|---|
| `custom` | 否 | 扩展自己的持久状态 |
| `custom_message` | 是 | 扩展注入模型上下文 |

源码：`session-manager.ts:90-137`。

这是非常重要的边界：需要恢复的插件状态不一定应该消耗模型 context。

## Compaction

Pi 的 compaction：

- 总结旧消息；
- 保留最近消息；
- 原始 JSONL 不删除；
- active context 用 compaction summary；
- `/tree` 仍可回到完整历史；
- extension 可以覆盖 summary 行为。

`AgentSession` 区分：

- manual；
- threshold；
- overflow。

overflow 可 compact 后自动重试一次；threshold 只 compact，不默认继续重发用户请求。

源码线索：`agent-session.ts:1767-2188`。

为什么区分：

- overflow 是当前请求失败后的恢复动作；
- threshold 是为未来请求腾空间；
- 两者若都自动重试，可能重复执行用户动作。

## ExtensionRunner

ExtensionRunner 的事件面很广：

- input；
- before agent start；
- before provider request/headers；
- message end；
- before/after tool；
- session switch/fork/tree/compact；
- resources discovery；
- project trust；
- shutdown；
- UI。

源码：`core/extensions/runner.ts:14-65,113-159`。

扩展还能注册：

- tools；
- commands；
- provider；
- model；
- UI components；
- entry renderer；
- Skill/prompt/theme 路径。

这就是 Pi “激进扩展”的含义：核心少，但 extension API 很宽。

## Pi 主动不内置什么

README 明确列出：

- no built-in MCP；
- no built-in subagents；
- no permission popups；
- no plan mode；
- no built-in todos。

源码文档：`packages/coding-agent/README.md:489-499`。

这不是声称这些能力无价值，而是拒绝替用户决定唯一工作流：

- MCP 可由 extension 接入；
- 子 Agent 可由 tmux、多进程或 extension 实现；
- permission 可由 extension 或外部 sandbox 实现；
- plan/todo 可按团队习惯实现。

## 安全边界

### 没有内置 sandbox

根 README 明确：

> Pi 默认拥有启动它的用户和进程权限。

`docs/security.md` 进一步说明：

- built-in tools 可按宿主用户权限读写和执行；
- extension 是同权限 TypeScript 模块；
- project trust 不是 sandbox；
- 真隔离应使用 container、VM、micro-VM 或 policy-controlled sandbox。

### Project trust 是什么

它只决定是否加载项目本地：

- settings；
- extensions；
- packages；
- skills/prompts/themes。

它防止仓库在启动时静默加载可执行扩展，但不会限制已启动 Agent 的工具能力。

### Tool allowlist 是什么

`--tools`、`--exclude-tools`、`--no-builtin-tools` 控制向模型提供哪些工具。它减少能力面，但仍不是 OS security boundary。

## 为什么这种选择合理

### 适合

- 想读懂 Agent loop；
- 想快速做自定义 provider；
- 想把 harness 嵌入自己的 Node 应用；
- 已有容器/远程 sandbox；
- 不希望平台规定 plan、todo、subagent 工作流。

### 不适合

- 希望开箱即用的企业政策中心；
- 希望内建跨平台 sandbox；
- 希望统一的 permission UI；
- 希望无需评审就安装大量第三方 extension；
- 希望一个进程内原生管理复杂 subagent graph。

## 设计优点

1. Agent loop 可读、可测、可复用。
2. provider 抽象广且保留差异。
3. session tree 结构简单但支持 branch/compact/export。
4. extension API 足够宽，不必 fork core。
5. 安全文档诚实区分 trust、allowlist 与 sandbox。

## 设计代价

1. 默认运行权限很大。
2. 团队需要自己选择 permission、MCP、subagent 方案。
3. extension 拥有完整进程权限，供应链风险高。
4. 复杂工作流可能在不同 extension 中形成不一致实现。
5. “核心少”不等于产品层代码少，`AgentSession` 仍然很大。

## 初学者容易看错

### “没有内置”等于“不能做”

Pi 能通过 extension 或外部进程做这些能力，只是不规定唯一实现。

### project trust 等于执行权限

project trust 只控制项目资源加载；工具执行仍按宿主进程权限。

### session JSONL 是线性聊天

entry 有 `parentId`，一个文件可以保存树。

### Agent loop 已包含完整产品

`agent-loop.ts` 不负责持久化、TUI、auth、compaction policy 和 extension discovery，这些由 AgentSession/CLI 层完成。

## 思考点

1. Pi 为什么把 permission gate 留给 extension，而 Codex/OpenCode 放进 core？
2. `custom` 与 `custom_message` 为什么必须分成两种 session entry？
3. overflow compaction 为什么可自动重试，threshold compaction 默认不重试？
4. 如果你给 Pi 增加内置 MCP，会给最小内核带来哪些新状态？
